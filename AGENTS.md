## 这个项目是什么

这是一个 AI 剧本创作桌面软件，不是普通文本生成工具。

正式主链只认这一条：

用户输入
-> 确认信息
-> `storyIntent`
-> 粗纲
-> 人物小传
-> 详细大纲
-> 剧本

系统的职责不是替创作做裁判，而是：
- 帮用户把输入结构化
- 给 AI 清楚的工序边界
- 保证每一层只做自己的事
- 防止第二口径、双来源、旧旁路回流

## 当前工程主权（2026-04-18 起生效）

从这轮“前后端分离终极收口行动”开始，工程主权固定如下：

1. **HTTP API 是唯一正式业务入口**
   - 正式业务入口在 `server/`。
   - renderer 的业务请求只准走 HTTP API。
   - Electron IPC 不再是正式业务入口。

2. **PocketBase 是唯一项目真相存储**
   - 用户、积分、项目、粗纲、人物、详纲、剧本都要统一收进 PocketBase。
   - 本地文件、project-store、renderer 本地缓存都不能再冒充正式真相源。

3. **Electron main / preload 只保留桌面壳职责**
   - 只负责窗口、系统能力、必要的桌面桥接。
   - 不再掌管项目读写、生成流程、正式状态广播和业务 authority。

4. **HTTP 一旦对齐，对应 IPC 必须物理退场**
   - 不允许 HTTP 和 IPC 长期双管线并存。
   - 旧 IPC 最多短暂作为迁移阻断壳，不能继续偷偷执行业务。

## 七问框架（篇章级工具，agent-first）

七问是篇章级工具，不是集级模板。

### 核心原则

1. **七问只在篇章层用，不在集层用**
   - 一个剧本可以分成多个篇章（如西游记狮驼岭占 10 集）。
   - 整个剧也可以只有一个篇章。
   - 每一集只是这个篇章里的一步执行动作。

2. **七问在粗纲层确定，不在详纲层**
   - 用户在粗纲阶段划分篇章边界、填写七问答案。
   - 到详纲再补已经晚了，调不过来。
   - 七问是用户参与决策，不是系统自动生成。

3. **七问只挂篇章，不挂集**
   - 同一篇章内的所有集共享同一套七问答案。
   - 系统负责把七问答案往下传递到每一集的执行动作里。
   - 不能把七问逐集套用，否则会导致创作过度结构化。

4. **七问是 agent-first，不是普通 prompt**
   - 有一个专用的 `seven_questions_agent` 负责起草七问。
   - 七问生成走 Agent 协议：输入 -> 起草 -> 用户确认/修改 -> 确认 -> 落盘。
   - 不是普通 prompt 调用，是专用 Agent 单职责。

### 七问 Agent 职责

```
seven_questions_agent
  输入：粗纲全文 + 逐集粗纲 + 篇章边界（用户指定或建议）
  输出：每个篇章的七问答案 { goal, obstacle, effort, result, twist, turnaround, ending }
  禁止：为每集单独生成七问 / 逐集填充七问框架 / 展开到具体场景
```

### 七问内容

七问负责回答一个篇章的叙事骨架：

1. 目标：这一篇章要达成什么
2. 阻碍：阻止目标实现的核心障碍是什么
3. 努力：角色做了什么来克服阻碍
4. 结果：努力带来了什么（往往是代价或新问题）
5. 意外：计划之外发生了什么
6. 转折：意外如何改变了局势
7. 结局：最终走向什么结果

### 七问落地位置

- **粗纲层（agent-first）**：`outlineBlocks[].sevenQuestions` → 篇章划分 + 七问答案 + 用户确认
- **详纲层**：每集的 sceneByScene 骨架，感知篇章七问叙事目标，但不重复生成七问
- **剧本层**：每集的具体动作和对话，隐含篇章七问方向，不显式出现七问

### 七问 Agent 工作流

```
用户确认粗纲
    ↓
用户指定篇章数（如：60集分3个篇章）
    ↓
seven_questions_agent 生成七问初稿
    ↓
展示给用户 → 用户修改 → 确认
    ↓
七问写入 outlineBlocks[].sevenQuestions
    ↓
进入详纲层（七问往下传递，不重新生成）
```

### 七问关键约束

1. **七问只生成一次**：在粗纲层，用户确认后不再改动
2. **七问不过集**：每一集不单独套七问，只从篇章继承叙事方向
3. **七问是用户决策**：agent 起草，用户确认或修改
4. **七问 agent-first**：不是普通 prompt，是专用 agent

### 典型误区

- ❌ 把七问逐集套用 → 每集都问目标/阻碍/代价，内容会腻、节奏会拖
- ❌ 把七问放在详纲层 → 篇章边界已经在粗纲定好，详纲调不过来
- ❌ 系统自动生成七问不经过用户确认 → 七问需要用户参与决策
- ❌ 把七问当成普通 prompt → 七问是专用 agent，有自己的职责边界

## 长期硬规则

1. 用户输入的剧本意图是唯一真相。
2. 不允许第二写口、第二发布口、第二裁判口。
3. 发现双口径，不打补丁，直接收一套。
4. 不做兜底。失败就报错，回根因层修。
5. 不要把创作问题修成 gate、命中词校验、重型 recovery、抽名补名单。
6. AI 主体负责创作，系统只做引导、结构化输入、轻量校验和最小必要机制。
7. 能用提示词、脚手架、工序边界解决的，不改成硬限制。
8. 旧逻辑、兼容逻辑、fallback、support、自动推进，只要重新掌权，就必须汇报并清掉。
9. 任务完成时，如果发现两套逻辑并存，要顺手收口，不允许留着等下轮继续反弹。
10. 默认只修原有功能，不借机发明新功能。
11. HTTP API 是唯一正式业务入口；renderer 不准继续把 IPC 当正式业务口。
12. PocketBase 是唯一项目真相存储；本地文件和 Electron project-store 不准继续掌权。
13. 一旦某个业务阶段 HTTP 已对齐，对应 `window.api` / `ipc` 入口必须尽快物理退场，不留双口径。

## 文档规则

1. `docs/聊天记录.md` 只保留一条当前主线。
2. 每累计 3 轮有效来回，如果当前终端正在推进主线，就更新一次 `docs/聊天记录.md`。
3. `docs/项目原则.md` 只记长期原则、禁区、典型误判和识别信号。
4. `docs/AI自我学习.md` 只记 AI 对项目的理解、踩坑、错误判断、有效做法和后续提醒。
5. `docs/项目结构.md` 只记仓库结构、工序结构、主链落点和归档布局。
6. `docs/主线/YYYY-MM-DD-主线.md` 只记当前主线计划、任务清单、进度和完成定义；主线完成后移入 `docs/归档/主线/`。
6. 正在推进主线并完成一轮任务的终端，收口时必须顺手同步四件套：`docs/聊天记录.md`、`docs/项目原则.md`、`docs/AI自我学习.md`、`docs/项目结构.md`。
7. `docs/项目结构.md` 不是可有可无的整理活；只要主链落点、目录结构、风险区、归档布局变了，就必须同步更新。
8. `docs/聊天记录.md` 主要由正在干活的终端维护，不记录纯文档整理终端自己的整理过程。
9. `docs/归档/` 里的东西只保留历史证据，不再当现役入口。
10. 用户发“存档”时，这是额外强制整理口令；四件套都要同步检查、更新、压缩和归档。
11. 如果旧文档的结论已经被新事实推翻，不能继续保留在现役文档里。
12. 文档整理相关细则统一收在根目录 `文档管理.md`，整理文档终端默认先看它。

## 协作方式

1. 全程中文，讲大白话。
2. 先讲清本质，再讲动作。
3. 重大改动先给计划，再执行。
4. 发现之前判断错了，要明确写进文档，不要假装没发生过。
5. 不要拿“做了很多动作”冒充“解决了本质问题”。
6. 不要拿“某一层通过了”冒充“整条链通过了”。

## 进度汇报格式

当前任务：
进度：`x/5`

规则：
- 滚屏时，进度放前面。
- 收尾时，进度放最后。
- 一旦到 `5/5`，要明确提醒任务做完了。

## 测试和工具纪律

1. 测试按板块测，不测整条 E2E 当当前验收。
2. 用完 MCP、浏览器自动化、测试工具后必须关掉，不留残进程。
3. 不在仓库外生成文件或目录。
4. 新建文件不要超过 500 行。
5. 新建文件只负责一个功能。
6. 不要在当前指挥终端运行带 `vite-node` 的命令；这类命令必须换到独立执行终端跑，否则当前终端容易直接崩掉。
7. 只要是在验粗纲、人物小传、详细大纲、剧本的内容质量，就默认直接调用真实大模型 API 出结果；`mock`、单测、结构校验只负责验流程、合同和解析，不替代内容质量验收。
8. 内容质量真实验证时，`20` 集及以内默认可以按用户要求直接全量跑；更长任务默认先按用户指定范围、切片或抽样来跑，不擅自整季全写完。
9. 每次真实调用得到的结果，都要落成仓库内可复看的 `txt` 或 `md` 证据文件，方便指挥终端和执行终端一起看内容、判断质量、继续调提示词。

## 开工前自查

每次开始做方案或改动前，先问自己 4 件事：

1. 是不是在加兜底。
2. 是不是把建议写成死限制。
3. 是不是越过工序边界。
4. 是不是让系统替代创作裁判。

只要有一条答案是“是”，就先停下，回根因层重做方案。
