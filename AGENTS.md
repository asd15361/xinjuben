# AGENTS（首页导航）

做计划的标准是解决本质问题。什么是本质？推不翻的才叫本质。任务的交付标准就是有没有解决本质问题。

本文件只做 3 件事：
- 告诉人和 AI 先看哪份文档
- 按任务类型分流
- 只留最顶层总原则

## 总原则
- 做优秀软件，不做只会拦错的审查软件。
- 做软件先看用户为什么要用，再看功能怎么接。
- UI 不好看，产品就没真正成立。
- AI 可以实现系统，但不能重新发明系统。
- 先钉死交付标准，再允许 AI 开工。
- 先锁定义权，再开功能。
- 同一个意思，只保留一个最权威版本。
- 重复动作固定成套路，不要每次临场发挥。
- 重要检查固定成门禁，不要只靠 AI 想起。
- 解开补丁，找到本质。
- 能用大白话讲清楚的本质，才是真的本质。
- 默认只让 AI 自己完成能自动完成的事。
- 默认只和用户聊战略、本质、方向、结果和风险。
- 尽量并行，及时关代理进程，防 CPU 卡死。
- 用完工具就收尾，别留下代理、浏览器和 MCP 残留进程；清理时按进程树回收，不要只杀表面 PID。
- 每个自然日开工前先检查 `D:\vivecoding规则` 是否已有当天同步记录；没有就先同步文档，再开始当天正式工作。

## 使用方式
- 首页只做导航，不展开细节。
- 长期原则写进主手册。
- 项目过程写进任务卡、计划或 worklog。
- 系统定义权以 `docs/system-authority（系统定义权与主权规则）.md` 为准。
- 文档同步以 `docs/doc-sync（文档同步机制）.md` 为准。
- 过时内容移入归档，不继续堆在首页。

## 当前任务规则
- 不是一句话能做完的小修，先建当前任务卡。
- 每轮开始先看当前任务卡。
- 旧任务没完成，默认先接旧任务。
- 只有用户明确暂停，或任务已阻塞，才允许切走。

## 新增规则准入
- 先问：这是不是长期有效。
- 再问：这是不是一个新意思。
- 再问：它最该写在哪一份文档。
- 最后问：能不能压成一句动作句。
- 能改旧句解决的，不新增新句。

## 汇报规则
- 做整体计划时，同时汇报 `总任务进度` 和 `当前小任务进度`。
- 不要把小任务 100% 说成总任务 100%。
- 收尾先讲结论，再讲结果和原因，再讲进度，最后讲建议。
- 建议放在完成情况后面，不要抢到前面。
- 大任务默认一气呵成做到能交付，不主动拿“下一步”打断用户。
- 中途默认只显示进度，不额外展开战术汇报；用户打断再解释。
- 进度滚屏固定同时带：当前小任务、当前小任务进度、当前大任务、当前大任务进度。
- 百分比后面要跟一句简单说明，让用户不用翻前文也知道现在在干什么。
