## 一、这份计划现在解决什么问题

这份计划不再回答“接下来随便做什么”，  
而是只回答一件事：

**怎么把新仓库做成一套不会再次长回旧项目那种大文件、高耦合、真假边界混乱的系统。**

这份计划的交付标准只有一个：

- 有没有解决本质问题

什么叫本质问题：

- 推不翻的才叫本质

所以这份计划只围绕 6 件不可推翻的事展开：

1. 正式事实定义权必须唯一
2. 四工序流动边界必须刚性
3. 主链运行态必须只在主进程收口
4. 旧项目只迁业务资产，不迁旧结构
5. 当前先单模型可用，但架构必须随时能恢复交叉路由
6. 仓库不能再重新长成大文件

## 二、这次重做的唯一真相

### 1. 我们在做的不是普通生成器

我们做的是：

- 以创作者主权为前提
- 以正式事实和戏剧推进链为双主轴
- 以运行态、ledger、audit、repair 为闭环
- 以旧项目深层业务逻辑为能力来源

的 AI 剧本创作赋能引擎。

### 2. 当前不可推翻的边界

- 正式事实定义权只属于创作者
- 正式事实正式声明动作只允许发生在 `outline`
- 下游工序只能消费 confirmed formal facts
- `renderer` 只做展示、编辑、引导
- `preload` 只做桥接
- `main` 才能拥有运行编排、模型调用、审计修补执行权
- `shared` 只放纯规则、合同和可复用语言

### 3. 新增的不可推翻体验底线（从今天开始写进主线）

我们不是在做“给开发者看的面板”，而是做“给创作者用的工具”。  
所以从今天开始，这条体验底线正式写进总主线，和边界门禁同级：

- 默认入口必须是 **Chat-first**：像 GPT 一样先聊天，系统引导用户聊完 1 轮，把信息收束成唯一真相对象。
- “唯一真相”不是让用户填一堆表单，而是让系统把聊天内容抽成：`storyIntent -> outlineDraft -> characterDrafts -> (后续详纲/剧本)` 的受控链路。
- UI 必须可读、可用、可解释：用户一眼能看懂“我现在要说什么 / 系统会产出什么 / 我下一步该做什么”。

这不是表层优化，而是黄金链路能否跑通的前置条件。

### 3. 当前真正的工程判断

不是“骨架没搭”，而是：

- 骨架方向已经对了
- 但还有几处本质没封死
- 如果现在不收口，后面还是会重新长歪

## 四、这次之后只按 7 个大任务推进

以后不再把零散小计划当总主线。  
主线只剩下面 7 个大任务，且必须按顺序推进。

### 大任务 1：封死定义权与工序边界

本质问题：

- 谁能定义正式事实
- 正式事实怎样受控流到下游

必须做到：

- `outline` 成为唯一正式声明入口
- `character / detailed_outline / script` 只能消费 confirmed formal facts
- stage contract 和 input contract 变成服务级门禁
- UI 不得靠临时状态偷偷补真相

交付标准：

- 下游不能偷补上游
- 事实定义权、校验权、升格建议权彻底分清
- 文档、合同、UI 提示一致

### 大任务 2：锁定 DeepSeek 单通道运行，但保留交叉路由架构

本质问题：

- 当前产品要先稳定可用，不要一边重构一边多模型漂移

必须做到：

- 当前运行时默认只启用 `DeepSeek`
- `Gemini Flash / Gemini Pro` 保留为可恢复 lane，不删除结构
- router 仍然存在，但“当前激活策略”和“未来多 lane 能力”分离
- UI 只展示当前可用 lane，不假装所有 lane 都在线

交付标准：

- 现在能稳定单模型跑
- 将来恢复交叉路由不需要重构目录和合同
- `.env`、runtime policy、provider summary 三处口径一致

### 大任务 3：先拆结构债，再继续加能力

本质问题：

- 债务文件一旦继续加功能，就会回到旧项目那条死路

这轮必须优先处理的债务点：

- `src/renderer/src/App.tsx`
- `src/renderer/src/features/script/ui/ScriptStage.tsx`
- `src/main/application/script-generation/start-script-generation.ts`
- `src/main/application/script-generation/build-execution-plan.ts`
- `src/preload/api/workflow.ts`
- `src/shared/domain/policy/*` 平面堆叠问题
- `renderer/src/components/*` 与 `renderer/src/features/*/ui/*` 双轨过渡问题

交付标准：

- 债务文件不再继续长
- 大职责被拆成稳定目录
- 旧过渡壳文件开始退场
- 之后新增功能只能落在拆好的位置

### 大任务 4：继续迁旧项目最值钱的业务资产

本质问题：

- 新项目不能只结构更漂亮，必须继承旧项目为什么能写出好剧本

必须按这个顺序迁：

1. 正式事实主权和故事契约层
2. 戏剧推进链和关系施压规则
3. audit issue 到 repair action 的映射体系
4. prompt 收缩策略和运行时上下文治理
5. 模型路由经验和 env 语义

交付标准：

- 迁进来的是政策、合同、规则、映射
- 不是把旧仓库 service 原封不动复制进来
- 每迁一轮都能解释“这条能力为什么能让剧本更好”

### 大任务 5：把旧项目经验沉成新仓库自己的正式资产

本质问题：

- 学到了但没沉淀，等于没真正拥有

必须做到：

- `runtime / audit / repair / progression / pressure` 变成正式 policy 资产
- 每条资产带版本、来源、适用范围、变更原因
- 关键策略能追溯到旧项目来源或新仓库演化来源
- repair mapping 并入统一 lineage 体系

交付标准：

- 新仓库不再依赖“记得旧项目怎么写”
- 策略资产可追溯、可版本化、可逐步替换

### 大任务 6：打通一条真人可走通的黄金链路

本质问题：

- 不走真人链路，系统永远可能只是在局部演示

必须做到：

- Chat-first 入口：像 GPT 一样引导用户聊天输入，聊完 1 轮后系统能把内容收束为 `storyIntent`（唯一真相对象）。
- 一键生成：基于 `storyIntent` 自动生成粗纲与人物小传（草稿），并写入项目真相中心。
- 正式事实主权：系统只能生成“候选正式事实（draft）”，必须由用户在粗纲工序确认后才可进入下游。
- 从粗纲到人物
- 从人物到详纲
- 从详纲到剧本生成
- 从剧本生成到审计修补

形成一条真实可验证链路。

交付标准：

- build 通过
- typecheck 通过
- 一条样本项目能完整走通
- 不靠手工补字段、不靠 UI 偷传状态
- 坏内容能被 gate 拦住，而不是只打分提醒
- 用户一眼看懂：入口是聊天，引导清楚，生成结果与“唯一真相对象”对齐，不再要求用户理解内部结构名词才能用
- 关键门禁证明：用户确认正式事实会转成 `user_declared + confirmed`，下游 stage contract 能稳定消费；Electron e2e 已验证 Script Gate 可从 disabled 变为 ready（`MOCK_AI_ENABLE=1`）

### 大任务 7：收口仓库活文档，只保留单一行动真相

本质问题：

- 计划越来越多，最后就会变成“代码一份真相，文档三份真相”

必须做到：

- 只保留一份主计划活文档
- 只保留一份当前任务卡
- `worklog` 只记过程，不再充当第二个计划库
- 过时方案全部归档
- 每天先做 `D:\\vivecoding规则` 对齐，再开工

交付标准：

- 开工时找得到唯一主计划
- 找得到唯一当前任务
- 旧计划不再和当前代码打架

## 六、每个大任务的硬门禁

以后任何开发都必须同时满足下面门禁。

### 结构门禁

- 新文件硬上限 `220` 行
- `180` 行进入预警区
- 老文件超过 `260` 行只允许拆和修，不允许再叠新功能
- 单函数硬上限 `40` 行
- 根装配文件只允许做组合，不允许承载业务总控

### 边界门禁

- `renderer` 不得拥有运行编排、provider 路由、ledger 计算、audit 核心规则
- `preload` 不得拥有业务判定和兼容补丁
- `main` 才能承受执行权
- `shared/domain` 必须保持纯规则

### 主权门禁

- AI 不得写 confirmed formal facts
- repair 不得改写正式事实
- compatibility 不得重新掌权
- Prompt 只有表达权，没有定义权

### 文档门禁

- 同主题不再并列新增多份“活计划”
- 过时方案进归档
- 当前任务只看 `active-task`
- 总主线只看这份计划
