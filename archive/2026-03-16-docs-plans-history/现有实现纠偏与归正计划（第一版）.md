这份文档不是否定前面做过的工作。

这次纠偏的目标是：

- 保留已经搭好的本质骨架
- 把不够纯的地方降回正确层级
- 后续开发只围绕本质继续推进

## 三、当前需要纠偏的地方

### 纠偏点 1：部分 runtime 推导还在 renderer

当前问题：

- `renderer` 仍在调用 hook 组合生成 runtime 预览
- `useProjectStagePersistence` 同时承担“保存草稿”和“推导 runtime”

为什么这不够本质：

- runtime 推导属于应用层，不属于表现层
- 前端应尽量展示结果，不应长期承担运行规则推导

归正方向：

- 把 runtime 推导逐步收回 `main/application`
- `renderer` 只拿最终 DTO 和状态

### 纠偏点 2：输入合同虽然已经有 gate，但还不是服务 gate

当前问题：

- 页面按钮会拦住
- 但主进程还没有一个“真正拒绝执行”的统一启动入口

为什么这不够本质：

- 真正的工序边界必须由服务层把守
- UI gate 只能算表现层辅助

归正方向：

- 新增真实执行入口
- 所有生成都必须先过 `main` 的 gate

### 纠偏点 3：运行态对象存在，但还不是完整状态机

当前问题：

- 已有 plan / progress / resume / failure
- 但它们还没有统一的事件推进机制

为什么这不够本质：

- 现在更像“状态对象集合”
- 还不是“状态规则系统”

归正方向：

- 下一步把状态推进规则做成正式状态机
- 让 batch start / batch fail / batch pause / batch resume 都有统一规则

### 纠偏点 4：状态台账还没进入核心层

当前问题：

- 现在有合同，有计划，有运行态
- 但还没有连续性台账

为什么这不够本质：

- 旧项目真正能稳定写长链的底层能力之一就是 ledger

归正方向：

- 尽快为 ledger 留正式核心位置
- 不能再把它往后排成“增强项”

### 纠偏点 5：审计修补还没进主链

当前问题：

- 当前还停在前置约束和运行控制
- 后置质量闭环还没进主链

归正方向：

- 后续执行器一旦接入，就要同时为审计修补预留接口

## 四、纠偏后的四段执行顺序

### 第一段：总纲锁死

目标：

- 用不可推翻的本质约束全仓库

要做的事：

- 写本质总纲
- 写纠偏计划
- 以后所有开发围绕这两份文档

### 第二段：把前四条本质做硬

目标：

- 真相源
- 工序边界
- 合同
- 状态机

要做的事：

- 把 gate 服务化
- 把 runtime 状态机化
- 把阶段合同和输入合同进一步统一

### 第三段：把旧项目最深层价值迁进来

目标：

- ledger
- audit/repair
- runtime policy

### 第四段：最后再做表现层优化

目标：

- 让 UI 只承担展示和交互
- 让表现层可换而不伤本质层

1. 把真实批次执行器接到 gate 后面
2. 把 runtime 对象推进成正式状态机
3. 开始第一版状态台账

这是下一轮开发的主线。
