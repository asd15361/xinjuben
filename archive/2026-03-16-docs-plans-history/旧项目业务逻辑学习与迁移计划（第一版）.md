# 旧项目业务逻辑学习与迁移计划（第一版）

## 一、这份计划的目标

这份计划的目标不是“参考一下旧项目”。

而是要系统完成三件事：

1. 深入学习旧项目 `D:\project\juben` 里真正让剧本质量变高的业务逻辑
2. 把这些业务逻辑拆成可迁移的能力模块
3. 按新项目 `D:\xinjuben` 的正确架构，逐步迁入，不重演旧项目的维护灾难

一句话：

**我们这次不是复刻旧项目代码，而是继承旧项目的创作能力和产品智慧。**

## 二、总原则

这份计划全程遵守下面 6 条原则。

### 1. 学逻辑，不抄结构

旧项目最值钱的是：

- 它为什么能写出好剧本
- 它每一步怎么把用户需求收紧
- 它怎么让粗纲、人物、详纲、剧本逐步变好
- 它怎么用规则、提示词、技能、对齐器和市场规律协同工作

旧项目最不该继承的是：

- 超大页面总控
- renderer 层服务爆炸
- 一堆补丁式修复链

### 2. 先学“为什么有效”，再决定“怎么迁”

对每个旧模块都要先回答：

- 这个模块是干嘛的
- 它为什么能提升剧本质量
- 它依赖什么输入
- 它输出什么
- 它应该归到新项目哪一层

### 3. 搬“能力”，不搬“历史包袱”

旧项目里很多 API、env、提示词、路由、阶段逻辑都能继承。

但继承方式必须分三类：

- `直接继承`
- `继承思路，重写实现`
- `只保留经验，不搬代码`

### 4. 先主链，后外围

优先学习和迁移这些：

1. 对话链
2. 粗纲链
3. 人物小传链
4. 详细大纲链
5. 剧本生成链

再迁这些：

6. 审计修订
7. 市场情报
8. 学习反馈
9. 恢复续跑
10. 导出治理

### 5. 每迁一层，都要验证“质量有没有保住”

新项目不能只是结构更漂亮。

它必须保住旧项目真正的优势：

- 剧本质量
- 创作逻辑强度
- 需求贴合度
- 结构稳定性

### 6. env 和 API 可以迁，但要做治理

旧项目的 API、模型路由、环境变量配置都很有价值。

但迁移时必须做到：

- 配置归位
- 文档清楚
- 不泄露敏感信息
- 不把 provider 逻辑写死在 UI

## 五、每个模块怎么判断值不值得学

对旧项目里的每个模块，都按这个模板判断。

### 模块判断模板

1. 这个模块属于哪条业务链
2. 它解决了什么创作问题

4. 它的输入是什么
