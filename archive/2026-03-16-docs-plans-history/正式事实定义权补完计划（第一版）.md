# 正式事实定义权补完计划（第一版）

## 一、为什么这份计划必须立刻补

“正式事实”不是普通字段。  
它是后面这些能力共同围绕的结构根：

- 粗纲结构成立与否
- 人物是不是在承载核心事实
- 详细大纲是不是在推动事实升格
- 剧本生成是不是在围绕正确冲突推进
- ledger 记录的是不是正确事实
- audit / repair 修的到底是不是该修的点

如果“谁有资格定义正式事实”没有被系统正式锁死，后面所有能力都有可能围着一个系统自己偷偷发明出来的“伪正式事实”运转。

那样的话，项目就算功能很多，也还是不本质。

---

## 二、这件事现在做到哪一步

### 已经做到的

- `FormalFact` 作为概念已经存在于粗纲合同里  
  见：`src/shared/contracts/workflow.ts`
- 已经有最小“正式事实有效性校验”  
  见：`src/shared/domain/formal-fact/definition-engine.ts`
- 文档里多次强调“正式事实很重要”“正式事实要绑定主题和主线冲突”

### 还没做到的

- 没有正式定义“谁有定义权”
- 没有正式定义“谁只有校验权”
- 没有正式定义“谁能判定升格”
- 没有正式定义“谁能确认升格”
- 没有把这些权责写入 contract
- 没有把这些权责写入 outline 工序边界
- 没有防止 AI / ledger / audit 偷偷创造正式事实

### 当前结论

现在的系统有“正式事实概念”，但还没有“正式事实权力结构”。

这件事如果不补上，后面很多功能虽然看起来在围绕正式事实工作，实际上并没有守住最核心的边界。

---

## 三、必须立刻锁死的四权分离

这件事以后只能按“四权分离”来做。

### 1. 定义权

定义：

谁有资格把一个创作元素正式声明为“正式事实”。

必须锁死为：

- **只有创作者有定义权**
- **定义动作只能发生在粗纲工序**
- **系统、AI、审计器、ledger 都没有定义权**

系统后果：

- 只有用户在 `outline` 工序里正式声明的事实，才允许进入 `confirmed formal facts`
- 其他所有生成、建议、推测出来的事实，都只能算：
  - 候选事实
  - 建议事实
  - 推断事实

但不能自动进入正式事实主链

### 2. 校验权

定义：

谁有资格判断“这个被声明的事实像不像一个合格正式事实”。

必须锁死为：

- **系统拥有校验权**
- **校验权不包含定义权**

系统能做：

- 检查事实是否绑定主线冲突
- 检查事实是否绑定主题
- 检查事实是否具有跨工序价值
- 输出评分、问题、建议

系统不能做：

- 自动把一个普通元素升成正式事实
- 自动替换用户定义
- 自动删除用户定义

### 3. 升格判定权

定义：

谁有资格判断“某一段情节是否构成正式事实的升格节点”。

必须锁死为：

- **系统拥有升格判定权**

系统能做：

- 检测某场戏是否推动了事实升格
- 判断升格是否同时命中冲突、人物弧光、主题落地
- 输出“升格成立 / 升格不足 / 升格断裂”

系统不能做：

- 自动替用户决定“这就是核心升格点”

### 4. 升格确认权

定义：

谁有资格确认某个升格节点被正式采纳为结构真相。

必须锁死为：

- **创作者拥有升格确认权**

也就是说：

- 系统可以判定
- 用户才能确认

---

## 四、以后系统必须遵守的一句话原则

### 正式事实定义原则

**正式事实只能由创作者在粗纲工序中声明；系统只允许校验、追踪、判定，不允许代替创作者定义。**

这句话以后必须同时写进：

- contract
- domain
- 工序边界规则
- UI 文案
- 审计与修补规则

---

## 五、要补的核心数据结构

当前 `FormalFact` 太薄，只是“一个事实长什么样”。

以后必须升级为“一个正式声明过的事实长什么样”。

## 5.1 `FormalFact` 合同升级

当前只有：

- `id`
- `label`
- `description`
- `linkedToPlot`
- `linkedToTheme`

必须补成至少：

```ts
type FormalFactAuthorityType = 'user_declared' | 'ai_suggested' | 'system_inferred'
type FormalFactStatus = 'draft' | 'confirmed' | 'rejected'
type FormalFactLevel = 'core' | 'supporting'
type FormalFactDeclaredStage = 'outline'

interface FormalFact {
  id: string
  label: string
  description: string
  linkedToPlot: boolean
  linkedToTheme: boolean
  authorityType: FormalFactAuthorityType
  status: FormalFactStatus
  level: FormalFactLevel
  declaredBy: 'user' | 'system'
  declaredStage: FormalFactDeclaredStage
  createdAt: string
  updatedAt: string
}
```

### 为什么必须这样补

因为系统后面要能明确区分：

- 这是创作者正式声明的事实
- 这是 AI 仅供参考的候选事实
- 这是系统从文本里推断出来但没有资格入主链的事实

如果不分层，定义权永远是糊的。

---

## 六、工序边界必须怎么锁

### 6.1 粗纲工序

粗纲工序必须成为：

- 正式事实的唯一声明入口

粗纲工序允许：

- 新建正式事实
- 编辑正式事实
- 删除未确认正式事实
- 确认正式事实级别

粗纲工序不允许：

- AI 自动写入 `confirmed` 正式事实

### 6.2 人物工序

人物工序不能定义正式事实。

人物工序只能：

- 引用已确认正式事实
- 指定某人物如何承载某正式事实

### 6.3 详细大纲工序

详细大纲工序不能新定义正式事实。

它只能：

- 使用已确认正式事实
- 设计升格路径
- 标记候选升格节点

### 6.4 剧本工序

剧本工序不能新定义正式事实。

它只能：

- 引用已确认正式事实
- 展示事实推进是否落实
- 标记升格是否成立

## 七、哪些模块以后绝不能拥有定义权

下面这些模块以后都不能拥有定义权：

- `shared/domain/formal-fact/definition-engine.ts`
  - 只有校验权
- ledger 相关模块
  - 只有追踪权
- audit 模块
  - 只有识别问题权
- repair 模块
  - 只有修补权
- AI 路由与执行模块
  - 只有执行权
- renderer 提示层
  - 只有展示权

## 九、这件事的落地顺序

### 第一步：锁 contract

先把“正式事实到底是什么、谁声明、什么状态”写死。

### 第二步：锁权责

把“定义权 / 校验权 / 升格判定权 / 升格确认权”落实到 domain 和 application。

### 第三步：锁工序边界

让粗纲成为唯一声明入口，其他工序只能引用。

### 第四步：锁 UI

让 renderer 的文案、交互、按钮全部跟权责一致。

### 第五步：锁审计与 ledger

让它们只能围绕 `confirmed formal facts` 运转。

---

## 十、做完之后的验收标准

满足下面这些，才算真的补完：

1. 系统里能明确区分：
   - 用户声明的正式事实
   - AI 建议的候选事实
   - 系统推断的非正式事实
2. 只有粗纲工序能创建 `confirmed formal facts`
3. 其他工序尝试创建正式事实时会被系统拒绝
4. 校验器只能给建议，不能自动定义
5. ledger / audit / repair 只消费 `confirmed formal facts`
6. 文档、contract、UI 文案完全一致

---

## 十一、最后一句结论

后面系统所有能力都可以围绕正式事实生长，  
但前提必须是：

**正式事实先被谁定义这件事，要比“怎么围绕它生成内容”更早、更硬、更不可动摇。**

如果这件事没锁死，后面一切都只是围着一个可能是假的中心在转。
