# 新项目架构本质总纲（不可推翻版）

## 一、这份文档的地位

这不是阶段性建议，也不是某次讨论的临时方案。  
这份文档定义的是新项目的“不可推翻架构本质”。

后面无论：

- 换模型
- 换 prompt
- 换 UI
- 换交互
- 换工作流细节
- 换榜单策略

都不能破坏这里定义的 7 条本质。

如果某个实现和这 7 条冲突，优先推翻实现，不推翻本质。

---

## 二、七条不可推翻本质

### 本质 1：单一项目真相源

系统里所有创作事实，都必须归属于同一个项目真相中心。  
不能出现：

- 页面各自维护一套真相
- 工序之间互相猜状态
- 运行态和创作态分裂

项目真相中心至少统管：

- 故事意图
- 粗纲
- 人物
- 详纲
- 剧本
- 剧本运行态

这条本质决定了：  
“项目”是第一实体，页面只是项目的观察窗。

### 本质 2：四工序刚性边界

粗纲、人物、详细大纲、剧本是四道刚性工序，不是四个标签页。  
它们之间只能按正式边界流动，不能乱流。

允许：

- 上游产出正式交给下游
- 下游引用上游已确认内容

不允许：

- 下游偷偷补上游结构
- 页面直接跨工序写入另一工序真相
- 某工序未完成时让后续工序假装可执行

### 本质 3：阶段合同与输入合同强绑定

每道工序必须同时具备：

- 阶段合同：上游正式产出长什么样
- 输入合同：进入本工序前必须满足什么条件

阶段合同保证“引用的是谁”。  
输入合同保证“能不能开始”。

这两者必须是系统级规则，不是界面提醒。

### 本质 4：剧本生成运行态必须是单一状态机

剧本生成不能只是“一次调用模型”。  
它必须是一个正式运行系统，至少有：

- 执行计划
- 进度板
- 续跑解析
- 失败恢复

这四者必须共享同一套状态来源。  
不能各自推导、各自判断、各自记忆。

### 本质 5：状态台账必须存在

系统要想稳定写长链路剧本，必须有状态台账。  
它不是锦上添花，而是连续性的根。

状态台账至少要承接：

- 人物状态
- 关系状态
- 正式事实状态
- 道具与身份状态
- 未解钩子
- 关键冲突推进状态

没有台账，就不可能真正做到“可维护、可扩展、还能持续写好”。

### 本质 6：审计修补闭环必须进入主链

生成不是结束。  
真正的主链应该是：

输入合同 -> 执行 -> 审计 -> 修补 -> 再确认

也就是说，质量控制不能停留在“打个分”或“显示建议”。  
系统必须具备：

- 问题识别
- 问题分类
- 定向修补
- 修补结果回写

### 本质 7：AI 受控执行，策略全部外置

AI 只是一种执行能力，不是系统本体。  
所以：

- 模型调用只能走受控执行层
- key 和 provider 只允许留在 `main`
- 路由、回退、重试、超时都属于策略层
- prompt 可以调整，但不能承担架构职责

这条本质决定：  
AI 可以换，但系统边界不能跟着塌。

---

## 三、什么不是本质

下面这些都重要，但都不是架构本质：

- 当前按钮长什么样
- 当前页面布局
- 当前推荐 lane
- 当前默认 targetEpisodes
- 当前文案
- 当前建议展示形式
- 当前 prompt 细节

这些都属于：

- 策略层
- 表现层
- 阶段实现层

它们可以被推翻，只要不破坏 7 条本质。

---

## 四、以后所有开发的判断规则

每次开发前都先问 3 个问题：

1. 这次改动是在加强本质，还是只在增加策略？
2. 这次改动会不会让真相源分裂？
3. 这次改动会不会让工序边界变软？

如果答案不清楚，就先不要继续堆功能。

---

## 五、当前执行优先级

按照本质优先级，当前顺序必须是：

1. 单一项目真相源
2. 四工序刚性边界
3. 阶段合同与输入合同强绑定
4. 运行态单一状态机
5. 状态台账
6. 审计修补闭环
7. AI 受控执行与策略外置

这 7 条就是新项目后面所有开发的最高约束。
