# 第三阶段第一刀发布主权阻断重排（第一版）

## 这份文档只回答一件事

既然第三阶段要收“发布主权”，那到底先收什么、为什么先收、按什么顺序收。

## 先说结论

第三阶段第一刀只按下面 4 个顺序推进：

1. 先收正式发布身份
2. 再收正式环境边界
3. 再收内部测试入口边界
4. 最后收对外发布责任边界

这个顺序只看一个标准：

**哪一层最像“开发模板还没退场”，就先收哪一层。**

## 为什么现在轮到第三阶段

因为第二阶段已经把“质量承诺”收成正式边界了。

现在最前面的本质问题，已经不再是“质量边界说不清”，
而是：

**这个项目看起来、跑起来、发布起来，还不像一个正式产品。**

## 第三阶段真正要解决的本质问题

第三阶段不是简单改几个名字。

第三阶段真正要解决的是：

**把“开发中的系统”推进成“正式对外产品”。**

## 当前最前面的 4 类阻断

### 一、正式发布身份还没成立

为什么排第一：

- 现在安装包身份、发布信息、产品身份还有明显模板味
- 这是用户、平台、分发渠道第一眼就能看见的问题

当前表现：

- `author` 还是模板值
- `homepage` 还是模板站
- `appId` 还是模板值
- 自动更新地址还是示例地址
- Windows app user model id 还是模板值

这一步要解决什么：

- 先让产品身份不再像一个拿来练手的空壳

### 二、正式环境边界还没成立

为什么排第二：

- 就算身份改了，如果正式环境边界还是开发态，发布主权还是站不住

当前表现：

- 主窗口还没把正式环境边界收紧
- 运行依赖还明显偏开发态、本地态

这一步要解决什么：

- 让正式环境和开发环境不再混成一团

### 三、内部测试入口边界还没成立

为什么排第三：

- 内部测试入口直接露给外部用户，会直接打穿正式产品感
- 但它排在身份和环境后面，因为它更像“露在外面的开发痕迹”

当前表现：

- 头部还挂着明显内部测试入口
- 内部身份和测试动作还直接暴露在前台

这一步要解决什么：

- 把“内部人看的东西”和“外部人该看到的东西”切开

### 四、对外发布责任边界还没成立

为什么排第四：

- 前三项更偏“产品像不像正式版”
- 这一项更偏“正式版到底该怎么担责”

当前表现：

- AI 运行强依赖本地环境变量密钥
- 对外发布后的接入责任边界还没钉死

这一步要解决什么：

- 正式说清楚：外部发布时，什么能力由产品承担，什么不能再靠开发环境硬撑

## 第三阶段第一刀的正式执行顺序

### 第一步：先把发布主权阻断重新排成唯一顺序

目标：

- 后面不再东改一点西改一点
- 只收最前面的正式产品阻断

### 第二步：给每一类阻断写清当前到底怎么暴露出来

目标：

- 不再泛泛说“还有开发味”
- 要说清楚到底哪里还像模板、哪里还像内测工具

### 第三步：给每一类阻断写清先从哪一刀下手

目标：

- 从“知道哪里不对”推进到“知道先收哪一刀”

### 第四步：做这一刀的总判断

目标：

- 明确回答：发布主权是不是开始成立了

## 这一刀验收标准

只有下面 4 件事同时成立，才算这一刀真的解决本质问题：

1. 发布主权阻断已有唯一顺序
2. 每一类阻断当前暴露点已经写清
3. 每一类第一刀从哪下手已经写清
4. 总判断能说明项目是不是比现在更像正式产品了

## 当前正式判断

当前正式判断是：

- 第三阶段已经开始
- 当前最该先收的是正式发布身份
- 发布主权现在还不能说已经成立

## 最后结论

第三阶段第一刀当前单一口径就是：

**先把产品身份从模板味里拉出来，再把正式环境收紧，再把内部测试入口收回去，最后钉死对外发布责任边界。**
