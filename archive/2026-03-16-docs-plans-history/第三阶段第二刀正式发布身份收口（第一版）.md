# 第三阶段第二刀正式发布身份收口（第一版）

## 这份文档只回答一件事

第三阶段既然先收“正式发布身份”，那这一刀到底先收了什么、为什么先收这些。

## 先说结论

第三阶段第二刀当前先收了 4 个最容易露馅的点：

1. 把系统级产品身份从模板值改回项目身份
2. 把安装包发布身份从模板值改回项目身份
3. 把前台最明显的内部测试露馅点先藏回开发态
4. 把产品展示名称收回同一口径

这一步解决的不是全部发布主权，
但已经先把最明显的“开发模板味”压下去了一层。

## 为什么先收这 4 个点

因为它们最容易一眼露馅。

大白话说：

- 用户一打开
- 平台一识别
- 安装包一生成

最先暴露项目还不像正式产品的，
就是这 4 个地方。

## 这一刀先收了什么

### 一、系统级产品身份先收回项目身份

这一步先解决：

- 系统识别还像模板工程

当前动作：

- 把系统级产品身份从默认模板值，收回项目自己的身份口径

这一步的意义：

- 先让系统层不再把自己认成一个通用模板壳

### 二、安装包发布身份先收回项目身份

这一步先解决：

- 安装包、发布配置里还残留模板味

当前动作：

- 把安装包身份、维护者信息，从模板口径收回项目口径
- 把明显假的示例发布地址先退场

这一步的意义：

- 先避免“外表像在发布，实际还是示例壳”这种假正式状态

### 三、前台最明显的内部测试露馅点先藏回开发态

这一步先解决：

- 最明显的内部测试按钮直接露在正式脸上

当前动作：

- 把明显只给开发和测试用的入口，先收回开发态可见

这一步的意义：

- 先把“外部人一眼就会怀疑这还是测试版”的露馅点压住

### 四、产品展示名称收回同一口径

这一步先解决：

- 前台、系统信息、产品展示名还不完全是一套话

当前动作：

- 把产品展示名统一回 `XINJUBEN`

这一步的意义：

- 先让产品对外说自己是谁时，只剩一套口径

## 这一刀还没解决什么

这一刀还没解决的是：

1. 正式环境边界还没收紧到终态
2. 内部测试入口边界还没整体收完
3. 对外发布责任边界还没钉死

所以这一刀现在可以算“先压住最明显露馅点”，
还不能算“发布主权已经成立”。

## 这一步解决了什么本质问题

这一步真正解决的是：

**项目已经开始从“模板工程的脸”往“正式产品的脸”转。**

这件事不容易被推翻。

## 当前正式判断

当前正式判断是：

- 第三阶段第二刀已经起步并拿到第一批结果
- 项目最明显的模板身份和测试露馅点已经先压住
- 发布主权还没有彻底成立，还要继续往正式环境边界推进

## 验证结果

当前验证已经通过：

- `npm run typecheck`
- `npm run build`

## 最后结论

第三阶段第二刀当前正式口径就是：

**先把最明显的模板身份和测试露馅点收掉，让项目先开始像一个正式产品，而不是像一个开发壳。**
