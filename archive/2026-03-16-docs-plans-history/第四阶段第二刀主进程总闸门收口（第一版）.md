# 第四阶段第二刀主进程总闸门收口（第一版）

## 这份文档只回答一件事

第四阶段既然先收“主进程总闸门混装点”，那这一刀到底先收了什么、为什么先收这些。

## 先说结论

第四阶段第二刀当前先收了 1 件关键事：

**把剧本生成总闸门从“自己包办很多活”，收成“只挂门牌、把具体活分出去”。**

这一步解决的不是全部结构问题，
但已经先把最危险的主路口往单一职责上拉回了一层。

## 为什么先收这个口子

因为它是主路口。

大白话说：

- 只要总闸门自己又接单、又调度、又记失败、又做预览
- 后面任何新动作都还会继续往这个门口堆

所以不先收这个口子，第四阶段后面拆别的地方也会一直被它拖回去。

## 这一刀先收了什么

### 一、把“计划和进度”挂回计划口

这一步先解决：

- 总闸门不该继续把计划、进度、恢复这些事全揽在自己身上

当前动作：

- 把计划、进度、恢复相关动作，收回到独立挂口里

这一步的意义：

- 让总闸门回到“挂门牌”，不再自己兼做计划台

### 二、把“开工和预览”挂回运行口

这一步先解决：

- 总闸门不该继续自己管开工、失败收口、账本预览

当前动作：

- 把真实开工和账本预览收回到独立挂口里

这一步的意义：

- 让总闸门不再兼做调度室和故障台

### 三、让总闸门回到薄壳

这一步先解决：

- 一个总闸门文件继续胖下去，后面谁都不敢动

当前动作：

- 现在总闸门只负责把计划口和运行口挂起来

这一步的意义：

- 先把“接单”和“下场干活”分开

## 这一刀还没解决什么

这一刀还没解决的是：

1. 工作区总入口还没收
2. 审计总表还没收
3. 账本后收口还没收
4. 前台这边的几个危险口子还没真拆

所以这一刀现在可以算“先把主路口拉薄”，
还不能算“第四阶段已经完成”。

## 这一步解决了什么本质问题

这一步真正解决的是：

**主进程总闸门已经不再继续一边接单、一边自己下场干很多活。**

这件事现在不容易被推翻。

## 验证结果

当前验证已经通过：

- `npm run typecheck`
- `npm run build`

## 当前正式判断

当前正式判断是：

- 第四阶段第二刀已经拿到结果
- 最危险的主进程总闸门已经先拉回薄壳
- 第四阶段下一刀应该继续收工作区总入口

## 最后结论

第四阶段第二刀当前正式口径就是：

**先把主进程总闸门从“大柜台”拉回“挂门牌的薄壳”，不让它继续自己包办很多活。**
