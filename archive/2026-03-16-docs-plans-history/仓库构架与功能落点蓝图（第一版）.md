# 仓库构架与功能落点蓝图（第一版）

## 一、这份蓝图现在解决什么问题

这份蓝图不再泛泛而谈“架构要好”，  
而是把三件事钉死：

1. 仓库以后按什么边界长
2. 旧项目功能应该精确落到哪里
3. 哪些地方绝对不能再长成大文件

这份文档是仓库级施工图。  
以后写代码前先对照这里落位，不准想到哪塞到哪。

---

## 二、先说结论：当前仓库方向是对的，但纪律还不够硬

现在已经有这些正确骨架：

- `src/shared`
- `src/main`
- `src/preload`
- `src/renderer`

也已经开始把这些东西收口：

- 正式事实
- stage contract
- input contract
- runtime
- ledger
- audit
- repair

但现在真正的风险不是“没有架构”，而是：

- 债务文件已经接近临界线
- 过渡壳文件还没退场
- policy 目录开始变平面杂糅
- 文档活页太多，容易形成多份真相

所以这次不是重做骨架，  
而是把骨架的纪律补硬。

---

## 三、仓库只按 5 个边界组织

以后仓库只允许按下面 5 个边界组织。

### 1. `src/shared`

职责：

- 合同
- 纯规则
- 纯语言
- 不依赖运行时的领域逻辑

禁止：

- React
- Electron
- IPC
- fetch
- 文件系统

### 2. `src/main`

职责：

- use case 编排
- provider 执行
- runtime 状态
- ledger / audit / repair 执行
- 持久化

禁止：

- 直接依赖 renderer store
- 直接写 UI 分支

### 3. `src/preload`

职责：

- 暴露白名单 API
- 只做 IPC 桥接

禁止：

- 业务规则
- 兼容补丁
- 默认值推断
- 数据修正

### 4. `src/renderer`

职责：

- 展示
- 编辑
- 引导
- 本地交互态
- feature hook 组合

禁止：

- 模型调用执行权
- router 决策
- ledger 计算
- audit / repair 核心规则
- 正式事实定义权

### 5. `docs`

职责：

- 长期规则
- 当前任务
- 单一主计划
- 工作记录
- 归档

禁止：

- 同主题并列多份活计划
- 多份现行真相

## 五、每类能力精确落点

| 能力 | 必须放哪里 | 绝不能放哪里 | 说明 |
|---|---|---|---|
| 正式事实定义、校验、升格规则 | `src/shared/domain/formal-fact/` + `src/main/application/formal-fact/` | `renderer` 页面文件 | 规则是纯的，执行是服务层的 |
| 故事契约、用户锚点 ledger | `src/shared/domain/story-contract/` | `renderer` / `preload` | 这是驱动 prompt、audit、repair 的主资产 |
| 戏剧推进链、关系施压 | `src/shared/domain/drama-progression/` + `src/shared/domain/policy/progression/` + `pressure/` | `renderer/components` | 这是业务真本事，不是 UI 文案 |
| 模型路由、单通道激活策略 | `src/main/application/ai/router/` | `renderer` | 当前只启用 DeepSeek，也必须放这里 |
| provider 配置和 env | `src/main/infrastructure/runtime-env/` | `renderer` | 密钥和路由执行权不能进前端 |
| 执行计划 | `src/main/application/script-generation/plan/` | `ScriptStage.tsx` | UI 只看结果 |
| prompt 组装 | `src/main/application/script-generation/prompt/` | `renderer` | prompt 是执行输入，不是页面逻辑 |
| 生成运行态 / 续跑 / 失败恢复 | `src/main/application/script-generation/runtime/` | `renderer hook` | 运行态必须只有一套真相 |
| ledger | `src/main/application/script-generation/ledger/` | `renderer/services` | ledger 是主链资产 |
| audit | `src/main/application/script-generation/audit/` | 面板组件 | 面板只展示 |
| repair 映射和执行 | `src/shared/domain/policy/repair/` + `src/main/application/script-generation/repair/` | `renderer` | repair 不得偷偷补定义 |
| IPC handler | `src/main/ipc/` | `main/index.ts` | `index.ts` 只做装配 |
| preload API | `src/preload/api/` | `preload/index.ts` 一坨大对象 | 以领域拆，不以“所有 workflow”堆一起 |
| feature UI / hook | `renderer/src/features/*` | `renderer/src/components` | feature 建好后，业务 UI 不得再堆根目录 |
| 通用组件 | `renderer/src/shared/ui/` | `features` 或 `components` 混放 | 只放无业务语义组件 |

## 六、当前必须盯死的结构风险点

### 1. 债务临界文件

- `src/renderer/src/App.tsx`
- `src/renderer/src/features/script/ui/ScriptStage.tsx`
- `src/main/application/script-generation/start-script-generation.ts`
- `src/main/application/script-generation/build-execution-plan.ts`
- `src/preload/api/workflow.ts`

这些文件现在还没爆，但已经进入“再长就歪”的区间。  
后续规则很简单：

- 只能拆
- 不能继续叠加新职责

### 2. 过渡双轨问题

当前仓库还有这种形态：

- `renderer/src/components/CharacterStage.tsx`
- `renderer/src/features/character/ui/CharacterStage.tsx`

以及：

- `renderer/src/components/DetailedOutlineStage.tsx`
- `renderer/src/features/detailed-outline/ui/DetailedOutlineStage.tsx`

这类“新路径只是转发旧路径”的文件只允许短期过渡。  
一旦 feature 化稳定，就必须删掉旧壳。

### 3. `renderer/components` 目录语义过松

以后这里不再允许放整块业务舞台。  
规则改成：

- `renderer/src/components` 只保留兼容过渡件，且必须逐步清空
- 通用组件进入 `renderer/src/shared/ui`
- 业务组件进入 `renderer/src/features/*/ui`

### 4. `shared/domain/policy` 目录平面堆叠

以后这里不再接受“都算 policy 就先塞进来”的做法。  
必须按子域继续分层：

- `runtime/`
- `audit/`
- `repair/`
- `progression/`
- `pressure/`

### 5. 文档也属于结构风险

当前 `docs/plans` 活文档过多。  
以后只保留：

- 单一主计划
- 正在生效的少量专项计划
- 当前任务卡
- 归档

旧版本、旧阶段、已失效计划必须归档。

## 七、以后新增功能的落位顺序

每次新增一个能力，必须按下面顺序走：

1. 先判断是不是跨层合同，如果是，先写 `shared/contracts`
2. 再判断是不是纯规则，如果是，写 `shared/domain`
3. 再写 `main/application`
4. 再接 `main/ipc`
5. 再接 `preload/api`
6. 最后再写 `renderer/features/*`

禁止反过来：

- 先写页面
- 页面里临时拼逻辑
- 长大以后再想办法塞回 main

## 八、文件和目录硬门禁

### 文件门禁

- 新文件硬上限：`220` 行
- `180` 行以上进入预警区
- 老文件超过 `260` 行视为结构债务文件
- 债务文件默认不允许继续追加新功能

### 目录门禁

- 每建一个新目录，必须先回答一句话：它的唯一职责是什么
- 回答不出来，就不准建

### 根文件门禁

下面这些根装配文件只允许做组合：

- `main/index.ts`
- `renderer/src/App.tsx`
- `preload/index.ts`

不允许写：

- 运行编排
- 审计修补
- 模型路由
- 复杂条件分发

### 过渡门禁

- 纯 re-export 壳文件不得长期保留
- 过渡文件必须在任务卡里写清退场点

## 十、以后判断是不是又在堆，只看这 8 个问题

1. 这段逻辑是不是放错层了
2. 这个文件是不是已经有两个以上主职责
3. 这是不是本该进 `shared/domain` 的纯规则
4. 这是不是本该进 `main/application` 的执行逻辑
5. 这是不是只因为现在写得顺手，才想塞进页面
6. 这个文件三周后会不会超过 `220` 行
7. 这个目录有没有唯一职责
8. 这是不是会让旧项目那种大 service 重演

只要有一个问题答不上来，就先别写代码。

---

## 十一、最后结论

新仓库不是要“功能先跑，再慢慢整理”，  
而是要从现在开始就把规则钉死：

- 功能放对层
- 文件不过大
- 主权不漂移
- 旧项目只迁灵魂，不迁旧壳

后面所有开发，都必须先看这份蓝图，再决定代码落哪里。
