# 文档管理

这份文档只给“整理文档的终端”看。

下次如果用户让你整理文档、对齐文档、归档文档，或者直接让你“看 `文档管理.md`”，你就先只看这份文档，不要自己重新猜流程。

## 1. 这份文档是干嘛的

它只解决 4 件事：

1. 哪些文档该看，哪些文档不要当现役入口。
2. 每个现役文档各自负责什么。
3. 什么内容该保留，什么内容该压缩，什么内容该归档。
4. 整理文档终端到底能改什么，不能改什么。

## 2. 整理文档终端默认怎么做

整理文档终端默认只做文档治理，不接管代码主线。

默认动作：

1. 看这份 `文档管理.md`。
2. 按需要查看现役文档和相关归档文档。
3. 整理 `项目原则.md`、`AI自我学习.md`、`项目结构.md`、`docs/归档/`。
4. 输出文档整理结果和剩余风险。

默认不要做的事：

1. 默认不要更新 `docs/聊天记录.md`。
2. 默认不要把自己这次“整理文档的过程”写进 `聊天记录.md`。
3. 默认不要顺手去改代码逻辑。
4. 默认不要把历史材料重新抬成现役入口。

只有两种情况，整理文档终端才可以接管 `docs/聊天记录.md`：

1. 用户明确点名让你来整理或重写 `docs/聊天记录.md`。
2. 用户明确说“存档”，并且就是让你这个终端执行这一轮文档归档和压缩。

## 3. 现役文档各自负责什么

### `AGENTS.md`

仓库最高口径。

负责：
- 全仓通用规则
- 主线工作规则
- 文档维护硬规则
- 读取顺序

不负责：
- 记录当前主线细节
- 记录历史长过程

### `docs/聊天记录.md`

这是干活终端的主线记录。

负责：
- 当前正在做什么
- 最近遇到了什么问题
- 怎么解决的
- 解决了没有

不负责：
- 记录整理文档终端自己的过程
- 长期原则总结
- 仓库结构说明

### `docs/项目原则.md`

这是长期原则和禁区。

负责：
- 长期有效的原则
- 禁区
- 验收底线
- 已经坐实的典型误判

不负责：
- 记录某一轮细节过程
- 记录临时任务步骤

### `docs/AI自我学习.md`

这是 AI 视角下的经验沉淀。

负责：
- 我对项目的理解
- 为什么过去反复失败
- 这轮为什么成、为什么败
- 下次最容易再犯什么错

不负责：
- 代替聊天记录记录当前主线滚动过程
- 代替项目结构描述仓库目录

### `docs/项目结构.md`

这是结构说明和维护职责说明。

负责：
- 仓库结构
- 主链落点
- 风险区
- 归档布局
- 四件套维护职责
- 干活终端每轮收口清单

不负责：
- 记录某一轮对话细节
- 代替项目原则记录长期禁区

### `docs/主线/`

这是当前主线计划区。

负责：
- 当前唯一一份主线计划文档
- 当前任务清单
- 当前进度和完成定义

不负责：
- 长期原则
- 历史过程堆叠
- 同时并列多份主线

### `docs/归档/`

这是历史证据区。

负责：
- 保留旧文档
- 保留历史计划
- 保留旧任务卡
- 保留测试说明和旧工具文档

不负责：
- 当现役入口
- 替代现役文档发声

## 4. 什么东西该留在现役文档里

满足下面任意一种，才值得留在现役文档：

1. 它是当前主线还在用的事实。
2. 它是长期原则，后面还会反复用到。
3. 它是已经反复踩过的坑，必须明确提醒下一终端。
4. 它是当前仓库结构和主链落点的重要变化。
5. 它会直接影响下一轮怎么判断、怎么执行、怎么验收。

## 5. 什么东西该压缩

满足下面这类情况，就不要展开写长过程，应该压缩成摘要：

1. 问题已经解决了。
2. 当时的中间试探过程已经没用了。
3. 这段内容只是某一轮为了得出结论走过的路径，不再影响当前主线。
4. 同一结论已经在多个地方重复出现。

压缩时只留 3 件事：

1. 当时在解决什么。
2. 最后结论是什么。
3. 现在还影响不影响当前主线。

## 6. 什么东西该归档

满足下面这类情况，就归到 `docs/归档/`：

1. 旧计划、旧任务卡、旧规则文档。
2. 已经被新事实推翻的旧结论。
3. 已经不再当现役入口的 README、ARCHITECTURE、worklog。
4. 历史测试说明、旧工具 README、旧阶段报告。
5. 只具备历史证据价值，不再参与当前判断的材料。

## 7. 如果几份文档写得不对齐，怎么处理

按这个顺序判断：

1. 最新用户明确要求优先。
2. `AGENTS.md` 的仓库硬规则优先。
3. 当前主线事实以 `docs/聊天记录.md` 为准。
4. 当前主线任务清单和完成定义以 `docs/主线/` 下当前唯一一份主线文档为准。
5. 长期原则以 `docs/项目原则.md` 为准。
6. AI 经验和成败原因以 `docs/AI自我学习.md` 为准。
7. 仓库结构、入口、风险区以 `docs/项目结构.md` 为准。

如果发现一件事写在了不该写的文档里：

1. 有用的内容移到正确文档。
2. 重复的内容压缩。
3. 过时的内容归档。
4. 不保留第二口径。

## 8. 整理文档终端的标准动作

每次整理文档，默认按这个顺序：

1. 先确认这次用户到底是要“整理文档”还是要“推进代码主线”。
2. 如果只是整理文档，先看这份 `文档管理.md`。
3. 盘点现役文档有没有职责串位、重复、冲突、过时内容。
4. 把有用的长期内容吸收到正确文档。
5. 把过时、重复、历史过程挪到 `docs/归档/`。
6. 如果这轮主线已经完成，把 `docs/主线/YYYY-MM-DD-主线.md` 移到 `docs/归档/主线/`，再新建下一份。
7. 做一次自检：
   - 当前现役入口是不是还清楚
   - 有没有第二口径
   - 有没有把整理终端自己的过程写进 `聊天记录.md`

## 9. “存档”到底是什么意思

“存档”不是随便收一下。

它的意思是：

1. 当前主线保留展开。
2. 已经收口的旧过程压缩成摘要。
3. 不再影响当前主线的旧材料归到 `docs/归档/`。
4. `项目原则.md`、`AI自我学习.md`、`项目结构.md` 都要同步检查有没有新内容该补。

如果这次“存档”明确就是让整理文档终端来做，那么这时你可以接管 `docs/聊天记录.md`；否则默认还是由干活终端维护。

## 10. 整理文档终端最后要交付什么

整理完之后，至少要说清楚：

1. 改了哪些现役文档。
2. 归档了哪些内容。
3. 清掉了哪些重复或冲突。
4. 还有哪些地方没法判断，得等干活终端继续推进后再补。

## 11. 最后一句

整理文档终端的目标，不是多写文档。

而是让下一个打开仓库的人，少看、看准、不被旧口径带偏。
