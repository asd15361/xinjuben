# system-authority（系统定义权与主权规则）

本文件只回答一件事：
**在任何项目里，到底谁有权定义什么，谁不能越权。**

如果你想看别的，请去这里：
- 首页导航：`AGENTS.md`
- 基础硬规则：`2.rules.md`
- 长期协作哲学：`3.agents.md`

## 总判断
- AI 最大的风险，不是偶尔写错代码，而是在项目早期偷偷重新发明系统。
- 一旦定义权没锁死，系统就会慢慢滑成多中心解释架构。
- 局部看起来能用，不代表整条链路真的稳。

## 总原则
- 先锁定义权，再开功能。
- 先锁主链路，再接能力。
- 先阻断坏内容，再谈评分展示。
- 先收口唯一真相源，再谈兼容扩展。

## 六类权力

### 1. 正式事实定义权
- 任何项目只允许一个正式事实源。
- 正式事实必须收口到结构化合同里。
- 兼容字段和临时推断可以存在，但不能掌权。

硬规则：
- 没有唯一事实源，不得开主功能。
- 同一件核心事实不准同时有多个版本。

### 2. Prompt 表达权
- Prompt 负责表达，不负责定义事实。
- Prompt 可以展开合同，不能反向替合同补事实。

硬规则：
- 不能把 Prompt 当数据结构。
- 需要靠 Prompt 临场猜的事实，默认说明结构没建完。

### 3. UI 展示权
- UI 是操作面，不是真相源。
- UI 只展示正式结果、必要风险和下一步。

硬规则：
- UI 不得自己拼业务真相。
- UI 状态不得反向改写正式事实。

### 4. 兼容层边界权
- 兼容层只负责过渡，不负责掌权。
- 旧字段可以活着，但不能继续主导结果。

硬规则：
- 兼容层不得回写正式事实。
- 兼容层不得继续主导准入、生成、校验和用户可见决策。

### 5. Repair 补救权
- repair 只修执行，不修事实。
- repair 一旦开始补结构，系统就会重新变脏。

硬规则：
- repair 不得改写正式事实。
- 根因在事实层，就回合同层修，不准只打补丁。

### 6. 准入与阻断权
- 质量系统先做门禁，再做诊断。
- 真有害的内容，不准只打分提醒，必须先拦住。

硬规则：
- 事实不完整，不放行。
- 引用关系不清，不放行。
- 连续性不成立，不放行。
- 内容不可执行，不放行。

## AI 最容易犯的错
- 还没定义事实源，就先堆功能。
- 还没定义主链路，就先接模型和能力。
- 把 Prompt 当结构。
- 把 UI 状态当真相。
- 把兼容层慢慢做回主逻辑。
- 用 repair 补结构缺口。
- 把局部成功误判成全链路可用。
- 把评分系统误当质量闭环。

## 新项目开工前先查 8 件事
1. 唯一事实源定了没有。
2. 兼容字段退到边上没有。
3. Prompt 只剩表达权没有。
4. UI 只剩操作和展示权没有。
5. repair 只剩执行补救权没有。
6. 阻断条件定了没有。
7. 用户真实目标路径定了没有。
8. 旧链路退场点定了没有。

少一项，都不该直接进入大规模实现。

## 判断有没有被 AI 带偏，只看这 7 个信号
1. 同一件事在不同模块里有不同答案。
2. 业务真相只能靠 Prompt 或 UI 状态反推。
3. 修一个问题要在多个地方同时补丁。
4. 旧字段名义上兼容，实际上还在掌权。
5. 分数很好看，坏内容照样进主链路。
6. 单点演示没问题，真人链路一走就露馅。
7. 每次修复都像加判断，不像收口真相源。

## 最后一句
- 只要正式事实源不唯一，坏内容不能先阻断，Prompt 和 UI 还在偷偷定义真相，系统就还没稳。
