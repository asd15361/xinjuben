## 问题总结

### 一句话判断

当前卡顿不是单点 bug，而是“开项目会话重复加载 + CharacterStage 派生写回级联 + DetailedOutlineStage render 期重计算 + ScriptSceneIndex 全量同步派生 + ScriptPlan 访问碎片化”五条链叠加，最终把 Electron 渲染线程长期压满。

### 当前已确认的根因排序

1. `StageSessionTruth` 没收口：打开项目时同一批数据被 IPC 获取、hydrate、切页后再次 IPC 获取、再次 hydrate。
2. `ActiveCharacterBlocksTruth` 没收口：角色页把派生数据当成 render 后副作用写回，制造跨页面级联更新。
3. `DetailedOutlineViewModelTruth` 没收口：详纲页在 render 期直接构建重量视图模型，任何订阅变动都重做。
4. `ScriptSceneDerivedTruth` 没收口：剧本页列表和搜索仍依赖全量同步派生，script 引用变化就把 CPU 压在主线程上。
5. `ScriptPlanAccessTruth` 没收口：计划构建入口碎片化，JSON 序列化和多调用方让同一份计划被重复建模。
6. `PerformanceEvidenceTruth` 没收口：日志存在，但标签、边界和预算不统一，执行人容易在“build 过了”时误报完成。
7. `ExecutionDocumentTruth` 没收口：计划、任务卡、worklog 不一致，会把执行人重新带回旧主线。

### 现象与根因对应

- 打开项目最卡：优先看 `StageSessionTruth`，其次是 hydrate 后立刻触发的 `ActiveCharacterBlocksTruth` 与 `DetailedOutlineViewModelTruth`。
- 点侧边栏切页慢：优先看 `StageSessionTruth` 与 `DetailedOutlineViewModelTruth`。
- 剧本页滚动、输入、展开慢：优先看 `ScriptSceneDerivedTruth` 与 `ScriptPlanAccessTruth`。
- 拖窗口都发虚：说明渲染线程持续有长任务，不是单个按钮 onClick 慢。
- 之前“不怎么卡，现在突然卡”：说明最近几轮重构把几条重计算链叠加了，不是电脑性能退化。

## 分线与依赖

### 1. StageSessionTruth

- 本质问题：项目打开与切页没有统一成“单次会话加载”，导致一次进入项目要经历两轮 IPC 和两轮 hydrate。
- 为什么容易漏：执行人只会看到“现在有 payload 了”，但不会继续追 renderer 是否还在重复拿 payload。
- 不修会继续出现什么假完成：切页看似能跳，实际上打开项目仍然最卡；第一下交互仍被重复 hydrate 吃掉。
- 依赖关系：这是当前轮第一阻断项；不先做，后续页面级优化只能缓解局部，不会解决“打开项目最卡”。
- 上游：`workspace-project-handlers.ts`、`shared/contracts/workspace.ts`、`preload/api/workspace.ts`。
- 下游：`useHomePageActions.ts`、`AppSidebar.tsx`、`AppHeader.tsx`、各 stage 页切换入口。
- 调用方：renderer 里所有 `window.api.workspace.openProjectShell()`、`getStagePayload()`、`changeProjectStage()` 直接调用者。

### 2. ActiveCharacterBlocksTruth

- 本质问题：`activeCharacterBlocks` 的正式真相没有钉死，导致角色页在 render 后把派生结果写回 store。
- 为什么容易漏：看起来只是一个 `useEffect`，但它会把 outline、characters 的变化扩散到详纲页、剧本页、计划构建链。
- 不修会继续出现什么假完成：人物页似乎不卡了，但项目打开、切页、详纲页仍然会因为写回级联发虚。
- 依赖关系：在 `StageSessionTruth` 收口后立即处理；它会直接影响 `DetailedOutlineViewModelTruth` 和 `ScriptPlanAccessTruth`。
- 上游：`outline.summaryEpisodes`、`outline.planningUnitEpisodes`、`characters[*].activeBlockNos`。
- 下游：`DetailedOutlineStage`、`useDetailedOutlineStageActions`、`useScriptStageActions`、`useScriptGenerationPlan`。
- 调用方：所有读取 `useStageStore((s) => s.activeCharacterBlocks)` 的页面和 hook。

### 3. DetailedOutlineViewModelTruth

- 本质问题：详纲页没有独立视图模型层，render 期直接拼装整页展示数据。
- 为什么容易漏：执行人很容易只盯子组件，把上游容器里的 `.flatMap/.find/.filter` 当成“普通映射”。
- 不修会继续出现什么假完成：去掉虚拟化、拆小组件以后仍然慢，因为大计算还在容器 render 里。
- 依赖关系：依赖 `ActiveCharacterBlocksTruth` 先稳定，否则视图模型会被无意义刷新。
- 上游：`outline`、`detailedOutlineBlocks`、`activeCharacterBlocks`、`script`。
- 下游：`DetailedOutlineActsPanel`、详纲页统计、去剧本页 gate 提示。
- 调用方：`DetailedOutlineStage.tsx` 及其传给面板的 `acts` 数据。

### 4. ScriptSceneDerivedTruth

- 本质问题：剧本页列表、搜索、状态提示混用同一份重量派生结果，导致每次 script 变动都做全量同步计算。
- 为什么容易漏：有 `useMemo` 和 module cache，执行人容易误以为“已经做了性能优化”。
- 不修会继续出现什么假完成：列表滚动看似用了虚拟化，但只要 script 引用变动就照样长任务阻塞。
- 依赖关系：可以和 `DetailedOutlineViewModelTruth` 并行推进，但必须在 `ScriptPlanAccessTruth` 前完成摘要/质量拆分。
- 上游：`script` 原始草稿、`screenplay-format`、`screenplay-quality`。
- 下游：`ScriptSceneList`、`ScriptSceneNavigator`、`useScriptSceneFilter`、脚本诊断 UI。
- 调用方：`ScriptStage.tsx` 及其下游组件。

### 5. ScriptPlanAccessTruth

- 本质问题：同一份 script generation plan 在多个 hook、页面和 action 里重复构建，且用整包 JSON 序列化做 memo key。
- 为什么容易漏：每个调用点单看都“说得过去”，但全局叠加就是重复 IPC + 重复序列化。
- 不修会继续出现什么假完成：script 页一处变快了，另一处还在偷跑同一份计划；切页和诊断仍然慢。
- 依赖关系：依赖 `ActiveCharacterBlocksTruth` 和 `ScriptSceneDerivedTruth` 先切出正式输入边界。
- 上游：`outline`、`characters`、`activeCharacterBlocks`、`detailedOutlineBlocks`、`script`、`runtime.failureHistory`。
- 下游：`ScriptStage`、`DetailedOutlineStage`、`stage-navigation-truth`、`useScriptStageActions`。
- 调用方：renderer 中所有直接 `buildScriptGenerationPlan()` 的调用点。

### 6. PerformanceEvidenceTruth

- 本质问题：已有零散 perf 日志，但没有统一标签、统一预算、统一证据归口，执行人容易凭感觉宣布完成。
- 为什么容易漏：日志一旦存在，看起来就像“已经可观测”；但标签不统一时，前后版本无法比。
- 不修会继续出现什么假完成：每个人都说“我这里看起来快了”，没有任何一条能复核的链路证据。
- 依赖关系：必须跟随每一波代码改动同步落地，不能拖到最后补。
- 上游：main IPC、renderer 会话、视图模型、脚本索引、计划服务。
- 下游：用户手测定位、后续回归排障、worklog 记录。
- 调用方：`performance-logger.ts`、`workspace-project-handlers.ts`、各 service/hook。

### 7. ExecutionDocumentTruth

- 本质问题：计划、任务卡、worklog 还在写旧任务，执行人会按旧文档继续施工。
- 为什么容易漏：大家会默认“文档晚点改也行”，但这里的执行人就是按文档逐条照做。
- 不修会继续出现什么假完成：代码按性能主线改了，文档却要求恢复链/E2E/质量主线，下一轮马上分叉。
- 依赖关系：本轮一开始和收尾都要做；不是附属任务，是当前轮阻断项。
- 上游：本文、`active-task（当前任务卡）.md`、`4.worklog.md`。
- 下游：后续执行顺序、验收口径、归档动作。
- 调用方：所有接手这轮任务的人和 AI。

### 执行波次与强依赖关系

1. Wave 0：前提校正与文档切主线。
2. Wave 1：`StageSessionTruth`。这是当前轮阻断项，没过不许报“打开项目已解决”。
3. Wave 2：`ActiveCharacterBlocksTruth`。依赖 Wave 1，完成后才能判断页内级联是否真正停下。
4. Wave 3：`DetailedOutlineViewModelTruth` 与 `ScriptSceneDerivedTruth` 可以并行，但两者都要在 render 期重计算完全移出后才算通过。
5. Wave 4：`ScriptPlanAccessTruth`。依赖 Wave 2 和 Wave 3；否则输入边界还在抖，计划服务会继续反复建模。
6. Wave 5：`PerformanceEvidenceTruth` 与 `ExecutionDocumentTruth` 收口、grep 清场、最小验证、交用户手测。
7. 任何发现“切页只切标识不切数据”“冷启动只 hydrate 当前页壳子但缺上游依赖”的情况，一律视为当前轮阻断项，不得写成“后续优化”。

## 消费矩阵

### 正式 / 兼容 / fallback 分层

- 现役正式口径：只认当前正式会话对象、当前正式派生对象、当前正式计划服务对象。
- 历史兼容口径：只允许单向 adapter 读取旧字段，禁止旧字段反写现役正式对象。
- 临时展示 fallback：只允许用于 UI 占位和提示，不得参与正式判断、持久化和切页决策。

## 逐条施工单

### A. StageSessionTruth

1. 问题定义  
   打开项目与切页没有统一会话协议。当前 renderer 仍然在首页打开项目时先拿 shell，再拿 stage payload，hydrate 一次后又调用 `changeProjectStage()` 再拿一次 payload，形成双读双 hydrate。

2. 为什么这是根因  
   用户反馈里“打开项目最卡”是最高优先级现象；如果会话入口本身重复，后面所有页面优化都会被第一下重复加载抵消。

3. 受影响的文件 / 函数 / 字段 / 状态
   - `src/renderer/src/features/home/ui/useHomePageActions.ts`：`openProject()`、`enterProject()`
   - `src/main/ipc/workspace-project-handlers.ts`：`workspace:open-project-shell`、`workspace:get-stage-payload`、`workspace:change-project-stage`
   - `src/shared/contracts/workspace.ts`：stage session / 切页返回 DTO
   - `src/preload/api/workspace.ts`：renderer 可用 API
   - `src/renderer/src/app/store/useWorkflowStore.ts`：`applyStageTransition()`
   - `src/renderer/src/store/useStageStore.ts`：`hydrateStagePayload()`

4. 调用方清单
   - `useHomePageActions.ts`
   - `AppSidebar.tsx`
   - `AppHeader.tsx`
   - `ProjectGenerationBanner.tsx`
   - `OutlineStage.tsx`
   - `CharacterStage.tsx`
   - `DetailedOutlineStage.tsx`
   - `ScriptStage.tsx`
   - `RuntimeConsoleStage.tsx`
   - `ChatStage.tsx`
   - `useChatStageActions.ts`

5. 必须改成什么
   - 新增单一 stage session 合同：`openProjectSession(projectId)` 与 `changeProjectStage(projectId, targetStage)` 都返回同一结构：`project + nextStage + payload`。
   - 首页打开项目只允许一次 IPC 会话获取，不再分拆 `openProjectShell + getStagePayload`。
   - renderer 新增单一 `stage-session-service`，所有 stage 进入与切换都只走这个服务；服务内部完成 `hydrateStagePayload -> applyStageTransition` 的顺序控制。
   - `useHomePageActions.ts` 不再直接调用 `openProjectShell` / `getStagePayload`。
   - 保留 `openProjectShell`、`getStagePayload` 仅作为兼容 adapter 时，必须明确标记为历史兼容，不允许 renderer 现役主链继续直连。

6. 明确禁止什么
   - 不允许 renderer 继续同时直连 `openProjectShell()` 和 `getStagePayload()`。
   - 不允许 `changeProjectStage()` 只切 stage 标识，不切目标数据。
   - 不允许打开项目时先 hydrate 一次，再因为切页再 hydrate 第二次同类数据。
   - 不允许恢复链绕开 stage session service 直接本地改 stage。

7. 完成定义
   - renderer 里 stage 进入与切页统一只剩一个 service 入口。
   - 首页打开项目只发生一次 stage session 获取和一次 hydrate。
   - stage 切换只发生一次目标 payload hydrate。
   - 冷启动、切页、恢复进入 script 页都共用同一会话协议。

8. 验收证据
   - grep：`window.api.workspace.openProjectShell(` 在 `src/renderer/src` 清零。
   - grep：`window.api.workspace.getStagePayload(` 在 `src/renderer/src` 清零。
   - grep：`window.api.workspace.changeProjectStage(` 只允许保留在单一 stage session service 文件。
   - perf：打开项目时控制台只出现一组正式链路标签，不再出现“双 session / 双 hydrate”证据。

9. 如果不这样改会怎样继续漏  
   只要首页还保留旧 split chain，执行人就会在“侧边栏点得快了”时误判完成，但用户真正最痛的“打开项目最卡”仍然存在。

10. 冷启动 / 切页 / 恢复链怎么处理

- 冷启动：直接 `openProjectSession(projectId)` 返回当前推荐 stage 的正式 session，一次 hydrate 完成。
- 切页：统一 `changeProjectStage(projectId, targetStage)`，返回目标 stage session，不再本地拼。
- 恢复链：恢复按钮、失败跳转、banner 跳转都必须复用 stage session service，不得走旧本地切 stage。

### B. ActiveCharacterBlocksTruth

1. 问题定义  
   `CharacterStage.tsx` 当前在 render 期重新生成 `availableBlocks` 和 `activeCharacterBlocks`，然后在 `useEffect` 里调用 `replaceActiveCharacterBlocks(activeCharacterBlocks)` 写回 store。

2. 为什么这是根因  
   这不是简单的重复计算，而是“派生 -> 写回 -> 触发订阅方重渲染 -> 再派生”的级联源头。它会把人物页的变化放大到详纲页、剧本计划和切页检查链。

   - `src/renderer/src/features/detailed-outline/ui/useDetailedOutlineStageActions.ts`
   - `src/renderer/src/features/script/ui/useScriptStageActions.ts`
   - `src/renderer/src/app/hooks/useScriptGenerationPlan.ts`

4. 调用方清单
   - `CharacterStage.tsx`
   - `DetailedOutlineStage.tsx`
   - `useDetailedOutlineStageActions.ts`
   - `useScriptStageActions.ts`
   - `useScriptGenerationPlan.ts`

5. 必须改成什么
   - 把 `activeCharacterBlocks` 定义成正式派生对象：来源只认 `outline.summaryEpisodes + outline.planningUnitEpisodes + characters`。
   - 新增纯函数 builder，集中产出 `availableBlocks` 与 `activeCharacterBlocks`，禁止在组件内写 IIFE 派生。
   - `CharacterStage.tsx` 只读 builder 结果，不再在 `useEffect` 里写回 store。
   - 如业务仍需要 store 中缓存 `activeCharacterBlocks`，缓存只能在 hydrate 或保存成功后由单一 authoritative path 写入一次，renderer render 周期不得写入。
   - `useDetailedOutlineStageActions.ts`、`useScriptStageActions.ts`、`useScriptGenerationPlan.ts` 改为读 builder / service 的结果，不再依赖角色页 effect 产物。

6. 明确禁止什么

   - `CharacterStage.tsx` 不再有 render 后派生写回 store。

   - 最小定向测试：新增 builder 测试，验证同一输入不会生成不同语义结果；角色编辑只在输入变化时更新。

9. 如果不这样改会怎样继续漏  
   只做 `useMemo` 而不收正式真相，执行人很容易留下另一个写口；后续任何页面只要还订阅 `activeCharacterBlocks`，级联就会回来。

10. 冷启动 / 切页 / 恢复链怎么处理

- 冷启动：角色页或详纲页拿到 session 后一次性读取正式派生结果，不再靠角色页挂载补写。
- 切页：切到详纲 / 剧本页时，直接从会话或 builder 读取，不再等待角色页 effect 写入。
- 恢复链：恢复回 script 时直接消费同一派生结果，不通过角色页中转。
