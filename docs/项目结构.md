# 项目结构

## 先查图谱

- 回答架构问题前先看 `graphify-out/GRAPH_REPORT.md`
- 需要追调用、依赖、owner、异常连接时再看：
  - `graphify-out/graph.html`
  - `graphify-out/graph.json`
  - `graphify-out/cache/`

## 正式主链

```text
renderer -> server/http api -> server/application -> src/shared -> PocketBase
```

- `src/renderer/`：前台交互和展示
- `server/src/api/`：正式 HTTP 路由
- `server/src/application/`：核心业务和生成逻辑
- `src/shared/`：前后端唯一共享合同与领域逻辑
- `server/src/infrastructure/pocketbase/`：项目真相落盘

## 当前 owner

- HTTP API：`server/`
- 共享合同：`src/shared/`
- 桌面壳：`src/main/`（只保留 IPC 事件桥接、导出、系统信息）
- 剧本生成 authority：`server/src/application/script-generation/`
- 剧本审计（只读）：`src/main/application/script-generation/audit/`（通过 IPC 暴露给 renderer）
- 七问生成：`server/src/api/routes/generate.ts`（HTTP `/api/generate/seven-questions`）

## 当前危险点

- 任何 renderer 里残留的 `window.api.workspace` 都是双口径风险
- 任何本地文件、project-store、renderer 本地缓存重新掌权，都会冲撞 PocketBase 真相源
- 任何没先看 `GRAPH_REPORT.md` 就下架构结论的行为，都容易误判 owner 和依赖
- 任何在 main 端新增剧本生成逻辑的行为，都会重新长出第二 authority
- server 端 `/api/script-audit/execute-repair` 仍是 TODO 占位——renderer 已切 HTTP，但 server 实现待补
