# 项目结构

这份文档只做一件事：

让人看一遍就知道这个仓库现在有哪些正式线、每条线负责什么、哪里是观察口、哪里是原型线、哪里已经是历史壳，出了问题先去哪里查。

这不是进展播报，不是 worklog，也不是主线日报。

## 1. 这份文档要回答什么

看完这份文档，应该能直接回答下面 6 个问题：

1. 这个项目的正式主链是什么。
2. 每一层工序分别负责什么，不负责什么。
3. `src/shared / src/main / src/preload / src/renderer / tools/e2e` 各自干什么。
4. 哪些目录是正式生产线，哪些只是观察口、原型线、历史壳。
5. 出现具体问题时，第一时间先去看哪几个文件。
6. 当前哪些地方相对稳定，哪些地方还是风险区和噪音区。

## 2. 一眼总图

### 2.1 业务总链

正式主链只认这一条：

用户输入
-> 确认信息
-> 七问篇章
-> `storyIntent`
-> 粗纲
-> 人物小传
-> 详细大纲
-> 剧本

当前产品页签上，七问已经升成独立一级 stage：

`chat`
-> `seven_questions`
-> `outline`
-> `character`
-> `detailed_outline`
-> `script`

### 2.2 工程总链

正式工程流转是：

`renderer`
-> `server/http api`
-> `server/application`
-> `shared/contracts + shared/domain`
-> `PocketBase`
-> 返回项目快照
-> `renderer` 重新 hydrate

判断边界时记住一句话：

- `shared` 管真相和共享合同。
- `server` 管正式业务、正式生成、正式持久化。
- `PocketBase` 管正式项目真相。
- `main/preload` 只管桌面壳和系统桥接。
- `renderer` 只管页面、交互、前台派生和展示。
- 旧 `main/ipc + project-store` 现在属于待退场历史壳，不再当长期正式主链。

### 3.3 代码区

- `src/shared`
  - 跨层合同、真相源、纯规则、共享领域逻辑。
- `server/`
  - 当前唯一正式业务入口。
  - 项目 CRUD、生成流程、积分扣费、认证鉴权，后续都统一收口到这里。
  - `server/src/infrastructure/pocketbase/` 是后续正式项目仓储落点。
- `src/main`
  - Electron 主进程桌面壳。
  - 当前仍残留旧业务链，但身份已降级为待退场历史壳，不再视作长期正式入口。
- `src/preload`
  - 桌面壳桥接层。
  - 允许保留窗口和系统能力，不再长期承载正式业务 authority。
- `src/renderer`
  - 前台工作台和页面逻辑。
  - 项目大厅和工作台初始化读链已开始改走 `src/renderer/src/services/api-client.ts`，不再默认认 `window.api.workspace`。
  - 七问确认保存也已开始改走 HTTP，保存后直接吃服务端返回项目快照。
  - `useWorkflowStore.projectEntityStore` 现已承接项目快照里的 `entityStore`，`OutlineStage.tsx / CharacterStage.tsx` 不再各自偷读旧 IPC 项目副本。
  - 顶层登录弹窗当前统一走 `src/renderer/src/components/LoginModal.tsx`，并通过 `createPortal(..., document.body)` 脱离局部组件树，避免被 `AppHeader` 这类 header 容器的 stacking context 污染。
- `server/src/api/routes/`
  - 当前 HTTP 路由入口。
  - 已有 `auth / credits / generate / outline-characters / projects`。
  - `projects.ts` 已承接项目列表、创建、读取、删除和阶段保存接口。
  - 七问确认保存也已并入 `projects.ts`，通过 `/api/projects/:projectId/seven-questions/confirm` 落到项目 outline 真相。
- `server/src/application/`
  - 当前和后续的正式业务服务层。
  - `workspace/build-outline-character-entity-store.ts` 负责把 faction matrix + V2 人物结果缝成项目 `entityStore`，供人物页 / 粗纲页观察口直接复用。
- `server/src/infrastructure/pocketbase/`
  - PocketBase 客户端与后续项目仓储层。
  - `project-snapshot-mapper.ts` 负责把拆表后的项目数据重新缝成 `ProjectSnapshotDto`。
  - `project-repository.ts` 负责项目 CRUD、阶段读写和基于 `version` 的乐观锁。
- `server/pb_migrations/`
  - PocketBase 正式项目表结构的迁移入口。
  - 当前项目核心表已定为：
    - `projects`：项目元信息 + 项目级 JSON 真相
    - `project_chats`：聊天记录
    - `project_outlines`：粗纲
    - `project_characters`：人物与 active character blocks
    - `project_detailed_outlines`：详细大纲 blocks / segments
    - `project_scripts`：剧本与运行时状态
  - 迁移执行入口：
    - `npm run migrate:up`
    - `npm run migrate:down`
- `tools/e2e`
  - 验收、观察、分析、原型脚本和证据目录。
  - `tools/e2e/observation/ui-outline-character-http-parade.mjs` 是当前 UI 阅兵脚本，会起服务、登录 Electron、点“生成粗纲和人物”、校验积分刷新与人物页回显。
  - `tools/e2e/utils/e2e-output.mjs` 负责统一修剪 `tools/e2e/out`，包括 `userdata-* / evidence-* / e2e-tmp-* / Cache_Data`。

## 4. 正式主链分层图

### 4.1 用户输入与确认信息线

作用：

- 把聊天输入收成正式可生成的确认信息。
- 统一集数、人物口径、角色卡名单、生成 brief。
- 在这一层把世界底账上游材料拆出来。

正式真相：

- `storyIntent`
- `storyIntent.generationBriefText`
- 确认后的角色名单和项目级创作控制

主要落点：

- `server/src/application/` 下后续对应的确认信息服务

- `src/shared/domain/short-drama/short-drama-constitution.ts`

- 直接生成粗纲

## 5. 代码分层职责图

### 5.1 `src/shared`

这是全仓“单一合同和共享真相”层。

- `src/shared/contracts/`
  - DTO、合同、跨层字段定义。
  - 同一个字段如果多处解释冲突，先以这里为准。
- `src/shared/domain/workflow/`
  - 阶段真相、authority、batching、激活包、阶段推导。
- `src/shared/domain/entities/`
  - `entityStore` 相关映射、归一化、人物底账桥接。
- `src/shared/domain/script/`
  - 剧本格式、质量、污染检测、repair guard。
- `src/shared/domain/script-generation/`
  - 剧本控制包和生成合同收口。
- `src/shared/domain/short-drama/`
  - 项目级短剧创作宪法、人物行为边界、集级控制卡。
- `src/shared/domain/formal-fact/`、`story-contract/`、`policy/`
  - 事实确权、故事合同、运行时政策。

判断标准：

- 只要某件事属于“跨层都要认的一套规则”，优先放这里。
- 如果 renderer 和 main 各自写一套同名规则，通常就已经开始出问题了。

### 5.2 `src/main`

这是正式业务执行层。

- `src/main/application/ai/`
  - AI provider 调用、路由、超时、模型入口。
- `src/main/application/workspace/`
  - 确认信息、七问、粗纲、人物、详纲相关 use case。
- `src/main/application/script-generation/`
  - 剧本 plan、prompt、runtime、rewrite、postflight。
- `src/main/application/input-contract/`
  - 阶段输入合同和前置判断。
- `src/main/infrastructure/storage/`
  - 项目快照读写、normalize、authority merge、read repair。
- `src/main/infrastructure/runtime-env/`
  - provider 配置和运行时环境。
- `src/main/infrastructure/diagnostics/`
  - 诊断和观察，不是主链真相源。
- `src/main/ipc/workflow/`
  - renderer 到 main 的正式请求入口。

判断标准：

- 业务流程跑偏、真相没落盘、IPC 调错、重写没保住，优先查 `src/main`。

### 5.3 `src/preload`

这是桥接层。

- `src/preload/api/`
  - 对 renderer 暴露白名单接口。

判断标准：

- 它只负责传，不负责编排。
- 如果桥接层开始做阶段判断或改写业务真相，就是越界。

### 5.4 `src/renderer`

这是前台工作台层，所有正式业务请求已切换为 HTTP 接口，不再依赖 IPC。

- `src/renderer/src/app/`
  - 会话打开、hydrate、切页、全局 hooks、顶层服务。
- `src/renderer/src/features/`
  - 每个 stage 自己的 UI、model、api、前台 guard：
    - `script/ui/useScriptStageActions.ts`：已完全重写为 HTTP 调用（开始/暂停/恢复/停止/重写）+ 3 秒轮询模式，移除所有旧 IPC 依赖
- `src/renderer/src/services/`、`store/`
  - 前台状态和服务层：
    - `api-client.ts`：统一 HTTP 客户端，已新增剧本生成全接口
- `src/renderer/src/components/`
  - 通用组件。

`src/renderer/src/features/` 当前各目录职责：

- `chat`
  - 聊天输入和对话入口。
- `intake`
  - 确认信息阶段 UI。
- `seven-questions`
  - 七问起草、确认、保存回读。
- `outline`
  - 粗纲页、关键设定、世界底账观察口。
- `character`
  - 人物页三轨：完整小传、轻量卡、势力与人物位。
- `detailed-outline`
  - 逐集详纲页、前台缺口提示、重生入口。
- `script`
  - 剧本页、整批生成、单集重写。
- `workspace`
  - 工作台外壳、notice、项目级交互。
- `formal-fact`
  - 关键设定相关支撑模块。
- `runtime-console`
  - 运行时观察口。
- `home`
  - 首页和非主生产阶段入口。

判断标准：

- 页面点不动、展示不对、用户路径绕、notice 不清，优先查 renderer。
- 但 renderer 不该自己发明第二套业务真相。

### 5.5 `tools/e2e`

这是验证和观察层，不是业务主链。

- `tools/e2e/utils/runner-taxonomy.mjs`
  - `tools/e2e` 官方口的真实分类源。

- `tools/e2e/runners/observation/`
  - 会真实拉起执行链的观察 runner。
- `tools/e2e/utils/foundation-verdicts.mjs`
  - 官方质量门的共用 verdict builder，不是独立 runner 入口。
- `tools/e2e/utils/`
  - 官方基础模块和旧辅助脚本收口区。
- `tools/e2e/observation/`

判断标准：

- `tools/e2e/out/` 里的证据可以用来复盘，但它不是正式逻辑来源。

### 5.6 根目录噪音区

这轮已经物理收过一刀。

- 根目录不再保留 `tmp-* / check_* / fix_* / debug_* / find_*` 这类一次性排障脚本。
- 已确认无正式消费方的根目录噪音，统一退到：
  - `scripts/archive/root-noise-2026-04-13/`

判断标准：

- 根目录只留正式入口、配置、依赖声明和少量项目级说明文件。
- 一次性排障脚本即使暂时想留证据，也不能继续留在根目录冒充现役入口。

## 9. 维护规则

1. 这份文档先写总图，再写各线职责，再写问题定位口，不写流水账。
2. 只要正式主链、目录结构、正式入口、风险区变了，就更新这份文档。

7. 这份文档的标准不是“写了多少文件名”，而是“新来的人能不能看一遍就知道哪里该动、哪里别动、哪里先查”。
