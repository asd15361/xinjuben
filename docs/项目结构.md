## 正式主链

```text
renderer -> server/http api -> server/application -> src/shared -> PocketBase（轻量 metadata）
  |
  v
本地 content store（剧本正文、运行时状态真相源）
```

- `src/renderer/`：前台交互和展示，收到 server 结果后写本地内容真相源
- `server/src/api/`：正式 HTTP 路由，只鉴权/扣积分/AI 调用并返回结果，不再写完整剧本大 JSON 到 PB
- `server/src/application/`：核心业务和生成逻辑
- `src/shared/`：前后端唯一共享合同与领域逻辑
- `server/src/infrastructure/pocketbase/`：账户与计费真相源（账号、登录、积分、订单、权限、项目轻量索引）
- `src/main/infrastructure/storage/local-content-store.ts`：本地内容真相源（剧本正文、草稿、运行时状态）

创作工序命名：
```text
灵感对话 -> 创作信息总结/故事梗概 -> 人物小传/底账 -> 剧本骨架 -> 详细大纲 -> 剧本
```

- 用户可见统一叫“剧本骨架”，不再叫“粗略大纲”
- 内部 `rough_outline` 类型名可暂时保留，但不得新增用户可见“粗略大纲”文案
- 七问候选 session 是草稿态和审核辅助；人物小传/剧本骨架当前由 `storyIntent` 直驱，旧锁定七问只作为兼容约束注入

## 当前 owner

- HTTP API：`server/`
- 共享合同：`src/shared/`
- 桌面壳：`src/main/`（只保留 IPC 事件桥接、导出、系统信息）
- 剧本生成 authority：`server/src/application/script-generation/`
- 剧本审计（只读辅助计算）：`src/main/application/script-generation/audit/`（当前仍是本地辅助链，不是正式写入口）
- 七问生成：`server/src/api/routes/generate.ts`（HTTP `/api/generate/seven-questions`）
- 剧本修稿执行 authority：`server/src/application/script-generation/repair/`（已补实，通过 HTTP `/api/script-audit/execute-repair` 调用）

## 前端性能边界

- renderer 只做展示、交互和轻量派生；大文本拼接、大 JSON 构建、全页复制内容生成不得在 render 阶段执行。
- 复制类 UI 统一使用惰性 `getText`：用户点击复制时才调用 `buildXxxCopyText`，不要把长字符串提前算好塞进组件 props。
- `CharacterStage`、`OutlineStage`、首页工作台属于高风险页面：它们同时承载卡片列表、生成状态、导出/复制、编辑入口，新增交互后必须做真实滚动和点击体验验证。
- 首页项目列表有本地摘要缓存：刷新时先显示 `localStorage` 中的项目摘要，再由 `/api/projects` 后台刷新；不要为了“权威数据”把首屏项目卡片卡在远端列表请求后。
- 项目内阶段导航是“本地先切 + 后台补快照”：`switchStageSession` 必须先更新本地 `currentStage`，再等待 `apiGetProject` 补水；不要重新引入等待后端才切页的路径。
- 大列表后续如果继续膨胀，优先拆卡片组件、收紧 store selector、用 `useMemo` 固定派生边界；如果仍卡，再考虑虚拟列表。

## 人物小传/剧本骨架链路

```text
storyIntent
  -> faction matrix / character profiles
  -> rough outline bundle
  -> OutlineCharacterGenerationBundleDto
  -> legacy outlineDraft + characterDrafts + entityStore projection
```

- `src/shared/contracts/outline-character-generation-bundle.ts` 是人物/骨架链路新的正式交接合同，统一承载 `storyIntent`、`outlineDraft`、`sevenQuestions`、`characterLedger`、`diagnostics`、`warnings`。
- 人物小传/剧本骨架/详细大纲生成只认 `server/src/application/workspace`；`src/main/application/workspace` 不再保留旧生成 prompt、旧 agent、旧骨架实现和旧 IPC handler。
- `server/dist` 是 ignored 编译产物，不是当前源码真相；如果里面残留旧模板，不得用它判断当前逻辑。本轮已清理旧 `server/dist`，开发和测试以 `cd server; npm run dev` / `server/src` 为准。
- `warnings` 当前承载人物/骨架题材策略污染信号：人物/骨架 bundle 会先用 `GenerationStrategy.repairStrategyContaminationValue` 清洗 `outlineDraft`、人物底账和 `entityStore`，返回 `generation_strategy_contamination_repaired`；如果清洗后仍残留禁词，再返回 `generation_strategy_contamination`。
- 详细大纲出口也复用 `GenerationStrategy.repairStrategyContaminationValue`：`generateDetailedOutlineFromContext` 在四幕合并和完整性校验后清洗 `DetailedOutlineSegmentDto[]`，通过 diagnostic 记录清洗结果；详细大纲不再复制人物/骨架里的递归清洗函数。
- `generateOutlineCharacterBundleFromConfirmedSevenQuestions` 是新的 canonical 生成函数；旧 `generateOutlineAndCharactersFromConfirmedSevenQuestions` 只保留为兼容 wrapper，负责把 bundle 投影回旧接口形状。
- `characterLedger` 是人物底账边界：`factionMatrix`、`characterProfilesV2`、`normalizedDrafts`、`fullProfileDrafts`、`visibleCharacterDrafts`、`entityStore` 必须作为一组向下传递。
- `getCharacterContractIssues` 现在同时校验旧展示字段和 V2 结构字段，避免“展示能看但底账残缺”或“底账有字段但前端不可用”的半成品进入骨架。
- 人物小传先于剧本骨架生成，骨架批次失败不能丢人物小传。
- `rough_outline_batch_parse_failed` 走可见临时骨架恢复，返回 `outlineGenerationError` 给前端提示。
- 20 集项目的势力矩阵走轻量编制；60 集以上再使用完整多势力/多分支/多人物要求。
- 20 集项目完整小传上限为 8；超出或缺失的势力人物位必须进入轻量人物卡/世界底账，不能只显示为“待补席位”导致轻量卡为 0。
- 人物补全链禁止统一规矩人模板；爪牙、忠诚卧底、家族合作者等不同功能人物必须生成不同压力、目标和弧线。
- 人物补全链的家族名、主角名、暗线名必须从当前项目推断；不得在模板里硬编码上一轮项目实体。
- V2 到旧人物小传映射负责清理小传拼接标点；实体底账负责对 goals/pressures 去重，避免轻量卡复制文本重复。
- 轻量 slot 目标由 `buildSeatSlotGoals` 按席位职责生成，避免护法/弟子/情报/执行全部写成同一条泛目标。
- `character-profile-v2-agent` 是人物内容质量第一入口：Prompt 必须要求自然小传、具体可拍触发、行动抓手和结构化弧线。
- `parseCharacterProfileV2Response` 与 `mapV2ToLegacyCharacterDraft` 是人物文本兜底清洗层：负责修掉硬拼 biography 和只写结果的 arc，保证下游骨架 Prompt 和前端展示拿到同一套人物底账。
- `mapV2ToLegacyCharacterDraft` 必须拒绝字段硬拼小传：`身份是/性格底色/在戏里/牵动他的软肋` 等短语出现时直接重建自然 biography；结构化弧线若终局重复整条链，必须压成真正终局句。
- `parseCharacterProfileV2Response` 还负责从带解释文字或 fenced markdown 的响应中提取 JSON；如果连续重试仍是坏 JSON，`generateCharacterProfileV2ForFactionWithAdaptiveSplit` 会按势力占位生成 fallback V2 人物并写 `faction_parse_fallback` 诊断。
- V2 映射层是人物质量防火墙：`biography` 必须自然、包含人物名并融合五维；`publicMask` 要清掉重复“表面是”；`arc` 缺五段结构就强制重写。
- `enrich-character-drafts.ts` 是缺字段补全防火墙：守护女主、亲传大弟子、欺凌压迫者、普通执事兜底必须分流；默认兜底只能写现场职责、证据和站位后果，不得输出“旧账/外部靠山/可调动关系/自己这条线”这类系统模板腔。
- 权力反派也必须分流：本人是大长老/长老派领袖且目标是夺血脉、架空掌门、夺权时，要走“野心大长老/夺权反派”分支；“大长老的爪牙/亲信/走狗”仍走 enforcer，不能混成同一个模板。
- `build-outline-character-entity-store.ts` 是人物归属防火墙：势力矩阵占位名只能提供候选，V2 `factionId` 决定实名人物主归属；同名实体必须合并，错误势力成员必须剔除。
- 势力矩阵 Prompt 明确禁止把同一实名人物复制进多个势力成员表；跨势力利用、卧底、安插关系只能写在 `crossRelations` / sleeper 字段。
- `derive-character-stage-sections.ts` 是展示兜底：用 masterEntityId + 人物名双重识别完整人物，避免旧脏 entityStore 把同名完整人物显示成轻量卡。
- `src/shared/domain/entities/faction-seat-characters.ts` 与 server 侧席位生成必须保持同口径；护法、弟子、情报、执行、执事、长老等席位目标不能退回泛模板。
- `src/renderer/src/features/character/model/character-stage-copy-text.ts` 是人物小传验收面，复制完整小传必须带出外在形象、性格、身份、价值观和剧情作用。
- `server/src/shared/` 本地副本已退役；server 的 `@shared/*` 只通过 `server/tsconfig.json` 指向根 `src/shared/*`。
- 七问不是这条链路的第二账本；如果旧项目已经锁过七问，只把它折叠成兼容约束。

## 学习型创作资产层

写作逻辑分三层：
```text
A Stable Writing Kernel（稳定内核）
B Market Playbook（市场打法包）
C Project Intent（项目自身设定）
```

- A 放稳定合同：平台安全、剧本格式、七问、剧本骨架、连续性、每集施压/反击/钩子
- B 放市场变化：优秀剧本样本提炼出的开局、爽点、钩子、反派压迫、主角反击、道具信息承载
- C 放用户项目：题材、受众、垂类、人物、集数、故事梗概、锁定七问
- MarketPlaybook 必须版本化、可审核、可选择；不能覆盖稳定内核
- 导入优秀剧本时先提取打法包草案，不长期把全文塞进 Prompt

MarketPlaybook Prompt 注入顺序（P8）：
```text

  ↓

  ↓

```

注入位置：
- 七问：`seven-questions-agent.ts` → `buildSevenQuestionsPrompt` 内，关键人物之后、七问格式说明之前

- 正式剧本：`create-script-generation-prompt.ts` → MarketProfile 之后

阶段特定输出：
- `seven_questions`：opening_pressure, payoff, villain_pressure, hook
- `script_skeleton`：payoff, protagonist_action, villain_pressure
- `episode_script`：opening_pressure, villain_pressure, protagonist_action, payoff, hook, dialogue_style
