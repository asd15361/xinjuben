# Prompt Writing（提示词写作方法）

这个文件只回答一件事：
**提示词怎么写得短、准、狠。**

如果你想看别的，请去这里：
- 首页导航：`AGENTS.md`
- 基础硬规则：`2.rules.md`
- 长期协作方式：`3.agents.md`
- 工程排障与根因修复：`docs/engineering-workflow（工程排障与根因修复）.md`

## 核心方法

### 1. 先排除
- 几句话是一个意思，就删到只剩一句。
- 不看字面像不像，只看是不是同一个意思。

### 2. 再串行
- 剩下的不同意思，要串成动作句，不要并排堆 checklist。
- AI 要一眼知道先做什么，不要自己猜顺序。

## 判断标准
- 一句话能讲清，就不要写两句。
- 一句话最多放 3 到 5 个信息点，超过就拆句。
- 字数太多也是干扰，能更短就继续压。
- 提示词先驱动作，不先解释概念。

## 压缩公式
- 先删重复意思。
- 再把剩下的意思串成动作。
- 最后继续压，压到再短就会丢意思为止。

## 补充原则
- 重复会做的事，优先写成固定动作句。
- 能固定顺序的，不要写成开放讨论题。
- 能触发检查的，不要只写成“记得做”。

## 例子

### 例子 1
原句：
- 要有证据
- 不是随便猜
- 要能证明
- 要能说服用户

本质句：
- 拿出证据说服用户。

### 例子 2
原句：
- 找到根本原因再开始干
- 没找到前不准开始干
- 不能立马开始干
- 用户认可了你才能开始干

本质句：
- 先证明根因，用户认可后再开工。

### 例子 3
原句：
- 所有打补丁的修复都不是根本原因
- 解开所有的补丁找到本质
- 有补丁就不对
- 看到补丁就继续往前找

本质句：
- 解开补丁，找到本质。

### 例子 4
原句：
- 要用大白话讲
- 用户要能听懂
- 不能拿代码逻辑来证明
- 能用大白话讲清楚的本质才是真的本质

本质句：
- 能用大白话讲清楚的本质，才是真的本质。

### 例子 5
原句：
- 能并行就尽量并行
- 不要默认只靠一条单线硬串
- 但用完代理要及时关掉
- 不要让空进程把 CPU 卡死

本质句：
- 尽量并行，及时关代理进程，防 CPU 卡死。

### 例子 6
原句：
- 先找到根源问题
- 再证明它是不是根源
- 自己推翻自己
- 推不翻才可以做计划

本质句：
- 推翻自己找到的根源问题，证明这不是根源，直到推不翻，才可以做计划解决问题。

## 最后自检
- 有没有两句其实是一个意思？
- 有没有并排堆要求，没有串成动作？
- 有没有超过 3 到 5 个信息点？
- 有没有还能再压短？
