# AI自我学习

这个文件只记录我从 AI 视角对项目的理解、踩坑、成败原因和后续提醒。

断线恢复时，先看 `AGENTS.md`，再看 `docs/聊天记录.md`，然后看这份文件。

## 1. 我对这个项目的当前理解

这是一个 AI 剧本创作桌面软件。

它不是让 AI 直接一把写完全本，而是把创作拆成明确工序：

用户输入
-> 确认信息
-> `storyIntent`
-> 粗纲
-> 人物小传
-> 详细大纲
-> 剧本

这个项目真正难的不是"怎么不报错"，而是：

- 用户输入必须始终是唯一真相。
- 上游一旦确认，下面只能展开，不能偷偷改写。
- 系统工程保护不能越界长成第二口径。

## 2. 为什么过去做了十几轮重构，还是反复失败

### 2.1 不是没努力，是没有唯一掌权的记忆系统

过去不是没计划，也不是没人记录。

真正的问题是：

- 同时有太多文档在发声。
- 旧任务卡、旧 worklog、旧计划总表没有退场。
- 新开的终端会先看到一堆过期但还像现役的东西。

结果就是：

- 第二天很像重新开了一个项目。
- 明明昨天已经判断错过，今天又按同一套错法再判断一轮。

### 2.2 过去常把"这轮范围内通过"当成"本质问题已解决"

这类假完成特别多：

- 一个入口能跑了，就写"主链已完成"。
- 一个测试通过了，就写"本质问题已解决"。
- 一轮结构搜索清零了，就写"唯一真相源已收口"。

但后面新事实又证明：

- 旧旁路没死。
- 旧兼容还在。
- 旧自动推进还在。
- 新主线还没覆盖到下游。

所以以后我必须先问：

- 这次解决的是局部范围，还是整条主链。
- 旧影子口有没有物理退场。
- 这个完成定义会不会被下一轮新事实直接推翻。

### 2.3 过去常把阶段问题误判成单点 bug

老错很明确：

- 剧本炸了，就只盯剧本。
- 详纲炸了，就只盯详纲。
- parser 报错，就只盯 parser。

但很多时候，下游问题只是上游越界、双口径、补丁链残留的结果。

以后我不能再只盯最先报错的点，必须先追第一次越界的位置。

### 2.4 过去常把局部真测当整链成立

老项目里很多问题都是这样被误判的：

- 首场顺了，不等于整季顺。
- 单层顺了，不等于整链顺。
- 结构能出，不等于定义已经对。

以后我必须先看：

- 当前层是不是只吃了一套真相。
- 上游是不是已经收死。
- 旧数据和旧项目有没有污染判断。

### 2.5 计划本身也会制造第二口径

过去不是只有代码会越界，计划也会。

常见问题：

- 分计划各讲各的，没有一份真正掌权的总入口。
- 计划里混了哲学、执行、验收、补丁，职责不清。
- 有些计划已经被事实推翻，但还保留"已完成"口气。

以后我不能再把所有计划都当现役，只能把结论吸收进现役 5 文档，原文一律归档。

## 3. 当前已经确认有效的做法

### 3.94 Express.js 路由注册顺序会导致同路径冲突，必须用显式路径区分（2026-04-17）

这轮 HTTP API 前后端分离遇到一个很典型的路由冲突：

- `generateRouter` 和 `outlineCharactersRouter` 都在某层注册了 `/`
- 结果 `/api/generate/outline-and-characters` 返回 404 “接口不存在”
- 真正原因：两个路由在同层级抢占同一路径，Express 只认第一个

修复方式：

1. 把 `outline-characters` 路由从 `/` 改成 `/outline-and-characters`
2. 确保路由路径唯一，不靠注册顺序碰运气

以后遇到”接口明明定义了但返回 404”，先查：

1. 有没有多个路由在同层级抢同一路径
2. 注册顺序是否决定了谁优先匹配
3. 是否需要显式路径区分

### 3.95 `optional chaining + fallback` 是防御性访问的标准模式（2026-04-17）

这轮粗纲+人物生成服务遇到空指针：

- `sevenQuestions.sections` 直接访问会报 undefined
- 但业务逻辑允许 sections 为空（空数组也能正常处理）

修复方式：

```typescript
const sections = sevenQuestions?.sections || []
```

这个模式要记住：

1. `?.` 防止 undefined/null 报错
2. `|| []` 提供安全的空值兜底
3. 后续逻辑不用再判 null，直接迭代

以后遇到”字段可能不存在但业务需要继续”，优先用这个模式，不要用 `if (x && x.y)` 冗长判断。

### 3.96 HTTP API 前后端分离后，积分扣费必须在路由层做，不能在服务层做（2026-04-17）

这轮设计积分系统时，我差点把扣费逻辑塞进业务服务。

但正确位置应该在路由中间件：

1. 认证中间件先验证 JWT，确保 `req.user` 存在
2. 积分中间件检查余额，不足直接返回 402
3. 业务服务执行成功后，再调用扣费

这样分层的好处：

- 业务服务不关心用户身份和积分
- 路由层统一处理认证和计费
- 测试业务服务时不用 mock 积分逻辑

以后 HTTP API 设计，认证/计费/限流这类横切关注点，优先放在中间件层。

## 3.97 HTTP API 前后端分离后，前端必须统一封装 api-client，不能散落各处（2026-04-17）

这轮前端迁移 HTTP API 时，我确认了正确的做法：

- 所有 HTTP 调用统一收敛到 `api-client.ts`
- Token 管理由 api-client 统一处理（localStorage + Bearer header）
- 错误处理统一返回 `{ success, data, error }` 格式
- 各 feature 目录只调用 api-client，不直接 fetch

好处：

1. 换 API base URL 只改一处
2. 换认证机制只改一处
3. 错误文案统一管理
4. 方便后续加 retry、cache、log

以后 HTTP API 调用，不准在组件里直接 `fetch('/api/...')`，必须走统一 api-client。

### 3.98 HTTP API 服务端和 Electron IPC 主进程可以共存，但要明确分工（2026-04-17）

这轮项目有两套后端入口：

1. `src/main` - Electron IPC 主进程（桌面软件）
2. `server/` - HTTP API 服务端（前后端分离）

分工原则：

- Electron IPC 负责桌面软件的本地生成流程
- HTTP API 负责浏览器/远程调用的生成流程
- 共享的业务逻辑放在 `server/src/application/`，两边都可以引用
- 不共享的逻辑各自独立

以后遇到”一套代码要跑两个环境”，先问：

1. 哪些逻辑是共享的
2. 哪些入口是环境特有的
3. 共享逻辑是否抽成独立服务

### 3.99 “先挂一个简化版 HTTP 旁路，后面再慢慢补齐”会制造第二正式主链（2026-04-18）

这轮我需要记住一个已经坐实的坑：

- HTTP API 刚接进来时，看起来像“已经前后端分离了”
- 但如果它只覆盖了登录、积分、七问，粗纲/人物还是简化版，项目存储还是本地文件
- 本质上就不是正式迁移，而是多长了一套旁路

最危险的地方在于：

1. 前台会出现“有些走 HTTP，有些走 IPC”的双入口
2. 存储会出现“用户和积分在 PocketBase，项目真相还在本地”的双真相
3. 团队很容易把“已经有 API”误判成“已经完成前后端分离”

以后只要遇到“迁一半”的方案，我要先问：

1. 这条 HTTP 链是不是正式等价能力
2. 它是不是已经接管正式真相存储
3. 对应旧入口有没有明确退场计划

三条里只要缺一条，就不能写“前后端分离已完成”。

### 3.100 项目云端落盘不能照搬本地 `projects.json` 整包思路（2026-04-18）

这轮做 PocketBase 项目表设计时，我又确认了一件事：

- 本地文件存储可以一把写整份 `ProjectSnapshotDto`
- 但云端如果也照抄成“一个项目一整包 JSON”
- 后面会马上撞上局部更新、并发覆盖、单阶段重跑、脚本大体积这些问题

这次更稳的做法是：

1. `projects` 主表只放项目元信息、`storyIntent`、`entityStore`、`generationStatus` 这类项目级真相
2. 聊天、粗纲、人物、详纲、剧本各自独立分表
3. 每张阶段表只保留当前一条真相，并带自己的 `version`

以后只要做“本地文件 -> 云端存储”迁移，我要先问：

1. 哪些是项目级字段
2. 哪些是阶段级大 blob
3. 哪些更新会高频局部发生

如果不先拆这三层，后面的 Repository 一定会越写越脏。

### 3.101 前端不该知道数据库拆了几张表，快照缝合必须留在服务端（2026-04-18）

这轮做阶段 3 时，我确认了一个非常关键的边界：

- PocketBase 可以拆成 `projects + chats + outlines + characters + detailed_outlines + scripts`
- 但前端仍然只认一份 `ProjectSnapshotDto`
- 所以“把 6 张表重新缝起来”这件事，必须放在服务端 mapper，不准推给前端自己拼

如果把数据库拆表事实直接外泄到前端，后果会很差：

1. 前端会开始感知存储结构，而不是业务结构
2. renderer 会重新长出第二份拼装逻辑
3. 后面一改表结构，前后端一起震

以后只要做存储重构，我都要先问：

1. 数据库存储形态是不是只留在仓储层
2. 服务端返回 DTO 有没有继续保持稳定
3. 前端是不是仍然只吃业务快照

只要第 3 条没保住，迁移就不算稳。

### 3.102 项目 CRUD 换轨时，要先切“读项目真相”的入口，再谈生成链迁移（2026-04-18）

这轮阶段 4 又把一个顺序问题钉死了：

- 用户要求这一步只做项目 CRUD 数据流换轨
- 不能顺手去碰粗纲/人物生成链
- 真正该先收的是“项目大厅”和“工作台初始化”这两个读项目真相的入口

因为只要这两个入口还走 `window.api.workspace`：

1. 前台看到的项目列表和项目快照就还掌握在本地 IPC 手里
2. 后面即使服务端已经有 Repository 和快照 mapper，产品体感上也还是本地项目
3. 很容易误把“API 已加上”当成“正式主链已接管”

这轮更对的顺序是：

1. 先把 `api-client.ts` 补成正式项目 HTTP 客户端
2. 再把项目大厅的 `list/create/delete`
3. 再把工作台初始化的 `getProject`
4. 再把常用保存接口切过去

以后再做前后端分离收口，我要先问：

- 用户最先看到的项目列表是谁给的
- 打开项目时第一份正式快照是谁给的
- 这些入口是不是已经统一走 HTTP

如果没有，那就还没真正换轨。

### 3.103 七问确认这类前置真相，不能只“保存动作走 HTTP”，保存后的成功判断也必须走云端回读（2026-04-18）

这轮阶段 5 又确认了一个很容易漏掉的半截工程：

- 表面上把“确认并保存七问”按钮改成调 HTTP，不算真的收口
- 如果保存后前端还是：
  - 先本地把 `isSaved=true`
  - 再去 IPC 读项目
  - 或者干脆不回读，只当自己已经保存成功
- 本质上还是双管线

真正收口要同时满足三件事：

1. 保存动作走 HTTP
2. 保存后的成功判断也走 HTTP 返回的项目快照
3. 七问正式真相仍然只落在 `outlineDraft.outlineBlocks[].sevenQuestions`

以后遇到这种“前置确认步骤”的换轨，我要同时检查：

- 写口是不是换了
- 回读口是不是也换了
- UI 成功态是不是依赖服务端真相，而不是本地乐观幻想

### 3.92 人物 V2 这类结构化批量任务，正确顺序不是”先重试更多次”，而是”先退避，再缩包” （2026-04-14）

这轮又把一个很实战的点坐实了：

- `terminated / 502 / 504` 不一定说明合同错了
- 很多时候只是这一包太重、这一秒太挤，provider 先把连接掐了

如果我只做“同样大小的 prompt 继续秒重试”，结果通常是：

1. provider 还没恢复
2. 下一次还是同样大小
3. 重试只是把总耗时拉长

这次更稳的顺序是：

1. 先做有限次指数退避，让 provider 有呼吸时间
2. 如果同一势力还连续炸，就别再整势力硬扛
3. 直接顺着天然边界拆成单人物请求

以后遇到“结构化批量生成 + provider 断流”这类问题，我要先问：

1. 有没有退避
2. 有没有天然切块边界
3. 连续失败后会不会自动继续缩包

三件事少一个，线上就还不算稳。

### 3.93 `tools/e2e/out` 这类证据目录，如果只靠手动清理，最终一定会变成垃圾场（2026-04-14）

这轮现实证据很直白：

- 目录里已经堆出大量 `userdata-* / evidence-* / e2e-tmp-*`
- 还有浏览器缓存 `Cache_Data`
- 即便仓库已经 `.gitignore`，它也还是会拖慢排查、污染判断

我得记住：

- “不进 git” 不等于 “不会失控”
- 这类目录必须在 runner 层默认自清
- 人工 `cleanup` 只能当补刀，不该当主方案

以后只要碰到长期生成证据目录，我都要优先查：

1. 新运行前会不会先修剪旧目录
2. 浏览器缓存会不会被一起带走
3. 修剪规则是不是只认一种命名，导致别的残留越滚越多

### 3.90 `faction_matrix` 这类重世界观任务，如果还借 `rough_outline` 的壳跑，我会同时丢掉预算、日志和前台错误语义（2026-04-14）

这轮我又被一个很典型的“假复用”提醒了：

- 势力矩阵明明是独立任务
- 但代码里却一直拿 `task: 'rough_outline'` 去跑
- 同时还在 agent 里硬编码 `120000ms`

结果就是三层一起乱：

1. timeout 预算不对
2. runtime 诊断分类不对
3. 前台只能收到一个很笼统的超时或白板结果

以后只要遇到这类天然重任务，我要先问：

1. 它有没有自己独立的 `AiTaskKind`
2. 它有没有自己独立的 timeout 预算
3. 它失败时前台能不能直接看出“到底是哪一锅炸了”

少任意一个，后面排查成本都会被放大。

### 3.91 “记录七问握手状态”值钱，不是因为它能拦人，而是因为它能直接证明不是空锅发车（2026-04-14）

这轮用户要求我确认“米到底下锅没有”，这个提法非常对。

真正该做的不是：

- 再加一层硬 gate
- 或者在入口再搞一套二次校验

而是：

- 在正式入口把每个 section 的 `goal / obstacle / effort / result / twist / turnaround / ending` 是否有值直接打进日志

这样一来，后面只要又出白板或超时，我第一时间就能分清：

1. 是不是空七问发车
2. 还是七问明明在，真正死在下游调用

这类握手日志是证据口，不是新门禁。这个边界以后不能再弄混。

### 3.89 后端已经解耦，不等于产品体验已经解耦；前台按钮链如果还绑死，用户体感上就还是一个黑箱（2026-04-13）

这轮我得记住：

- 七问后端其实早就有三段正式链：生成初稿、保存确认版、基于确认版继续生成粗纲和人物
- 但前台一直把它们塞在一个 `handleConfirmAndGenerate()` 里

结果就是：

- 代码口径上说“七问已独立保存”
- 用户体感上却还是“我一确认就直接掉进大任务黑洞”

以后不能只看“后端有没有接口”，还要再问：

1. 用户能不能明确感知到“这一步已经保存成功”
2. 用户能不能停在这一步不继续
3. 用户能不能晚一点再启动下一条长任务

做不到这三条，产品体验上就还没真正解耦。

### 3.88 人物 V2 这种天然可按 faction 切片的任务，不要继续幻想“一次 JSON 吐完整张表”还能长期稳住（2026-04-13）

这轮我把人物 V2 再往下扒开后，确认了一件事：

- 它原来不是高并发洪峰
- 而是单次把整个 `factionMatrix` 都塞进 prompt
- 再要求模型一次吐完整 `characters[]`

这在人物少的时候也许还能混过去，但人物和字段一多，就会稳定长成：

1. prompt 变大
2. response 变大
3. JSON 更容易截断或缺字段
4. retry 再整轮重来
5. 总时长直接被拖爆

这次更对的做法是：

- 按 `faction` 切块
- 每块单独生成
- 用并发池把最大并发压住
- 任一块失败时直接带势力名报错

以后遇到类似“整张结构化大表”的任务，我要先问：

- 有没有天然的业务切片边界
- 能不能先按边界拆块
- 再决定是否需要并发池

先拆块，再谈并发，不能顺序反过来。

### 3.87 人物 V2 Agent 只要还是“裸 parse + 裸错误”，它就一定会变成粗纲链新的盲区（2026-04-13）

这轮又补了一刀很具体的教训：

- 前面 `faction-matrix-agent` 已经做了 parse issue 诊断和二次纠偏
- 但后面的 `character-profile-v2-agent` 还停在：
  - 一次 `JSON.parse`
  - 缺任一字段就直接 `character_profile_v2_parse_failed`
  - 不记录缺了谁、缺了哪维、是不是截断

结果就是：

- 前台只能看到一个很笼统的报错
- 我如果不去翻最新主进程日志，就会分不清它是不是旧残留
- 下一轮还得从头猜

以后只要是这种结构化 Agent，我要默认检查三件事有没有同时到位：

1. 失败时能不能给出 parse issues
2. 失败后有没有最多一次纠偏重试
3. 最终抛错时，错误码尾部有没有带上具体 issue 摘要

少一个，线上排查成本都会被直接放大。

### 3.86 用户发“存档”时，不能只口头说“现在状态差不多清楚了”，而要做物理退场（2026-04-13）

这轮文档整理又把一个老坑坐实了：

- 旧主线明明已经写到 `5/5`
- 非现役参考文档也明明不该继续挂在 `docs/` 根目录

3. `characterDrafts`
   - 完整人物小传
   - 只留给真正关键的人
4. `active character package`
   - 当前 `5` 集批次活跃人物
   - 让详纲和剧本只吃当前需要的人

这次还要额外记住两个边界：

- “20 集约 10 人、40 集 20+ 人”这种话只能当经验区间，不能写成硬限制
- 新人物允许出现，但必须有来源：要么来自 `entityStore`，要么来自势力人物位，要么来自当前批次活跃人物包

以后再做人物系统，我先问的不是“还差几份人物小传”，而是：

- 势力底账有没有
- 当前批次活跃人物够不够
- 哪些人必须升级成 full，哪些人 light 就够

### 3.69 人物系统第二刀先把页面分轨做成“派生面”，不要为了好看立刻改存储合同（2026-04-09）

这轮我必须记住，第二刀最容易做歪的地方不是 UI 难，而是我很容易顺手去改存储合同。

但这次更对的顺序是：

1. `entityStore` 继续当世界底账
2. 人物页把数据拆成：
   - 完整人物小传
   - 轻量人物卡
   - 势力与人物位
3. 轻量卡只做派生和升级入口
4. 持久化仍然先只认现有 `characterDrafts`

这轮有效做法有两个：

- “谁已经升级成完整小传”优先认 `masterEntityId`，没有时再按名字回查主实体
- “升级成完整小传”直接复用 `fromMasterEntity()`，不要再手写第三套人物映射

以后只要做人物扩容，我先问自己：

- 这层是不是其实只需要派生页面数据
- 这轮是不是非改保存合同不可
- 轻量人物和完整人物是不是又要被我做成双真相

如果答案不是“非改不可”，那就先派生，先升级，别急着动存储。

### 3.68 结构性设计做完以后，第一刀要先落“正式底账 + 最小观察口”，不要一上来把编辑层全做爆（2026-04-09）

- “先做观察口”很重要，因为没看见就不知道底账有没有真的落到项目快照
- 第一刀先只读是对的，不要一轮里同时把势力位编辑、轻量卡编辑、完整小传升级全揉在一起

### 3.66 如果我要用户“回粗纲补设定”，那粗纲页就必须真的有这个入口，而且这条不能再卡死详纲（2026-04-09）

### 3.65 详细大纲前置条件如果已经知道，就在 renderer 先翻成动作，不要再让用户先撞一次失败（2026-04-09）

这轮用户现场说得很直接：

### 3.29 这轮七问联调又提醒我：要先分清“用户在看哪一页”和“项目正式推进到哪一层”

### 3.32 这轮又提醒我：集数口径不能只看“哪种字段更正式”，还要看聊天里谁说得更晚

### 3.34 这次“30 集只剩 10 集可见”提醒我：不要把“显示归一化”写成“默认截断”

这轮最开始我差点又往“是不是主进程只生成了 10 集”“是不是切页又拿旧项目盖回来了”这些方向继续钻。

- 持久化也已经有 `30` 集

结果就是：

- 详纲页也只会跟着看到前 `10` 集

这次我要记死两件事：

- **“归一化”只能补形状，不能在没有明确授权时缩真相**
- **共享 helper 只要同时服务“显示层 + 编辑层”，默认参数就必须非常保守**

以后再看到“明明后端对，前台只剩一部分”这种问题，我要第一时间检查：

- 共享 normalize helper 有没有偷偷带默认上限
- store 的编辑入口是不是复用了这个 helper
- UI 文案里有没有还在发出旧的固定集数口径

### 3.24 这次 10 集 fresh run 教我的，不是“模型又不行了”，而是“Agent 工序还没收干净”

刚才这次 fresh 10 集最值钱的事实，不是又多了一次 timeout，而是我终于把 timeout 卡点拆开了：

- `tools/e2e/out/evidence-v11-10ep-mngi0yj6/` 下已经有 `ep1` 到 `ep10` 的 evidence 文件。
- 这说明 batch 生稿阶段实际上已经一路跑到了第 10 集。
- 但同一轮 `workspace/projects.json` 仍然是：
  - `generationStatus=script`
  - `scriptDraft=[]`
  - `visibleResult=pending`

这证明：

1. 卡住点不在“10 集没生成出来”。
2. 卡住点在“整轮 startScriptGeneration 调用没有回到保存链”。
3. 真问题不是一句“超时了”，而是**正式 Agent 工序还没收干净，导致链路重复串行，整轮返回过慢。**

我这次必须记住两层具体根因：

1. **batch 内已经有当前集 rewrite**
   - 当前集胖了/瘦了/污染了，batch 第 2/3 次机会已经会调 `episode_rewrite`
2. **batch 后又整批再跑一次 repair orchestrator**
   - `start-script-generation.ts` 结束后还会调用 `repair-script-quality-with-agents.ts`
   - 这意味着同一批 10 集，内容问题可能被前后两层都串行打一遍

如果我再把 `hook_weak` 这种用户已经明确放掉的问题也塞进自动返修触发里，整轮时长只会更夸张。

这次真正该记住的话是：

- **Agent-first 不是“加了一个 repair 文件就算落地”**
- **真正落地，必须把“哪些问题该自动修、在哪一层修一次、修完何时返回保存链”全部收成一套**

### 3.25 用户这次把项目的底层方法论彻底讲透了：只准调 Agent，不准四处补门禁

这次用户不是在给一个“局部优化建议”，而是在重写这个项目的质量治理方法论。

我必须把这套方法论记死：

1. 只要问题属于内容质量：
   - 太长
   - 太短
   - 场景不稳
   - 人物不对
   - 对白不行
   - 污染残留
   - 节奏不对
   - 这些都优先交给对应 Agent 修
2. Agent 必须是**窄职责**：
   - 一个 Agent 只管一个问题
   - 字数 Agent 只管胖瘦
   - 场景 Agent 只管场景结构
   - 人物 Agent 只管人物口风和关系
3. Agent 必须**基于原稿修**：
   - 要把原稿全文发给它
   - 要把当前具体错误告诉它
   - 要把明确目标告诉它
   - 不准零基础重写冒充返修

用户这次真正想推翻的，不只是某个 hook 或某个 guard，而是这套老思路：

- 这里加一点规则
- 那里补一点门禁
- 再换一个阈值
- 再改一个 parser

这套老思路最大的问题不是“有时不准”，而是：

- 工程越改越复杂
- 链路越跑越慢
- 结果还是不稳定

所以我以后必须先问自己：

- 这个问题是不是其实应该新增一个单职责 Agent？
- 我是不是又在本能地想去别的地方补一条硬规则？

### 3.26 第 11 刀提醒我：解析保真和污染观测是两回事，不能混成一个判断

这次 placeholder stub 问题，我前一刀把 parser 修对了：

- duplicate stub 不再掌权
- 正式场头能被保住
- 第 28 集本地复喂后只剩真实 `28-1`

但我差点就把这当成“污染已经修完了”。

其实不是。

2. quality / batch guard 的职责是告诉系统“原文里有没有混进脏头”

如果 parser 一边把 `人物：人物`、`△# 第28集` 这种 placeholder stub 剥掉，quality 另一边又直接吃 parser 结果，那污染信号就会跟着一起消失。

这轮我必须记住：

- **被 parser 清掉，不等于没发生过**
- **原文污染信号要单独看 raw screenplay**
- **正式结构真相和污染观察信号可以同时成立，不冲突**

- **检测到污染，不要只停在记账**
- **要立刻把污染稿送进窄职责格式清理 Agent**

### 3.28 这轮 `Phase 4` 复验提醒我：最容易漏掉的不是大逻辑，而是“改名后还剩一个旧变量”

这轮我本来以为 `episode_engine / arc_control / emotion_lane` 已经基本收干净了。

### 3.29 这轮用户把“Agent-first”的落地方式讲得更直白了：不是多 Agent 叠加，而是一个专家写、一个专家按问题单改

我这轮最该记住的，不是“又加了一个手动按钮”，而是用户把逻辑重新讲顺了：

1. 批量写还是按 `5` 集一批跑，这样用户不用一集一集等。
2. 但改稿不要再搞一串 Agent 轮番上。
3. 检查器只先抓硬毛病：
   - 格式不对
   - 场次不对
   - 字数不对
   - 明显占位词
4. 抓到问题后，不是让另一个模型整集重写，而是把“哪儿不好、为什么不好”喂回给同一个专家，让它在原稿上改。
5. 自动最多改一次；改完继续往后写，别卡住整轮。
6. 真不满意就让用户点“重写这一集”。

这件事对我最大的纠偏是：

- **Agent-first 不等于 Agent 越多越高级**
- **对这个项目，用户现在要的是“一个专家 + 一个问题单 + 一次自动改 + 用户可手动再点”**
- **如果我又本能地去补第二轮、第三轮自动返修，或者再加一批并行专家，我就是又把系统做回旧路了**

我这次必须记住的新判断是：

- 如果 `epN` 正常，而 `epN+1` 的 `promptLength` 突然暴涨到前一集的 2-3 倍，就不要再只盯承接块
- 还要继续查 `ledger / momentum / recap` 有没有把刚生成的正文整段又灌回去

这轮真实证据就是：

- `ep11 promptLength=4149`
- `ep12 promptLength=12281`
- `ep12` 基本复写 `ep11`

所以以后遇到这种“批内第二集突然整集复写”的形态，第一反应要改成：

1. 先看 `promptLength` 是否异常膨胀
2. 再查 prompt 哪个 recap 块在回灌正文
3. 最后才考虑 detector 或 parser

### 3.28 这轮又纠正了我一个判断习惯：`promptLength` 暴涨不等于 base prompt 暴涨

我前面有一个偷懒判断：

- 某集 `promptLength` 很大
- 我就直接把它当成 base prompt 里 recap 块太肥

这次我把 `run-script-generation-batch.ts` 重新对了一遍，确认不是这么回事。

`evidence.promptLength` 记录的是：

- **这一次尝试真正发出去的 prompt**

如果 episode 已经进入第 2/3 次尝试，它发出去的可能是：

- `buildEpisodeEditPrompt()`
- 或 `char-count-agent` 的改稿 prompt

那它天然就会比首发 base prompt 长很多。

所以以后我必须先分两层：

1. 这是首发 prompt 变大了
2. 还是 episode 自己进入改稿尝试后变大了

不分这两层，就很容易把“改稿 prompt 长”误判成“前情 recap 长”。

### 3.29 最新 20 集 evidence 还提醒我：有时候内容问题已经过去了，但收口链没过去

这轮最新真实 `20` 集，有一个比“duplicate 还在不在”更重要的新事实：

- evidence 已经一路跑到 `ep20`
- 但项目落盘里还是：
  - `generation in progress`
  - `scriptDraft=[]`
  - `visibleResult=pending`

这说明什么？

- **内容生成不一定没出来**
- **有可能是生成已经出来了，但保存链 / 返回链 / 状态写回没收口**

所以以后我看到“命令超时 + 项目里没稿”时，不能立刻说“模型没写出来”。

我要先分清：

1. evidence 有没有已经写到后面
2. 如果 evidence 已经写到后面，是不是保存链没落

因为如果我让 continuity / character / theme 这些内容 Agent 直接吃 placeholder stub 脏稿：

1. 它们会把注意力浪费在脏格式上
2. 内容修复目标会被稀释
3. 最终又容易回到“检测能报，返修不落地”的半成品状态

所以现在更对的顺序是：

- 先 `format-pollution-agent`
- 再内容质量 Agent

### 3.27 这次主链复扫提醒我：别被“下面还能继续优化”带偏，先看正式入口有没有收住

这次用户追问得对。

如果我只是顺着当前手感继续往下补：

- `scene_structure-agent`
- 更多内容 Agent
- 更多检测口

那很容易又回到“下面越补越细，上面正式入口其实还没收住”的老路。

这轮真正值钱的发现不是再补出一个下游细节，而是：

1. `startScriptGeneration()` 之前默认同步等 `repair_batch`
2. 已有部分剧本但后续批次失败时，`ledger/postflight` 会直接空掉

这两个问题都属于**主链编排问题**，优先级高于继续补新 Agent。

所以我这次必须记住：

- 当用户问“还有没有必要继续往下面做”时，不能顺着惯性继续下钻
- 要先回头看正式入口、返回路径、落盘路径、可见状态是不是已经真的收住

这轮最后的正确判断是：

- placeholder / pollution 这条线已经基本收住
- 当前没必要继续往下扩新 Agent
- 更应该先把 `startScriptGeneration` 这条正式主链收口

### 3.28 这次 agent-first 总改造提醒我：首稿 prompt 太胖，本身就是波动源

这轮我已经确认一件事：

- rewrite guard
- rewrite 拒收
- Phase A / Phase B 编排

这些都开始收住了，但真实 10 集仍然波动很大。

这说明我之前有一个惯性错误：

- 总觉得“后面再补一个 Agent / 再补一个 guard / 再补一条拒收”就会稳

实际上，如果首稿 prompt 还在同时要求模型：

- 控结构
- 控字数
- 控人物弧光
- 控主题落点
- 控尾段余波
- 控制度场分布
- 控世界观表达

那首稿本身就已经被我写成“大一统写作裁判”了。

这和用户要的 agent-first 是冲突的。

我这次必须记住：

1. 首稿不是最终成品判定口
2. 首稿先写出稳定可修稿就够了
3. 高级工艺不该继续堆回首稿 prompt
4. prompt 一旦过胖，本身就会制造真实长季波动

以后再看长季波动，我必须先问：

- 是不是首稿 prompt 又在替后续 Agent 干活了？
- 是不是把“后续可修”误写成了“首稿一次写满”？

### 3.29 这次还学到一件事：没有主线文档，我很容易又陷入“继续补一点”

用户这次指出的问题很准：

- 我就容易边做边扩
- 做着做着就只剩“继续”

1. 当前主线必须单独落成 `docs/主线/YYYY-MM-DD-主线.md`
2. 任务清单和完成定义写在那里，不散落在聊天里
3. 主线做完就归档，下一轮新建新的日期文档
4. 没有主线文档时，不要假装自己知道“下一步自然该干嘛”

### 3.30 这次真实 10 集复验提醒我：关键验证过了，就该先考虑收口

这轮最重要的新事实不是“还能不能继续再补一刀”，而是：

3. 下一步应该先问“这条主线是不是该归档、切新主线”，而不是下意识继续往下钻

### 3.31 这次继续跑到 20 集提醒我：新失败类型出现后，要果断切主线

这次我继续往下跑，不是白跑。

因为它明确告诉我：

- 上一条主线解决的是前段稳定性
- 新冒出来的是后半程批次失败
- 失败码非常具体：`duplicate_scene_detected:scene_16_to_17`

1. 不能把“前 15 集已经全过”误写成“20 集也稳了”
2. 也不能把这个新问题继续硬挂在旧主线下面
3. 一旦失败类型变了，就该切新主线，不要怕麻烦

### 3.28 这轮 tie-break 修正提醒我：局部根因打中了，也可能只是“第二层根因”

这次我补 `bestAttempt` 的 tie-break，不是瞎试，方向其实是对的：

- 旧逻辑确实只按失败码分数选稿
- 如果两版都只剩 `char_count`，系统会机械保留第一次更胖的旧稿
- 这会让后续更接近合同的改稿白改

这刀最小单测和组合回归都过了，说明这个 bug 是真 bug，也确实修住了。

但真实 `10` 集复跑后，只从 `3/10` 到 `4/10`，而且仍有：

- 第 `7` 集 `4150` 字
- 第 `10` 集 `3440` 字
- 平均字数 `2462`

这说明我这次要记住另一层经验：

- **单个根因修对了，不等于主问题就只剩这个根因**
- **真实 evidence 如果还在大面积偏胖，就不能继续沉迷“这刀理论上已经修好了”**
- **下一步要直接查 Agent 的真实改稿效果，而不是继续围着入口和 tie-break 自证**

### 3.29 这轮又提醒我：Agent 改了稿，不等于系统就该把它收下

我这次本来在查“是不是 content agent 把稿又写胖了”，结果补出一个更工程化的洞：

- continuity / character / theme / format 这 4 个 Agent
- 之前判断 `changed` 的逻辑都是：
  - 候选稿 failures 变少，或者
  - 候选稿正文不一样

第二条其实非常危险。

因为这等于：

- 只要模型重写了
- 哪怕重写后 guard 更差
- 哪怕新长出 `char_count`
- 系统也会把它收成“有变化”

这不是内容问题，是正式收稿逻辑问题。

这轮把它补掉以后，真实 `10` 集虽然还没恢复到可用，但我至少多确认了一件事：

- **后续 content agent 不会再把更差的稿硬覆盖掉原稿**

同时我也得记住：

- 真实结果从 `maxChars=4150 / avg=2462` 到 `maxChars=3332 / avg=2226`
- 说明这刀不是没效果
- 但 `passRate` 仍只有 `3/10`

所以这轮不能写成“胖稿问题已解”，只能写成：

- **反向收坏稿的系统漏洞已修**
- **真正剩下的是 char-count-agent 自己还压不够狠**

### 3.30 这轮按场硬压证明：真实效果终于上来了，但单向加压会带来新的结构副作用

这次把 `char-count-agent` 改成“按场硬压最胖场”之后，真实 `10` 集第一次出现了比较像样的变化：

- `passRate` 到了 `5/10`
- `avgChars` 降到 `1788`
- 最坏胖稿也从上一轮 `3332` 继续往下掉到 `2993`

这说明：

- **按场打最胖场** 这个方向是对的
- 之前那种“整集抽象压一点”确实不够

但新的坑也很明确：

- 第 `4` 集被压成 `72` 字
- 场号还出现了 `4-1 / 4-1`

这提醒我一件很重要的事：

- **Agent prompt 不能只写“必须压到合同内”**
- **还必须写“最低保真边界”**

不然模型为了完成压缩任务，会直接把正文削成壳，甚至把场次结构也压坏。

所以下一刀我不能再继续单向加“多删一点”“再狠一点”，而是要补：

- 最低场次保真
- 最低单场内容保真
- 禁止重复场号
- 禁止把剧本压成提纲壳

### 3.31 这轮继续补“反过压保护”后，我得承认：只改 prompt 到这里已经到收益递减点了

这次我把 `char-count-agent` 的 prompt 又补了一层：

- 场号必须全保留且不能重复
- 每场至少保留 1 条有效动作 + 2 条有效对白
- 禁止 markdown 壳、placeholder 壳

本地回归全过，但真实 `10` 集又掉回 `2/10`。

这说明一个很关键的事实：

- **prompt 里写了，不等于模型就稳定做到**
- **模型偶发做不到时，系统当前也没有二次拒收这类结构坏稿**

所以我这轮必须记住：

- 当 prompt 已经写到很细，但真实样本还反复出现“重复场号 / placeholder 壳 / 极瘦稿”
- 就不要再指望“再补一句 prompt 就稳了”
- 该切到**程序侧验收 rewrite 结果结构真相，不合格就拒收候选稿**

### 3.32 这次连程序侧 rewrite 拒收都补了，真实样本还是反弹，我不能再假装“下一刀还在这条线上”

这轮我已经把 rewrite 侧能补的两类东西都补了：

- prompt 约束
- 程序侧拒收

而且本地测试、组合回归都继续是绿的。

但真实 `10` 集却掉到了 `1/10`。

这说明我必须承认一个事实：

- **rewrite 链不是最后主根因**

也就是说：

- 单集改稿怎么选
- 候选稿怎么拒收
- prompt 怎么写硬一点

这些已经不是决定性矛盾了。

真正还没打到的，是更前面的层：

- 首稿本身是不是已经严重失衡
- 当前批上下文是不是把后几集越喂越肥
- 上游详细大纲 / 承接脚手架是不是在真实链里把单场回合数推得过密

所以我之后不能再继续假装“只要把 char-count-agent 再补半刀就能收口”。

这次又补了一条很重要的新记忆：

- **主链断层修好，不等于真实质量自动回来**

我把正式 IPC 入口显式切回 `waitForRepairBatch: true` 后，真实 `10` 集复跑确实不再出现那种：

- 第 `10` 集只剩 `66` 字
- placeholder 脏头直接落盘

但整体结果还是只有 `3/10` 过线，而且大量是胖稿。

这说明：

1. 之前那个“默认不等 repair 导致首稿直接落盘”的问题，确实存在
2. 但它不是当前唯一主问题
3. 当前真正还没收住的是：**字数返修链在真实长季里没有把大量胖稿有效压回合同**

以后我必须避免再犯一个判断错误：

- 不能因为修好了入口返回路径，就急着宣布“剧本主链已恢复”
- 真实 evidence 只要还显示大量胖稿，主战场就还是字数返修效果本身

### 3.13 正式链 / 原型链并存时，不把边界写进代码，我自己下一轮也会看走眼

这轮代码质量整治又提醒我一件事：

- 我脑子里虽然知道 episode-level 才是正式链，scene-level 还是 prototype。
- 但文件头如果自己写着 `Production assembly`、`Single-scene generation prompt` 这种口气，下一轮非常容易又被我或别人误读成“这套已经接生产了”。

1. `create-scene-generation-prompt.ts` 明写 prototype-only。
2. `assemble-episode-scenes.ts` 明写 not wired into current production chain。
3. `run-script-generation-batch.ts` / `create-script-generation-prompt.ts` 明写自己才是正式入口。

同时我还确认了另一条工程纪律：

- 这种文件继续挂着，只会制造“它是不是还算现役”的噪音。

### 3.12 死代码判断不能只信静态搜索，真实编译链才是最终裁判

这轮清死代码，我又补了一条很值钱的教训：

- 我先用 grep 看 `resolve-lane-strategy.ts`、`select-script-generation-batch.ts`，表面像 `0` 外部引用。

  1. `build-execution-plan.ts -> resolve-lane-strategy.ts`

这说明：

1. 静态搜索只是候选发现器，不是最终裁判。
2. 死代码清理必须小步改，然后立刻跑真实编译验证。
3. 真正稳的删除对象，往往是那种“同文件内也没人再用”的分析残留、格式 helper、诊断导出，而不是看起来孤儿的正式生产文件。

### 3.13 依赖分层不只看 `src/`，还要把根目录脚本 / tools / 历史文档一起纳入判断

这轮做代码质量整治计划时，我又补了一条很值钱的教训：

但它们很多时候只是手工调试入口、prototype 验证入口或历史设计证据，不是正式生产依赖。

2. prototype 依赖
3. 测试依赖
4. 根目录临时脚本依赖

### 3.14 长测前别被旧测试文案带偏，先看真实 prompt 和 build

### 3.15 用户说的“三道工序”不是比喻，是正式重试策略

这次我又差点按自己的惯性把 `char_count` 重试继续做成“再生成一遍”。

但用户说得很准：

- 第一遍可以放飞写。
- 第二遍、第三遍必须基于第一遍成稿改稿。
- 如果第二遍第三遍还当新稿写，本质上就不是稳步收敛，而是一次次抽卡。

这次真正该记住的是：

1. 长季控长失败时，不能只在原 prompt 后补一句“太长了/太短了”。
2. 必须把上一版完整 screenplay 原文塞回去，让模型直接在原稿上做减法或加法。
3. 要把“当前字数、目标区间、比例、最胖/最瘦场”明确给它，不然模型不知道从哪下手。

以后只要用户明确说“改稿”，我就不能再偷偷做成“重生成”。

### 3.16 用户把合同定死后，我不能只改 guard，不改 prompt

这次发车前检查又补出一个很典型的自伤点：

- `contract-thresholds.ts` 和 batch guard 已经收成正式 `800-1800`
- 但 `create-script-generation-prompt.ts` 里还在拿旧口径 `700-1200 / 600-1300` 教模型写

这不是“小地方没改到”，而是系统自己一边让模型写薄，一边再用更高门槛把它拦死。

这次正确动作不是继续怪模型，也不是先去跑 30 集，而是先把三处口径收成一套：

1. 正式硬红线：`800-1800`
2. prompt 首稿稳态：压在中间带 `950-1450`
3. 场级预算：只做中间带参考值，不再把 `1800` 当常态，也不再拿 `700-1200` 误导模型

以后只要用户把某个合同定死，我必须机械检查三件事：

1. threshold 常量是不是改了
2. 正式 prompt 是不是还在说旧话
3. 测试断言是不是还在盯旧口径

少看一处，长测就可能白跑几个小时。

### 3.17 发车前把口径收成一套后，10 集真实 API 可以直接看出链路是不是还在炸

这次我把正式字数口径统一成：

1. 硬红线：`800-1800`
2. 首稿稳态：`950-1450`
3. 场级预算：中间带参考，不再让模型按旧的 `700-1200` 去写

然后直接跑真实 API 10 集：

- 结果 `10/10` 全过
- 最少 `1151`
- 最多 `1773`
- 平均 `1551`
- 没有 thin / fat
- 没有阻断、没有模板污染、没有 95 字错接稿

这说明一件事：

- 这次 10 集的主要改善，不是“模型突然变聪明了”
- 而是我先把系统内部互相打架的口径收掉了

以后如果用户让我先小样本发车，我要优先看：

1. 有没有完整跑完
2. 有没有中途链路炸裂
3. 字数是不是整体回到合同中间带

这三件事如果都成立，才配说“可以继续扩大样本”。

### 3.18 10 集全过后，20 集第一刀要盯“中段异常膨胀”，不是回头怀疑 10 集结论

这次继续扩到 20 集后，我学到一条更细的判断：

- 前 10 集全部过线
- 但第 11 集直接冲到约 `4374` 字
- 报错是 `char_count`，不是模板污染，不是 95 字错接稿，也不是运行时阻断

这说明：

1. 10 集结论没有被推翻，链路收口这件事仍然成立
2. 新暴露出来的是“中段上下文一放大，单集会异常膨胀”
3. 这种失败不能再偷懒写成“又炸了”或“整条链又不行了”

以后看到这种结果，我要先拆开写：

- 小样本链路是否已收住
- 扩样后第一次失控出现在第几集
- 失控是污染、阻断，还是异常膨胀

只有这么拆，后面才能真去打第 11 集这个点，而不是又把整条链重判一遍。

### 3.19 用户说“胖了就让它按原稿压到合同内”，我不能自作聪明加一个会误导模型的比例下限

这次第 11 集查到底，最关键的不是“模型怎么会这么笨”，而是我自己的改稿提示词把目标写歪了：

- 对胖稿，我之前用的是压缩比例
- 但又给这个比例加了一个最低 `65%` 的钳制

结果就是：

- `2050` 字时，`65%-70%` 还勉强能压回线内
- `4374` 字时，`65%` 还是 `2843` 字，根本还在失败区

也就是说，提示词表面在说“去改短”，实际上是在要求模型继续超线。

这次真正该记住的是：

1. 用户要的是更简单的做法，不是更花哨的算法
2. 胖稿就直接告诉模型：原文在这里，改到 `1500-1800`，目标约多少字
3. 瘦稿就直接告诉模型：原文在这里，补到 `1000-1400`，目标约多少字
4. 比例可以作为辅助说明，但绝不能再反过来压过绝对合同

以后只要是“基于原稿改稿”的任务，我都要先问：

- 这个目标是不是直接可执行
- 还是我又加了一个看起来聪明、实际会把模型带偏的中间规则

### 3.20 守门合同和提示词目标要分层：系统守 `1800`，模型只看 `1000` 左右

这次用户又把一件事说得很透：

- 正式合同当然还是 `800-1800`
- 但提示词里如果把 `1800` 直接亮给模型，它会把这个上沿当成“可以继续冲”

所以现在更合理的分层是：

1. **系统守门**：`800-1800`
2. **提示词执行目标**：`1000` 字左右，尽量 `900-1200`

这套分层的价值在于：

- 用户要的是最终合同稳定
- 模型看到的是更保守、更容易收住的写作目标

以后只要合同上沿明显高于模型的稳态输出，我都要先问：

- 这个上沿是不是该继续留在系统层
- 而不是直接暴露给模型层

### 3.21 用户说“代理就是 Agent”时，我不能再用工程黑话把简单工序说复杂

这次用户把字数返修的本质说得很准：

- 代理就是 Agent
- Agent 当然要调大模型
- 但它应该是一个单职责 Agent，只管胖瘦，不管剧情裁判

我之前的问题不是没想到“可调用大模型的返修口”，而是还在用自己的工程分层语言说“这不是那种复杂 Agent”，结果把一件很简单的事说绕了。

这次真正该记住的是：

1. 用户要的不是大而全多轮规划 Agent，而是**专门改字数的窄职责 Agent**
2. 这个 Agent 最重要的不是“名字”，而是：
   - **基于原稿改**
   - **胖稿/瘦稿分开提示**
   - **和首稿任务口分开**
3. 只要 `char_count` 重试还跟 `episode_script` 混在一起，系统口径就还是不干净

所以这次正式落地时，我做对的点应该固定下来：

- 新增独立 `char-count-agent.ts`
- batch 里显式区分：
  - `episode_script` = 首稿
  - `episode_rewrite` = 字数代理改稿
- prompt 里直接把角色写成“字数代理（char-count-agent）”

以后用户再说“搞个 Agent 专门干这个”，我不能再先从“是不是复杂 Agent 系统”去理解，而要先看：**是不是一个单职责、可调用大模型、能把当前工序收干净的窄口子。**

### 3.22 用户已经把项目核心战略重新钉死：质量问题不是拿来卡死的，是拿来继续派 Agent 改的

这次用户把更底层的战略说得很直接：

- 不要门禁
- 不要判官
- 不要卡着
- 不要为了“判断不过关”就停在那里
- 遇到质量问题，就继续派 Agent 去改

我必须记住，这不是一句情绪话，而是这个项目下一阶段真正的产品方向。

它的本质是：

1. **系统失败** 和 **内容没写好** 不是一回事
2. `runtime / parse / API / persist` 这类真失败，才该报错
3. 胖瘦、污染、格式、钩子、对白、节奏这些内容问题，本质上都是“下一道工序的输入”
4. 所谓 judge / guard / 质量检测，以后主要负责：
   - 告诉系统哪里不好
   - 帮系统决定该派哪个 Agent
   - 不负责最后拍死作品

这次我最该修正的思维是：

- 以前老想着“先定义一套门，再判断它过没过”
- 用户现在要的是“别老想着拦，想着怎么继续改”

所以以后我做方案时，第一反应应该变成：

1. 这是什么问题类型
2. 应该新增或调用哪个单职责 Agent
3. 这个 Agent 要基于原稿怎么改
4. 改完之后再看还剩什么问题

而不是：

1. 先上 gate
2. 先上 judge
3. 先上 hard fail
4. 再想怎么补

一句话记住：

**这个项目的核心不是“判得准”，而是“改得动”；不是“会不会卡死”，而是“能不能用一串 Agent 把作品越改越好”。**

### 3.23 光把理念写进文档不够，必须把“掌权位置”真的换掉

这轮我又补到一个很关键的事实：

- 只要 `run-script-generation-batch.ts` 里还保留“质量失败就 throw”
- 只要 `finalize-script-postflight.ts` 里还把 `screenplay_quality_failed` 写进 issues
- 只要 `persisted-generation-truth.ts` 里还把 `quality.pass=false` 写成 blocked

那项目嘴上就算说“不要门禁”，代码实际掌权的还是门禁。

这次真正完成收口的动作，是把 3 个掌权点同时换掉：

1. **batch 退位**：只挑当前最佳稿，不再判死内容问题
2. **repair 接管**：batch 后统一走 `repair-script-quality-with-agents.ts`
3. **persisted truth 改口**：只要已有稿，就保持可见；没真失败时直接 released

我这次最该记住的是：

- 用户说“不要门禁”，不是让我少说点 gate 这个词
- 而是要我把**真正决定结果的代码分支**从“拦住”换成“继续改”

以后只要项目方向说“改成 Agent 串行返修”，我就要立刻追问：

1. 现在哪个函数还在 `throw`
2. 哪个持久化分支还在 `blocked`
3. 哪个前台提示还在说“不过线先卡着”

不把这 3 类掌权点一起收掉，方向就还没真的落地。

### 3.11 timeout 修复必须用同口径 fresh evidence 验，不然很容易被旧产物骗

这次 30 集详纲 timeout 修复之后，真实 fresh run 又提醒我一件事：

- 只看源码里 `DETAILED_OUTLINE_TIMEOUT_MS = 150000`，不够。
- 只看 `npm run build` 成功，也不够。
- 真正能坐实“新 build 已生效”的，是 fresh evidence 里的 `runtime-diagnostics.log`。

这次有效证据就是：

1. `opening / midpoint / climax / ending` 四个 `act_start` 都写明 `timeoutMs=150000`
2. 之前最危险的 `climax (15-22)`、`ending (23-30)` 都有 `act_finish`
3. 两段 `responseChars` 都大于 0，没有 timeout

这说明以后我遇到“超时修复已重建”这种任务时，不能提前默认修好了，必须按同一季长、同一链路再跑一轮 fresh evidence，把日志和后半段结果一起钉死。

### 3.1 先收主权，再修细节

这轮真正有效的不是"再试一下"，而是按顺序收口：

1. 删掉粗纲入口二次 `summarizeChatForGeneration()`。

这个顺序要记住，因为它证明：
先收主权，再看超时和质量，才是真的顺序。

### 3.2 先清文档主权，再继续大改代码

当前这一步做 5 份现役文档，不是形式主义。

这是在修过去反复失忆、反复误判、反复重开的根因。

如果这一步不做，后面不管再修多少代码，下一轮还是可能按旧任务卡和旧计划重来。

### 3.2.1 用户已经把批次口径定死时，不要再自作聪明搞动态尾段

这次又踩到一个很典型的坑：

- 我先看到 `20` 集扩面失败，就本能地往 “`targetEpisodes-aware` 动态尾段规则” 去想。
- 但用户要的根本不是“随总集数动态伸缩尾段”，而是更简单也更强的口径：**永远 5 集一批，前文压缩后喂下一批**。

这个坑的本质不是技术判断错了，而是我又试图替用户发明一套“更聪明”的规则。

以后遇到这种场景，要先做 2 件事：

1. 先看用户是不是已经把批次单位说死了。
2. 如果用户已经说死，就只收 prompt 口径，不再往动态区间、尾段推导、`targetEpisodes-aware` 这些方向发散。

这轮真正对的动作不是重写 batching，而是把正式 prompt 里所有 `第7-10集 / 最后一集 / 后四集` 全部收成：

- `当前 5 集批次`
- `当前批次末集`
- `当前批次末两集`
- `当前 5 集批次的收口段`

### 3.3 旧文档必须物理退场

只靠嘴上说"这个已经过期了"没用。

只要旧文档还像现役入口，新开的终端就会自然相信它。

所以以后：

- 新主线一旦成立，旧文档就要退到 `docs/归档/`。
- 旧结论一旦被推翻，就不能再在现役文档里出现。

### 3.4 文档收口一定要带全仓自检

这轮又验证了一次：

- 只要仓库里还能轻易看到旧 README、旧 plans、旧当前工作区，新开的终端还是会被带偏。

以后做完文档整理，我必须补一刀机械自检：

1. 全仓搜 Markdown，排除 `node_modules` 后，确认 `docs/归档/` 外只剩 5 份现役文档。
2. 再看 `docs` 根目录，确认除了 4 份现役文档外，只剩 `归档/`，不再挂旧目录空壳。
3. 如果旧目录里还有非 Markdown 附件，也要一起并进 `docs/归档/`，不能留下半套旧入口。

### 3.14 用户已经把字数战略定死时，我不能再用“放宽合同”替代“当前集重写”

这轮我又差点犯老毛病：

- 看见 50 集不稳，就本能想把合同放宽到 `600-2200`
- 这其实是在迁就结果，不是在执行用户战略

用户真正要的是：

1. 正式合同固定 `800-1800`
2. 当前集字数不对，就给当前集重写机会
3. 当前集没过线，就停在当前集，不继续往后写

我这次必须记住：

- “先让它跑完”不是正式目标
- 放宽合同会直接掩盖真实问题
- 真正该做的是把 `char_count` 接成正式阻断项，让生产入口按当前集重写策略工作

### 3.15 “每 5 集一批”不等于“每批输入压力一样”，我不能再凭感觉判断长季链路

这次 30 集第 19 集卡死，又补了一条很关键的教训：

- 批次虽然固定是 5 集
- 但正式运行时的上下文压力并不是固定的
- `estimateEpisodeContextTokens()` 会随着已生成剧本持续增长

更严重的是，我还发现正式承接链一直带着一个隐藏 bug：

1. 测试 helper 给 scene 塞了 `episodeNo`
2. 正式产物只有 `sceneNo`
3. `build-scene-progression-directives.ts` 却一直按 `episodeNo` 找上一集

结果就是：

- 测试看起来“连续性正常”
- 正式生产里第 6/11/16/19 集的承接块却一直在吃“上一场待补”

我这次必须记住：

- 长季问题不能只看“还是 5 集一批，所以逻辑应该一样”
- 要直接看正式产物形状、正式 prompt、正式承接块到底吃到了什么
- 只要测试 helper 比生产对象多字段，就必须高度怀疑自己是不是又测出了假连续性

### 3.5 每轮收口不补四件套，下一轮还是会失忆

现在还要多记一条：

- 干活终端每完成一轮任务，不能只停在"代码改完了"。
- 必须顺手把 `聊天记录.md`、`项目原则.md`、`AI自我学习.md`、`项目结构.md` 一起补上。
- "先不存档"只是不做额外压缩归档，不代表这四份可以不更新。

尤其是 `项目结构.md`，它不是可有可无的整理活。

如果结构已经变了，这份文档不更新，下一轮新终端还是不知道主链现在落在哪、风险区在哪、哪些目录已经退场。

### 3.6 详纲这轮不是单根因，是"双根因"

这次详细大纲终于收住后，我要记清楚：

- 第一层根因是单次请求太重，一口气让模型吐整季四段逐场景，超时概率很高。
- 第二层根因是详纲自己还绑着另一套解析口径，模型即使回了内容，也可能被系统自己解析坏。

真正有效的动作不是只调 timeout，而是：

1. 改成四段顺序生成。
2. 解析口径统一到 `tryParseObject`。
3. 给每段补 `act_start / act_finish / act_fail` 日志。

这三步缺一不可。少一步，后面还会回到"看起来像模型不稳定"的假判断。

### 3.7 详纲单测卡住，不是测试问题，是依赖越界

这次 `generate-detailed-outline.test.ts` 跑不起来，真实根因是：

- `generate-detailed-outline.ts` 直接 import 了 Electron 诊断日志模块。
- 单测还没开始跑业务逻辑，模块加载阶段就先被 Electron 绊倒。

这说明我以后不能再把"能独立测试的业务逻辑"直接绑进运行时依赖。

正确做法是：

- 业务模块只吃注入进来的 logger 接口。
- Electron 的 `appendRuntimeDiagnosticLog` 留在 IPC / 主进程入口接进去。
- 就算 logger 自己写盘失败，也只能吞掉，不能覆盖正式业务错误。

这样既保住真实软件里的运行日志，也保住纯 Node 单测。

### 3.8 纯 Node 回归不只盯业务模块，shared / renderer 运行时文件也得写成 Node 能直接吃

这次 `detailed-outline-generation-notice.test.ts` 和 `resolve-detailed-outline-persistence.test.ts` 又说明：

- 光把业务模块拆掉 Electron 依赖还不够。
- 只要测试链会直接 import 到 shared / renderer 运行时文件，extensionless runtime import 和 parameter property 一样会让 `node --test` 在加载阶段先死。

以后我要固定这么做：

- 需要被原生 Node 直接 import 的运行时 TS 文件，运行时 import 全部带 `.ts` / `.tsx` 扩展名。
- 不要在这些文件里写 Node strip-only 不支持的语法，尤其是 constructor parameter property。
- 先让测试能加载，再谈业务判断。

### 3.9 "算了但没用"比"完全没算"更难发现

这次 `buildScriptGenerationExecutionPlan()` 证明了一种新型假放行：

- 函数内部明明先调用了 `validateStageInputContract('script', payload)` 得到了真实的 stageValidation。
- 但 return 时直接硬编码 `ready: true / blockedBy: []`，完全丢弃结果。
- 表面上有校验动作，但校验结果对输出没有任何影响。

我以后看到"先调用了 X 但最后硬编码返回"这种模式，要直接当成假放行，不只是技术问题，是设计问题。

这次剧本入口三处同时存在这类问题（plan 构建 / evaluateStageAccess / start-script-generation），说明这不是偶发，是开发过程中没有意识到"算了就要用"这条纪律。

## 3.10 链式验证时不能只相信"操作成功了"，还要验证"读到的路径和写入的路径是不是同一个"

这次 `script-quality-matrix.mjs` 发现：

- `saveScriptDraft` 和 `saveScriptRuntimeState` 都执行成功了（通过 poll `scripts=10` 可以证明数据已落盘）。
- 但 `loadProjectStore(caseOutDir, projectId)` 始终返回 null。
- 根因：`caseOutDir`（`tools/e2e/out/`）和 `userDataDir`（`tools/e2e/out/userdata-sq-{caseId}-{ts}/`）只差一个子目录名字，但写入的目标是 `userDataDir`，读的是 `caseOutDir`。

教训：链式操作（写→等→读）里，我习惯性相信"操作成功了，验证也应该能读到"，但实际 bug 经常出在参数传错（传了名字相似的不同变量）。以后链式验证时，要先打印出读写路径是否真的相等。

## 3.11 judge 读到错误数据会给出错误的 quality 判断

这次发现 `judgeScript` 读 `scriptDraft[0]` 作为 firstScene。如果 rewrite 模式的拼接顺序错了（`[...base, ...newScenes]` 而不是 `[...newScenes, ...base]`），firstScene 会变成 overflow episode（没有 dialogue），导致 quality=弱，但其实内容是好的。

教训：`judgeScript` 本身没问题，是数据源错了（拼接顺序）。当 quality 判断不符合预期时，要先验证 `scriptDraft[0]` 是不是期望的那个 scene，再怀疑 judge 的质量标准。

## 3.12 judgeScript 捕获的是"数据质量"，不是"系统链路质量"——要区分

这次 RS-A 真实模型验证又上了一课：

- judgeScript 报 quality=弱，firstScene.totalLength=107，hasPressure=false，hasStrongHook=false。
- 我第一反应是"postflight 有问题"或者"prompt 没对齐"。
- 但 plan/gen/save/runtime 全部通过，说明系统链路没问题。
- 根因是 seed ep1 内容太薄，不满足 judgeScript 的质量门槛。
- 修复 seed 后，RS-A real 立即 PASS。

教训：judgeScript 是质量裁判，不是系统裁判。当 quality=弱 且 issues=0 时，要先问"输入数据本身是否满足质量门槛"，而不是直接跳到"系统哪里有问题"。

## 3.13 历史 PASS 会过期；不按当前口径复跑，就会被旧结论骗

这次 2026-03-28 凌晨复核 2026-03-27 条目，我又学到一件很痛的事：

- 我原本以为“3/27 写进聊天记录的 build/typecheck 通过、3 个 real case PASS”可以先信。
- 结果当前代码一复跑，`npm run typecheck` / `npm run build` 先被新的未入库测试文件打红。
- 再按当前 official quality 复跑 `fs-b` / `rs-a` / `rw-b`，三例全部 FAIL。

这说明：

- 历史绿灯不是现役绿灯。
- 旧 PASS 只能当历史证据，不能直接指挥今天的判断。
- 只有“当前代码 + 当前口径 + 当前命令”三件套同时再过一次，才配写成现役可信结论。

## 3.14 真实 evidence 一旦证明“系统误伤修住了”，我就必须马上把注意力切回内容层

这次真实 API 全链又补了一刀经验：

- 我先修了 `VO`、`sceneCount`、parser 对无时间词标题的容错。
- 新 evidence 一出来，`VO` 漏出没了，`Ep10` 也不再被压成 1 场。
- shared 质量从上一轮 `6/10` 提到这轮 `9/10`。

这说明：

- 系统误伤已经不是主战场了。
- 如果我还继续围着 parser / 检测器打转，就是在错过真正的问题。

真正剩下的，是剧本内容厚薄失衡：

- 前段个别集偏瘦，例如 `Ep1=826`
- 中后段个别集过肥，例如 `Ep3=1357`、`Ep6=1561`、`Ep9=2196`

以后看到这种局面，我要立刻收口成一句话：

- 上游层基本稳，系统口基本稳，当前主问题只剩 script 内容层。

不能再把“全链都可能有问题”当安全说法。

## 3.15 后半程发胖，根因通常不是“字数规则不够狠”，而是单场里塞了太多回合

这次新的真实样本让我更确认一件事：

- 真正发胖的集数，不是因为每句都废，而是因为**一场里塞了两三轮追打、翻盘、解释**。
- 比如 `Ep9-2` 这种长场，本质上是把“发现线索 -> 使者堵门 -> 黑气灌入 -> 暗道追杀 -> 钥匙暴露 -> 退路断掉”几拍戏挤进了一场。

所以后面再调 prompt，我要优先压这件事：

1. 一场只打一轮。
2. 结果落地就切场。
3. 重复追打、重复逼问、重复翻盘，直接删。

而不是继续写“写短一点”“不要太长”这种空话。

## 3.16 parser 要尽量保住真实多场，别因为模型少写半个格式词就把结构全判死

这次 `Ep10` 的真实问题很典型：

- 模型写了 `10-1 玄玉宫山门外石阶`、`10-2 医庐内室`
- 场号、地点、人物表、正文都在
- 只是少了时间词

旧 parser 直接只认最后一个 `10-3 午后...`

这类问题如果不修，后面所有质量判断都会被带偏：

- sceneCount 错
- 字数合同错
- 钩子判断窗口也会错

所以我以后要记住：

- parser 首先要保住真实结构。
- 格式不完美，可以交给 prompt 和质量门去纠偏。
- 不能一上来就把真实 3 场压成 1 场。

## 3.17 10 集项目里调顺的尾段 prompt，不能默认拿去跑 20 集

这次 20 集真实 API 全链把一个之前没暴露彻底的问题坐实了：

- runner 改成 `--episodes=20` 后，上游真的出了 20 集
- 但正式 prompt 和详纲 prompt 里还残留大量固定文案：
  - `第7-10集`
  - `第8-10集`
  - `第9-10集`
  - `最后一集`

这些规则在 10 集项目里是对的，在 20 集项目里就会错位：

- 真正的后半程应该落在 18-20 或至少更靠后
- 结果系统却在 7-10 就开始用“终局/余波/制度确认最短/不要再开新口”的约束
- 到 15-20 集时，反而失去正确的尾段收束和压缩规则

真实结果也很一致：

- 20 集 shared 质量只有 `12/20`
- 失败集集中在 `15-20`
- `VO` 在后段重新漏出
- 后段字数重新暴涨

以后我看到“10 集口径固定写死”的 prompt，必须第一时间问一句：

- 这是不是 targetEpisodes-aware？

如果不是，就不能拿它去验更长季长。

## 3.18 长季质量下滑，有时候不是模型不听话，而是 compact 逻辑先把最值钱的脚手架删了

这次 T6 我又踩到一个很典型的坑：

- 我先盯的是“VO 太多、对白不足、弱钩子、场数少、厚薄不均”，直觉很容易继续怪模型或怪 prompt 不够狠。
- 但把正式链读透后发现，长季一进 compact mode，系统自己先删了两块最关键的东西：
  1. `【当集最后一场结果落点】`
  2. `【对白口风】`

这两块一删，后果非常直接：

- 弱钩子更容易回升，因为最后一场没有被硬推到“结果已落地”
- 对白更容易变薄，因为人物口风和对白抓手被系统自己抽掉了
- 长季越容易进 compact，越容易刚好在最难的时候失去这些约束

以后我看到“长季比短季更飘”时，要先查：

1. compact mode 删了什么
2. 被删的是不是正好影响质量主问题的脚手架
3. 这是不是系统自己放松了质量合同，而不是模型突然更差

## 3.19 详纲压缩不是越狠越好；30 集以内如果把逐集可见性压没，剧本层会一起失真

这次另一个很值钱的新认识是：

- `compact-script-context.ts` 之前会把长季逐集覆盖压成大段范围占位
- 这对省 token 有利，但对剧本层理解“当前是第几集、这一集在整批里的密度和节奏该怎样”并不友好

特别是在 `30` 集这种用户明确要看的长度里：

- 如果 prompt 里逐集信息被压成大范围占位
- 模型就更容易把场数写松、把厚薄写飘、把某些集写成泛化推进

所以以后我要记住：

- `30` 集及以内，优先保逐集可见
- 先压细节，不先压 episode-level visibility
- 不要为了“压缩得很漂亮”把真正对长季节奏有用的信息一起压掉

## 3.14 剧本 batch 里的后续集，必须看到本批次前面刚生成的内容

这次 P3-3 又学到一件很关键的事：

- 我一开始以为“只要 rewrite 前情改成按 `episodeNo` 查就够了”。
- 但 batch 里如果 episode 2、3、4 都还在吃 batch 开始前那份旧 `existingScript`，continuity 还是会飘。
- 这时表面像“有些集完整，有些集只剩标题和一两行”，很容易被我误判成模型本身忽然不稳定。

正确做法是：

- prompt 生成 episode 2 时，要能看到 episode 1 的新内容。
- prompt 生成 episode 3 时，要能看到 episode 1、2 的新内容。
- batch 不是一次冻结上下文，而是要滚动更新。

## 3.15 rewrite 同一集有旧版和新版并存时，必须显式 latest wins

这次真正把 rewrite 收干净，靠的不是“我感觉应该拿新的”，而是把规则写死：

- `existingScript` 很可能同时存在 `旧版第3集` 和 `新版第3集`。
- 如果还按第一条命中去取上一集，第二批生成就会把旧稿又扶正。
- 改成“取最后一个匹配版本”以后，跨批次 rewrite 才真正能承接第一批已经重写出来的新内容。

这条以后我必须记死：

- 同一 `episodeNo` 多版本并存，不允许交给数组顺序碰运气。
- 一定要明确谁是最新版本，默认规则就是 `latest wins`。

## 3.16 mock 口的 postflight FAIL，不等于流程没通

这次 6/6 mock 全流程又提醒了我：

- `plan/gen/save/runtime` 全绿，说明流程已经通了。
- 但 mock stub 文字极短，`postflight` 一定会按质量门把它拦下。
- 如果我把这个 `FAIL` 直接讲成“剧本流程失败”，那就是把质量问题又拿去冒充流程问题。

以后我必须先问：

1. 这次失败是流程层断了，还是质量门在正常工作？
2. mock stub 的失败，能不能拿来判断真实流程？
3. 当前到底是在验“能不能跑通”，还是在验“写得好不好”？

答案要分开记，不能再混。

## 3.17 生成格式合同如果同时要多场 heading 和 A/D/E 三段，最后一定会互相打架

这次剧本 `sceneCount=1` 终于让我看清一件事：

- prompt 明面上要求 2-4 场、每场 scene heading；
- 但主生成入口长期先走 A/D/E；
- repair 入口也只改 A/D/E，不刷新 `screenplay / screenplayScenes`。

结果不是“模型偶尔不稳”，而是：

- 模型写出来的多场信息会在入口路由里先被压扁；
- quality 再把这件事报成“场次数不在 2-4 场”。

以后我必须先问：

1. 当前正式交付格式到底是一套什么合同？
2. prompt、parse、quality 是不是都在吃这一套？
3. 如果入口默认单场重建，那就不能再嘴上要求多场 screenplay。

## 3.18 parse 入口选择比 parser 本体更关键

这次一个很容易看错的点是：

- 我一开始总想去怀疑 `parseScreenplayScenes()` 会不会不认多场；
- 但真正的问题不是 parser 本体，而是 `parse-generated-scene.ts` 和 `parse-ai-repaired-scene.ts` 到底先走 screenplay 还是先走 A/D/E。

如果入口选错了：

- parser 再强也接不到原始多场信息；
- 后面的 quality 和 repair guard 看到的都只是被压扁后的单场结果。

以后我必须先查：

1. 原始输出里的关键信息是在哪一步丢的；
2. 是 parser 不会认，还是根本没机会看到；
3. 真正该修的是 parser 规则，还是入口路由优先级。

## 3.19 repair 只改 A/D/E、不刷新 screenplay 真相，会把旧场次结构继续带下去

这次又补上一课：

- 以前我会觉得 repair 只要把 `action / dialogue / emotion` 改掉就算完成；
- 但后面真正参与 quality、compact、guard 的，往往还是 `screenplay / screenplayScenes`。

如果 repair 后不刷新这两份：

- 主生成也许已经是多场新口径；
- repair 出来的 scene 却还挂着旧的单场结构；
- 最后项目里会并存两套行为，只是表面上看不出来。

以后只要 repair 改了正文，我就要顺手确认：

1. `screenplay` 有没有同步刷新；
2. `screenplayScenes` 有没有同步刷新；
3. 后面的 guard / compact / quality 是不是还在吃旧结构。

## 3.20 先查“哪条链在跑”，再谈提示词好坏

这轮我又踩到一个很蠢但很贵的坑：

- 看了很多 P0/P1/P2/P5 汇报，就默认 scene-level 已经是正式主线；
- 结果进代码一看，正式批量生成还在 `run-script-generation-batch -> create-script-generation-prompt`；
- `create-scene-generation-prompt.ts` 和 `assemble-episode-scenes.ts` 只被原型测试与 `tools/e2e/test-scene-p*.mjs` 用到。

教训：

1. 只要任务在调 prompt、验质量，第一步必须先查引用链，确认现在到底是哪条 prompt 在生产上跑。
2. prototype 跑通，最多说明“这条路值得继续”，不能说明“生产已经接上”。
3. 如果不先查引用链，我就会把大量精力花在一条根本没进正式链的 prompt 上。

## 3.21 production prompt 不能自己打自己

这轮生产 prompt 最脏的一处不是模型不听话，而是 prompt 自己冲突：

- 前面已经要求 screenplay-first，禁止 `Action:/Dialogue:/Emotion:`；
- 后面却还在用“Emotion 段”“Action 叙述”“Emotion 字段必须……”这套旧话术继续教模型写。

这会直接造成两层伤害：

1. 模型不知道到底该按 screenplay 写，还是按旧 A/D/E 结构去组织内容。
2. 我以为自己在“加强质量规则”，实际是在把旧格式残词重新塞回生产 prompt。

以后我必须死记：

1. 正式交付格式一旦收成 screenplay-first，所有写作规则也必须一起改成 screenplay-first。
2. 不准一边禁某种格式，一边继续用那种格式给模型上课。
3. prompt 测试也要防回流，不能再用断言把旧 `Emotion 段` 话术锁回去。

## 3.22 控长不是靠继续加长 prompt 去压

这轮我又试错了一次，而且是真样本直接打脸：

- 我先给 production prompt 加了一整块“按场次控长脚手架”；
- 结果 prompt 从 `4781` 涨到 `5097`，真实样本反而从 `1372` 涨到 `1683`；
- 还顺手带出了不可拍心理描写和弱钩子。

然后我把这块撤掉，换成更短的规则：

- 降低膨胀性最低要求（`2 条△动作 + 3 轮对打` 改成 `1-2 条关键△动作 + 2-3 轮有效对打`）
- 明确禁止同类动作、同义威胁、同一情绪反应换句重复写

结果第三次 production 真样本：

- prompt `4882`
- `charCount=1157`
- `sceneCount=2`
- `pass=true`
- 只剩 `集尾钩子偏弱`

教训：

1. 模型超长时，不是脚手架越多越好。
2. prompt 一旦继续变长，就要警惕它是不是在鼓励模型“写得更完整、更铺满”。
3. 控长最有效的 prompt 往往不是多给预算说明，而是少给膨胀空间，直接禁止重复动作和重复威胁。



### 3.26 这轮真正让我看清了"信号分类"才是 Agent-first 的真正门槛

这轮第一阶段的核心工作不是"加新 Agent"，而是把已经存在的质量信号做分类：

- **actionable**：会触发自动返修的问题（字数胖瘦、污染、格式、VO）
- **observe-only**：只展示给用户看，不触发自动返修（`hook_weak`）

这件事最难的不是技术实现，而是意识转变。

以前我会本能地觉得"所有问题都应该修"，于是把 `hook_weak` 也悄悄接进了返修链路。

但用户这次把方向说得很清楚：**系统不替代创作裁判**，弱钩子不是系统该拦截的东西。

所以这轮真正学到的是：**观察信号（看得到，但不动手）和自动返修信号（只对 actionable 生效）**这条边界一旦画清楚，后续每加一个新 Agent，我就先问："它是 actionable 还是 observe-only？"

### 3.27 "先返回、先落盘"不只是一句口号，是真正的架构约束

这次把 `waitForRepairBatch` 放进正式链之前，我反复确认了一点：

- `postflight` 永远跑（它只是观察）
- `repair_batch` 可以选择跳过（它可以异步）

这个顺序不能反。如果我把"等 repair 跑完"当成"正式返回条件"，系统就会因为内容问题永远卡在半路。

用户真正要的是：**先生成，再改；先生成，再改；先生成，再改**。

不是：先生成，等修完，才算返回。

## 4. 我最容易再犯的错

1. 看见超时、格式错、报错码，就直接追技术点，不先查第二口径。
2. 看见某一层顺了，就下意识觉得主链快好了。
3. 看见旧计划写得很完整，就把它当成今天还有效。
4. 看见"唯一真相源已收口"这种旧结论，没先核对后面有没有新事实推翻。
5. 把历史参考文档当成现役入口。
6. 以为"现役文档写完了"就等于文档系统已经收口，没有再做物理迁移和全仓搜索。
7. 以为只要 `聊天记录.md` 更新了就够了，没有把 `项目原则.md`、`AI自我学习.md`、`项目结构.md` 一起补齐。
8. 看见 judgeScript 报 quality=弱，就直接跳到"prompt 问题"或"postflight 问题"，而不是先验证 seed 内容本身是否满足质量门槛。
9. 看见昨天文档里写着 PASS，就偷懒不复跑，直接把旧结论搬成今天的指挥依据。
10. 看见碎片化，就先怪模型不稳定，没有先查 batch 滚动上下文和 rewrite latest-wins 有没有收住。
11. 把带 `vite-node` 的命令直接丢在当前指挥终端跑，结果把会话本身跑崩，连刚整理好的判断口径一起丢掉。

## 5. 我以后每次新开终端必须先做什么

1. 先读 `AGENTS.md`。
2. 再读 `docs/聊天记录.md`，确认当前主线。
3. 再读 `docs/项目原则.md`，确认底线和已知误判。
4. 再读这份 `AI自我学习.md`，确认过去为什么反复失败。
5. 最后读 `docs/项目结构.md`，知道仓库主链和结构落点。

如果我没按这个顺序，就很容易被归档里的旧计划、旧规则、旧任务卡带偏。

## 6. 当前风险排序

### P0

- 文档主权如果没收死，新开的终端会继续被旧计划、旧任务卡带偏。
- 详纲 / 剧本阶段还没有按这轮新主权做完整真实复测。
- 这轮详纲虽然已经真实跑通到"四段全部落盘"，但还不能宣布整链稳定。

### P1

- ~~`summarize-chat-for-generation-support.ts`、旧 fallback/support 还有回流风险~~ → P2-1 已物理退场（9 个 fallback 文件已删）。
- ~~generationStatus 第二写口~~ → P2-1 已收口（renderer 直写链全删，main 单一写口）。
- 自动推进后续怎么恢复，不能再长成后台第二主线。
- 剧本 `sceneCount` 主因已经在代码层收住，但 fresh real evidence 还没补齐；compact 路径和“有 heading 但无第X集标题”的残余路由还要继续盯。
- 详纲下游"详细大纲 -> 剧本"现在已经可以进入真实开测，但 official quality 还没闭环。

### P2

- 旧测试脚本、旧污染项目、旧阶段结论再次被拿来当当前判断依据。

## 7. 我以后做判断的固定顺序

1. 先问：现在的现役文档是不是只有 5 份在掌权。
2. 再问：当前主线是不是只有一条。
3. 再问：用户输入是不是还在当唯一真相。
4. 再问：旧兼容、旧 fallback、旧任务卡、旧自动推进有没有回流。
5. 最后才看具体技术点、提示词、结构，性能问题。
6. 只要命令会拉起 `vite-node`，就默认交给独立执行终端，不在当前指挥终端直接跑。

## 8. 最后提醒

- 这个项目最容易输，不是输在模型不够强，而是输在旧口径没死。
- 这个项目最容易假好，不是因为没人测试，而是因为太容易把局部通过写成整链通过。
- 这个项目真正要修的，不只是代码，也包括"AI 下一轮怎么不再按错路重开"。

## 9. P2-1 学到的（2026-03-28）

### 死代码退场要彻底

这次删 9 个 fallback 文件，发现：只要代码还在磁盘上，就永远有被重新引用的风险。物理删除比"标记废弃"有效得多。

### 单一写口是防止双口径的最有效机制

generationStatus 原来 renderer 和 main 都能写，结果 stale 检测在 renderer 侧、main 又不知道 renderer 写了什么——两个独立写口永远会慢慢长出不同步。这次把写口收到 main，事件通过 preload 广播给 renderer，renderer 只订阅不写，是正确的架构。

### 新文件放在哪很重要

stale 检测逻辑放在 `src/shared/domain/workflow/generation-state.ts`（shared），而不是 `src/main/` 或 `src/renderer/src/app/utils/`。这样 main 和 renderer 都能用，没有重复。

### 全量测试隔离运行和全量运行结果不一样

这次发现全量 `node --test` 时 53 个失败，但隔离运行 P2-1 相关测试时 82/82 全绿。说明有一些 pre-existing 的测试隔离问题，不影响 P2-1 本身。验收时应该先隔离验证核心改动，再全量。

## 10. P3-0 学到的（2026-03-28）

### 真实样本 runner 里的启发式指标，只能当观察口，不能当最终验收口

这次 P3-0 把 `script-quality-matrix` 的 verdict 从 `judgeScript` 收到 shared `inspectScreenplayQualityBatch()` 后，mock matrix 直接从旧口径的“看起来还能过”变成官方口径的 6/6 FAIL。

这说明：

- `judgeScript` 这种启发式观察口可以告诉我“首场有没有钩子、长度够不够、有没有 action/dialogue/emotion”。
- 但它不能替代正式质量裁判。
- 如果我让启发式观察口继续参与最终 PASS/FAIL，本质上就是又长出了一套第二裁判口。

以后我必须先问：

1. 这个 runner 里的指标是正式裁判，还是 debug 观察口？
2. PASS/FAIL 是不是只由当前现役 shared/domain 质量规则派生？
3. 如果 debug 指标和正式裁判冲突，最终听谁的？

答案必须固定：**听正式裁判，不听 debug 指标。**

## 11. P3-1 学到的（2026-03-28）

### prompt-only 没拉动官方质量时，不能继续盲加 prompt 条款

这次我只改了 `create-script-generation-prompt.ts`，把开场卖点、后果承接、场尾硬钩都写得更直接了，而且 prompt 测试全绿。

但跑完官方口径下的 mock matrix，结果还是 0/6 FAIL，且全部归因到 `postflight`。

这说明：

- prompt 的表达确实可以更清楚；
- 但如果官方质量结果完全不动，就不能继续靠“再加一条 prompt 指令”赌运气；
- 这时真正该做的是先确认 postflight/官方质量到底是在准确指出问题，还是把别层问题统一收成一个大 FAIL。

以后我必须记住：

1. prompt-only 做完，先看官方矩阵有没有真实改善；
2. 如果没有改善，不准继续无节制加 prompt 条款；
3. 先转去看裁判边界和归因层，而不是把所有问题都当 prompt 不够狠。

### 内容质量不能只靠 `mock`、单测和当场口述

这轮又补了一条很关键的教训：

- 粗纲、人物小传、详细大纲、剧本，只要是在验“写得怎么样”，就必须直接调真实模型 API 看成品。
- `mock`、单测、结构校验只能告诉我流程通没通、合同有没有对齐，不能告诉我内容到底写得好不好。
- 真实调用后的原始输出，必须落成仓库内可复看的 `txt` 或 `md` 文件；不然只有执行终端自己看过，指挥终端没法复盘，也没法据此调提示词。
- `20` 集及以内，默认可以按用户要求直接全量验证；更长任务先按用户指定范围或切片来跑，不擅自整季全写完。

### Plan B（逐场生成）原型证明：字数问题首先是生成粒度问题

这轮真正把字数问题拆开后，我学到：

- 旧 episode-level 一次生成整集时，字数合同就算写进 prompt，也会被 30+ 规则稀释。
- 改成 scene-level 逐场生成后，单场 350 字 budget 能明显守住，说明模型不是“完全不认字数”，而是“整集一次生成时会自然把每场写满”。
- 所以字数问题不能一直拿 prompt wording 或 postflight trim 当主战场；第一次失控点在生成粒度。
- 但这条结论目前只是原型证据，不是生产事实；正式链还没接上 `create-scene-generation-prompt.ts`。

以后我必须先问：

1. 当前预算是写成提示语，还是写进了生成工序？
2. 如果模型在单场里能守预算、整集里守不住，问题就在生成粒度，不在后处理。
3. 逐场生成只要最终仍然组装回同一份 screenplay 并走同一 parser，就不是发明新功能，而是在修工序。

### 旧合同会误伤新工序，必须跟着改

这轮我又学到一件很容易忽略的事：

- 旧的 `800-1200` 是按整集单次生成定的。
- Plan B 改成逐场生成后，每场都有 heading、人物表、空行这些 connective overhead。
- 如果质量门还继续拿 flat `800` 去判两场 scene-level 结果，就会把本来合理的 500-600 字整集误判成失败。

以后只要工序粒度变了，我就要立刻检查：

1. 质量合同是不是还在按旧工序算。
2. 阈值是不是已经 sceneCount-aware。
3. 当前失败是内容真的差，还是旧合同在误伤。

### 安全阀死代码和辅助阀门要分清

这轮 C2 让我彻底记住：

- 没接进生产主链的 safety valve，就是死代码，不算能力。
- 真接进主链以后，也不代表它就能解决主问题。
- 如果真实诊断表明超长主因是对白和动作骨架本身过密，而不是尾部解释/总结肥厚，那 trim 就只能是辅助阀门。

以后看到“生成后收束”这类方案，我必须先问：

1. 它有没有真的接进正式路径。
2. 它命中的问题类型是不是当前主因。
3. 如果主因是正文骨架过密，继续扩 trim 范围会不会越过工序边界。

### 弱钩子有时不是 prompt 问题，而是检测器词根太窄

这轮最容易误判的一点是：

- 我们已经把 hook prompt 改成“最后一句必须是已经发生的可见后果”。
- 模型也确实写出了“字帖被撕成两半”“手腕弯折”“墙上压痕”“握刀的手伸进来”这种视觉结果。
- 但 `hasConcreteHardHook()` 仍然把它们打成弱钩子，因为 `RESULT_MARKERS` / `ACTION_RESULT_PATTERNS` 只认自己那几个词根。

以后处理弱钩子，我不能再直接跳到“prompt 还不够狠”，必须先查：

1. 模型最后一句到底写了什么。
2. 检测器到底认什么词。
3. 是模型没写到，还是检测器没认出来。

### 如果 hook prompt 再加一轮都没用，就别继续原地堆 prompt

最新 5 集回归又补了一刀：

- 我们已经继续收紧了单场 prompt 的 hook 合同。
- 但弱钩子频率仍然是 `5/5`，一点没降。
- 同一轮里还能看到“字帖被撕成两半”“手腕弯折”“压痕出现”“刀手伸进来”这些已经发生的可见结果。

这说明：

- prompt 不是完全没作用，但它已经不是当前第一矛盾。
- 如果我还继续把精力放在“再加一条 hook 规则”，就是在重复无效动作。
- 当前第一动作应该是把 `hard-hook` 的 result marker、action result pattern 和真实样本重新对齐。

以后我必须记住：

1. 连续两轮 prompt 收紧都没把频率打下来，就要停。
2. 这时先查检测器，不先怪模型。
3. 真实样本已经写出的东西，检测器认不出来，就是检测口径问题，不是创作能力问题。

### 补弱钩子口径时，要只补“结果句”，不要把弱句一起放进来

这轮我又补了一条实操纪律：

- `露出一角纸边`、`露出缝隙的铜钥匙` 这类，是关键物件已经显形，属于结果句。
- `指甲崩裂渗出血丝`、`嘴角渗出血丝` 这类，是身体后果已经落地，也属于结果句。
- 但 `踏上台阶`、`盯着看`、`叩三下` 这些，还是逼近、观察、预备动作，不该因为“也有张力”就放进硬钩子。

以后我补检测器时要先问：

1. 这句话是不是某个状态已经改变了？
2. 还是只是危险靠近了、视线落过去了、动作准备开始了？
3. 如果只是后者，就继续判弱，不要为了提高通过率把口径冲塌。

## 12. P3-6 学到的（2026-03-29）

### Bold-markdown 包裹 scene heading 会导致 Route 1.5 完全失效

这次发现了一个新型路由失效：

- 模型把 scene heading 包裹在 `**bold**` markdown 里（如 `**6-1 日**`）
- Route 1.5 的 regex `/(?:^|\n)\d+\-\d+\s+/` 无法匹配 `**6-1 日**`（`\*\* 在数字前面）
- 结果：所有 heading 都被漏检，路由 fallback 到 A/D/E 路径，全部压成 1 场

教训：parse 路由里的 regex 假设如果和模型的实际输出格式不符，路由会悄悄失效。

模型用 markdown bold 包裹 heading 是合理的表达方式，但 parse 代码没有预处理这一步。

正确做法：任何文本标准化（如剥除 `**`）都要在 regex 检测之前做，不能假设输入是裸文本。

### sceneCount 质量门用哪份数据源

这次发现 quality gate 里的 `sceneCount` 读了 `parseScreenplayScenes(screenplay).length`，但 `screenplay` 可能是 A/D/E rebuild 出来的单场。

解决：quality gate 优先读 `screenplayScenes.length`（generation 原始解析结果），fallback 到 `parseScreenplayScenes(screenplay).length`。

教训：一个字段有多份数据源时，必须确认每一层在读哪一份，以及那份数据在当前场景下是否还是真的。

## 13. P3-7 学到的（2026-03-29）

### 三分类才是这轮最值钱的结果

这轮最核心的收获不是"检测器修好了"，而是把三类问题正式分开了：

1. **检测器误杀**：模型写出了真实后果，检测器词根没覆盖 → 修检测器
2. **产品口径边界**：某类结尾确实模棱两可 → 产品决策，不扩词根
3. **生成侧真弱**：结尾停在观察/逼近/环境描写，没有落地结果 → 改生成侧的 scene data（hookEnd / tension）

一旦分开，就不会把所有问题都混成"再调 prompt 试试"。

### 弱钩子正式定义已写入项目原则（原则 38）

弱钩子只认"后果已落地"，不认"线索显形"与"环境显形"：

- 认：锁扣落下、渗血、崩裂、撕裂、被撞开、弯折
- 不认：看见某物、逼近中、露出一点、泛出冷光、环境光影变化

教训：一条判断标准如果不写进正式文档，下轮一定会有人回来重新讨论。已写入 `docs/项目原则.md` 原则 38。

### problems=['集尾钩子偏弱'] 和 overall pass=true 可以共存

这是正常的非矛盾：

- hook 检查看 hook window **任一行**是否通过（`some(...)`），不是只看最后一句
- overall pass 看 blocking 问题（penalty=40），non-blocking 问题（penalty=8）只扣分不拦截
- 集尾钩子偏弱是 non-blocking 问题，记录在案但不阻止 episode 通过

教训：看到"既 pass 又 fail"的报告，先不要急着说它矛盾，先看这两层是不是在检查不同的维度。

### E2E regression 必须用当前代码实时重跑，不能信任旧 results.json

这轮发现 results.json 里的数据和当前代码重跑的结果完全不同（旧批次的 hard-hook.ts 版本和现在不同）。

教训：旧 E2E 结果只能当线索，不能当现役结论。指挥下一轮任务前，必须用当前代码、当前命令、当前数据重新跑一次。

### 产品口径决策必须由人来拍板，不能由检测器词根决定

Ep1 纸角露出是产品口径问题还是检测器 bug？这轮选择了"产品口径问题，不扩词根"。

理由：一旦把"角落显形"放进来，"钥匙露出一角/账册边露出/封皮角显出来"会整串要求进来，最后检测器会从"结果句裁判"变成"悬念句裁判"。

教训：检测器和产品边界是两回事；词根能覆盖多少样本是技术问题，要不要覆盖是产品决策。

## 14. P3-8 学到的（2026-03-30）

### 检测器一旦明显止血，就别继续拿它当主战场

这轮我又差点犯老毛病：

- 明明弱钩子检测器已经把误杀从 `12` 集降到 `8` 集；
- 还想继续围着“怎么再多捞几集”打转。

但用户这次明确给了新的主线：

- 结尾不用为了过检测器继续越写越狠；
- 先看真实写作质量；
- 如果剧情已经够用，就别在检测器上浪费时间；
- 如果剧情确实不行，再回头改内容层。

教训：

1. 检测器的职责是减少系统误伤，不是反过来指挥创作口味。
2. 一旦误杀已经明显下降，后面就要把注意力切回“剧情到底顺不顺、好不好看”。
3. 继续死抠检测器，很容易把非主问题重新抬成主问题。

### 检测器校准要补真实误杀句式，不补泛词

这轮真正有效的，不是“把更多词塞进 marker 列表”，而是：

- 对着真实误杀句式补结构：`全城搜`、`扫飞`、`点燃+焦黑`、`踏进巷口`、`封印要破`、`捂住口鼻/扼住咽喉`
- 同时把最危险的泛词口收回去：`消失` 不能裸放强结果

我这次如果顺手把 generic `消失` 放成 pass，表面像“多救回一批”，本质上是在把检测器重新做宽。

教训：

1. 检测器修正优先补“真实误杀句式”。
2. 泛词一定要警惕，它最容易把假钩子一起放进来。
3. 只要 `账册……消失在窗外` 这种边界句还被稳稳拦住，才说明这轮收口没放崩。

## 15. P3-9 学到的（2026-03-30）

### “质量不过”不能再被偷换成“整轮失败”

这轮我在软件层又踩到一个很典型的坑：

- 用户真正要的是：**内容已经写出来，就让他先看、先改、先继续走**；
- 但系统旧行为更像：只要 quality 不过，就把整轮结果打成失败。

这会把两件事混在一起：

1. **执行失败**：runtime / parse / guardian / persist 真的炸了
2. **质量未过**：内容已经出来，但还没正式通过验收

这两件事如果继续混写，用户体感就会变成：

- 明明下面已经有稿
- 系统却在上面告诉他“失败了”
- 结果大家又会反复重跑、反复误判“软件不稳定”

教训：

1. 只要剧本内容已经生成出来，系统就应该优先保 `visibleResult=visible`。
2. 正式 gate 没过，可以继续让 `formalRelease=blocked(QUALITY_NOT_PASSED)`，但不能把结果整块打成失败不可见。
3. `scriptFailureResolution` 只能留给真失败；一旦它开始承载“有结果但质量未过”，前台和持久化都会一起被带歪。
4. 前台提示也不能只做“成功/失败”二分，至少要补第三种：**内容已生成，但质量还没正式过线**。

## 16. P3-10 学到的（2026-03-30）

### 字数偏差惩罚不能太轻，否则模型不在乎

这轮我发现一个很直观的问题：

- 合同写了"全集不超过 1200 字"
- 但 23 集写了 1993 字（超了 66%）
- 原因是超限惩罚只有 12 分，太轻了

教训：

1. 超限惩罚应该和不足惩罚对齐（都是 24 分），让模型同等重视两端。
2. prompt 里用"尽量写到"这种词，模型很容易无视；改成"必须写到""超了就是不合格"，语气要硬。
3. 如果惩罚力度不够，模型就会按自己的舒服写，不会管你合同怎么定。

### VO 漏出不能只靠 prompt，要让 repair 能自动修

这轮还有一个事实：

- prompt 里已经有 `SCREENPLAY_NO_VO_RULE`
- 但模型还是写了 6 处 `（画外音）`
- 原因是规则太软，而且 repair 没有修 VO 的指令

教训：

1. prompt 规则如果已经被多次证明无效，就不能继续只靠 prompt。
2. 必须在 repair 指令里加明确的修复动作，让修复轮能自动把画外音改掉。
3. 检测端和修复端要配合：检测出来 → 修复指令 → 模型改掉，形成闭环。

## 17. P3-11 学到的（2026-03-31）

### 字数稳定的真正主问题不是“整集太长”，而是“单场太胖 / 骨架太薄”

这轮我一开始差点又走偏，只盯着整集 1200 字上限和罚分。

但把 50 集样本拆开后发现：

- 胖集（18/21/40）大多还是 2-3 场，不是场数超了；
- 真问题是单场里塞了太多回合、太多动作、太多对白；
- 瘦集（32/33/34/41）也不是简单写少了，而是上游 `sceneByScene` 本来就给得薄。

教训：

1. 胖集优先从“单场回合数失控”下手，不要误判成“场数太多”。
2. 瘦集优先回查 `sceneByScene` 骨架，不要先怪模型偷懒。
3. 只盯整集字数上限，会把真正的问题看扁。

### 字数稳定必须进入“场级预算”，不能只靠整集罚分

这轮我已经确认：

- 单靠整集 1200 字上限，再加一点罚分，不足以稳定长季输出；
- 必须把预算拆到每场，不然模型还是会在 2-3 场集里把一场写成 800-900 字的大胖场。

教训：

1. 2 场集、3 场集、4 场集都要有自己的每场预算。
2. prompt 不只要写“总字数别超”，还要明确“每场大概能写到多少、超过多少就算写胖”。
3. repair 也要按场级预算去削胖补瘦，而不是只按整集总量去修。

### 第一轮落地后，真正有用的是“预算 + 分流”，不是再加大惩罚

这轮正式把场级预算和胖瘦分流落地以后，我确认了一件事：

- 单纯提高超限罚分，只会让检测更凶；
- 真正让系统开始稳下来的，是把“每场该写多少”和“胖集/瘦集怎么修”写成可执行规则。

教训：

1. 惩罚只能当裁判，预算和分流才是控制器。
2. repair 一旦知道“这是胖集还是瘦集”，就不再是瞎修，而是有方向地收口。
3. 真正有效的修法不是继续抽象讲“控长”，而是把数字、阈值、动作写死。


### 3.28 char-count-agent prompt 里的"约"字会把合同变成建议（2026-04-02）

这次第4集胖稿的根因不是"Agent 没接入"，而是 prompt 文案本身太软：

- 旧写法：约保留原稿 48% 的体量 → 模型听起来像"参考比例"，不是硬指令
- 修复后写法：字数必须从当前约 4374 字压到 900-1200 字，这是合同硬红线，超 1200 字就是不合格
- 同时加：不准新增场景、不准添加新人物、不准改变情节点

教训：

1. 合同红线只有写进 prompt 时才是真的合同；写成"约"或"尽量"就变成软建议
2. "约保留原稿 X%" 这类比例词不如直接给绝对字数目标有效
3. 胖稿 prompt 还要加"不准新增场景/人物/情节点"，不然模型会把"改稿"听成"继续写新内容"
4. 改完 prompt 后，单测断言也要同步更新；旧断言盯旧文案，会把新 prompt 打成 FAIL

### 3.30 我之前把“剧本质量 Agent”想得太像“小说质量 Agent”了（2026-04-03）

这轮重看 continuity / character-arc / theme-anchor 三个 Agent，我补到一个很关键的认知偏差：

- 我之前虽然知道“不要靠 gate，要靠 Agent 修”
- 但真正写检测器和 prompt 时，脑子里还是偏小说评审：
  - 看有没有 `决定 / 终于 / 依然`
  - 看有没有 `不争 / 放下`
  - 看有没有某些项目私有循环词

这会有两个问题：

1. **它不够像剧本**
   - 剧本不是比谁更会说“我变了”
   - 剧本更看重：这一集有没有推进、每场有没有结果、人物有没有动作和关系变化
2. **它不够像通用工程能力**
   - 一旦把“谦卦”或某几个角色名写死，工程上就只是当前项目特例，不是真正可复用的 Agent 框架

所以这轮我必须记住，剧本质量要优先看 4 件事：

1. **局面有没有变化**
   - 不只是吵了一轮
   - 而是有人被带走、证据被抢、关系翻面、结果真的落地
2. **每场有没有引擎**
   - 有没有阻碍
   - 有没有选择
   - 有没有结果
3. **角色有没有戏剧功能**
   - 主角要做选择
   - 对手要施压
   - 配角要提供杠杆
4. **主题有没有通过选择-代价-后果落地**
   - 不是喊主题
   - 不是解释主题
   - 是角色为了这个主题真的付出一点什么，局面也因此变化

这轮真正有效的改法，不是再加几个关键词，而是：

- `screenplay-content-quality.ts` 新增
  - `dramaticTurnScore`
  - `sceneEngineScore`
  - `characterFunctionScore`
- `continuity-agent` 改成修“推进和结果”
- `character-arc-agent` 改成修“人物戏剧功能”
- `theme-anchor-agent` 改成修“主题动作”
- orchestrator 改成每次修完重测，不再按第一次检测结果把三个 Agent 机械串完

以后只要我再写“剧本质量 Agent”，先问自己一句：

- 我是在修戏，还是在修文？

如果答案更像后者，就说明我又偏回小说思路了。

### 3.32 主题 Agent 最容易犯的错，不是“没触发”，而是“触发太勤”（2026-04-03）

### 3.28 这轮我才真正理解"七问不是独立 stage，是粗纲 stage 的确认子步骤"（2026-04-04）

- 七问不是独立 stage，不是先跑完七问再进粗纲
- 七问是粗纲 stage 里的一步确认动作
- 正确流程：storyIntent 已确认 -> 生成七问初稿 -> 前端展示 -> 用户修改/确认 -> 写回 outlineBlocks -> 再生成粗纲

这让我避免了一个典型的工程陷阱：把"七问"做成一整套独立 stage，再把粗纲做成另一个 stage，结果把"storyIntent -> 七问 -> 粗纲"变成三条平行链路。

真正该做的是：

1. 七问生成后，先前端展示让用户确认
2. 确认后写回 `outlineDraft.outlineBlocks[].sevenQuestions`（唯一真相源）
3. 粗纲生成时直接吃 `outlineBlocks` 里的七问，不再重复调用七问 Agent

这轮另一个关键认识是：**七问真相源必须唯一**。之前有两个地方可以放七问（顶层 `project.sevenQuestions` 和 `outlineBlocks[].sevenQuestions`），这会导致双口径。现在已经收成：只以 `outlineBlocks` 为权威，顶层字段直接移除。

### 3.104 长时间 AI 任务 timeout 需要同步调整 server 端（2026-04-19）

### 3.106 HTTP 生成接口扣费必须在路由层，不在服务层（2026-04-19）

### 3.100 UI 阅兵先打掉的是“前台假象”，不是自动等于后端真相已经落好（2026-04-18）

这轮我一开始看到人物页只有 `8` 张完整人物卡，第一反应是前端还在偷读旧 IPC。这个判断只对了一半。

- 只查后端，不查前端，也会漏掉 renderer 继续偷读旧 IPC 的残根。

1. 页面当前读的是谁。
2. HTTP 快照里到底有没有那份数据。
3. 观察脚本的验收口径是不是还停留在旧 UI 结构。

### 3.101 人物门禁失败不能只记一个总错误码，要尽量把缺哪几个人、缺什么字段写清（2026-04-18）

这轮真实联调里，`character_contract_incomplete` 再次挡住了保存。

如果只留下一个总错误码，排查就会很慢，因为根因可能有三种：

1. 主角或对手没进人物名单。
2. 某个角色的旧版合同字段没补齐。
3. V2 基础五维或核心字段丢了。

所以我这轮补了新的诊断方向：

- 失败时尽量把 `protagonistCovered / antagonistCovered` 打出来。
- 还要把 `incomplete=[...]` 和本轮 `names=[...]` 记进 runtime log。

这类日志不是 gate，但它能把“门禁挡住了”从一句空话，变成下一轮能直接修的证据。

### 3.102 前端弹窗只要还挂在局部组件树里，`fixed + z-index` 也不一定真能顶到最上层（2026-04-18）

这轮登录弹窗错位又提醒了我一个很具体的前端坑：

- `LoginModal` 原来挂在 `AppHeader` 内部。
- `AppHeader` 自己带了 `backdrop-blur-sm` 和 `relative z-10`。
- 结果就算弹窗写了 `fixed inset-0 z-[99999]`，也还是会被祖先 stacking context 污染。

用户看到的现象很典型：

1. 弹窗像掉到页面后面。
2. 只能透着半透明看到一点。
3. 整个框还往上飘，明显不是正常居中。

这次正确动作不是继续硬抬 `z-index`，而是：

1. 把弹窗改成 `createPortal(..., document.body)`。
2. 让它从局部组件树里脱离出来，直接挂页面顶层。
3. 顺手补 `body overflow hidden`，避免打开时页面滚动继续干扰位置。

以后只要遇到“弹窗明明是 fixed，却 still 被挡 / 错位”的问题，我要先查：

1. 它是不是挂在带 `blur / transform / relative z-*` 的祖先树里。
2. 它是不是应该直接 portal 到 `document.body`。
3. 它是不是还缺滚动锁，导致视觉上像是定位错了。
