# Engineering Workflow（工程排障与根因修复）

本文件只管工程问题：调试、修 bug、找根因、评估影响、重构、agent team 并行排查。
不写长期哲学，不写榜单 SOP。

## 目标
- 不接受表面修复，只接受根因修复。
- 不再靠第七层、第八层、第九层补丁链维持表面稳定。
- 动手前先看清真正链路、真正源头、真正影响面。

## 根因第一性原则（开工前先看这个）
- 解开补丁，找到本质。
- 拿出证据说服用户。
- 能用大白话讲清楚的本质，才是真的本质。
- 先证明根因，用户认可后再开工。
- AI 自己守住本质，不准回到表层。
- 一直往前推，推到推不动再开工。

## 根因修复总原则
- 只要当前定位到的层不是根因层，就不能收工。
- 根因不是最近报错层、最好修的那层、看起来最明显的那层。
- 根因是第一次把错误写进系统真相、正式名单、正式契约、正式状态或持久化结果的那一层。
- 确认根因后，直接围绕根因出计划，不再让用户为已确认根因做无意义补测。

## 原则优先规则（最高优先级）
- 先看用户标准，再看代码现象；不能先按自己的工程习惯分析，再把用户标准当补充说明。
- 如果用户标准已经讲清，而结果还是偏掉，先怀疑分析方法没按标准来，再怀疑代码细节。
- 根因分析默认用原则倒推法，不用现象回溯法。

## AI 自检规则（必须执行）
- 每次根因分析前，先自问：我现在是不是又滑回表层了？
- 不能用大白话讲清楚，就继续找。
- 说服不了用户，就不能开工。

## 根因追问模板（强制执行）
- 每找到一层原因，都继续问：为什么会这样？为什么系统会允许这样？为什么一开始没防住？
- 只要还能继续问，就继续往前推。

## 调试顺序（固定）
1. 明确症状：到底错的是什么，不要只描述现象。
2. 缩小流程：先找到它属于哪条执行链。
3. 找关键符号：锁定真正参与该链路的函数、服务、契约边界。
4. 看影响面：判断哪些调用方、流程、位置会被波及。
5. 向上追源：继续追到第一次把错误写成正式事实的那一层。
6. 不是根因层：停止表面修补，直接改计划。
7. 已到根因层：出修复方案，必要时重构。
8. 改完后验证：验证根因是否被修掉，不只看表层症状有没有暂时消失。

## 什么时候必须重构
- 错误已经跨多流程扩散。
- 脏数据已经进入正式输入、正式输出、持久化结果或系统真相。
- 当前方案仍然依赖兜底、回退、后处理这类补丁层存活。
- 一处改动牵动多条主链路，而旧边界本身已经不成立。
- 同一类问题反复出现，说明上次修复没打到根因层。
- 单文件已经超过 800 行，或者明显还在继续膨胀。
- 已经出现“先塞进去，之后再拆”的习惯性堆积。

## 超大文件判断（直接当根因信号）
- 超过 800 行就停下来找根因，不准继续往里塞。

## 自动执行与重试规则
- 工程任务默认进入自动循环：分析 -> 修改 -> 验证 -> 发现问题 -> 修复 -> 再验证。
- 不要执行一次就停下，除非已经达到验收标准，或者遇到必须用户授权的卡点。
- 命令、构建、API、浏览器验证、部署失败时，先自己分析并自动重试。
- 默认最多自动重试 3 次；超过 3 次仍失败，再向用户汇报根因和可执行方案。

## 模拟真人用户验证
- 让用户验证前，AI 必须先自己走一遍真实操作路径。
- 不能只看底层数据正常，就说功能正常。
- 必须看页面有没有黑屏、空白、错位、按钮无效、链路断裂、关键反馈缺失。
- 涉及部署、配置或线上修复时，还要打开线上地址、验证关键接口、检查 console 错误。

## agent team 规则
- 能拆就并行，用完代理就关。
- 并行排查时，每个子代理都要有清晰职责边界。
- 子代理汇报时必须说明自己找到的是表层，还是根因层。
- 调试期如果临时拉起 MCP、Playwright、DevTools、Git MCP、浏览器或本地 dev server，验证完成后必须主动收尾。
- 用完工具后，要先关闭对应 MCP 会话，再做后台清理。
- 不重复启动同类 MCP 实例；发现已经有同类实例在跑，先查归属再决定复用还是回收。
- 收尾不是只关页面，还要检查相关后台进程有没有残留；特别是 `chrome-devtools-mcp`、`@upstash/context7-mcp`、`@playwright/mcp`、`mcp-server-git`、`chrome-headless-shell`、watchdog。
- 清理按进程树做，不只杀单个 PID；父进程、子进程、watchdog 要一起看。
- 如果宿主仍在保活，不算清理完成。
- 如果发现明显孤儿进程，优先清理当前任务拉起的那批，再继续下一轮开发，不能把机器越跑越卡。

## 与用户的沟通要求
- 要明确告诉用户：现在是在找源头问题，还是已经开始修源头问题。
- 不要把表面层发现包装成最终答案。
- 拿出证据说服用户。
- 能用大白话讲清楚的本质，才是真的本质。
- 汇报时优先用大白话，不要堆术语。

## 完成前自检
- 我改的是根因层，还是表面层？
- 第一次写脏数据的地方找到了吗？
- 影响面分析做了吗？
- 验证的是根因修复，还是表层现象暂时消失？
- 页面、接口、console 都看过了吗？
- 这轮拉起的代理、MCP、浏览器、watchdog、临时服务都收干净了吗？
