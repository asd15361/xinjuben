# 逐场生成（Per-Scene Generation）设计方案

> 状态：设计稿，待评审
> 目标：替换"整集单次生成"为"逐场独立生成再组装"，从根本上解决字数预算遵从率低的问题

---

## 1. 现有工序 vs 目标工序

### 现有工序（Episode-Level）

```
episode plan (sceneByScene[])
  -> createScriptGenerationPrompt()    [prompt/create-script-generation-prompt.ts]
     一次 prompt，背 30+ 条 craftRules，含模糊的字数合同
  -> generateTextWithRuntimeRouter()   [ai/generate-text.ts]
     一次 AI 调用，生成整集
  -> parseGeneratedScene()            [runtime/parse-generated-scene.ts]
     路由：多 scene heading → screenplay path
  -> ScriptSegmentDto
```

**问题**：字数合同（1200 字 / 300-400 字每场）混在 30+ 条 craftRules 里，模型把它当建议性文字而非硬约束。

---

### 目标工序（Scene-Level）

```
episode plan (sceneByScene[])
  -> 拆解为 N 个单场 scaffold
       [scene-by-scene scaffold]
         每场包含：setup / tension / hookEnd / 本场 budget / 上一场结尾钩子
  -> 逐场调用 generateTextWithRuntimeRouter()
       每次只生成「X-Y 日」scene 片段，不含「第X集」
       字数预算从全剧 1200 精确到每场 300-400
  -> 组装（assembleScenes())
       拼接：「第X集」+ 各场「X-Y」scene 片段
  -> parseGeneratedScene()
       现有 parser，覆盖 screenplay format
  -> ScriptSegmentDto（与现在完全一致）
```

---

## 2. 单场生成合同（Single-Scene Contract）

### 场景 scaffold 输入字段

```typescript
interface SceneScaffoldInput {
  episodeNo: number          // 集号
  sceneNo: number           // 场号（1-N）
  sceneCode: string        // "1-1", "1-2" 等
  timeOfDay: string         // 日/夜 等
  characters: string[]      // 本场人物名单
  setup: string            // 本场 setup（来自 sceneByScene）
  tension: string           // 本场 tension（来自 sceneByScene）
  hookEnd: string          // 本场 hookEnd（来自 sceneByScene）
  budgetChars: number      // 本场字数上限（如 300-400）
  prevSceneHook?: string    // 上一场结尾（来自上场 raw output 或 scaffold）
  prevSceneOutcome?: string // 上一场结局（供本场开头承接）
}
```

### 场景 prompt 骨架

```
你正在为短剧《{outlineTitle}》第 {episodeNo} 集第 {sceneNo} 场写剧本片段。

【本场人物】{characters}
【本场 Setup】{setup}
【本场 Tension】{tension}
【本场 Hook 结尾】{hookEnd}

【字数合同】（硬约束）
本场不超过 {budgetChars} 字。写完本场推进点就停，不在本场补充背景、不写回想、不加解释句。
冲突未完则留给下一场，不在本场硬塞。

【格式合同】（必须）
本场输出格式：
{X}-{Y} {timeOfDay}
人物：{characters[join]}
△动作行
角色：对白行
△情绪行（可选）

注意：不要输出「第X集」标题，只输出本场片段。

【承接要求】
{prevSceneHook ? `上一场结尾是：${prevSceneHook}` : '本场为第 1 场，开头直接进入动作。'}
```

### 场景输出格式（正式合同）

```
1-1 日
人物：林守钥，沈黑虎
△旧库院门被一脚踹开，门栓断裂。
林守钥：（后退半步）你交出来。
沈黑虎：（冷笑）还没到时候。
△林守钥手指收紧，指节发白。
```

## 3. 组装逻辑（Assembly）

### 签名

```typescript
function assembleEpisodeScenes(
  episodeNo: number,
  scenes: string[]       // 每场 raw 输出（不含「第X集」）
): string                // 完整 screenplay（含「第X集」标题）
```

### 组装前后示意

**输入（逐场输出，3 场）**：
```
[scene1]: "1-1 日\n人物：林守钥，沈黑虎\n△...\n林守钥：...\n沈黑虎：..."
[scene2]: "1-2 夜\n人物：林守钥，沈黑虎\n△...\n林守钥：...\n沈黑虎：..."
[scene3]: "1-3 晨\n人物：林守钥\n△...\n林守钥：..."
```

**组装后**：
```
第1集

1-1 日
人物：林守钥，沈黑虎
△旧库院门被一脚踹开，门栓断裂。
林守钥：（后退半步）你交出来。
沈黑虎：（冷笑）还没到时候。

1-2 夜
人物：林守钥，沈黑虎
△...

1-3 晨
人物：林守钥
△...
```

**关键设计决策**：
- 场景 raw 输出**不含**「第X集」标题，由 assemble 函数统一添加
- 各场之间插入空行分隔（`\n\n`）
- scene heading（如「1-2 夜」）保留为每场输出的第一行
- 组装结果直接进入现有 `parseGeneratedScene()`，无需修改 parser

### 沿用 vs 不动

| 组件 | 状态 | 说明 |
|------|------|------|
| `parseGeneratedScene()` | **沿用** | screenplay path 已能正确处理多 scene heading |
| `extractStructuredSceneFromScreenplay()` | **沿用** | 输入格式不变 |
| `screenplayScenes[]` | **沿用** | parser 输出结构不变 |
| `action/dialogue/emotion` | **沿用** | parser 的 A/D/E 提取逻辑不变 |
| `inspectScreenplayQualityBatch()` | **沿用** | 输入接口不变 |
| `finalizeScriptPostflight()` | **沿用** | 后处理逻辑不变 |

## 4. 风险对账

### 风险 1：逐场生成打碎 continuity

**风险内容**：每场独立生成，后生成的场次可能与先生成的场次在人物状态、道具位置上产生矛盾。

**防守方案**：
- `sceneScaffoldInput` 必须包含 `prevSceneOutcome`（上场结局），由 assemble 函数从上场 raw 输出提取后注入下场的 prompt
- 第一场使用 episode plan 的 sceneByScene[0].setup
- 最后一场的 `hookEnd` 来自 episode plan 的 sceneByScene 最后一项
- 不依赖"模型记住前文"——每场 prompt 必须包含明确的上一场结局状态

### 风险 2：钩子变弱

**风险内容**：每场各自生成，最后一场的钩子可能脱离整集最末 hookEnd 的设计意图。

**防守方案**：
- 最后一场（第 N 场）的 prompt scaffold 必须明确传入 `hookEnd`（来自 episode plan 的 sceneByScene 最后一项）
- `hookEnd` 作为字数合同外的"质量门"，在 prompt 里要求落地为具体可见动作，不接受"情绪收口"
- assembly 函数不修改任何场的结尾内容

### 风险 3：增加第二格式口

**风险内容**：逐场生成和整集生成两套格式合同并行，导致维护成本和规则漂移。

**防守方案**：
- **不引入新的格式合同**——逐场生成的格式要求（scene heading / 人物 / △动作 / 对白）就是现有 craftRules 里格式合同的具体化
- 逐场生成只改变**生成粒度**，不改变**输出格式合同**
- 组装后仍走现有 `parseGeneratedScene()` 路由，没有旁路 parser

### 风险 4：调用次数和失败点暴增

**风险内容**：每个 episode 从 1 次 AI 调用变成 2-4 次，失败概率增加；部分场次失败后的 recovery 逻辑复杂。

**防守方案**：
- 每场独立失败不影响其他场——部分失败可以只重跑失败的场次
- 引入 `sceneStatus[]`（pending/ok/failed）跟踪，与现有的 `episodeStatuses` 并列但粒度更细
- 单场失败 → 重试该场（最多 2 次）→ 若仍败则整个 episode 标记 failed（与现有 episode 级失败处理一致）
- 调用次数增加但**每次 token 消耗减少**（单场 300-400 字 vs 整集 1500+ 字），总 token 量预期下降

## 5. A 方案最小对照版本（保留，不当主方案）

**最小改动**：在 `createScriptGenerationPrompt` 的 craftRules 里，将模糊字数合同拆成更显眼的 per-scene scaffold 注释。

**预期能验证**：
- 更显眼的 scaffold 格式是否能略微改善遵从率
- 不作为主方案，用于判断"prompt 表达方式"对模型遵从率的影响

**为什么不是主方案**：
- 本质还是"一次生成整集"——模型同时背着"每场 300-400 字 + 全集 1200 字"的双重约束，对单场预算的感知仍然弱
- 诊断数据已证明：71% 超长集的主因是对白过密，这是生成粒度问题，不是 prompt 表达问题

## 6. 推荐实施顺序

### P0：Scene Prototype 单场验证（最小切口）

**改动范围**：仅改 `create-script-generation-prompt.ts`，新增一个 `createSceneGenerationPrompt(sceneScaffold)` 函数

**目标**：用现有 fs-a 案例，专门生成 Ep1 的第 1 场，验证：
- 单场 scaffold prompt 格式是否正确
- 300-400 字预算是否被模型遵从
- 输出是否可以直接组装

**不碰**：`runScriptGenerationBatch`、`startScriptGeneration`、`parseGeneratedScene`

### P1：Assembly + Episode 打通

**改动范围**：新增 `assembleEpisodeScenes()` 函数；修改 `runScriptGenerationBatch` 用 scene-level 生成代替 episode-level

**目标**：1 个 episode（Ep1）的完整逐场生成 → 组装 → parser → quality inspection

### P2：Batch 扩展 + 失败处理

**目标**：5 集逐场生成，验证 sceneStatus 追踪、部分失败 recovery

### P3：全量（10 集）
