# 60集剧本稳定生成专项修复计划（第一版）

更新时间：2026-03-19

## 这份计划解决什么本质问题
- 用户真正要的，不是“首场更像戏”，而是“60 集链路能连续写下去，并且不在中途炸掉”。
- 最新真测已经坐实：前段链路能写，第一场也不算差，但 60 集链路第二轮就会断。
- 所以这份计划只解决一件事：把 60 集正文从“偶尔能落一场”升级成“能连续跑完，结构不乱，失败也不原地打转”。

## 为什么上一份 P7 做完，用户体感还是一样
- 上一份计划主要解决的是正文口味、前端状态和局部样本，不等于 60 集全链路稳定。
- 真正被证明的，是“首场和局部样本变好了一些”；没有被证明的，是“60 集多轮批次能稳定跑完”。
- 只要 60 集第二轮还会 `parse_interrupted`、字段串段、失败后回到原地，用户体感就还是“没写出来”。

## 当前真相（先钉死，不再自我安慰）
1. 粗纲 / 人物：连续 3 轮真测通过，平均约 156.2 秒，质量是好。
2. 60 集剧本：真实跑了约 341.4 秒，在第 2 轮失败，只落出 1 场。
3. 第一场虽然有戏味，但字段已经错位：
   - `action` 混入大段对白
   - `dialogue` 混入场景说明
   - `emotion` 继续写剧情推进
4. 第二轮直接原因已经坐实：模型吐出 `**Action** ... **Dialogue** ...` 这类旧格式，解析器没接住，直接 `parse_interrupted`。
5. 当前 recovery 只会把失败历史写回计划，不会把这轮原文先救回来再续跑。
6. 当前 audit 能抓明显标记污染，但抓不住“没带标记、却已经串段”的场景。
7. 当前 runtime profile 更关心“上下文要不要瘦身”，不够关心“长剧格式要不要更稳”。

## 这轮不做什么
- 不把问题重新说回大架构、唯一真相源、前端交互。
- 不再用首场成功冒充 60 集稳定。
- 不新长第二写口、第二裁判口、第二解释口。
- 不把创作问题改成一堆命中词硬拦截。
- 不开新的云端大工程。

## 最终验收标准
- 必须同时成立：
1. `tools/e2e/electron_real_script_60_full_export.mjs` 能真实跑完，成功导出 60 场。
2. 过程中不再出现 `parse_interrupted`、`section_polluted`、字段角色错位。
3. 失败如果发生，必须先在单场合同内自救，不再整批归零。
4. 首场、中段、后段各抽查 1 场，`action / dialogue / emotion` 职责正确。
5. 60 集 full export 结果文件真实落地，不再停在“有一场样本”。
6. 粗纲、人物、详细大纲质量不回退。

## 执行阶段

### 第一阶段：撤销假收口，统一验收口
- 目标：
  - 把“上一份计划只证明了局部质量，不等于 60 集稳定”正式写死。
- 动作：
  - 当前主线切到这份专项计划。
  - 任务卡、worklog、旧计划同步加更正口径。
  - 后续所有“完成”只认 60 集 full export，不再认首场单点样本。
- 完成标准：
  - 仓库里不再同时存在“P7 全部完成”和“60 集根本没跑通”两套说法。

### 第二阶段：收紧单场输出合同
- 目标：
  - 让模型输出和解析口只认一套稳定结构，不再靠运气接。
- 动作：
  - 收紧剧本 Prompt：三段输出格式写死，禁止把段头写成带星号的旧格式。
  - 在解析前加一层统一归一化，把 `**Action** foo **Dialogue** bar` 这类旧格式转成同一套可解析形态。
  - 不再把“猜哪一块像对白、哪一块像情绪”当主路，只把它留作最后兜底。
- 完成标准：
  - 同一条场景就算吐出旧段头，也能先被归一化，再进入正式解析。

### 第三阶段：修掉字段角色错位
- 目标：
  - 不让 `action / dialogue / emotion` 看起来都不空，但职责已经串了。
- 动作：
  - 在 `parseGeneratedScene` 边界增加字段角色检查：
    - `action` 不能主要由人物对白构成
    - `dialogue` 不能主要由场景叙述构成
    - `emotion` 不能继续当剧情推进正文
  - 一旦发现串段，先在单场合同内做一次重整或同场修复，不直接落库。
  - 把“串段但没带标记”的情况补进 audit 口径。
- 完成标准：
  - 第一场这种“内容有了、字段错了”的假成功不再算通过。

### 第四阶段：把 parse 失败改成单场自救，不再整批炸掉
- 目标：
  - parse 问题只影响当前场，不再让整批直接归零。
- 动作：
  - 对 `parse_interrupted` 增加单场救援链：
    1. 先归一化再解析
    2. 还不行就走一次低温同场修复
    3. 只有两步都失败，才正式记失败
  - 失败时保留原文、失败类型和当前场号，续跑从当前场继续，不回到抽象“再试一次”。
- 完成标准：
  - 第二轮再出 parse 问题时，系统优先救当前场，而不是让整批白跑。

### 第五阶段：补长剧专用运行档
- 目标：
  - 让 60 集模式和首场样本模式不再共用一套“看起来差不多”的策略。
- 动作：
  - 给 `targetEpisodes >= 40` 加长剧档位：
    - 更稳的温度
    - 更紧的段头合同
    - 更明确的失败恢复提示
    - 更保守的上下文打包策略
  - recovery history 不只影响“上下文瘦身”，还要直接影响“格式稳定优先级”。
- 完成标准：
  - 长剧模式下，第二轮开始不再频繁回到旧格式。

### 第六阶段：按 10 / 20 / 60 三层真测收口
- 目标：
  - 不再靠一轮样本宣布胜利，必须逐层证明。
- 动作：
  - 第 1 层：10 集三连真测，确认前段没有回退。
  - 第 2 层：20 集多轮批次真测，确认 batch/resume/repair 能跑。
  - 第 3 层：60 集 full export，确认能真正导出 60 场。
  - 每层都同时看：
    - 有无 parse 失败
    - 有无字段错位
    - 有无 section pollution
    - 首段 / 中段 / 后段抽样是否还像戏
- 完成标准：
  - 三层都过，才允许重新说“60 集稳定生成成立”。

- 当前只完成了第一阶段里的“真相冻结”和“专项计划立项”。

## 这轮第一棒先做什么
1. 先统一文档口径，撤销“P7 已经收口”的假完成。
2. 立刻进第二阶段：先收紧单场输出合同和解析归一化。
3. 然后做第三、第四阶段：先把“串段”和“parse 炸整批”这两个最硬问题砍掉。

- 只证明 10 集没回退，就说 60 集也稳了。
- parse 还是会炸，只是报错文案变好看。
- 字段还是串的，只是 audit 没看出来。
- 又为了自救长出新的第二写口、第二裁判口。

## 最后一句
