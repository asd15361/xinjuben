# E2E重建准入标准（2026-03-22）

## 用途

- 这份文档只回答一件事：以后什么样的 E2E 才有资格进入正式验收名单。
- 不满足准入标准的脚本，哪怕本机偶尔跑绿，也只能算调试工具。

## 准入红线

任何新 E2E 只要踩中任一条，直接不得入库为正式验收脚本：

1. 不允许直接改 `projects.json`
2. 不允许通过 `page.evaluate()` 或 preload 旁路调用内部生成 API
3. 不允许默认绑定历史 `tools/e2e/out/userdata-*`
4. 不允许使用旧字段、旧 DTO、旧术语、旧状态字符串做判断
5. 不允许用 `waitForTimeout()` 当主等待手段
6. 不允许 `.catch(() => null/false)`、bare `catch {}` 后继续判业务结论
7. 不允许自带第二质量合同

## 正式脚本必须满足的 8 条

1. 真实 UI 主链
   - 从用户可见入口进入
   - 点击真实按钮
   - 等待真实页面信号

2. 固定 seed 入口
   - seed 必须来自版本化 fixture
   - seed 构造脚本必须单独受控
   - 不能吃上一次测试跑出来的目录

3. 单一正式合同
   - 只认当前正式质量合同
   - 字数、场次、污染文本、钩子、恢复口径都必须和当前正式合同一致

4. 黑盒证据完整
   - 每次结论都要同时记录：
   - 页面状态
   - 项目快照状态
   - 运行耗时
   - 失败原因或完成证据

5. 错误必须显式分类
   - 产品失败
   - 采样失败
   - 环境失败
   - 三者必须分开，不能混判

6. 事件等待优先
   - 优先等按钮可见、按钮可点、文本出现、状态变更、文件落盘
   - 禁止靠“多等 5 秒”掩盖竞态

7. 选择器唯一
   - 选择器必须能明确指向当前页面唯一目标
   - 禁止模糊正则同时匹配多个按钮

8. 输出可复验
   - 同一脚本在新机器、新目录、新会话下可以独立运行
   - 不依赖本机历史残留

## 正式验收最小集合

当前建议只重建 4 类：

1. `electron_launch_smoke`
   - 目标：壳能否启动
   - 不负责产品功能验收

2. `ui_mainline_minimal`
   - 目标：真实 UI 主链最小可用闭环
   - 只测一条最短正式路径

3. `resume_mainline`
   - 目标：真实恢复链能否从正确集数继续
   - 必须验证前端恢复动作真的发生

4. `script_60_export`
   - 目标：60 集正式长测入口
   - 必须建立在前 3 类脚本已可信的前提上

## 新脚本过审流程

1. 先过红线检查
2. 再过静态审查
3. 再过固定 seed 首轮运行
4. 再过重复运行稳定性检查
5. 最后才允许进入正式验收名单

## 当前决策

- 在新脚本重建完成前，仓库不存在“正式 E2E 产品验收脚本”。
- 当前只保留基础工具，不保留旧验收链。
