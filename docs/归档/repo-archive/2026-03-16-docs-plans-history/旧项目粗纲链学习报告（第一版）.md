# 旧项目粗纲链学习报告（第一版）

## 一、这份报告回答什么问题

这份报告只回答一件事：

**旧项目的粗纲链为什么有效，重建时到底该保留什么。**

## 二、先说结论

旧项目的粗纲链不是“点一下生成一个大纲”。

它实际是一条完整的阶段链：

- 需求收集
- 粗纲任务会话
- 粗纲生成
- 粗纲校验
- 命名锁定
- 连续性检查
- 失败阻断 / 重试 / 修复
- 放行到人物小传和详细大纲

这条链值钱的地方，不是某一段 prompt，而是：

- 它把粗纲当作正式中间产物
- 它不允许“假成功”继续污染下游
- 它让粗纲成为后续工序的正式上游合同

## 四、粗纲链的真实作用

粗纲链不是“生成第一稿剧情梗概”，它至少承担 5 个职责。

### 1. 把用户需求压缩成可生成的故事骨架

它把：

- 项目配置
- 对话答案
- 自由输入
- 补充要求

压成粗纲阶段的正式输入。

### 2. 形成正式中间产物

旧项目的粗纲不是一段自然语言，而是要支撑这些结果：

- `episodes[]`
- `mainCharacters[]`
- `keyEpisodes[]`
- `endingDesign{}`

也就是说，粗纲必须是“能继续往下喂”的结构化真相。

### 3. 锁定关键命名和主线骨架

一旦粗纲通过，它不仅是内容草稿，还是：

- 人物命名来源之一
- 后续角色图谱的种子
- 详细大纲和剧本生成的故事主线锚点

### 4. 阻断假成功

旧项目真正值钱的是这点：

它不接受“看起来像大纲，其实是空心模板”的结果继续往下走。

### 5. 给后续工序提供可修复而不是可污染的上游

如果粗纲失败，它应该：

- 明确失败
- 可重试
- 可修复
- 不能静默污染下游

## 五、粗纲链的成功标准

旧项目里真正应该保留的，不是“生成出 10 集”，而是这些成功条件：

1. 用户核心需求在粗纲里可定位
2. 主线冲突是明确的，不是空泛主题句
3. 人物主轴已经浮现，不只是名单
4. 关键情节点可用于后续工序
5. 结局方向被锚定
6. 输出能作为正式中间产物进入下游

## 六、粗纲链最容易出错的地方

### 1. 看起来完整，其实空心

典型表现：

- 集数对了
- 结构像大纲
- 但内容抽象、模板化、不可继续使用

### 2. 用户需求没有真正落进去

如果用户要求：

- 特定题材
- 特定主线冲突
- 特定人物关系

但这些没有在粗纲结构里显式落位，后面就会越跑越偏。

### 3. 粗纲成功标准过松

只要“有字”就放行，会导致人物小传、详纲、剧本都吃到脏上游。

### 4. 页面多头判断粗纲状态

旧项目后期很痛苦的一点，就是：

- 页面文案
- 布尔状态
- 任务提示
- 恢复逻辑

都可能各说各话。

## 七、重建时必须保留的东西

### 必须保留 1：粗纲阶段合同

粗纲要成为正式工序合同，不是临时文本。

### 必须保留 2：失败阻断机制

粗纲不通过，不能继续污染人物小传和详细大纲。

### 必须保留 3：命名和主线锚点

粗纲必须承担角色名、主线骨架、结局方向的上游真相职责。

### 必须保留 4：可修复性

失败时要能：

- 重试
- 修复
- 继续

而不是“一失败就死”或者“失败也当成功”。

### 必须保留 5：人工可干预

用户必须能改粗纲，而不是只能接受一次生成结果。

## 八、重建时不要照搬的东西

### 不要照搬 1：页面总控式编排

粗纲链不能再挂在一个大页面里靠几十个状态拼出来。

### 不要照搬 2：renderer 中心化业务编排

粗纲阶段合同、失败判断、放行判断应该放在应用层或共享规则层。

### 不要照搬 3：补丁式兜底堆积

应该先把 contract 立住，再谈 repair，而不是靠 repair 假装 contract 存在。

## 九、迁入新项目时的建议落点

### 新项目应该这样落

- `shared/contracts/outline.ts`
- `shared/domain/formal-fact`
- `shared/domain/stage-boundary`
- `application/outline`
- `features/outline`

### 最小重建顺序

1. 先定义 `OutlineDraftDto`
2. 再定义 `OutlineStageContract`
3. 再定义粗纲成功 / 失败条件
4. 再接 UI 编辑与保存
5. 最后接旧项目生成逻辑

## 十、最终结论

旧项目粗纲链真正厉害的地方，不是“能写一版粗纲”，而是：

- 它把粗纲变成了正式上游合同
- 它把失败和放行做成了质量门
- 它给下游提供了稳定故事骨架

新项目重建时，必须先保留这套能力，再决定具体 prompt 和模型怎么迁。
