# 唯一真相驱动阶段路线图（第一版）

## 一、这份文档解决什么问题

这份文档只回答一件事：

**在“唯一真相驱动”前提下，新项目的大阶段应该按什么顺序推进。**

它不负责定义对象本身，  
对象定义和权责请看：

- `唯一真相对象与权责表（第一版）.md`

---

## 二、重排后的七段主线

### 第一阶段：锁死真相对象

目标：

- 把五类唯一真相对象全部定义清楚
- 把正式来源、写权限、确认权、读取边界锁死

验收标准：

- contract 能表达这些对象
- rules 文档与代码一致
- 不再存在“概念有了，权责没锁死”的情况

当前状态：

- 已基本完成第一版

### 第二阶段：锁死工序流动边界

目标：

- 四工序之间只能按正式边界流动
- 下游只能引用上游已确认内容
- input contract 和 stage contract 变成服务级约束

验收标准：

- 服务 gate 存在
- 合同不是 UI 提示，而是系统约束
- 页面无法绕开主进程偷改主线真相

当前状态：

- 已完成大半
- 还需要把 confirmed facts 的下游引用展示做显式化

### 第三阶段：锁死运行态主链

目标：

- 剧本生成的计划、进度、续跑、失败恢复都纳入单一状态机

验收标准：

- runtime 真相只存在一套
- 所有生成启动都先过服务 gate
- renderer 不再长期承担运行推导

当前状态：

- 已完成第一版

### 第四阶段：锁死 ledger 主链地位

目标：

- 把 ledger 变成连续性核心，而不是附属结构

验收标准：

- 生成、审计、修补都能读取 ledger
- ledger 至少覆盖正式事实、人物、关系、知识边界、钩子、动量

当前状态：

- 已完成第一版
- 还需要继续向旧项目的深层连续性能力靠拢

### 第五阶段：锁死审计修补闭环

目标：

- 让“执行 -> 审计 -> 修补 -> 回写”成为主链

验收标准：

- audit 不是只打分
- repair 不是只给建议
- 修补结果会正式回写真相中心和 ledger

当前状态：

- 已有第一版闭环

### 第六阶段：迁移旧项目深层业务策略

目标：

- 学旧项目真正值钱的地方
- 只迁业务逻辑和质量机制，不迁旧组织方式

注意：

- 旧项目能力不是到这一段才第一次开始吸收
- 前五段主线里就允许按边界“挂载式迁移”
- 这一段的重点是把前面零散吸收过的深层策略系统化收口
- 不能把这一段理解成“前面先自己做，最后一次性整体搬旧项目”

主要迁移对象：

- 正式事实相关 prompt 策略
- 戏剧推进链策略
- 角色施压和弧光策略
- 剧本运行时质量信号
- 更完整的连续性与修补经验

验收标准：

- 新项目能力变强
- 仓库结构不变形

当前状态：

- 已进入阶段性收口（第 4-6 段阶段收束记录已沉淀）

### 第七阶段：最后才做表现层增强

目标：

- 在不破坏真相对象和主链边界的前提下优化体验

包括：

- 可视化
- 非侵入式建议展示
- 更强的工序引导
- 更好的编辑器体验

验收标准：

- UI 可以换
- 真相和主链不能塌

当前状态：

- 表现层第一版已完成（不破坏真相与边界前提下的可读性与工作台能力）

## 三、这次对现有大主线的四个关键调整

### 调整 1：把“正式事实”升级成总主线锚点

以前它只是重要模块之一。  
现在它是四工序流动、ledger、audit、repair 的共同锚点。

### 调整 2：把“真相对象定义”放到一切功能前面

以后没锁对象和权责，就不继续往下接功能。

### 调整 3：把“旧项目迁移”后置到主链稳定之后

以后只能先锁主链，再迁旧项目业务逻辑。

### 调整 4：把“表现层优化”压到最后

页面可以增强，但不能为了页面方便把规则塞回 renderer。

## 四、接下来最近两轮的实际执行入口

### 最近第一轮

目标：

- 建立 Chat-first 入口：像 GPT/微信一样引导用户聊天收集输入，并把聊天收束成唯一真相对象（StoryIntent）

要做的事：

- 在工作台引入一个聊天框式入口，系统用问题引导用户聊完 1 轮（不让用户理解一堆表单字段）。
- 把聊天内容结构化为 `storyIntent` 并写入项目真相中心。
- 基于 `storyIntent` 一键生成粗纲与人物小传草稿，并写入真相中心（粗纲里的正式事实必须为 draft 候选，待用户确认）。

为什么它现在最优先：

- 现在系统边界和主链骨架已具备，但入口体验不够“真人可用”；用户看不懂，就无法验收黄金链路。
- 入口不统一，会导致“真相只能靠 UI 状态/临时表单反推”，这会反向破坏唯一真相目标。

当前状态：

- 已完成第一版（项目首页 -> 进项目默认聊天 -> 一键生成粗纲+人物 -> 粗纲确认事实 -> 下游推进）

### 最近第二轮

目标：

- 把 confirmed formal facts 显式接到人物 / 详纲 / 剧本工序界面，并让用户清楚“下游只能引用、不能再定义”

为什么它排第二轮：

- Chat-first 入口把“唯一真相对象”立住之后，才有必要把下游围绕 confirmed facts 的约束做成用户可见。
- 否则用户只会看到一堆规则提示，仍不知道自己该怎么用系统走通链路。

## 五、以后每轮任务前的判断顺序

1. 这轮任务加强的是哪一个唯一真相对象？
2. 它属于七段主线中的哪一段？
3. 这轮任务是在补真相、补流动，还是补表现？
4. 如果现在做它，会不会打乱主线顺序？

只要第 1 条和第 2 条不清楚，就不进入实现阶段。
