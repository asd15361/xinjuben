# 唯一真相驱动重做总执行计划（第三版）

> 主线状态：启用中（自 2026-03-14 起，取代第二版，作为唯一现行总计划）  
> 当前总进度：100%（第三版总计划已正式完成）  
> 当前阶段：第六阶段｜做最终上线判断  
> 当前判断：第三版总计划原来要回答的阶段问题、上线门禁、质量闭环、架构纪律、性能体验和最终上线判断，现在都已经形成正式结论；当前项目可内部自用和小范围内测，但还不建议外部正式上线。

## 一、为什么旧计划现在要停掉

第二版计划解决了当时最关键的问题：

- 真相主权先锁住
- 工序边界先锁住
- 主链先跑起来
- 旧项目深层业务资产先迁进来
- Chat-first 与黄金链路先走通

这些动作本身没有错，而且很多已经真的做进代码了。

但现在继续沿第二版推进，会出现 3 个问题：

1. 第二版更像“结构重建计划”，不再完全对应当前阶段
2. 主线进度在第二版里写成了 100%，会误导后续判断
3. 当前最紧要的问题已经从“先把骨架搭起来”，变成“把成熟度、上线口径、质量闭环、文档真相重新收口”

所以从今天开始：

- 第二版停止作为现行主计划
- 第三版成为唯一现行总计划
- 后续任务、任务卡、进度、收尾，一律按第三版口径汇报

---

## 二、我们现在到底处在哪个阶段

### 1. 现在不是哪个阶段

现在已经不是：

- 概念验证阶段
- 架构还没立住的早期阶段
- 页面能点就算完成的演示阶段

### 2. 现在也还不是什么阶段

现在还不是：

- 外部正式上线阶段
- 质量可以稳定承诺阶段
- 文档、策略、恢复、质量样本都完全收口的成熟产品阶段

### 3. 现在真实处在哪

现在处在：

**“中后段收口阶段”**

更准确地说，是：

- 架构主方向已经对
- 核心主权已经清楚
- 黄金链路已经跑通
- 系统已经能写剧本
- 但产品成熟度、质量稳定性、长期真相、上线口径还没最终封死

所以这轮新计划解决的本质问题只有一句话：

**把一个“已经能跑的系统”，收成一个“知道自己做到哪、还缺什么、怎么走向成熟”的系统。**

---

## 三、现在已经真正解决了什么

这些不是设想，而是已经落进代码和验证链路的东西。

### 1. 真相主权已经基本锁住

- 正式事实定义权属于创作者
- 正式事实正式声明动作收口在 `outline`
- 下游只能消费 confirmed formal facts
- Prompt 只有表达权，没有定义权
- UI 只有展示和操作权，没有定义权
- repair 只修执行，不修事实

### 2. 主链已经基本成立

- Chat-first 入口已经成立
- 项目首页 -> 进项目默认聊天 已成立
- story intent -> 粗纲 -> 人物 -> 详纲 -> 剧本 的主链已经可走
- gate / ledger / audit / repair / postflight 已进入主链

### 3. 旧项目最值钱的能力已经开始变成系统资产

已经进入新仓库主链的能力包括：

- 故事契约
- 用户锚点
- 戏剧推进链
- 关系施压
- trait-binding
- 失败历史驱动恢复
- 记忆回声与硬锚点
- 审计问题到修补动作映射

### 4. 架构已经不再是空壳

- `shared / main / preload / renderer / docs` 五层边界已立住
- 多个高风险大文件已经拆过
- root 装配文件大体回到组合职责
- renderer 两刀性能债已经压住首包

---

## 四、现在最关键的问题是什么

下面这些问题，就是第三版计划要解决的全部问题。

### A. 认知与口径问题

1. 我们已经能跑，但还没有一份新的主计划准确描述当前阶段
2. 第二版写成 100%，容易把“骨架完成”误当“产品成熟”
3. 文档活页仍偏多，后续容易再次出现多份真相

### B. 产品成熟度问题

1. 系统已经能写剧本，但还不能稳定承诺“每次都能写出好剧本”
2. 能跑通，不等于能对外上线
3. 缺少“内测可上线 / 外部正式上线 / 商业交付上线”三档口径

### C. 剧本质量问题

1. 当前更偏“结构正确 + 主链闭环”，还不是“质量稳定封顶”
2. continuity / tone / hook 承接等更深规则还没彻底收完
3. repair mapping 已经有了，但还没到最终态
4. 真实高质量样本还不够形成强证明

### D. 架构收口问题

1. 架构方向对，但还没到最终成熟态
2. 不是每一个文件都已经达到理想的一文件一职责
3. 仍有历史过渡件、遗留壳、后续可继续纯化的位置

### E. 性能与体验问题

1. renderer 首包已明显下降，但 vendor 依赖仍偏重
2. 表现层已经有第一版，但还不是最终交付态
3. 用户能走通，不代表体验已经足够顺手、足够稳、足够清楚

### F. 上线与交付问题

1. 现在适合内部使用和小范围内测
2. 还不适合直接当外部正式成熟产品上线
3. 缺少一份明确的上线前缺口清单与门禁清单

---

## 五、第三版总目标

第三版不是再回答“怎么把骨架搭起来”，而是回答：

**怎么把这个已经能跑的系统，收成一套真正可判断成熟度、可控质量、可控上线、可长期维护的产品。**

第三版交付标准只有 5 条：

1. 只有一份现行总计划
2. 只有一份现行当前任务卡
3. 产品当前阶段、上线口径、成熟度口径说的是同一套话
4. 系统不只会跑，还能更稳定地写出更像样的剧本
5. 到最终阶段时，能清楚回答“现在到底能不能正式上线”

### 第六阶段：做最终上线判断

本质问题：

- 前面做了很多事，如果最后不能正式判断“现在能不能上线”，前面就没有真正收口

本阶段目标：

- 基于前五阶段的结果，做一次正式上线判断

交付标准：

- 能明确回答：现在能不能正式上线

## 七、阶段顺序为什么这样排

顺序不能乱。

### 先收口项目真相

因为现在最危险的不是“少一个功能”，而是大家对项目阶段理解不一致。

### 再收口上线口径

因为如果不上来先把“能跑”和“能上线”分清，后面所有质量判断都会飘。

### 再收口剧本质量

因为这个项目最终值不值钱，还是看能不能稳定写出更好的剧本。

### 再收口架构纪律

因为质量逻辑收纯以后，才更容易判断哪些文件、哪些层还混职责。

### 再收性能和体验

因为前面先把方向和门禁钉死，后面的体验优化才不会回到表层修补。

### 最后才做最终上线判断

因为上线判断必须建立在前面都已有结果的前提上。

## 八、第一阶段要做什么

第一阶段只做 4 件事，不准跑偏。

1. 停掉第二版现行口径，启用第三版
2. 统一当前阶段、成熟度、上线口径的正式说法
3. 更新当前任务卡，切到第一阶段 0%
4. 把后续每个阶段的目标、验收标准、阻断点写清楚
5. 把“内部自用 / 小范围内测 / 外部正式上线”三档口径写成正式门禁文档

第一阶段明确不做：

- 不继续补新业务功能
- 不继续加新 UI 花活
- 不把性能债当当前主任务继续推进
- 不再沿用旧计划的 100% 口径

## 九、每个阶段都必须回答的 6 个问题

以后每推进一个阶段，都必须明确回答：

1. 这阶段要解决的本质问题是什么
2. 为什么现在轮到它
3. 现在真实卡点是什么
4. 完成后会获得什么
5. 验收标准是什么
6. 如果不做，会留下什么后果

只要有一个问题答不清，就不能算真正进入执行。
