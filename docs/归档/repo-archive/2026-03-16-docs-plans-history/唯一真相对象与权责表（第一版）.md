# 唯一真相对象与权责表（第一版）

## 一、这份文档解决什么问题

这份文档只回答一件事：

**新项目里，哪些对象属于“唯一真相对象”，它们分别从哪里来，谁能写，谁能确认，谁只能建议或读取。**

以后任何新功能，只要碰到主线数据，都先对照这份表。

---

## 二、五类唯一真相对象

### 真相对象 1：项目真相中心

它是第一真相容器。

必须统一承接：

- `storyIntent`
- `outlineDraft`
- `characterDrafts`
- `detailedOutlineSegments`
- `scriptDraft`
- `script runtime state`
- `scriptStateLedger`

写权限：

- 只能通过 `main/application/workspace/*` 或正式 use case 写入

读权限：

- 各工序可以读

禁止：

- 页面自己长期维护一套平行项目真相

### 真相对象 2：正式事实

这是当前最核心的业务真相对象。

正式定义：

- 只有创作者拥有定义权
- 正式声明只能发生在 `outline`
- 进入下游主链的只允许是 `confirmed formal facts`

写权限：

- 粗纲工序中的创作者正式声明
- 粗纲工序中的创作者正式确认

系统权限：

- 校验权
- 升格判定权
- 追踪权

禁止：

- AI、ledger、audit、repair、下游工序偷偷发明 confirmed facts

### 真相对象 3：四工序正式产出

四工序不是草稿堆，而是四段正式产出：

- 粗纲产出
- 人物产出
- 详纲产出
- 剧本产出

它们都必须属于项目真相中心，且只能按合同流动。

禁止：

- 下游直接补写上游结构
- 用页面状态伪装正式产出

### 真相对象 4：剧本运行态

剧本运行态不是 UI 临时态，而是正式业务对象。

至少包括：

- execution plan
- progress board
- resume resolution
- failure resolution
- state machine

写权限：

- 只能在 `main/application/script-generation/runtime/*`

禁止：

- renderer 自己长期推导运行态真相

### 真相对象 5：连续性台账

ledger 不是增强项，而是系统连续性真相对象。

至少承接：

- 人物状态
- 关系压力
- knowledge boundaries
- open hooks
- story momentum
- 正式事实相关状态

禁止：

- 用零散 prompt 补充代替 ledger
- 把 ledger 退回“只是显示信息”

## 三、唯一真相对象权责总表

| 真相对象 | 正式来源 | 谁能写 | 谁能确认 | 谁只能建议 / 判定 | 下游谁能用 |
|---|---|---|---|---|---|
| 项目真相中心 | `workspace` 主进程存储 | `main/application/workspace/*` | 系统持久化确认 | renderer 只能触发 | 全系统 |
| 正式事实 | 创作者在 `outline` 正式声明 | 创作者 | 创作者 | 系统可校验、判定升格 | 人物 / 详纲 / 剧本 / ledger / audit / repair |
| 工序正式产出 | 各工序正式保存结果 | 当前工序 use case | 系统按合同收口 | UI 只能编辑与展示 | 下一工序 |
| 剧本运行态 | runtime use case | `script-generation/runtime/*` | 系统状态机 | UI 只能展示 | 生成 / 续跑 / 失败恢复 / 修补 |
| 连续性台账 | ledger use case | `script-generation/ledger/*` | 系统回写 | 审计和修补读取 | 审计 / 修补 / 后续生成 |

---

## 四、最容易踩坑的 5 件事

### 坑 1：把“系统校验”误当成“系统定义”

系统可以判断一个事实像不像正式事实，  
但不能替创作者把它定义成正式事实。

### 坑 2：把“候选事实”误当成“confirmed fact”

AI 建议、系统推断、审计识别出来的事实，  
都不能自动入主链。

### 坑 3：把页面临时态误当成真相对象

页面草稿可以存在，  
但正式主线对象只能以项目真相中心和正式 use case 为准。

### 坑 4：把运行态放回 renderer

运行态可以在页面展示，  
但不能长期由 renderer 推导和维护。

### 坑 5：把 ledger 当成附属说明

ledger 不是注释，也不是报表。  
它是连续性真相对象。

---

## 五、以后新增功能的对照顺序

以后任何新功能都先对照这 4 个问题：

1. 我碰到的是哪一个唯一真相对象？
2. 这个对象现在的正式来源是否已经锁死？
3. 我这次是在合法写入，还是在偷偷开口子？
4. 下游消费的是不是 confirmed 或正式收口后的内容？

只要有一条答不清，就先不要动代码。
