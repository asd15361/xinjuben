# 旧项目人物小传链学习报告（第一版）

## 一、这份报告回答什么问题

这份报告只回答一件事：

**旧项目的人物小传链为什么重要，重建时应该保留哪些概念。**

## 二、先说结论

旧项目的人物小传不是一个独立表单。

它是一条正式的中间工序链，位置在：

- 粗纲之后
- 详细大纲之前

它的真正价值不是“补人物资料”，而是把角色从：

- 名单

升级成：

- 可驱动后续剧情的叙事锚点

并且它还带着：

- 确认门
- 质量门
- 修复层
- 超时回退
- 下游放行约束

## 四、人物小传链真正做了什么

### 1. 人物预算治理

旧项目不是让模型无限写人物。

它会先决定：

- 最少几个人
- 种子名从哪里来
- AI 还能补几个
- 上限多少
- 超时多久

这说明人物链本身承担“治理”职责。

### 2. 角色身份真相层

人物小传链不只是卡片，还承担：

- 手动锁名
- 改名同步
- 角色图谱快照
- 角色身份 bundle

也就是说，它是人物命名和身份的正式真相层之一。

### 3. 质量门

真正被放行的不是“字段填满”，而是：

1. 正文不能是占位草稿
2. 不同层级角色要有最低正文长度
3. 关键角色必须有身份锚点
4. 关键角色必须有关系证据
5. 名字不能非法或重复

### 4. 修复层

旧项目默认承认：

- 模型首轮输出不稳定

所以专门有 repair 层去做：

- 别名归一
- 白名单过滤
- 缺失角色补齐
- 正文补强
- 互串修正

### 5. 可降级、可继续

超时时旧项目不是整链停摆，而是：

- 自动生成 fallback draft
- 让用户先补关键角色
- 再继续推进

### 6. 下游放行门

没有人物小传，或者人物小传没确认：

- 不能继续详细大纲
- 不能继续剧本

这说明它不是旁路功能，而是正式工序门。

## 五、这条链为什么能提升剧本质量

因为很多剧本问题其实不是“文笔不行”，而是：

- 人物不稳定
- 主角没有驱动力
- 对手没有施压点
- 关系不形成杠杆
- 后续详纲和剧本没有明确的人物锚点

人物小传链的作用，就是在这些问题真正放大之前把人物站住。

## 六、旧项目里最值钱的成功标准

重建时真正该保留的是下面这些成功条件：

1. 至少形成可读人物正文，不是白卷
2. 主角 / 对手等关键角色有身份锚点
3. 关键角色有关系杠杆证据
4. 角色数量满足季长治理要求
5. 角色名稳定，不和锁名、图谱、别名冲突
6. 结果被确认后，能作为详纲 / 剧本上游合同使用

## 七、旧项目暴露出的风险

### 1. 过度结构化导致白卷

旧项目自己已经证明，人物小传如果做成“十几个字段一次性结构化输出”，模型一旦缺字段，就容易全卡退化成：

- 待补充
- 占位
- 空心卡片

### 2. 名字漂移和身份污染

人物链会同时吃这些来源：

- 手动锁名
- 别名映射
- 关系图
- 粗纲提名
- 模型自由补名

如果没有单一真相层，就很容易打架。

### 3. 超时后整链停摆

如果没有 fallback 与可继续机制，一超时就会卡死整季流程。

### 4. 工序边界被绕过

如果人物链只是“可有可无的资料页”，详纲和剧本质量会直接塌掉。

## 八、重建时必须保留的概念

### 必须保留 1：人物小传是阶段合同

它不是展示页，而是粗纲到详纲之间的正式中间产物。

### 必须保留 2：先保住可读正文

正文质量比字段齐全更重要。

### 必须保留 3：关键角色必须有身份锚点和关系证据

这是旧链最值钱的质量门。

### 必须保留 4：允许不完美，但不能整卡白卷

所以 repair 和 timeout fallback 的思想要保留。

### 必须保留 5：角色命名要有单一真相层

最后都要收敛到稳定 canonical name。

### 必须保留 6：结果必须能继续喂给下游

人物小传不是保存卡片，而是形成下游输入约束。

### 必须保留 7：失败要可降级、可恢复、可继续

这比“一失败就报错”更接近真实创作生产链路。

## 九、重建时不要照搬的东西

### 不要照搬 1：碎到过度的字段壳子

人物小传不该退化成表单录入页。

### 不要照搬 2：大量 repair service 名字本身

要保留 repair 思想，但不应该复制旧项目那种 service 爆炸式形态。

### 不要照搬 3：页面内 orchestrator 总控

编排应该归应用层，不该继续绑在大页面里。

## 十、迁入新项目时的建议落点

### 新项目建议这样落

- `shared/contracts/character.ts`
- `shared/domain/quality`
- `application/character`
- `features/character`

### 推荐迁移顺序

1. 先定义 `CharacterProfileContract`
2. 再定义成功 / 阻断条件
3. 再定义 canonical name 机制
4. 再定义 fallback / repair / confirm 机制
5. 最后接 UI 和下游放行

## 十一、最终结论

旧项目人物小传链最值钱的，不是它拆了多少 service，而是这四件事：

- 单正文主字段
- 质量门
- 修复层
- 阶段放行

新项目重建时如果保住这四件套，人物链就不会退化成“好看但没用的资料页”。
