# 旧项目剧本生成链学习报告（第一版）

## 一、这次学习的核心结论

旧项目真正值得迁移的，不是某一段 prompt，也不是某个页面长什么样，而是它为了“稳定写出高质量长链路剧本”建立的几套业务机制。  
这些机制一旦丢掉，新项目就算界面更干净、模块更漂亮，也很容易重新掉回“能跑，但一长就乱、一续跑就错、一返修就崩”的坑里。

这次先锁定 5 个最值得迁移的机制骨架：

1. 阶段合同指纹机制
2. 剧本生成前输入合同机制
3. 断点续跑与重写计划机制
4. Season State Ledger 状态台账机制
5. 审计闭环 + 定向修补机制

## 二、五个最该迁移的旧项目机制

### 1. 阶段合同指纹机制

#### 机制价值
- 把“详细大纲 + 已确认人物小传”的上游真相压成一个可校验的合同指纹；
- 下游剧本生成每次启动、续跑、重跑前，都确认自己吃到的是不是同一套上游输入；
- 这本质上是在程序层强制工序边界，不靠人脑记忆。

#### 为什么必须学过来
- 它直接解决“上游改了，下游还拿旧数据继续写”的隐性事故；
- 它是“工序边界刚性”的工程化落地，不是口号；
- 新项目后面做人物、详细大纲、剧本链时，都会依赖这个机制。

- `D:\project\juben\src\renderer\src\services\detailedScriptStageContractService.ts`
- `D:\project\juben\src\renderer\src\services\generationService.ts`

### 2. 剧本生成前输入合同机制

#### 机制价值
- 在真正生成剧本前，先严格检查详细大纲是否完整；
- 缺哪一集、哪一段、哪种必要信息，明确报错，不允许系统偷偷“脑补”；
- 保证剧本阶段永远是基于合格输入工作，而不是靠模型临场补结构。

#### 为什么必须学过来
- 这是避免“下游看似生成成功，实际上骨架早就坏了”的核心保险；
- 它把“结构线正式事实升格权”和“内容线戏剧推进链”保护在生成前；
- 很适合作为新项目剧本工序的硬边界。

- `D:\project\juben\src\renderer\src\services\scriptGenerationInputContractService.ts`
- `D:\project\juben\src\renderer\src\services\scriptGenerationExecutionPlanService.ts`

### 3. 断点续跑与重写计划机制

#### 机制价值
- 把“首次生成、失败续跑、局部重写、问题集返修”拆成不同执行模式；
- 系统自动计算起始集、已锁定前缀、剩余集数和下一步策略；
- 让长季生成不是一次性赌博，而是可以管理的生产过程。

#### 为什么必须学过来
- 这是旧项目能支撑复杂剧本生产的重要原因；
- 一旦没有这层机制，新项目后面做剧本批量生成时很快会乱；
- 这个机制也非常适合拆成新架构里的独立应用服务，而不是揉在页面里。

### 4. Season State Ledger 状态台账机制

#### 机制价值
- 把角色生死、伤势、羁押、身份状态、道具归属、未解钩子、张力变化、别名绑定等信息累计成一套账本；
- 每次生成前做 preflight 检查，每次生成后做 postflight 修正；
- 让“连续性”从感性经验变成可以核对、可以补丁、可以追责的数据层。

#### 为什么必须学过来
- 这是旧项目业务含金量最高的机制之一；
- 它决定了为什么旧项目能把很多长链路内容维持住，不轻易写崩；
- 新项目以后如果要做到“可维护、可扩展、还能稳定写好”，账本层必须尽早设计。

### 5. 审计闭环 + 定向修补机制

#### 机制价值
- 不是生成完只给个分数，而是先审计问题；
- 再根据问题类别拼接针对性的修补块；
- 必要时重试，并把反馈沉淀下来，形成真正的质量闭环。

#### 为什么必须学过来
- 这层机制和“柔性赋能、不替代创作”非常契合；
- 它不是硬改内容，而是把问题定位出来，再给定向修补能力；
- 很适合放进新架构的“质量检测 / 审计修补层”。

## 三、对新项目的直接启发

### 1. 新项目不要先复制“页面流程”，要先复制“业务真相层”

如果先搬旧项目页面和调用外壳，而不先重建上面这 5 套机制，新系统最多只是更整洁，但不一定更强。  
真正决定剧本质量和长链路稳定性的，是：

- 上游合同有没有锁住；
- 下游输入有没有被严格校验；
- 续跑计划是不是可控；
- 连续性账本是不是存在；
- 质量问题是不是能形成审计闭环。

### 2. 这 5 套机制分别应该落到新架构哪里

建议映射如下：

- 阶段合同指纹机制：落到 `shared/contracts + main/application/stage-contract`
- 输入合同机制：落到 `main/application/script-generation/input-contract`
- 续跑与重写计划机制：落到 `main/application/script-generation/execution-plan`
- 状态台账机制：落到 `main/domain or main/application/ledger`
- 审计闭环与定向修补：落到 `main/application/audit` 和后续赋能层

### 3. 新项目的迁移顺序建议

先后顺序建议如下：

1. 先补项目真相中心和各工序的项目级持久化
2. 再补粗纲、人物、详细大纲的阶段合同
3. 然后重建剧本输入合同和执行计划
4. 再接入状态台账
5. 最后补审计闭环与定向修补

## 四、当前判断

旧项目确实非常值得学，而且不是“学皮毛”，是“学它为什么能稳定写出好剧本”。  
从目前反拆结果看，它最值得继承的不是耦合方式，而是这些经过实战打磨出来的业务机制。  
新项目如果按这个方向推进，是有机会做到：

- 比旧项目更好维护；
- 比旧项目更容易扩展；
- 同时尽量保住旧项目写剧本很强的核心能力。

这也是这次重构最该坚持的方向。
