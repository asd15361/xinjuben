# 旧项目对话链学习报告（第一版）

## 一、这份报告回答什么问题

这份报告只回答一件事：

**旧项目的 intake / chat 链真正值钱的地方是什么，重建时该迁哪些能力。**

## 二、先说结论

旧项目里真正有价值的不是“聊天 UI 本身”，而是：

**把用户零散表达压成可继承的故事意图包，再稳定喂给粗纲 / 人物链。**

也就是说，这条链最值钱的是：

- 统一 intake 真源
- 动态追问补洞
- 语义清洗
- 实名锁定
- story intent bridge
- 项目级恢复

## 四、它到底收集了什么

旧项目不是只问“你想写什么题材”。

它实际收集的是一整包故事约束。

### 基础问答

- 题材
- 集数
- 单集时长

### 常规追问

- 风格 / tone
- 受众
- 主角
- 对手
- 核心冲突
- 结局方向
- 要素偏好

### 系统补题

旧项目很值钱的一点是，它会继续追问这些直接影响戏剧性的内容：

- 中段最大反转
- 关系杠杆
- 集尾钩子类型
- 赢的代价
- 反派目标底线
- 标志性 A 镜头

### 手动强覆盖

- 主角名
- 对手名
- 锁定角色名清单
- 手动需求总表
- free chat 最终总结

### 最终中间产物

最后它会把这些压成：

- `onboardingContextLines`
- `freeChatContextLines`
- `manualRequirementNotes`
- `freeChatFinalSummary`
- `semanticUserLines`

再进一步桥接成：

- `officialKeyCharacters`
- `characterCards`
- `themeAnchors`
- `methodAnchors`
- `worldAnchors`
- `relationAnchors`
- `dramaticMovement`

这就是它真正的“故事意图包”。

## 五、这条链为什么能提升剧本质量

### 1. 双入口，单真源

旧项目允许：

- 引导问答
- 自由聊天

同时存在，但最后必须汇到同一份需求真相。

这是很值钱的，因为它兼顾了：

- 用户自由表达
- 系统结构化收束

### 2. 动态追问补洞

不是只收集基础信息，而是追到真正影响粗纲质量的戏剧约束。

### 3. 语义清洗再入链

旧项目会去掉聊天废话、运行日志噪声，保留真正硬锚点。

### 4. 实名锁定与角色主键治理

它会从手动填写、总结文本、需求总表中继续抽实名，避免角色名漂移。

### 5. story intent bridge

这一步特别重要。

旧项目不是把聊天记录直接塞给粗纲模型，而是先桥接成：

- `factBlock`
- `dramaticMovement`
- `worldBrief`
- `characterBibleContext`

这会让粗纲更稳。

### 6. 项目级持久化恢复

对话链不是一次性聊天，而是可恢复的项目资产。

## 六、这条链最该保留的部分

### 必须保留 1：统一 intake 真源

所有输入最后必须汇到同一份正式真相。

### 必须保留 2：动态追问策略

真正影响戏剧质量的补题机制必须保留。

### 必须保留 3：语义清洗

聊天废话不能直接进入粗纲 / 人物链。

### 必须保留 4：实名锁定治理

人物主键必须稳定，不能让 AI 猜测角色名反复漂移。

### 必须保留 5：story intent bridge

对话链必须输出可以稳定喂给下游的故事意图包。

### 必须保留 6：项目级恢复

对话断了以后，锚点和意图不能丢。

## 七、不要照搬的东西

### 不要照搬 1：分裂的 hook / handler / orchestrator / service 结构

旧项目现在这条链已经有明显重复，不适合直接照搬。

### 不要照搬 2：free chat 收口现有细节实现

它里面仍有草稿和占位逻辑，不适合作为新项目标准答案。

### 不要照搬 3：大量正则推断堆在 UI 层

理念可留，组织方式不能留。

### 不要照搬 4：分散的 localStorage key 体系

状态真相应该统一收口。

## 八、迁入新项目的建议落点

### 新项目建议这样落

- `shared/contracts/intake.ts`
- `application/intake`
- `features/intake`
- `application/outline` 中的 story intent bridge

### 推荐迁移顺序

1. 先定义 `IntakeAnswer`
2. 再定义 `StoryIntentPackage`
3. 再定义动态追问策略
4. 再定义锁名治理
5. 最后接 UI 和项目恢复

## 九、最终结论

旧项目 intake / chat 链最值钱的不是“聊得多”，而是：

- 统一真源
- 动态补洞
- 锁名治理
- story intent bridge

新项目只要把这四件事保住，对话链就不会退化成“好看但没法稳定喂下游”的聊天面板。
