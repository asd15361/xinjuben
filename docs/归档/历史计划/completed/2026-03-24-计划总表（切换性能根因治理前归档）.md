# 计划总表

更新时间：2026-03-21

## 一句话结论

从现在开始，`docs/plans/计划总表.md` 是项目主计划文档和总入口之一，用来承接当前主线、统一口径和挂总索引。

本文件中凡是历史阶段留下的“测试 / E2E / 回归 / 长测 / 验收脚本”描述，一律只作为历史背景，不再构成当前执行计划。  
当前执行口径统一为：`代码实现 -> 必要的仓内收口检查与最小范围回归 -> 代码收口 -> 交由用户真人手测`。

我们负责三件事：

- 判断本质问题
- 更新唯一计划
- 验收执行人 的执行结果

执行人负责两件事：

- 严格按本计划执行代码、收口、清理和落文档
- 执行后把结果、风险、卡点按本计划回填

除本文件外，`docs/plans/` 还可以并行存在其他主题计划；但已完成的计划必须及时移入 `docs/plans/completed/` 或 `docs/archive/`，不要继续留在主目录混淆当前执行。

## 当前协作方式

### 我们怎么做

- 我们默认不再逐步审批执行人 的每一步执行。
- 我们的主要工作改成：
  - 持续巡检项目全局
  - 把发现的新问题写进本计划
  - 定义优先级、边界和验收标准
  - 定期抽查任务卡、worklog 和实际代码状态

### 执行人怎么做

- 执行人 默认按本计划持续往下执行，不需要每完成一步都等我们放行。
- 只有遇到下面 4 种情况，才需要停下并显式回报：
  - 遇到新的主阻塞，无法继续推进
  - 发现计划和代码现实明显冲突
  - 需要跨越当前阶段去改更大的链路
  - 准备宣布一个阶段完成

### 计划维护原则

- 本文件不是一次性计划，而是持续增长的问题总表和执行总表。
- 后续我们发现新问题，不单独开新计划，直接加进本文件。
- 已完成的事情，不从本文件删除，而是转移到“已完成项”区，保留项目全貌。

## 当前事实

### 1. 当前项目阶段

- 主链第一轮结构统一已完成，唯一工序仍是：`10集规划 / 10集详纲 / 5集写作 / 逐集细纲真源`
- 历史上已有过小批次通过记录，但那只说明某一轮曾经过门，不等于当前链路已经稳定可交付
- `D:\问题\剧本问题.txt` 已把当前更本质的问题坐实：真正压制稳定优秀产出的，不是 `2-3场 / 字数下限 / 集尾钩子 / 可拍格式` 这些工业底线，而是围绕这些底线附加的写法级微操过密

### 2. 当前必须保留的底线

- 每集 `2-3场`
- 字数下限与波动一致性
- 集尾必须有硬钩子
- 剧本必须可拍、可解析
- 正式事实与逐集细纲真源不能被 runtime / UI / repair 改写

### 3. 当前真正阻塞上线的真问题

- 主生成 prompt 把“结果标准”写成了“操作说明书”，AI 负载过高，容易写僵、写模板、写成只过门不出戏
- hard hook 裁判过于依赖词表/例句，存在误杀有效钩子、反向逼模型模板化的风险
- repair prompt 和 repair 执行链同时承担太多教学职责，修复不够聚焦，容易继续放大上下文负载
- 审计 / 修复 / 状态机 / batch 失败策略里仍有“一个点出错拖垮整批”与“静默失败”的结构问题
- 当前计划和任务卡里仍残留“先拿 key 跑通 real E2E”式旧口径，会误导执行顺序

## 项目本质问题

当前项目的本质问题，不再是“要不要继续围绕历史测试噪音打转”，而是：

> 标准已经基本找对了，但系统把写法管得太碎、修法堆得太满、失败传播得太重，结果变成“更容易过门，不更容易稳定出优秀作品”。

本轮必须解决的本质只有 4 个：

1. 把“结果标准”和“写法微操”彻底分开，只保留跨故事成立的硬底线
2. 把主生成、hook 裁判、repair 三处最重负载减下来，让模型先写活，再由裁判卡底线
3. 把“单点失败拖垮整批 / parse 失败直接炸 / 静默吞错”的工作流硬伤收掉
4. 把计划、任务卡、交付顺序改成“先调整、先收口、再交由用户真人手测”，不再被旧 key 和历史噪音带偏

所以当前策略固定成：

- 我们负责定义底线、阶段顺序、验收标准，并验收执行人 的结果
- 执行人负责按专项拆解并执行代码、文档回填
- 在完成这轮“质量系统减载”前，不安排 E2E、Playwright、浏览器自动化验收、长链路 smoke、验收脚本；只保留必要的仓内收口检查和最小范围回归，代码收口后交由用户真人手测

## 当前未完成项总表（执行人直接执行）

### 当前执行说明

- authority、foundation baseline、四层测试等内容，全部降为历史背景，不再驱动当前执行顺序
- 当前正式主线只做三件事：
  - P0：首稿质量代码收口
  - P1：恢复链代码收口
  - P2：长链入口与导出链代码收口
- 当前不再安排任何自动化验收、长链 smoke、分档放量或浏览器自动化

下面这些就是“整个项目还没搞完”的剩余事项。  
执行人不需要再自己判断优先级，直接按顺序执行、回填、收口。

### P0：首稿质量收口

#### 目标

- 把当前主问题从“能写但不稳”收成“能稳定过真实质量门”
- 重点打穿第 `11-15 集` 这类真实失败样本里的两个高频问题：
  - 字数低于 `500`
  - 集尾钩子偏弱

#### 执行人负责做什么

1. 先只改首稿生成，不先重改 gate、真源、上下文架构
2. 把近期补学的爆款规律真正前移到首稿：
   - 开场直接砸冲突，不慢热解释
   - 每场形成 `动作 -> 对打 -> 后果`
   - 爽点必须可拍、可见、可定格
   - 集尾必须停在已经发生的结果或直接威胁，不停在气氛和预感上
   - 字数靠推进撑满，不靠说明句灌水
3. repair 只保轻修，不准承担“重写主戏”的职责

#### 收口标准

- 代码层面把首稿质量相关根因收住：
  - prompt 不再过度微操
  - hook 判定不再词表化误杀
  - repair 不再承担主创作职责
  - 失败传播不再放大局部问题
- 同时只保留必要的仓内收口检查和最小范围回归，确认当前改动没直接写炸
- 改动完成后，由用户用真实项目手测确认：
  - `500` 字以下问题是否显著下降
  - `集尾钩子偏弱` 是否显著下降
  - 场次、可拍性、对白密度没有反向变坏
  - 首稿改善不是靠 repair 硬补出来

#### 禁止事项

- 不要把创作问题改成更重的 gate
- 不要让 repair 继续背锅成第二本创作说明书
- 不要为过质量门硬加模板腔和词表腔

### P1：失败恢复链收口

#### 目标

- 把“后端能恢复”变成“前端流程也真的能恢复”
- 证明失败后可以从正确集数继续写，不是假状态

#### 执行人负责做什么

1. 把失败恢复链代码与当前真流程彻底对齐
2. 只收代码里的真实恢复入口：
   - seeded 项目入口
   - sceneByScene 真源
   - 当前 ScriptStage UI
   - 当前“剧本草稿”术语
3. 不再编排 E2E / Playwright / 浏览器自动化恢复测试；只保留必要的仓内收口检查和最小范围回归，代码收口后交由用户真人手测：
   - 失败后能否看到恢复入口
   - 是否真能从 `resumeEpisode` 接着继续生成

#### 收口标准

- 不再走旧 chat / 旧粗纲 / 旧四段详纲 / 旧“剧本定稿”入口
- 代码中必须能明确看见并保证这几项真实语义：
  - `failure.kind`
  - `resume.canResume`
  - `resume.resumeEpisode`
  - 恢复后下一批从正确位置继续
- 最终由用户在产品里真人手测确认

### P2：长链入口与导出链收口

#### 目标

- 让长链入口与导出链只认当前真流程，不再依赖过时入口和旧测试脚本
- 把长链相关代码收口到可由用户直接操作确认的状态

#### 执行人负责做什么

1. 清掉长链相关旧测试入口和旧假设
2. 保留真正有产品价值的结果产物语义：
   - batchHistory
   - screenplayReport
   - export summary
3. 改成当前真流程入口：
   - seeded 项目
   - sceneByScene 真源
   - 当前 ScriptStage

#### 收口标准

- 旧黄金链入口不再作为主入口
- 代码与导出链必须能支持用户直接判断：
  - 是否稳定跑完
  - 哪一档开始掉质量
  - repair 触发率与残留问题
  - 是否仍存在批量故障放大

### P3：最终上线前口径统一

#### 目标

- 不让“代码改完”和“项目可交付”混成一个结论
- 把对外口径、任务卡、worklog、计划总表统一成一个真相

#### 执行人负责做什么

1. 每完成一轮代码收口，就同步回填：
   - `docs/plans/计划总表.md`
   - `docs/当前工作区/active-task（当前任务卡）.md`
   - `4.worklog.md`
2. 禁止再用旧的通过记录冒充新主链结论
3. 明确区分：
   - 代码结构是否完成
   - 真实生成质量是否完成
   - 恢复链与长链入口是否完成

#### 最终签收条件

项目只有同时满足下面 4 条，才允许报“基本完成”：

1. 首稿质量相关根因已完成代码收口
2. 失败恢复链已按当前 UI 真流程完成代码收口
3. 长链入口与导出链已完成代码收口
4. 上述三项都已交由用户手测
5. 计划、任务卡、worklog 三处口径完全一致
