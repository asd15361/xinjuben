# 60集详细大纲稳定性交付计划（第一版）

## 这轮要解决的本质
- 不是“详细大纲偶尔能不能出”。
- 真正的本质是：系统现在还会把“不完整的 60 集详细大纲”当成正式结果保存下来。
- 只要系统还允许 `60 集项目` 落成 `18` 集或 `24` 集的详纲，交付标准“稳定产出优效作品”就没有成立。
- 所以这轮不是继续补页面，也不是继续做表面兜底，而是要把“短详纲被写成正式结果”这件事从系统里彻底关掉。

## 已确认现象
- `60 集` 真实链路连续复测里，详细大纲出现过 `beatCount=24`、`beatCount=18`。
- 同一口径连续 3 次结果出现过 `好 / 中 / 弱`、`中 / 中 / 弱`、`弱 / 好 / 弱` 这类波动。
- 更早有一部分误差来自“改了 src/main 但没 build 就跑 Electron，测试吃旧产物”；这类假象已经识别。
- 但在重新 `npm run build` 之后，`60 集详细大纲` 仍然持续波动，所以现在剩下的就是产品真问题，不是测试假问题。

## 已排除的表层解释
- 不是前端入口没走通：当前真实 UI 主路已经能走到详纲生成。
- 不是测试脚本还在走旧路径：详纲和 60 集脚本都已经追到当前前端交互路径。
- 不是简单保存函数在裁切：`saveDetailedOutlineSegments` 当前只是整包写入，没有主动裁短逻辑。
- 不是“偶发慢一点”这么简单：现在的问题不是时间长短，而是结果本体有时根本不满 `1-60`。

## 当前根因判断
- 第一层本质：`60 集约束` 现在还不是系统里的硬合同，而更像“尽量遵守”。
- 第二层本质：系统缺少 `fail-closed` 门，导致不完整详纲也能升格为正式结果。
- 第三层本质：详纲链路里还存在“上游拿到短范围输入”或“生成后被较短版本回写覆盖”的可能，必须追到第一次把短结果写成系统真相的那一层。

## 根因追查范围
1. 生成前输入
- `outlineDraft.summaryEpisodes`
- `storyIntent.generationBriefText`
- `resolveProjectEpisodeCount(...)`
- 任何会把 `60` 集降成短范围的归一化逻辑

2. 生成与归一化
- `generateDetailedOutlineFromContext(...)`
- `normalizeDetailedOutlineSegmentsForProject(...)`
- act 范围切分和 `episodeBeats` 合并逻辑

3. 保存与回写
- IPC 出口保存链
- renderer `replaceSegments(...)`
- `useProjectStagePersistence` 的自动保存
- 项目重新打开或切页后的 hydrate / 回写顺序

## 修复原则
- 不接受“短了再补一点”的表面修复。
- 不接受“模型有时会飘，所以多跑几次”的解释。
- 必须把 `60 集详细大纲` 的完整覆盖，改成正式保存前的硬门。
- 对 `targetEpisodes >= 60`：
- opening / midpoint / climax / ending 四段都必须存在。
- `episodeBeats` 合起来必须完整覆盖 `1-60`。
- 任一集缺失、重复覆盖异常、段范围错位，都不得保存为正式详纲。

## 执行计划
### 阶段 1：钉死第一次写脏结果的层
- 给详纲生成链补最小必要日志和检查点。
- 记录生成前 `targetEpisodes`、生成后各 act 的范围、保存前总覆盖集数、renderer 收到后的总覆盖集数。
- 用同一套 `60 集` 输入连续复现，确认短结果第一次出现在哪一层。

### 阶段 2：把 60 集约束升成硬合同
- 统一 `60 集项目` 的 episode count 来源，避免不同层各算各的。
- 在保存前做严格校验，不再接受“有四段但只覆盖 18/24 集”的残缺详纲。
- 必要时把失败直接抛回前台，提示“这版详纲没有覆盖完整 60 集，不能当正式结果”。

### 阶段 3：切断短结果回写链
- 如果根因在 renderer 自动持久化或 hydrate 回冲，就修掉回冲源头。
- 确保生成完成后，不会再被较旧、较短、较残的 segments 覆盖回 store。

### 阶段 4：按交付标准重新验收
- 每次改完 `src/main` 先 `npm run build`，再跑真实 Electron。
- 连续 3 次 `60 集详细大纲` 真实复测。
- 验收口径不是“有一次好”，而是“连续 3 次不再掉成短详纲”。
- 最低标准：3 次都至少 `中`。
- 交付标准：3 次都 `好`，且不再出现 `beatCount=24/18`。

## 验证口径
- 真实脚本：`tools/e2e/electron_real_detailed_outline_check.mjs`
- 目标参数：`E2E_TARGET_EPISODES=60`
- 必看指标：
- `segmentCount`
- `beatCount`
- 各 act 覆盖范围
- 是否完整覆盖 `1-60`
- 连续 3 次质量等级
