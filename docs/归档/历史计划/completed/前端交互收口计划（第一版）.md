# 前端交互收口计划（第一版）

更新时间：2026-03-16 18:24

## 这条主线解决什么本质问题
- 不是再证明后端能不能产出。
- 真正要解决的是：现在软件虽然已经能生成粗纲、人物、详纲和剧本，但用户在前端实际操作时，根本搞不清“现在在生成什么、有没有生成完、下一步该点哪里、生成结果到底在哪”。
- 如果交互主路不收口，用户就会把“系统能写”体验成“系统在后台自己乱跑、界面不会说人话、阶段结构不让人改”，这会直接推翻“已经能交付”的判断。

## 当前已确认的问题
- 打开已有项目时，会自动开始生成详细大纲，用户没有明确触发。
- 顶部和页内进度条使用估算时间硬算百分比，导致长期卡在 `99%`、显示 `0 秒`，但用户不知道到底完成没有。
- 生成完成后没有明确提示，也没有稳定跳转；用户只能自己猜要不要点到下一页。
- 顶部和阶段说明文案偏系统脑内话，比如“当前故事收束”，用户看不懂，也没有关闭或下一步动作。
- 详细大纲当前是“四阶段摘要”，不符合用户“逐集查看、逐集修改”的使用方式。
- 剧本页当前更像内部生成工作台，不像“明确看到当前生成到哪、从哪继续写、每一场怎么改”的用户界面。

## 这条主线不做什么
- 不再动结果层写法本身，不把这轮问题重新改成提示词主线。
- 不借机重做整个视觉系统。
- 不把交互问题硬塞进自动链路和隐藏逻辑里继续糊弄。

## 修复顺序
1. 先停掉错误的自动链路
- 打开项目时不再自动继续生成详细大纲和剧本。
- 详细大纲、剧本都必须由用户主动点击触发。

2. 再重做生成状态反馈
- 去掉当前 `99%` 假进度逻辑。
- 改成“生成中 / 已完成 / 失败”三态。
- 完成后必须给明确提示，不准让用户猜。

3. 再修阶段跳转
- 粗纲完成后给出明确入口：去粗纲 / 去人物。
- 详纲完成后给出明确入口：去详细大纲。
- 剧本完成后给出明确入口：去剧本。
- 必要时允许自动切页，但不能偷偷切得让人不明白。

4. 再清顶部和说明文案
- 顶部提示改成人能看懂的话。
- 增加关闭或收起能力。
- 每个阶段只说“你现在在做什么、下一步怎么做”。

5. 再重做详细大纲结构
- 不再只停在四阶段摘要。
- 至少要让用户能按更细颗粒看和改，不能出现“粗纲按集，详纲反而不能逐集改”的倒挂。

6. 再重做剧本页结构
- 让用户明确看到当前生成结果、场次结构、继续生成入口和手动修改入口。
- 不再出现“进了剧本页还是 99%，但剧本在哪不知道”的情况。

7. 最后走一遍真实 UI 验证
- 按用户真实路径测试：打开旧项目、生成粗纲、确认详纲、进入剧本。
- 专门验：按钮响应、状态反馈、跳转、可编辑性。

## 验收标准
- 用户打开旧项目时，不会被系统偷偷推进到下一阶段。
- 用户点击生成后，能明确知道“正在生成什么”。
- 生成完成后，用户能明确知道“已经生成完了”和“下一步去哪”。
- 不再出现长期卡 `99%`、显示 `0 秒` 却不知道是否完成的体验。
- 详细大纲和剧本结构都能让用户看懂、改得动，不再像内部工作面板。
- 用户第一次上手，不需要靠猜，也能顺着前端把粗纲、详纲、剧本走通。

## 当前判断
- 这条主线不是可选优化，是当前真正阻断交付的一层问题。
- 如果不先把前端交互主路收口，再好的结果层也会被体验问题抵消掉。

## 当前进展
- 第 1 步已完成：项目打开后不再自动继续生成详细大纲和剧本。
- 第 2 步已完成大半：已经去掉 `99%` 假进度，改成“生成中 / 已完成 / 失败”三态，并补上完成后的明确提示和入口。
- 第 3 步已完成大半：粗纲、详细大纲、剧本三段都已补上下一步入口；粗纲生成后会直接切到粗纲页。
- 第 4 步已完成一部分：聊天页误导悬浮块已移除，详细大纲页和剧本页的空状态与下一步提示已改成人话。
- 第 5 步已完成：详细大纲底层已升级成“阶段总览 + 逐集细纲”，现在每个阶段下面都能按集查看和修改。
- 当前剩余主尾巴：还差真实 UI 复测，确认这套逐集细纲在生成完成提示、跳转和实际编辑手感上都顺。
