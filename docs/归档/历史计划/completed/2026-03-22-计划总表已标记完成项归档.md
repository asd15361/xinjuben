## P1-1 输入校验/合同/UI gate统一（已解决）

1. ~~输入校验、合同校验、UI gate 三套口径没有完全统一~~ ✅ 已解决

- 表现（最新排查结论，2026-03-20）：
  - UI 层“生成剧本”入口现在已经固定走 `workflow:build-script-generation-plan`，而 `buildScriptGenerationExecutionPlan` 内部会同时调用：
    - `validateStageInputContract('script', payload)` 作为 stage gate
    - `buildScriptGenerationContract` 作为合同 gate
  - 也就是说，现在 UI gate 不再自己发明一套条件，而是直接吃输入合同 + 脚本合同的联合结果。
  - 但 `buildScriptGenerationContract` 里仍保留 `segments` 作为“结构来源之一”；当 `detailedOutlineBlocks` 为空时，会退回用 `segments` 填 `structuralActs`，这等于给了旧口径一个“兜底真相”的位置。
- 影响：
  - 真正的风险不再是“UI 没有走 gate”，而是“合同这层本身还允许 `segments` 当结构来源”，会让后来的人误以为旧分段仍是主真相。
  - 一旦后续某次修 gate 时只改了 `validate-stage-input.ts`，忘了改 `contract-policy.ts`，又会重新形成“双口径”问题。
- 结论（2026-03-20 复核）：
  - `useScriptStageActions.handleStartGeneration` 已固定走 `workflow:build-script-generation-plan`；
  - `buildScriptGenerationExecutionPlan` 同时吃 `validateStageInputContract('script', payload)` 和 `buildScriptGenerationContract(input)`；
  - `buildScriptGenerationContract` 现已只基于 `detailedOutlineBlocks + activeCharacterBlocks + confirmed facts` 判定 `ready`，`segments` 不再参与主判断。
