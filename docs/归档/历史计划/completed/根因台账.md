# 根因台账

更新时间：2026-03-20

## 使用规则

- 本表是唯一根因视图。每一行代表一个已确认的根因。
- **OPEN** = 尚未解决或解决不彻底；**CLOSED** = 已正式封口，有自动化守卫，不会回弹；**MONITORING** = 已解决但需持续观察
- **自动化守卫**：`tools/e2e/root_cause_guards.mjs` — 14项检查，根因1-4和6各有关键守卫；任何代码变更后必须运行此脚本验证
- 遇到新问题时：**先查本表**——是否是这个表里旧问题的回弹？不允许直接起新名字。
- 每个根因必须有：唯一名字、第一次写错真相的层、已做修复、防回弹守卫（自动化检查或脚本）

## 每次遇到新问题时的检查流程

1. 查本表：这个问题是否已有记录？（现象别名匹配）
2. 如果匹配 → 这是旧问题回弹，不准起新名字；直接用已有根因记录
3. 如果不匹配 → 新根因，需要填新条目，包含：现象别名、真正根因名、第一次写错真相的层、证据、已做修复、防回弹守卫
4. 新根因创建后，同步更新计划总表

---

## 本表与计划总表的关系

- 根因台账记录"为什么这个问题会出现"
- 计划总表记录"哪个计划在解决什么问题"
- 一个计划可能同时解决多个根因
- 根因关闭条件：必须有自动化守卫或脚本验证，不能只靠"这次看起来好了"
