# 唯一真相源收口计划（第一版）

更新时间：2026-03-17

## 这轮到底要解决什么

- 这轮不是继续补一个结果，不是继续调一次提示词，也不是继续做单点通过。
- 这轮要解决的本质问题是：系统里还残着第二套权力。
- 只要系统里还存在第二写口、第二发布口、第二裁判口，系统就不是唯一真相源，验收就不成立。

## 用户给出的验收标准

- 系统里不能有第二个写口。
- 系统里不能有第二个发布口。
- 系统里不能有第二个裁判口。
- 找到一个，砍一个。
- 直到系统里真的只剩一套权力，不剩第二套。
- 只要还有一个斜口、影子口、兼容口还能偷偷掌权，就不算通过。

## 这一轮的正式判断

- 当前项目的物理写盘口只有一个：`project-store-core.writeStore()`。
- 但“能把内容送进正式结果”的上游入口不止一条，所以系统主权仍然没有真正收口。
- 目前已确认的最大根因，不是模型不稳，而是主链外还留着影子入口。

  - 只有 `outline + user` 可以声明、确认、删除正式事实
  - `ai_suggested + draft` 不会自动进入主线

  - 这意味着 formal fact 的定义权是单的，写口却仍受 outline 通用写口牵连

## 当前执行拆法

### 保留的口

- `useChatStageActions` 触发粗纲与人物生成
- `useDetailedOutlineStageActions` 触发详细大纲生成
- `useScriptStageActions` 触发剧本生成
- formal fact 专用动作 API

### 先砍的口

- `useProjectStagePersistence`
  - 这是“监听本地 store 自动升格正式结果”的根因层

### 第二批再收的口

- `useProjectAutoGeneration`
  - 这是自动链旁路，不该继续拥有正式发布权
- `useWorkspaceProjects`
  - 这是额外封装的通用保存入口

### 这轮执行原则

- 不是把所有保存都删掉。
- 是把“自动监听后偷偷保存”删掉，只保留“显式命令触发后正式保存”。

## 根因

- 根因不是“某个函数偶尔写错”。
- 根因是系统现在仍然允许多个层同时拥有这三种权力：
  - 写正式结果的权力
  - 向用户/下游发布正式结果的权力
  - 判断什么能过、什么不能过的权力
- 只要这三种权力没有各自收成唯一入口，系统就会反复出现“昨天好、今天又歪”的情况。

## 收口原则

- 正式结果只允许命令式保存，不允许监听式自动回写。
- UI 只展示正式结果，不再自己拼业务真相。
- Prompt 只表达，不补定义。
- 测试脚本只验证正式口，不再绕开主链代写正式结果。
- 质量判断只允许一套正式裁判，其他地方只能读结果码，不准重算。

## 执行计划

### 第一阶段：画出权力总图，冻结第二口

- 目标：把写口、发布口、裁判口全部列成单表。
- 动作：
  - 列出所有正式保存入口。
  - 列出所有直接发布到 UI/下游的入口。
  - 列出所有 ready/block/quality 判定入口。
  - 标出主链、兼容链、影子链。
- 完成标准：
  - 每个口都有文件路径。
  - 每个口都标清“保留 / 收编 / 删除”。

### 第二阶段：砍第二写口

- 目标：任何正式结果只能经一条命令式主链落盘。
