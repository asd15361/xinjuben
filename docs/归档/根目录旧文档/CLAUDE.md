## Node 进程管理

**不要杀全局 node.exe**，正确做法是先查目标，再按 PID 杀。

查看所有 Node 进程及命令行：
```powershell
Get-CimInstance Win32_Process |
  Where-Object { $_.Name -eq 'node.exe' } |
  Select-Object ProcessId, CommandLine
```

只杀占用指定端口的进程（如 3000）：
```powershell
Get-NetTCPConnection -LocalPort 3000 -ErrorAction SilentlyContinue |
  Select-Object -ExpandProperty OwningProcess -Unique |
  ForEach-Object { Stop-Process -Id $_ -Force }
```

# 现役文档主权

这个仓库当前业务现役文档只认 5 份：

1. 根目录 `AGENTS.md`
2. `docs/聊天记录.md`
3. `docs/项目原则.md`
4. `docs/AI自我学习.md`
5. `docs/项目结构.md`

除了这 5 份，其他文档一律视为历史资料，只能放在 `docs/归档/`，不再和现役文档并列发声。

另外单独保留 1 份根目录说明文档：

- `文档管理.md`
  - 这是整理文档终端的专用入口。
  - 它不负责主线业务判断，但负责告诉"整理文档的人"怎么整理、归档、压缩和对齐文档。

## 读取顺序

每次新开终端、断线恢复、主线切换前，都按这个顺序读：

1. `AGENTS.md`
2. `docs/聊天记录.md`
3. `docs/项目原则.md`
4. `docs/AI自我学习.md`
5. `docs/项目结构.md`

如果当前任务是"整理文档 / 对齐文档 / 归档文档 / 存档"，先看根目录 `文档管理.md`，按它的规则执行；不要自己重新发明一套文档整理流程。

`README.md`、`ARCHITECTURE.md` 以后都只当归档参考，不再当现役入口。

## 这个项目是什么

这是一个 AI 剧本创作桌面软件，不是普通文本生成工具。

正式主链只认这一条：

用户输入
-> 确认信息
-> `storyIntent`
-> 粗纲
-> 人物小传
-> 详细大纲
-> 剧本

系统的职责不是替创作做裁判，而是：
- 帮用户把输入结构化
- 给 AI 清楚的工序边界
- 保证每一层只做自己的事
- 防止第二口径、双来源、旧旁路回流

## 长期硬规则

1. 用户输入的剧本意图是唯一真相。
2. 不允许第二写口、第二发布口、第二裁判口。
3. 发现双口径，不打补丁，直接收一套。
4. 不做兜底。失败就报错，回根因层修。
5. 不要把创作问题修成 gate、命中词校验、重型 recovery、抽名补名单。
6. AI 主体负责创作，系统只做引导、结构化输入、轻量校验和最小必要机制。
7. 能用提示词、脚手架、工序边界解决的，不改成硬限制。
8. 旧逻辑、兼容逻辑、fallback、support、自动推进，只要重新掌权，就必须汇报并清掉。
9. 任务完成时，如果发现两套逻辑并存，要顺手收口，不允许留着等下轮继续反弹。
10. 默认只修原有功能，不借机发明新功能。

## 文档规则

1. `docs/聊天记录.md` 只保留一条当前主线。
2. 每累计 3 轮有效来回，如果当前终端正在推进主线，就更新一次 `docs/聊天记录.md`。
3. `docs/项目原则.md` 只记长期原则、禁区、典型误判和识别信号。
4. `docs/AI自我学习.md` 只记 AI 对项目的理解、踩坑、错误判断、有效做法和后续提醒。
5. `docs/项目结构.md` 只记仓库结构、工序结构、主链落点和归档布局。
6. 正在推进主线并完成一轮任务的终端，收口时必须顺手同步四件套：`docs/聊天记录.md`、`docs/项目原则.md`、`docs/AI自我学习.md`、`docs/项目结构.md`。
7. `docs/项目结构.md` 不是可有可无的整理活；只要主链落点、目录结构、风险区、归档布局变了，就必须同步更新。
8. `docs/聊天记录.md` 主要由正在干活的终端维护，不记录纯文档整理终端自己的整理过程。
9. `docs/归档/` 里的东西只保留历史证据，不再当现役入口。
10. 用户发"存档"时，这是额外强制整理口令；四件套都要同步检查、更新、压缩和归档。
11. 如果旧文档的结论已经被新事实推翻，不能继续保留在现役文档里。
12. 文档整理相关细则统一收在根目录 `文档管理.md`，整理文档终端默认先看它。

## 协作方式

1. 全程中文，讲大白话。
2. 先讲清本质，再讲动作。
3. 重大改动先给计划，再执行。
4. 发现之前判断错了，要明确写进文档，不要假装没发生过。
5. 不要拿"做了很多动作"冒充"解决了本质问题"。
6. 不要拿"某一层通过了"冒充"整条链通过了"。

## 进度汇报格式

滚屏和阶段汇报都用这个格式：

当前任务：
进度：`x/5`

规则：
- 滚屏时，进度放前面。
- 收尾时，进度放最后。
- 一旦到 `5/5`，要明确提醒任务做完了。

## 测试和工具纪律

1. 测试按板块测，不测整条 E2E 当当前验收。
2. 用完 MCP、浏览器自动化、测试工具后必须关掉，不留残进程。
3. 不在仓库外生成文件或目录。
4. 新建文件不要超过 500 行。
5. 新建文件只负责一个功能。
6. 不要在当前指挥终端运行带 `vite-node` 的命令；这类命令必须换到独立执行终端跑，否则当前终端容易直接崩掉。

## 开工前自查

每次开始做方案或改动前，先问自己 4 件事：

1. 是不是在加兜底。
2. 是不是把建议写成死限制。
3. 是不是越过工序边界。
4. 是不是让系统替代创作裁判。

只要有一条答案是"是"，就先停下，回根因层重做方案。
