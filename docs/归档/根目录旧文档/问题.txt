为什么 xinjuben 不能稳定产出优质内容
这是一份经过多维度交叉验证的分析——代码结构、计划文档、工作日志、与 wooo 的对标，全部看完了。
---
一、先破除一个数字幻觉
你说"完成了 100 个计划"，但真相是：
- docs/plans/ 里有 107 个计划文件
- docs/plans/completed/ 里只有 1 个文件（日期 2026-03-13）
这意味着：100+ 个计划被大量修改过、在 worklog 里被宣布"已完成"，但它们全部还躺在计划目录里，没有一个被正式关闭。 "100 个计划"实际上是"100+ 个被大量修改过的第一版计划"。
---
二、真实问题只有 5 个，它们在循环出现
从 worklog 的 628 行记录里，我追踪到了同一个循环跑了 20+ 轮：
根因	每次出现时叫什么	根因真的解决了吗
Prompt 块重叠	"5张卡叠在一起"、"故事补充/人物补充/阶段写法叠在一起"、"压缩块偷着重复定义"	解决了，但每次都要重新发现
AI 推测冒充正式事实	"observationSignals 当角色状态"、"关系张力=正式状态"、"chat 摘要越权主链"	部分解决，但没有固化
测试基础设施产生假失败	"36/35/45/50 集掉点"（实际是超时阈值、观察字段错误）	每次修完又会出现
三道工序边界串位	"剧本替粗纲补债"、"详纲替人物兜底"、"批内接力看不见自己"	解决了但没有守卫
内容太满/太解释性	"情绪段写成复盘"、"后段解释偏多"、"提要塞太多世界信息"	Ring 3 还没开始验证
这 5 个根因不是每次都真的被解决了——它们是被新的计划框架重新包裹了，然后又被宣称"这次解决了本质问题"。
---
三、最贵的幽灵：36 集问题
worklog 里大约 15+ 条记录在追逐"36/35/45/50 集掉点"这个所谓的产品问题。
最终结论：这不是产品边界问题，而是一条 测试测量链错误：
1. 超时阈值太短 → 报 timeout → 误判为"写不下去"
2. 观察口看错字段 → 误判为"项目状态错误"
3. 把以上假判断当证据 → 宣布"36 集是魔法数字"
真正的问题（16→25 中段发虚）到现在还没开始解决。
---
四、最核心的结构性问题：xinjuben 是原型，wooo 是工厂
这是两个项目最本质的差距：
 	wooo（能产出内容）	xinjuben（不能稳定产出）
架构	Worker 队列（BullMQ + Redis）	同步调用，主进程直连
任务	后台异步处理，可中断可续跑	同步等待，崩溃即丢失
质量门	60+ 个自动化检查	0 个
测试	60+ 个单元/集成测试文件	基本没有
编排层	Langgraph 风格，带重试+并发控制	简单线性批次
状态持久	MySQL + Prisma，任意节点可查	PocketBase 内存式
内容类型	多模态（文字→剧本→分镜→视频→语音）	仅文本
CI/CD	GitHub Actions + Docker 自动发布	本地开发
wooo 是一套生产内容工厂。xinjuben 是一个剧本桌面编辑器原型。
wooo 之所以能稳定产出，不是因为它的代码更聪明，而是因为它有：
- 任务队列（不丢活）
- 自动化质量门（不靠人眼验收）
- 完整测试（不靠试运行发现 bug）
- 编排层（出错可重试）
xinjuben 的 47 个剧本生成文件、5 张卡接力棒、账本系统、事后校验——这些设计都很聪明，但它们全都在一个没有质量守卫、没有重试机制、没有后台处理的同步调用链里跑。任何一步出错，整个链条就断。
---
五、更深的问题：计划循环，而不是知识积累
我观察到的是一个 "计划-发现-再计划"循环，而不是知识积累：
计划 A（第一版）
  → 工作中发现新问题
  → 宣布"A 已完成，发现本质问题 B"
  → 计划 B（第一版）
    → 工作中发现新问题
    → 宣布"B 已完成，发现本质问题 C"
    → 计划 C（第一版）
      ...循环 20+ 次
问题不是没有进展——代码确实在变好。问题是：
1. 同样的根因被不同名字反复出现。A+B 公式被"发现"了 3 次。批内接力问题被诊断了 4 次。
2. 没有固化机制。一个根因被解决后，没有自动化守卫防止它回弹。测试基础设施修好后会再次产生假失败。
3. 计划不关闭。11+ 个计划有"第二版"，但第一版从未正式移到 completed/。
4. 最终交付没有验证。Ring 3（剧本质量）从未正式跑过。Ring 1 和 Ring 2 的判断是"可用但不够好"——但这个判断是工作日志里的主观结论，没有被自动化测试固化为"已验证"。
---
六、本质诊断（一句话）
> xinjuben 有一个聪明的内容生成引擎，但没有生产质量守卫。
它可以产出一个好剧本——但它没有办法保证每一个剧本都好，因为它：
- 没有后台队列（出错即断）
- 没有自动化质量门（靠人眼判断"够不够好"）
- 没有测试（靠试运行发现问题）
- 没有环形守卫（根因被解决后会回弹）
---
七、建议（本质层面的）
如果你真的想稳定产出优质内容：
1. 立即可做：把 Ring 3 正式跑完
当前任务卡 Ring 3 进度 8%——这是唯一衡量"能不能稳定产出"的指标。跑完它。用自动化脚本跑，不是人眼验收。
2. 加上自动化质量门
wooo 的 60+ 个检查不是过度工程——它是让内容工厂能稳定产出的基础设置。至少要有：
- 输出格式验证（JSON schema）
- 重复检测（语义哈希）
- 字数/场数契约检查
- 端到端集成测试（Playwright）
3. 固化知识，而不是重新发现
下一个计划开始前，先回答：这个问题在之前哪个计划里出现过？结论是什么？为什么它会重新出现？
4. 关闭计划，而不是创建新计划
把 completed/ 文件夹用起来。一个计划只有被移到 completed/ 才算完成。
5. 长期：考虑后台任务架构
如果目标真的是"稳定产出"，而不是"原型演示"，需要把同步调用链改成后台队列。BullMQ + Redis 不贵，但它能解决"崩溃即丢失"和"无法续跑"这两个致命问题。
