## 1. 一句话结论

这次“点开项目就卡死、按钮要等很多秒才反应”的本质，不是单个按钮慢，也不是某一个 hook 慢，而是项目打开链在前几轮重构之后，仍然保留了错误的后半段结构：

- hydrate 后没有 clean/dirty 边界
- `currentStage` authority 没有彻底收死
- 存储仍然是单仓整对象思维
- 大剧本 / 大详纲页面的渲染模型还是一次性重挂大内容
- 性能观测口径没有形成统一标准

前几轮之所以看起来“修了又卡回来”，是因为修到一半就停了，停在了上层入口链和局部 provider，没有继续往下追到编辑态边界、stage 边界、存储边界、渲染边界、观测边界。

### 根因 1：hydrate 之后没有 clean/dirty 边界

这是一切“打开项目后又开始写盘”的上游根因。

前面已经清掉了 `DetailedOutlineStage` 的一段 mount auto-save，但系统层面仍然没有统一回答：

- 哪些数据是刚 hydrate 进来的只读状态
- 哪些数据是用户真实编辑后的脏状态
- 哪些保存属于 AI 生成链显式保存
- 哪些保存属于页面编辑态保存

没有这个边界，就算 `D1`/`D2` 把几个 effect 删了，后面仍然很容易重新长出：

- hydrate 即 dirty
- dirty 即保存
- 切页也保存
- 看一下也保存

这就是为什么 `D3` 不是“体验优化项”，而是后半段结构治理的第一步。

### 根因 2：`currentStage` authority 没有彻底形成唯一真相

前面虽然已经做过项目打开协议和 renderer hydration 第一刀，但当前 renderer 侧 `currentStage` 仍然没有真正收死：

- header、sidebar、viewport 都依赖它
- 多个页面都在发 stage 跳转请求
- 但没有统一 renderer 写口和统一同步入口
- 还存在“Store updated via IPC”这类事实上不成立的注释

这会直接导致两个后果：

1. stage 视觉状态和真实数据状态可能漂移
2. 页面还是容易带着“补读别页数据”的旧习惯生长

所以 `E1-E3` 的任务不是“让 stage 显示更准一点”，而是让 stage 重新成为真正的数据加载边界。

### 根因 3：存储层仍然鼓励整项目思维

虽然前面已经加过：

- lock timeout
- debounce flush
- store cache
- broadcast 节流

但底层仍然是：

- 单 `projects.json`
- 整个 `store.projects` 对象
- `updateProject(projectId, updater)` 这种整项目更新模型

这会带来一个长期性问题：

- 只要底层还是整项目对象思维，上层就会不断长出“读整份项目”“改整份项目”“写整份项目”的路径
- 于是前面做过的 session 化、payload 化、provider 化，很容易再次被打回旧习惯

所以 `F1-F2` 不是“顺手做个文件拆分”，而是把错误的地基拆掉。

### 根因 4：大页面渲染模型没有真正收口

即使数据链前面变轻了，大项目仍然会卡，原因在于 renderer 页面本身仍然偏重：

- 剧本列表还是全量挂载
- 详纲长表单还是全量挂载
- 长文本编辑器和全文展示还在放大 DOM
- 重型派生逻辑还会跟大数组一起重算

前面如果只改了数据读取、不改渲染模型，最后用户看到的还是：

- 鼠标能点，但滚动照样卡
- 进入页面没死，但展开就卡
- 页面切换恢复了一点，长内容操作依旧拖死

所以 `G1-G2` 不是锦上添花，而是结构治理的最后一刀。

### 根因 5：没有统一性能观测口径

前面重构几轮，有一个很明显的问题：

- 每一轮都说“应该快了一点”
- 但没有统一口径能看清楚到底慢在：
  - 主进程读慢
  - stage payload 慢
  - hydrate 慢
  - first interactive 慢
  - 还是大页面 mount 后慢

没有这个口径，后面每一轮都会退回靠体感争论。这样即便真修对了，也会浪费很多时间在“到底是不是这里”的争论上。

所以 `H1` 不是附属工具，而是防止下一轮再次回到“猜”的必要收口。

## 10. 工作包 D3：建立 dirty state 模型

### 10.1 目标

让系统明确区分：

- hydrate 装载
- 用户真实编辑
- AI 生成链显式保存

### 10.2 这包解决哪条根因

它解决的是“hydrate 后没有 clean/dirty 边界”这条根因。

如果不先做它，后面不管你怎么改 stage authority 或存储，都还会继续長出：

- 刚装载就脏
- 脏了就写
- 打开项目即写盘

### 10.3 必改文件

- `src/renderer/src/app/save-dirty-state/*`
- `src/renderer/src/features/outline/ui/OutlineStage.tsx`
- `src/renderer/src/features/character/ui/CharacterStage.tsx`
- `src/renderer/src/features/detailed-outline/ui/DetailedOutlineStage.tsx`
- 如有必要：相关 editor 组件

### 10.4 必做动作

1. 建统一 dirty registry
2. 每个 stage draft 都要有 `hydratedVersion` 或等价版本标识
3. hydrate 后默认 clean
4. 用户真实输入后才 dirty
5. dirty save 只能绑定：
   - 当前项目 id
   - 当前 stage
   - 当前 hydratedVersion
6. AI 生成成功后的持久化继续走生成链显式保存，不能混进页面 dirty-save
7. 清掉所有“刚 hydrate 就被误判成改过”的路径
8. 明确区分“replace from hydrate”和“user edit”两类 action

### 10.5 推荐做法

1. 在 stage store 或单独的 dirty manager 中建立：
   - `markHydrated(stage, version)`
   - `markDirty(stage, version)`
   - `isDirty(stage, version)`
   - `clearDirty(stage, version)`
2. editor 组件只在用户输入事件里调用 `markDirty`
3. hydrate 路径只调用 `markHydrated`，不能顺手触发保存
4. 保存完成后清理 dirty 状态，且必须校验版本仍匹配

### 10.6 主要失败点

1. 只加 dirty 标志，但 hydrate 和用户输入仍走同一 action
2. 保存后不校验版本，导致旧保存覆盖新状态
3. 只在一个 stage 做 clean/dirty，其他页继续保留灰色旧逻辑

### 10.7 完成标准

- 打开旧项目不编辑，不触发保存
- 切页不触发保存
- 查看现有内容不触发保存
- 真实编辑后才保存
- AI 生成后的显式保存链仍正常工作

### 10.8 必交证据

1. dirty registry 的入口文件
2. hydrate clean 和 user dirty 的调用点列表
3. 删掉的旧自动保存路径
4. 最小范围确认：打开项目后不编辑，确认无保存日志
