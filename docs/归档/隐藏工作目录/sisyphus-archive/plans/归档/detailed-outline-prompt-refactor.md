# 详细大纲 Prompt 根因重构计划

## TL;DR

这次不是改几个“不要/禁止”就能好。

详细大纲现在跑歪，根因有两层：

1. 这道工序职责写胖了  
它本来只该把“粗纲逐集事件 + 人物抓手”展开成“可写剧本的逐集细纲”，现在却被塞进了上层哲学、宣传语要求、四段总述、情绪兑现、卖点维护、阶段职责教学、反例教学。

2. 这道工序输出写重了  
它一次要同时交“四段阶段总述 + 全部逐集细纲”。这等于让模型一边写总表，一边写明细，还得两边互相对齐。60 集时最容易先空转、再偷懒、最后超时。

所以这份计划不再以“删禁止词”为目标，而是直接把详细大纲这道工序压回它该做的事：

- 输入只留这道工序真正要吃的事实
- 任务只写这道工序真正要完成的展开动作
- 输出先保逐集细纲，四段总述降级为派生物，不再和逐集细纲抢同一轮脑力

---

## 这次要解决的真问题

不是“prompt 里有禁止词”。

是这 4 个真问题：

1. `详细大纲` 没守住工序边界
2. prompt 不是 A+B+C，而是 A+上层哲学+写作说明书+反例教学+双重交付合同
3. 模型被要求同时写“阶段总结”和“逐集展开”，注意力被拆开
4. 60 集时输出体量太大，模型更容易偷成空话、漏集、缩集，甚至直接卡超时

---

## 证据

### 1. 当前 prompt 真实职责已经越界

当前 [`generation-stage-prompts.ts`](/D:/project/xinjuben/src/main/application/workspace/generation-stage-prompts.ts) 里的 `buildDetailedOutlinePrompt`，除了给输入和输出合同，还额外塞了这些东西：

- 让四段都持续吃住“设定成交句、核心错位、情绪兑现”
- 要写“最值钱的大白话”“最像宣传片剪辑的一句”
- 教模型四段分别该怎么写
- 教模型什么叫空话、什么叫说明书口气
- 同时要求“阶段推进图 + 逐集细纲”双交付

这些话不全是假，但很多不该放在这道工序里。

### 2. 当前消费链本身就说明“四段总述”不是主任务

[`generate-detailed-outline.ts`](/D:/project/xinjuben/src/main/application/workspace/generate-detailed-outline.ts) 里真正被下游需要的是：

- `episodeBeats`
- 四段 `content`

但从链路看，四段 `content` 本质上只是这段的概述；真正决定后面剧本怎么写的，是每集 `episodeBeats`。

更关键的是，代码已经有 fallback：

- `buildFallbackSegments`
- `buildFallbackEpisodeBeats`

这说明系统自己也默认承认：阶段总述和逐集细纲并不是同一个难度层，系统已经在偷偷帮模型兜这两份活。

### 3. 当前详细大纲输入不是“必要事实”，而是“混合讲义”

当前 prompt 输入同时塞了：

- 粗纲总摘要
- 更上层推进合同
- 人物执行说明书
- 四段职责说明
- 情绪兑现/卖点/错位要求

这不是“给详细大纲喂事实”，这是“把上游所有想法一起重新讲一遍”。

模型最容易出现的结果就是：

- 不是沿着粗纲逐集展开
- 而是沿着 prompt 里最响的那些口号去凑大话

### 4. 60 集时，双重交付天然更容易超时

当前合同要求一次性输出：

- opening.summary
- midpoint.summary
- climax.summary
- ending.summary
- 以及四段下全部 episodes

60 集时，这不是单纯“多写一点”，而是“同一轮同时维护两套结构”。

大白话说：

它一边要写总目录，一边要写 60 条分目录，还要保证两边能对上。  
这很容易先把脑子用在“怎么讲阶段大话”上，后面逐集就开始缩水、重复、偷懒，最后直接拖慢甚至超时。

---

## 根因判断

这次根因不是“文风不对”，而是“工序定义错位”。

第一次违反原则的地方，就在 `buildDetailedOutlinePrompt` 这里：

- 它没把自己当“展开工序”
- 它把自己写成了“上层创作哲学执行器 + 阶段总结器 + 逐集细纲器”

一旦这里定义错了，后面不管怎么删几个词、改几个句式，都还是同一个错。

---

## 重构目标

这次只做一件事：

把 `详细大纲` 压回“逐集展开工序”。

交付标准改成下面 3 条：

1. 模型看到 prompt 后，第一反应是“把每一集从粗纲展开一点”，不是“先写四段创作说明书”
2. 60 集时，输入强度和任务类型不变，只是集数变多，不再额外叠加教学负担
3. 下游剧本拿到的是稳定、连续、按集可改的 `episodeBeats`，不是一堆阶段大话

---

## 新的工序边界

### 粗纲负责什么

- 每集主事件推进
- 关键阻力
- 每集钩子
- 大方向升级

### 人物小传负责什么

- 前史
- 动机
- 怕失去什么
- 会怎么施压
- 会在什么条件下行动

### 详细大纲只负责什么

- 把粗纲每一集再展开半步
- 补出这一集更具体的动作、压强、代价、转折口
- 把人物小传里的“行动抓手”挂进这一集
- 让下游剧本知道“这集具体怎么写”

### 详细大纲不再负责什么

- 不再兼任上层创作哲学宣讲
- 不再兼任卖点口号训练
- 不再兼任四段写法教学
- 不再同时承担“总表 + 明细表”双主任务

---

## 新的 A+B+C

### A：只留必要事实

详细大纲真正该吃的 A，收口成两块：

1. `粗纲逐集`
   - 每集 1 条
   - 这是主输入

2. `人物行动抓手`
   - 只留会影响逐集推进的字段
   - 例如：`goal / fear / protectTarget / conflictTrigger / weakness`

以下内容不再作为详细大纲主 prompt 的主输入：

- 大段“更上层推进合同”
- 宣传语式说明
- 四段职责教学
- 大段反例教学
- 重复解释“设定成交句/核心错位/情绪兑现”

这些不是完全没用，而是不该挤在这道主工序里。

### B：只写当前动作

当前任务压成一句大白话：

把粗纲每一集展开成“可直接写剧本”的逐集细纲，并把人物抓手落进每一集的动作、压强、代价和钩子里。

B 段只保 4 个意思：

1. 按集展开，不漏集、不并集
2. 每集比粗纲更具体一点
3. 把人物抓手落进动作，不写空情绪
4. 每集结尾留出下一集的逼压口

### C：只保最小合同

主合同先收口到逐集结果，不让模型同时扛两张表。

推荐主合同：

```json
{
  "episodes": [
    {
      "episodeNo": 1,
      "summary": "string"
    }
  ]
}
```

如果 UI 或下游仍然需要四段视图，走派生路线：

- 方案 A：系统按集数自动分段
- 方案 B：逐集结果出来后，再做一轮轻量总结

但四段总述不再和逐集细纲绑成同一次主生成。

## 方案取舍

### 主方案：逐集细纲单主任务

做法：

- 详细大纲主 prompt 只生成 `episodes`
- 四段分组由系统根据集数自动切
- 四段摘要由系统派生，或第二轮轻量总结

为什么选它：

- 最对工序边界
- 最能减负
- 最利于 60 集稳定
- 用户按集改最直接

### 备选方案：保留四段，但降成系统派生物

做法：

- 主生成仍然只产逐集细纲
- 系统把逐集细纲按四段范围归组
- 每段 summary 用程序拼，或者在结果稳定后补一轮压缩总结

为什么它比当前方案强：

- 四段视图还在
- 但不再抢主生成的脑力

### 明确不选的方案

1. 只删“禁止写法”几个字
原因：表层修辞，不改职责错位

2. 继续在现有 prompt 上增删规则
原因：还是在写说明书，不是在减负

3. 继续让一轮模型同时生成“四段 + 逐集”
原因：60 集时天生更重，根因没动

## 执行步骤

### 第 1 步：先做工序收口

目标：

- 删掉详细大纲 prompt 里不属于这道工序的内容

具体处理：

- 删四段写法教学
- 删反例教学
- 删大段卖点/宣传语要求
- 删重复上层合同说明

保留：

- 粗纲逐集事实
- 人物行动抓手
- 逐集展开任务
- 最小输出合同

### 第 2 步：改输出合同

目标：

- 让详细大纲主输出只保 `episodes`

具体处理：

- `buildDetailedOutlinePrompt` 改成只要逐集数组
- `generate-detailed-outline.ts` 改为先接主结果，再做四段派生

### 第 3 步：把四段视图改成派生层

目标：

- 四段不再由主模型同时创作

具体处理：

- 用 `buildFourActEpisodeRanges()` 切段
- 每段 `episodeBeats` 由 `episodes` 归组得到
- 每段 `content` 先走稳定派生方案

派生优先顺序：

1. 先用程序从该段 episodes 压出一句摘要
2. 如果程序摘要太差，再单开一轮轻量 summary，不污染主链路

### 第 4 步：清 fallback 角色

目标：

- fallback 只做兜底，不再替主链路偷做主任务

具体处理：

- `buildFallbackEpisodeBeats` 保留
- `buildFallbackSegments` 降成真 fallback
- 不再让主 prompt 本身跟 fallback 的职责重叠

### 第 5 步：真测只看两件事

1. 60 集详细大纲能不能稳定全部产完
2. 每集是不是都能直接喂给剧本，而不是一堆空话

---

## 验收标准

### 结构验收

- `buildDetailedOutlinePrompt` 只保 A+B+C
- A 里没有上层讲义堆积
- B 里没有阶段写法教学
- C 里主合同只保逐集细纲

### 结果验收

- 10 集能完整产出全部逐集细纲
- 60 集能完整产出全部逐集细纲
- 不再出现只写出 4 段空概述、逐集却缩水的情况

### 质量验收

每集至少能直接看出这 4 件事：

- 这集具体发生什么
- 谁在施压
- 主角或关键人物付了什么代价
- 下一集为什么还要点开

### 根因验收

如果最后只是“禁止词变少了”，但：

- prompt 还是很胖
- 还是双重交付
- 还是靠四段教学在写

那就算没修到根上。

---

## 风险

### 风险 1

四段总述如果立刻取消，UI 可能一时少一层概览。

处理：

- 先做派生版四段视图，不让 UI 断

### 风险 2

旧代码很多地方默认拿 `opening/midpoint/climax/ending`

处理：

- 兼容期先保留四段结构
- 但四段内容来源改成派生，不再要求主模型同时生成

### 风险 3

如果粗纲本身质量太空，详细大纲再怎么展开也会空

处理：

- 这不属于这次详细大纲主根因
- 需要单独回粗纲工序验收，不在这里偷补

---

## 最后一句话

旧计划的问题，是把“详细大纲不会写”误判成“提示词措辞太硬”。

新计划打的是更前面那层：

不是先问“哪些词该删”，  
而是先问“详细大纲这道工序到底该干什么，不该干什么”。

只有先把这层掰正，后面的 prompt 才有可能真的稳定。
