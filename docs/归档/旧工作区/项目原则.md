# 项目原则

这个文件只记录已经确认的长期原则、禁区和验收底线。

如果出现新的稳定原则，就补进来；如果旧原则过时，就直接替换，不保留第二口径。

## 当前有效原则

1. **用户输入是唯一真相**
   - 用户在题材、集数、人物、冲突、关系上的最新明确输入，是唯一真相。
   - 不能同时保留第二写口、第二发布口、第二裁判口。

2. **系统不替代创作裁决**
   - AI 主体负责创作，系统只负责引导、结构化输入、轻量校验和最小必要机制。
   - 不要把创作问题修成门禁、兜底、抽名补名单、重型 recovery 或补丁链。

3. **不要兜底，错了就报错**
   - 这个项目追求优秀作品，不追求“表面不报错”。
   - 真失败就报错，回根因层修，不要用保底把问题盖掉。

4. **发现双口径，不打补丁，直接收一套**
   - 新逻辑已经成立后，旧逻辑、双路径、双入口、双真相必须删一个。
   - 不能在旧链上再叠补丁，否则系统会一直来回回退。

5. **AI 运行时也不能双主权**
   - AI provider、模型 lane、运行时策略都属于系统口径。
   - 当前正式运行时已经确定为 `DeepSeek` 单通道，就不能再保留另一套“随时切换”的正式路线。

6. **创作真相字段不能二次裁决**
   - 集数、主角、对手、核心冲突、关键关系这类创作真相，不能再用 `Math.max`、默认值、fallback 或猜值回填。
   - 上游缺失就报错，不要让系统替用户决定。

7. **确认信息阶段只能保留一份正式合同**
   - 确认信息不能再让模型同时输出两套都像正式真相的对象。
   - 正式持久化只留一份真对象；展示文本如果要有，就从这份对象本地渲染。

8. **粗纲/人物阶段不能二次确认聊天**
   - “生成粗纲和人物”只能吃已经确认过的 `storyIntent`。
   - 不能因为当前项目里没对上，就在生成入口里自己再 `summarizeChatForGeneration()` 一次。

9. **系统不能在落盘前补写正式创作事实**
   - 粗纲里的 `facts`、人物里的核心设定，要么来自用户真相，要么来自该阶段 AI 正式输出。
   - 不允许系统在落盘前再拼一批“看起来更完整”的正式事实。

10. **同一个字段只能有一个权威来源**
   - 如果某个字段已经声明“唯一来源”，那就不能再在磁盘、主进程、renderer、兼容层各留一份平行版本。
   - `activeCharacterBlocks` 这种字段尤其要收死；不能一边说 renderer 派生是唯一来源，一边又持久化和回读另一套快照。

11. **上游放行口径和下游验收口径必须一致**
   - 同一份阶段产物，不能在上游用宽口径放行，到下游再用更严的一套合同拦住。
   - 这样会制造“这一步明明成功了，下一步却说基础不够”的假稳定。

12. **自动推进不能跨越工序边界偷偷开工**
   - 自动推进如果会在用户还没确认阶段稳定前，就后台继续拉起下一轮重 AI 任务，那就是另一条隐性生成链。
   - 自动推进只能在明确边界内工作，不能制造后台第二主线。

13. **提示词优化优先于限制**
   - 生成质量问题优先改提示词、脚手架和工序边界。
   - 不要第一反应去加限制词、命中词校验和硬门槛。

14. **读取链不能悄悄改正式真相**
   - `repair / normalize / migrate / compatibility / alias` 这类逻辑只能修结构，不能顺手补写创作内容。
   - 发现读取链在改正式真相，要当成严重问题处理。

15. **当前工作区只保留三份有效文档**
   - 当前工作区固定只保留 `聊天记录.md`、`项目原则.md`、`AI自我学习.md`。
   - 旧任务卡、阶段验收单、一次性台账如果还有价值，先吸收结论，再归档原文。

16. **断线恢复顺序固定**
   - 恢复顺序只认：`聊天记录.md -> 项目原则.md -> AI自我学习.md`。
   - 如果三份文档不一致，以聊天记录当前主线和这份原则为准，立刻整理掉第二口径。

17. **真实复测必须先清污染**
   - 做真实软件复测前，先清掉旧测试项目、旧日志或改用隔离 `userData`。
   - 不允许拿已经混入旧粗纲、旧人物、旧角色块的数据继续当验收依据。

## 当前重点盯防

- 不要让粗纲入口重新长出“二次确认聊天”的隐性入口。
- 不要让 `outline-facts`、`enrich`、`fallback` 这类系统补写重新混进正式真相。
- 不要让 `generationBriefText` 再变成比正式 `storyIntent / outline` 更高一层的第二真相。
- 不要让 `activeCharacterBlocks` 继续一边派生、一边持久化、一边被下游消费。
- 不要让人物生成和详纲验收继续用两套合同。
- 不要让自动推进继续在后台制造第二条生成链。
- 不要在旧脏项目上反复测，然后把污染结果当成新代码问题。
