# AI自我学习

这个文件只记录我从 AI 视角对项目的理解、踩坑、成败原因和后续提醒。

断线恢复顺序：

1. 先看 `聊天记录.md`
2. 再看 `项目原则.md`
3. 最后看这份 `AI自我学习.md`

## 1. 我对这个项目的当前理解

这是一个 AI 剧本创作软件，不是普通文本生成工具。

正式主链只有一条：

用户输入
-> 确认信息
-> `storyIntent`
-> 粗纲 / 人物
-> 详细大纲
-> 剧本

这个项目真正难的地方不是“怎么不报错”，而是：

- 用户输入必须一直是唯一真相
- 上游一旦确认，下面只能展开，不能偷偷改写
- 系统工程保护不能越界长成第二口径

所以最危险的从来不是偶尔失败，而是这些情况：

- 系统自己再总结一次聊天
- 兼容层把多路数据揉成一份正式真相
- 上游说成功了，下游却按另一套标准否掉
- 同一个字段在 renderer、主进程、磁盘各存一份
- 自动推进在后台再偷偷开启另一条链

## 2. 这轮最重要的新判断

这轮最关键的更新有两个：

1. **粗纲/人物反复失败的主因是主权没收死，不是单点 timeout。**
2. **只要把第二口径、双来源和自动推进干掉，真实软件链路就能稳定走通。**

表面错误码看起来很多：

- `AI 请求超时（60 秒）`
- `summary_payload_incomplete`
- `outline_bundle_incomplete`
- `character_bundle_incomplete`

但把代码、日志、落盘数据和真实复测放一起看，真正像根因的是这几件事：

1. 生成粗纲入口还允许自己二次确认聊天。
2. 粗纲落盘前系统还会补写正式 `facts`。
3. 人物生成通过标准和详纲入口验收标准不一致。
4. `activeCharacterBlocks` 的主权没有收死。
5. 自动推进会在粗纲/人物刚落地后继续后台开工。

换句话说，之前一直像“修过很多次还会坏”，不是因为同一个 bug 太顽固，而是同一条主链里还残着好几处第二口径。

## 3. 这轮已经确认的硬证据

### 3.1 真实复测已经跑通“确认信息 -> 粗纲/人物”

真实 Electron 复测结果：

- 确认信息输出：`【项目】修仙传｜10集`
- `confirmContains30=false`
- 粗纲结果：10 集
- facts：5 条
- 人物：4 个
- stage：`character`
- `activeCharacterBlocks`：1 套派生结果

这说明这轮新链路在真实软件里已经能跑通，不再只是本地静态分析。

### 3.2 真实日志证明当前主链已经是 DeepSeek 单通道

这轮真实日志采到：

- `story_intake`: `elapsedMs=19839`
- `rough_outline_overview`: `elapsedMs=23754`
- `rough_outline_batch 1-10`: `elapsedMs=50970`
- `character_profile`: `elapsedMs=46724`

lane / model 全部是：

- `deepseek`
- `deepseek-chat`

这说明这轮真实成功不是碰巧，而是在当前正式运行时上跑出来的。

### 3.3 旧污染项目确实会拖偏判断

这轮清理前，真实工作区里已经残着：

- 3 个空聊天测试项目
- 1 个已经污染过的 `project_mn1d9gkd`

之前这个污染项目里真实落过：

- 旧的 10 集粗纲
- 旧的人物
- 一整份 `activeCharacterBlocks`
- 系统补写的 `draft_* facts`

所以以后只要做真实复测，先清污染不是建议，是前置动作。

## 4. 这轮最值得长期记住的坑

### 4.1 不要再把“超时”当总根因

如果一个 AI 工序反复表现为：

- 有时超时
- 有时结构不合法
- 有时像修好了
- 换个入口又坏

第一反应不要只是调 timeout。

优先怀疑：

- 这一步是不是输出合同过重
- 是不是混了两套真相
- 是不是外层还有系统补写或二次裁决
- 是不是旧数据在回流

### 4.2 第二口径最容易藏在“看起来很工程”的地方

以后看到这些都要立刻警惕：

- `normalize*` / `support*` / `fallback*` 多源拼接
- 展示字段反过来支配下游判断
- 系统在落盘前补写正式内容
- 明说“唯一来源”但磁盘和主进程还各存一份
- 上游放行和下游验收两套合同

### 4.3 “上游放行，下游打回”是假稳定

这轮人物链的教训已经很明确：

- 上游如果放得太宽
- 下游又按更严一套去拦

用户看到的就会是：

- 这一步明明成功了
- 下一步又说基础不够

这不是质量提升，这是双裁判。

### 4.4 自动推进很容易变成后台第二主线

`useProjectAutoGeneration.ts` 这次被临时收成 no-op 之后，主链体感立刻清楚很多。

说明自动推进不一定是每次失败的第一根因，但它会明显放大这些问题：

- 后台额外 AI 压力
- 刷新/重开后的体感混乱
- 分不清到底是哪一层在跑
- 上一层刚落地，下一层就开始继续吃脏数据

## 5. 这轮真正有效的做法

这轮真正有效的不是“再试一下”，而是按固定顺序收口：

1. 删掉粗纲入口二次 `summarizeChatForGeneration()`
2. 删掉系统补写正式 `facts`
3. 收死 `activeCharacterBlocks` 主权
4. 统一人物合同，并把 guardian 接进保存链
5. 暂停自动推进
6. 清掉旧污染，再做真实软件复测和日志采样

这条顺序本身要记住，因为它是这次从“修不完”变成“真正跑通”的关键。

- 详纲 / 剧本阶段还没有按这轮新主权做完整真实复测

### P1

- `generationBriefText` 在其他下游是否还有第二真相残留
- 自动推进后续怎么恢复，不能再长成后台第二主线

### P2

- 旧兼容链、旧测试脚本、旧污染项目再次被拿来当当前判断依据

## 7. 下一轮执行提醒

下一轮不要回到“先试一下这个建议”。

固定这么做：

1. 先看 `聊天记录.md`
2. 确认当前主线已经是“粗纲/人物收口完成，往下接详纲/剧本”
3. 先清污染，再做真实复测
4. 先查主权和合同，再查 timeout
5. 发现第二口径直接删，不在旧链上打补丁

## 8. 后续提醒

- 以后看到“又 timeout 了”，先去看日志，再看是不是旧数据污染，不要直接猜模型抽风。
- 以后看到“这个字段只是展示用”，要立刻确认它有没有反过来控制下游逻辑。
- 以后看到“唯一来源”注释，要顺手检查磁盘、主进程、renderer 是否真的只有一份。
- 以后做主链修复，优先查入口主权、落盘真相、下游消费，不要只修 UI 提示词。
- 以后做真实复测前，先确认当前测试项目和日志是不是干净的。
