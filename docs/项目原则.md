# 项目原则

## 核心原则

1. 用户输入是唯一真相，系统不替代创作裁判。
2. 发现双口径不打补丁，直接收一套。
3. 项目不要先补丁，要修源头问题。
4. 不允许第二写口、第二发布口、第二裁判口。
5. 系统只做引导、结构化输入、轻量校验和最小必要机制。
6. 写作逻辑采用 `A+B+C`：稳定内核 A 不随市场变化乱动，市场打法包 B 可版本化迭代，项目设定 C 来自用户输入。

## 工程原则

1. server 是唯一正式业务入口。
2. PocketBase 是唯一正式项目真相存储。
3. renderer 正式业务只准走 HTTP `api-client`。
4. IPC 只保留桌面壳职责；HTTP 对齐后，对应 IPC 必须物理退场。
5. 退役业务 IPC 不能只留 deprecated handler；必须同步删除 handler、main 旧实现、旧测试和零引用 DTO。
6. `src/shared/` 是前后端唯一共享合同层，严禁双轨副本。
7. 假迁移不算迁移——把 authority 从一个 main 文件挪到另一个 main 文件不是真迁移，必须后移到 server 或下沉到 src/shared/。
8. "server 有壳但 main 才是真逻辑"不算后移——server 端路由如果只是 TODO/占位，main 端旧实现就不是真正退场。
9. 允许精确进程管理；杀进程前必须确认 PID、命令行、用途，禁止广谱杀进程和误杀 AI 宿主/当前终端。
10. UI 性能是工程验收项，不是上线后再看的体验项；任何新增卡片、复制按钮、动效、生成状态、列表渲染，都必须确认没有明显卡顿。
11. renderer 不得在 render 阶段构建大文本、大 JSON 或全页派生数据；复制、导出、预览这类重计算必须点击时或 `useMemo` 边界内执行。
12. 全局弹窗（登录、充值、支付、错误拦截）必须 portal 到 `document.body` 并使用全局最高层级；不能挂在 header/卡片/阶段面板内部被局部 stacking context 压住。

## 创作架构原则

1. A 稳定内核包括平台安全、剧本格式、每集施压/反击/钩子、主角不窝囊、反派不降智、连续性、七问和剧本骨架合同。
2. B 市场打法包包括每月热门剧本样本提取出的开局压迫、爽点兑现、钩子、反派压迫、主角反击、道具承载信息等可变打法。
3. C 项目设定包括题材、男频/女频、垂类、人物、世界观、集数和用户特殊要求。
4. 优秀剧本样本不能整段长期塞进 Prompt；必须先提炼为可审核、可版本化的 MarketPlaybook，再接入生成和质检。
5. 新打法包不能覆盖稳定内核；市场变化只改 B，除非用户明确要求重构 A。
6. B 层当前已完成资产层生命周期（合同/注册表/版本化/草案提取/激活归档）、Prompt 集成（七问/剧本骨架/正式剧本三阶段注入）和质量观测（`playbookAlignmentScore` 纯观测信号，不进 overallScore，不触发修稿）。修稿链尚未接入 playbook——不得假装已完成修稿集成。
7. 市场学习闭环必须低成本先验证：样本→草案→人工审核→启用→Prompt→观测；未经过审核的 draft 不能自动进入 active，观测分不能自动变成修稿裁判。
8. 审核工作台生成的 active JSON 只是预览资产；没有数据库持久化和 server 解析前，不能宣传为“已全局启用”。
9. 20 集短剧不要强行套长篇势力矩阵；人物和势力只生成会直接制造冲突、递线索、压主角或反转立场的角色。
10. 人物小传/剧本骨架只维护一套账本：`storyIntent` 是正式输入，七问候选只能辅助审核，不能再成为第二套硬前置。
11. AI 返回坏 JSON 时可以保住已成功资产，但必须给用户可见 warning；临时骨架不能伪装成高质量正式骨架。
12. 人物字段兜底只能补结构缺口，不能把不同人物补成同一套模板；若出现重复句式，优先修补全源头。
13. 轻量人物卡是短剧人物编制的一部分；完整小传被限制后，剩余实名人物或势力空席必须可见，不能在 UI 上消失。
14. 人物补全模板禁止硬编码项目外实体名；所有家族、主角、对手、阵营名都必须来自当前项目输入或角色摘要。
15. 复制出来的阶段文本也是验收面；重复标点、重复字段、`表面是X是` 这类低级拼接问题必须按 bug 处理。
16. 功能席位不是纯占位文案；轻量卡至少要写出能被剧情调用的差异化职责目标。
17. 完整人物小传不能是字段硬拼；展示给用户的 `biography` 必须是自然段，人物弧线必须包含触发条件、心理摇摆、代价选择和终局变化。
18. 人物字段里的 `conflictTrigger` 与 `advantage` 必须能直接转成剧本场面，不能停留在抽象性格判断。
19. 人物小传自然段必须点名人物，并融合外在形象、身份处境、性格/价值观、隐藏压力和剧中动作方式；只生成字段但展示不可见，按质量失败处理。
20. 人物弧线只要缺少起点、触发、摇摆、代价、终局任一环，就必须被清洗层重写，不能把一句结局当弧线。
21. 复制/导出文本是正式验收面；人物五维字段、自然小传和结构化弧线必须在复制结果里可检查。
22. 功能席位的差异化职责必须在 server 与 `src/shared` 同步，不能让共享层重新生成“撑住这条线”式泛目标。
23. 势力成员表只能记录人物主归属；跨势力利用、卧底、安插和对抗必须写到关系层，不能把同一个实名人物复制到多个势力里。
24. entityStore 构建时 V2 `factionId` 高于势力矩阵占位名；若两者冲突，优先相信人物 profile 的主归属并去重同名实体。
25. 已升完整人物不能再出现在轻量人物卡；即使旧底账里 masterEntityId 不一致，也要按人物名兜底隐藏。
26. 人物小传/剧本骨架链路必须先产出 canonical bundle，再向旧展示结构投影；旧接口只能兼容调用，不能继续当生成真相。
27. 人物底账传递必须同时满足旧展示字段和 V2 结构字段；只满足一套字段就是半成品，不能进入下游骨架。
28. warnings、diagnostics、恢复态错误必须作为 bundle 的正式字段传递，不能散落在日志、临时字符串或前端猜测里。
29. 人物 V2 模型输出 JSON 外壳损坏不能直接 500；如果势力矩阵已生成，必须从 fenced/prose 中尽量提取 JSON，仍失败则按势力占位生成带诊断的 fallback 人物底账。
30. 任何出现在用户可见人物小传里的兜底模板句，都按链路污染处理；修 shared 映射、server 补全和 main 副本时必须同步清理。
31. 箭头链式弧线只能当原始素材；展示层必须拆成起点、触发、摇摆、代价、终局，不能把整条箭头链复制到每个阶段。
32. 轻量卡的 `想要/压力` 展示必须先去重归一；同一句去掉句号后相同，不得再用 `/` 重复展示。
33. 同族管家、暗卫、家臣等人物如果被上游放进错误势力，entityStore 要按已实名家族核心人物纠偏主归属；成员表仍只放主归属。
34. `主角/男主/女主/反派/对手/敌人` 这类泛称不能作为硬覆盖锚点触发整链重跑；没有具体姓名时只能做软提示或等待 outline/profile 反推。
35. 合同重试不能粗暴重跑整条链；重试条件必须区分“结构半成品”和“泛称锚点无法匹配”，否则一次人物/骨架失败会被放大成数倍耗时。
36. 守护女主、亲传大弟子、仙盟爪牙这类常见短剧角色必须有专属补全分支；不得统一塞进“旧账/位置/外部靠山”通用兜底。
37. `identity/biography` 明确写出“某势力盟主/长老/特使/掌门/弟子”时，可作为 entityStore 主归属纠偏依据；但纠偏后仍只能保留单一主归属。
38. 占位 fallback 只能写人物当前职责和取舍，不能出现“被更强的人替掉”“还能掌控的位置”这类系统兜底腔。
39. 题材规则不得写进全局生成函数；男频玄幻、女频古言、女频霸总、都市逆袭、都市律政等必须通过 `GenerationStrategy` 策略层分发。
40. `marketProfile.audienceLane/subgenre` 是题材策略选择的第一真相源；`genre` 只能作为旧项目兜底，不得反向覆盖用户选择。
41. 非当前题材禁用词必须可检测；例如都市律政生成宗门/仙盟/魔尊血脉，按策略污染处理，不按普通文案瑕疵处理。
42. 人物/骨架阶段的题材污染必须先修权威 bundle 再给下游；禁止只在前端展示层替换禁词，禁止让已知串味的 `outlineDraft`、人物底账或 `entityStore` 继续进入详细大纲。
43. 下游阶段如果需要清洗题材串味，必须复用 `GenerationStrategy.repairStrategyContaminationValue` 这类共享策略出口；禁止在详细大纲、正式剧本各自复制一份递归替换函数。
44. 正式剧本题材污染必须先走 batch guard/retry：`strategy_contamination` 属于 `EpisodeGuardFailure`，触发重写并进入问题单；postflight 只做最终兜底审计，不得再新增第二裁判口。
45. 用户可见 biography 不得出现 `身份是`、`性格底色`、`在戏里`、`牵动他的软肋` 这类字段拼接痕迹；出现一次就按 V2 映射层失败处理。
46. 缺字段默认兜底不得输出“旧账/外部靠山/可调动关系/自己这条线/筹码依赖外部靠山”等系统腔；即使只是兜底分支，也必须按产品可见文本标准写。
47. 压迫型弟子、走狗、欺凌者、首席打手必须进入 enforcer 分支，不能落到普通利益杠杆兜底。
48. 组织骨干位和功能席位的目标必须是职责动作，不得写“撑住某条线/守住自己这条线”这种无法入戏的泛目标。
49. 大长老本人和大长老爪牙必须分开：夺血脉、取代掌门、架空宗门的权力核心人物是反派主轴，不得被补成“程序慢半拍/站队代价”的普通执行人。
50. 人物弧线展示不得出现 `起点：起点是`、`终局变化：终局变化`、`代价选择 → 终局` 这类阶段标签泄漏；箭头链终局必须拆净再展示。
51. 用户可见自然小传不得出现 `让X信奉`、`在主线里的作用`、`行动抓手` 这类字段拼接说明腔；要改成自然人物传记句。
52. 男性特征明显的人名不得被情感杠杆兜底写成 `她/师妹/少女/大小姐`；性别不确定时优先中性化为同门、年轻弟子、人物名本身。

## 文档原则

1. 现役规则只认 `CLAUDE.md`；`AGENTS.md` 只做同步指针。
2. `docs/` 只保留 5 个活文档 + `当前主线.md`。
3. 其他文档全部归档到 `docs/archive/`，不再参与现行判断。
4. 每次代码改动后、每次出现新问题、每次出现新结论时，立即同步五件套。
5. 活文档尽量控制在 200 行以内，超过就压缩。

## 验证原则

1. 任何功能必须跑通再算完成。
2. 不许只看代码就说“完成”。
3. 默认至少看 `npm test`、`npm run build`、`npx tsc --noEmit --pretty`。
4. 提示词和文案质量测试，允许真实调用模型 API。
5. 第三方模型通道优先，第三方连不上才回退官方 DeepSeek。
6. 前端 UI 改动必须补真实页面体验验证；列表页、编辑页、工作台页至少检查滚动、输入、展开/折叠、复制按钮和生成中状态是否卡顿。

## 架构原则

1. 回答架构问题前先读 `graphify-out/GRAPH_REPORT.md`。
2. 新项目默认前后端分离：前端一个目录、后端一个目录、生成逻辑和提示词放后端。
3. 多项目共享 PocketBase 时，必须按 `appId` 和 `<appId>_xxx` 业务表前缀做物理隔离。
4. `user_apps` 管准入，`user_wallets` 管积分，`<appId>_transactions` 管项目流水。
5. 项目积分绝不能写在全局 `users` 表里。

## 典型误判

1. 把某一层通过当整条链通过。
2. 把旧截图、旧日志、旧 PASS 当现役结论。
3. 发现双口径后继续保留兼容分支。
4. 回答架构问题时不看图谱，靠印象拍脑袋。
5. 文档口径已经换了，但旧文档没有物理归档。
6. 把“1M 上下文”理解成“一次写完整部剧”，而不是用结构化状态快照和打法包提高单集稳定性。
