# 项目原则

只记录已确认的长期原则、禁区、验收底线和已坐实的典型误判。旧原则被新事实推翻，直接替换。

## 当前有效原则

1. 用户输入是唯一真相，系统不替代创作裁判。
2. 发现双口径不打补丁，直接收一套；同一字段只能有一个权威来源。
3. 提示词优化优先于硬限制；工序边界必须明确，哪层只做哪层的事。
4. 读取链、兼容链、repair、fallback 不能悄悄改正式真相；自动推进不能偷变成第二主线。
5. 内容质量问题不靠兜底和门禁卡死，优先交给对应 Agent 基于原稿返修；系统失败才报错。
6. 检测层只负责报问题和分发，不负责判死；质量信号分 actionable（触发返修）和 observe-only（只观察）两层。
7. 单职责 Agent 必须吃原稿，输入必须含：原稿全文 + 当前问题 + 明确目标 + 禁止事项。
8. 内容返修默认不停机；当前集写得不好就让返修 Agent 继续改，不做成门禁式停机。
9. 已有可见剧本时，后续失败也要产出 ledger/postflight；真失败可以阻止继续推进，但不能把现成稿抹成不可见。
10. repair_batch 是可选异步步骤，不能卡死同步返回路径；已有稿必须先落盘。
11. 首稿职责是"写出稳定可修稿"（场次稳、承接稳、冲突稳、结果稳），高级工艺交给后续单职责 Agent。
12. production 返修分两段：Phase A 逐集即时修结构真相和胖瘦；Phase B 批后轻量内容工艺修整。同类问题不跨两段重复掌权。
13. 字数合同在 threshold、prompt、测试断言三处必须共用同一口径；合同一旦定死，靠字数 Agent 改到位，不迁就现状放宽。
14. Agent prompt 合同写"必须"和绝对字数区间，不写"约""尽量"；模型会无视软词。
15. 场级预算写成硬数字；只盯整集总字数会漏掉单场塞爆（2-3 场集仍能靠单场写 1700-1900 字）。
16. 胖集主因优先按"单场塞太多回合"处理，瘦集主因优先按"sceneByScene 骨架偏薄"处理。
17. 字数稳定修复先消灭极端样本（1800+ 极胖和 500- 极瘦），再谈整体美化。
18. 剧本质量 Agent 修"戏剧工艺问题"不修"小说措辞"；continuity 修局面推进、character 修选择/代价、theme 修主题落到动作。
19. 主题 Agent 低频出手，只在关键集、批次末集、收口集；不准写死项目专属主题词，必须从 outline/theme 动态取。
20. 首稿 prompt 不准兼任 showrunner、弧光裁判、主题裁判和情绪裁判；prompt 堆成大而全就是退回 prompt-first。
21. 弱钩子正式定义：只认"后果已落地"（锁扣落下、渗血、崩裂、被撞开），不认"线索显形/环境显形/逼近/预备动作"；弱钩子是 non-blocking penalty。
22. 检测器校准优先补真实误杀句式，不补泛词；像"消失"这种泛动作词不能裸放进强结果口径。
23. 继续堆 prompt 规则有收益递减点；一轮 prompt 改动后通过率反而下降，就必须换打法。
24. parser 不能把"模型少写时间词"放大成场次数误判；场号+地点+人物表+正文结构成立时，尽量保多场结构。
25. 批次口径永远 5 集一批（粗纲/详纲/剧本统一），prompt 只写"当前批次"，不准写死尾段集号。
26. 长季 compact mode 不能删 hook landing block 和 dialogue voice block 这类直接影响质量的正式约束。
27. 30 集及以内详纲优先保逐集可见，不压成大段范围占位。
28. 承接摘要必须基于 screenplay/screenplayScenes，不能把拆分后的 A/D/E 再喂回模型。
29. 同一集多版本并存时必须 latest wins；batch 内新生成内容优先级高于旧稿。
30. rewrite 拼接顺序必须 [新内容, base]；验证 scriptDraft[0] 确实是新内容。
31. 重试/改稿必须基于上一版成稿，带原文、当前字数、目标区间、压缩/扩写比例；不准从零重写。
32. 字数返修是独立工序，任务名、prompt 角色、调用入口显式区分于首稿写作。
33. 内容 Agent 候选稿只有在 guard 维度更接近合同才允许覆盖原稿；改了但没变好就退回原稿。
34. 字数 Agent 必须保住最低结构真相；压缩出现极瘦稿、重复场号、placeholder 壳时，补反过压保护或程序侧拒收。
35. 污染类问题顺序：先清污染再修内容；template_pollution / voice_over 不先拿掉，后续 Agent 吃到脏稿会稀释返修目标。
36. repair 改了 action/dialogue/emotion 就必须同步刷新 screenplay/screenplayScenes。
37. 生成格式合同、解析入口和质量合同必须是一套；正式交付 screenplay-first，不准旧 A/D/E 口径混进生产。
38. 内容质量真实验证默认调用真实大模型 API，落成仓库内 txt/md 证据文件；mock/单测/结构校验只验流程不验内容。
39. 20 集及以内可全量真实验证，更长任务按用户指定范围/切片/抽样验证。
40. 流程验收和质量验收分开记；plan/gen/save/runtime 全绿不等于 official quality 已过。
41. 历史 PASS 只能当线索不能当现役结论；要指挥下一轮任务必须用当前代码/口径/命令复跑。
42. 前段样本过线不等于长季已稳；中后段新失败类型要单独立成新主线。
43. 长季 timeout 修复必须补对应季长的真实 fresh evidence，直接看 runtime-diagnostics.log；npm run build 不算自证。
44. 能纯 Node 单测的业务模块不要绑 Electron 运行时依赖；运行时诊断留在外层注入。
45. 原生 node --test 加载的共享文件，import 带扩展名，避免 Node strip-only 不支持的语法。
46. 死代码清理先 grep/LSP 找候选，再小步改动，最后必须用 typecheck/build 确认；grep 0 引用不等于真死代码。
47. 正式链和原型链边界写进文件头（如 CURRENT PRODUCTION ENTRY / PROTOTYPE-ONLY / LEGACY），不靠人脑记忆。
48. 能静态 import 的不要留动态 import；动态 import 放大搜索看不出的误判风险。
49. 无人消费的旧 hook/旧方法/停用空壳模块，按噪音处理直接物理退场。
50. 七问是篇章级 agent-first 工具：粗纲层确定、只挂篇章不挂集、用户参与决策、seven_questions_agent 起草后用户确认再落盘。
51. 七问真相源唯一权威是 outlineDraft.outlineBlocks[].sevenQuestions；顶层 project.sevenQuestions 已移除不准回流。
52. 七问工作流三接口：生成初稿 -> 用户确认 -> 写入 outlineBlocks -> 再生成粗纲；不准跳过确认或偷偷覆盖已确认七问。
53. 七问确认保存只改 outlineBlocks 一处正式真相；七问确认成功以服务端落盘返回为准，不由前端自己拨状态。
54. 确认信息里角色名必须在持久化前归一清洗；集数口径按聊天里最后一个正式声明取值。
55. 生成底稿有角色卡后，正式人物名单只认角色卡；角色卡外的人不准混进正式人物链。
56. 人物系统按"势力先行"建模；characterDrafts 只代表完整小传，功能人物允许只保留轻量人物卡。
57. 详纲和剧本只吃当前批次活跃人物包，不全量吃整季人物。
58. 确认信息收成 generationBriefText 后，实体拆解在这一层正式落成 entityStore；旧实体整包重置时优先保留已有稳定 id 和 richer fields。
59. 项目级创作真相源在 confirm_story_intent 时落盘，集级创作真相源在 detailed_outline 时落到 episodeBeats[]。
60. 剧本首稿 prompt 先看创作宪法和当前集控制卡，再看 recap；前情不能抢到控制包前面。
61. 创作宪法（快节奏/强冲突/稳情绪）只能派单给 Agent，不准实现成 hard gate。
62. 上游人物真相变化时，下游正在生成的剧本必须 stale warning 并停。
63. 前台切页和项目主进度不是一回事；project.stage 是正式进度，用户点回旧页时 currentStage 跟着切。
64. 上游阶段重跑成功后，下游旧结果必须同步让位清空。
65. 关键设定确认可以作为粗纲层显式能力，但不能当详纲硬门禁；粗纲正文已成立时详纲应允许先写。
66. server/ 是唯一正式业务入口；PocketBase 是唯一项目真相存储；src/main / src/preload 只保留桌面壳职责。
67. 前后端必须且只能共用工程根目录的 src/shared，服务端一律通过 @shared 别名导入，严禁存在双轨副本。
67. 前端 HTTP API 调用统一收敛到 api-client.ts；Token 管理、错误处理、重试逻辑都在 api-client 统一处理。
68. HTTP 与 IPC 不准长期双管线并存；HTTP 对齐后对应 window.api/ipc 业务入口必须物理退场。
69. 认证/计费/限流等横切关注点放路由中间件层；业务服务不关心用户身份和积分。
70. 积分扣费在业务执行前检查余额（不足返 402）、业务成功后执行；失败不扣费。扣费时机和金额写在路由层。
71. HTTP API 错误响应统一格式 { error, message }，错误码 snake_case，message 中文用户可读。
72. PocketBase 项目建模用"主表 + 分阶段子表"，每张表带独立 version 整数版本号；不在现役表里并列多份旧版本。
73. 云端 Repository 先拼快照再给前端，返回继续对齐 ProjectSnapshotDto，不把数据库拆分细节甩给前端。
74. 乐观锁以 version 冲突直抛为准，检测到版本不一致报 version_conflict，不偷偷覆盖，仓储层不加兜底 merge。
75. PocketBase JSON 字段直接传对象，不需要 stringifyJson() wrapper；double-stringify 会导致 400 错误。
76. 多项目共用 PocketBase 必须按 appId 前缀隔离业务表（<appId>_xxx），严禁无前缀表名。
77. 全局共享表（users/user_apps/user_wallets）由 SSO 架构统一管理，项目只读写自己的数据。
78. 用户注册写 registerSource=appId；跨项目登录时静默补绑 user_apps 和 user_wallets。
79. 积分钱包按 appId 独立核算；<appId>_transactions 记录项目专属流水。
80. 新项目接入必须申请唯一 appId（全小写不重复），按规范创建业务表和配置认证流程。
81. 长时间后台任务必须用异步模式：POST /start -> 202 Accepted -> 后台 Worker -> 前端轮询。
82. 剧本生成进度条由轮询数据驱动，不做本地时间估算；轮询间隔推荐 3 秒。
83. 剧本生成逻辑完全从 Electron 迁到 server 端，前端只做 HTTP 调用不执行生成逻辑。
84. AI timeout 扩到 15 分钟时，server.setTimeout()/keepAliveTimeout/headersTimeout 必须同步调整。
85. Episode control 缺失时优先用 buildEpisodeControlCard() fallback，而非阻断整条链。
86. 所有生成类 HTTP 接口必须先验 JWT 确保 req.user 存在。
87. 部署口径收成阿里云单线后，Cloudflare Tunnel/Vercel/Sealos 旧地址只能算历史资料。
88. 多项目共用支付宝当面付时，通道可共用但订单/钱包/回调/退款/对账按项目隔离；ALIPAY_NOTIFY_URL/RETURN_URL 按项目显式配阿里云正式地址。
89. 商业计费链的账本字段必须跟真实业务对象一一对应；api_call_logs 要记 project 和 durationMs。
90. 前端切到 HTTP 主链后，余额刷新也必须跟服务端取真相，不靠本地状态猜扣费结果。
91. 页面展示依赖项目快照的字段（如 entityStore）必须跟 HTTP ProjectSnapshotDto 一起 hydrate，不单独走 IPC 偷读旧本地项目。
92. 共享"归一化"函数不能偷带默认集数改写 outline 真相；未显式传入 count 时必须先信任 outline 自身真实集数。
93. 正式任务名收口后，错误码、日志前缀、UI 文案也必须同步收口。
94. 唯一真相源确认后，项目主快照合同里不保留同名顶层字段；deprecated 不是收口，移除才是。
95. 结构化大表生成优先按上游业务分块，不要求模型一次吐完整张大表；并发池限流保稳（如锁 3），不放回无上限 Promise.all。
96. 倒计时结束后白板是失败不可见，不算成功等待态；已明确 parse/runtime 失败的分块必须带对象名直接抛给前台。
97. 天然重任务必须有独立 task 身份和 timeout 预算，不能借别的 stage 的壳跑。
98. 正式发车前留握手证据（如确认版七问），但握手日志是证据口不是 gate。
99. 前后端分离收口时项目 CRUD 先整体换轨，不长期并存半截主链。
100. 局部保存接口必须保住未改字段；runtime-state 不能顺手清空 scriptDraft，detailed-outline segments 不能顺手清空 blocks。
101. 旧项目已有确认版 generationBriefText 时，读项目必须先按已确认信息回填 entityStore，不能因历史空值显示零人物。
102. 势力拆解优先认真实势力名，不认糊成一整串的描述文本。
103. 会拉起 vite-node 的命令不在当前指挥终端跑，必须换独立执行终端。
104. 根目录临时脚本/运行残留目录不长期混在主仓根；check-*/debug-*/minimal-repro.* 等不用就删或归档。
105. E2E 证据目录默认自清，不把 tools/e2e/out 当永久仓库。
106. 5 份现役文档之外只准进 docs/归档/；docs/ 根目录只允许 5 文档 + 主线/ + 归档/ 两个正式分区。
107. 每轮任务收口后更新四件套：聊天记录.md、项目原则.md、AI自我学习.md、项目结构.md；"存档"口令额外强制整理一次。
108. 主线完成后立刻归档并补下一份当前主线文档，不准靠口头滚动推进。
109. 判断主线是否继续要靠最新真实 evidence，不靠感觉；关键验证项跑通就先考虑收口。
110. 项目结构.md 是正式交付的一部分，至少写清主链、分层、线身份、风险区。
111. docs/聊天记录.md 记干活终端主线推进，不记纯整理文档终端的过程。
112. 质量门三关（test/lint/build）必须同时绿才算收口完成，不能把 build 失败按"老问题"口头划走。

## 当前项目必须死守的创作底线

1. 不要第二写口、第二发布口、第二裁判口。
2. 不要让系统替用户定义人物、关系、集数、核心冲突。
3. 不要把创作问题修成 gate、命中词校验、重型 recovery、抽名补名单。
4. 不要哪层炸了就让别层补根因；不要拿某一层通过冒充整条主链通过。
5. 不要拿"做了很多动作"冒充"解决了本质问题"。
6. 官方 authority gate 只依赖现役文件；已归档文档继续挂进 gate 也是第二口径。
7. prompt 调优先改已有指令的表达和排序，定位根因前不加新条款。
8. 剧本正式交付格式只能有一套。
9. 不拿 mock/单测/静态验证冒充"写得怎么样"。
10. 不把生成后规则化改写钩子当主方案，否则会长成第二写口和第二裁判口。
11. prompt 连续无效时不在原地继续堆条款；先换打法。

## 已坐实的典型错误

1. **把阶段完成当问题解决**：只证明一层能跑一次/一个入口/一个样本/静态搜索清零，不算解决。
2. **把局部真测当整链成立**：首场好不等于整剧好；验收先问上游是不是唯一真相、下游是不是只吃这一套、旧旁路有没有真退场。
3. **哪层炸了只修哪层**：先看第一次越界的是哪层，再看上游是不是交了不该交的债，再看有没有第二口径偷跑。
4. **把优化做回限制**：系统越来越像审查器而不是创作脚手架，方向就歪了。
5. **旧文档继续像现役入口**：旧任务卡/README/ARCHITECTURE 还在对外发声，必须物理退场。
6. **假完成藏在"结构收口"里**：生成入口还有二次 summarize、系统还补写 facts、自动推进还在后台开工，就不准写"唯一真相源已收口"。
7. **只改现役文档不做物理退场**：旧文档不搬走、旧入口名不退场、全仓不搜确认，等于没收口。
8. **只改代码不补四件套**：代码收住了但项目结构/原则/学习/聊天记录没同步，下一终端会接错方向。
9. **运行时依赖绑进业务模块**：纯逻辑模块只依赖合同和注入接口，Electron 日志/IPC/文件路径放外层入口。
10. **"算了但没用"假放行**：先调了验证但最后硬编码返回 ready:true，等于没校验。
11. **链式操作读写路径不一致**：参数名相似不等于值相等，"操作成功但验证失败"先查路径。
12. **rewrite 拼接顺序反了**：新内容必须在前面，改 rewrite 逻辑时验证 scriptDraft[0] 是新内容。
13. **旧绿灯当成今天还能背书**：历史 PASS 只能当线索，指挥下一轮必须用当前代码/口径复跑。
14. **同一集多版本默认取第一条**：多版本并存时必须 latest wins，find()/第一条命峰会把旧稿扶正。
15. **生成合同/解析入口/质量合同互相打架**：先收一套正式交付格式，入口路由和质量合同必须一致。
16. **grep 0 引用当死代码结论**：必须再跑 typecheck/build 确认真实 import 链。
17. **"有人手工用"误判成"正式生产在用"**：调试脚本/原型测试的引用不算正式消费方，先问引用属于哪种身份。

## 以后看到这些信号，立刻停下来

1. 同一件事在不同文档里有不同答案。

## 有效做法顺序

1. 先确认当前现役文档是不是只有 5 份在掌权。
2. 先确认当前主线是不是只有一条。
3. 先确认用户输入是不是还在当唯一真相。
4. 先查旧兼容、旧 fallback、自动推进、旧任务卡有没有回流。
5. 再查具体技术点、提示词、超时、格式问题。
6. 只要命令会拉起 vite-node，就默认交给独立执行终端。
