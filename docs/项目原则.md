# 项目原则

这个文件只记录已经确认的长期原则、禁区、验收底线，以及过去已经坐实的典型误判。

如果旧原则被新事实推翻，就直接替换，不保留第二口径。

## 当前有效原则

1. 用户输入是唯一真相。
2. 系统不替代创作裁判。
3. 真正的系统失败才报错；内容质量问题不靠兜底和门禁卡死，优先交给对应 Agent 基于原稿继续返修。
4. 发现双口径，不打补丁，直接收一套。
5. 提示词优化优先于限制。
6. 工序边界必须明确，哪层只做哪层的事。
7. 上游放行口径和下游验收口径必须一致。
8. 读取链、兼容链、repair、fallback 不能悄悄改正式真相。
9. 自动推进不能偷偷变成后台第二主线。
10. 同一个字段只能有一个权威来源。
11. 当前仓库只有 5 份现役文档，其他文档只准进 `docs/归档/`。
12. 只要新事实已经成立，旧逻辑、旧文档、旧任务卡就必须退场，不能继续并列存在。
13. 文档收口不能只改说法，必须做物理迁移和全仓自检。
14. 每轮任务收口后，干活终端必须顺手更新四件套：`聊天记录.md`、`项目原则.md`、`AI自我学习.md`、`项目结构.md`。
15. `项目结构.md` 是正式交付的一部分，不是可有可无的整理活。
16. `聊天记录.md` 主要记录干活终端的主线推进，不记录纯整理文档终端自己的过程。
17. 用户发"存档"时，四件套必须额外强制整理一次。
18. "先不存档"不等于四件套可以不动；干活终端每轮任务收口后，四件套照样要同步。
19. 已经写到 `5/5` 的主线文档不能继续留在 `docs/主线/` 里冒充当前主线；一旦确认完成，就要立刻归档并补下一份当前主线文档。
20. `docs/` 根目录只允许现役 5 文档和 `docs/主线/`、`docs/归档/` 两个正式分区；非现役参考材料必须放进 `docs/归档/`。
21. 能做纯 Node 单测的业务模块，不要直接绑 Electron 运行时依赖；运行时诊断要留在外层注入。
22. 诊断日志是旁路证据，不是业务 gate；日志写盘失败不能反过来打断正式生成，也不能覆盖原始业务错误。
23. 只要某个共享/运行时 TS 文件会被原生 `node --test` 直接加载，运行时 import 就必须带扩展名，也不要写 Node strip-only 不支持的语法（例如 constructor parameter property）。
24. 历史验收结论只有在当前代码、当前口径、当前命令下复跑仍成立，才算现役可信；旧 PASS 不复核，不能继续拿来指挥当前任务。
25. 流程验收和质量验收必须分开记；`plan/gen/save/runtime` 全绿，不等于 `official quality` 已过。
26. 同一集如果存在旧版和新版，续写 / 重写上下文必须显式选择最新版，不能按"第一条命中"碰运气。
27. batch 生成不是"冻住上下文一次写完"；后一集必须能看到本批次前面刚生成的集数。
28. 会拉起 `vite-node` 的命令不能在当前指挥终端跑，必须换独立执行终端；否则连当前会话和验收口径都会一起崩。
29. 生成格式合同、解析入口和质量合同必须是一套；不能一边要求 2-4 场和 scene heading，一边让正式入口默认走单场 A/D/E 重建。
30. repair 只要改了 `action / dialogue / emotion`，就必须同步刷新 `screenplay / screenplayScenes`；否则旧场次结构会继续冒充当前真相。
31. 验粗纲、人物小传、详细大纲、剧本的内容质量时，默认直接调用真实大模型 API 看成品；`mock`、单测、结构校验只能证明流程或合同，不算内容质量结论。
32. 内容质量真实验证默认要把原始输出落成仓库内可共享的 `txt` 或 `md` 证据文件；只让执行终端自己现场看、不留材料，不能当正式验收。
33. scene-level 新工序上线后，质量合同也必须跟着 sceneCount-aware；旧的 flat threshold 继续拿来判新工序，属于合同错位。
34. 安全阀如果没接进正式生产路径，就按死代码处理；"理论上会触发"不算现役能力。
35. 生成后 trim 只能处理尾部解释、回忆、总结类污染；如果真实超长主因是对白和动作骨架本身过密，主战场必须回到生成粒度，不能继续扩 trim 去碰正文骨架。
36. 弱钩子判断要先对照检测器词根和真实结尾样本；如果模型已经写出可见后果但检测器没认出来，先修检测口径，不要继续怪 prompt 不够狠。
37. 如果 hook prompt 已继续收紧，但弱钩子频率没有下降，就不再把"再加一条 prompt 规则"当主方案，直接回到检测器口径和真实样本对齐。
38. `20` 集及以内的内容质量验证，默认可以按用户要求直接全量调用真实模型；更长任务默认先按用户指定范围、切片或抽样验证，不擅自整季全写完。
39. 弱钩子检测口径可以补"显形类物件结果"和"渗血类受伤结果"，但不要顺手放宽成"逼近、观察、预备动作"也算硬钩子。
40. **弱钩子正式定义**：只认"后果已落地"，不认"线索显形"与"环境显形"。具体判断标准：
    - **认**：锁扣落下、渗血、崩裂、撕裂、被撞开、弯折、结果已发生（纸页被抽出、物件被控制）
    - **不认**：看见某物、逼近中、露出一点、泛出冷光、环境光影变化
    - **产品口径边界**：纸角露出一角 = 线索显形，不算硬钩子（2026-03-29 确认）；这类结尾应由生成侧推进成更具体的结果句，不扩词根覆盖
    - **当前状态**：弱钩子是 non-blocking 问题（penalty=8），记录在案但不拦截 overall pass；生成侧负责把弱钩子结尾推成结果落地句
39. 检测器修到能明显减少误杀后，就不要继续把主精力耗在"为了过检测器把结尾越写越狠"上；接下来先看真实写作质量，如果剧情已经够用，就不再围着检测器打转。
40. 检测器校准优先补"真实误杀句式"，不补泛词；像 `消失` 这种泛动作词，不能裸放进强结果口径，必须保留具体后果结构约束。
41. 同一份真实 evidence 回放如果已经证明某轮检测器修正把误杀从 `12` 集降到 `8` 集，就说明检测器不再是第一主问题；后面优先判断内容层，而不是继续把检测器当主战场。
42. 正式 prompt 不能一边禁止某种交付格式，一边继续用那种格式教模型写；既然正式交付是 screenplay-first，就不准再拿 `Emotion 段`、`Action 叙述` 这类旧 A/D/E 口径混进生产合同。
43. 原型 runner、实验 prompt、测试组装链，只有在正式生产代码真正引用后，才算现役主线；`tools/e2e/test-scene-p*.mjs` 或 prototype 文件自己跑通，不等于生产链已经接入。
44. 生产链控长优先靠"减膨胀、减重复、减同义反复"，不要靠继续加长预算脚手架；如果一轮 prompt 变长后真实样本更肥，就立刻回退，改打反重复和反散漫。
45. parser 不能把"模型少写了时间词"直接放大成场次数误判；只要 `场号 + 地点/标题 + 人物表 + 正文` 结构成立，就要尽量保住多场结构，再把格式纠偏留给 prompt。
46. 真实内容质量验收要把"系统问题"和"内容问题"拆开记：`VO / sceneCount / parser` 这类系统误伤修住后，就不要继续拿它们掩盖真正的内容厚薄失衡。
47. 后半程剧本发胖时，优先收"单场回合数"，不是继续给"写厚一点"的鼓励；一场里连续追打、连续翻盘、连续解释，本质上就是把三场硬塞成一场。
48. 只要真实 evidence 已经证明粗纲、人物小传、详细大纲基本稳住，当前主线就应收缩到剧本层，不再把上游一起重判成"全链都不行"。
49. 正式生产链的批次口径已经定死为"永远 5 集一批"；prompt、详纲、粗纲都只能写"当前 5 集批次 / 当前批次末集 / 当前批次末两集 / 当前批次收口段"，不准再回到 `第7-10集 / 第8-10集 / 第9-10集 / 第10集 / 最后一集 / 最后两集 / 后四集` 这种写死尾段口径。
50. 长季一旦进入 compact mode，不能为了省 token 把真正控制质量的脚手架一起删掉；`hook landing block` 和 `dialogue voice block` 这类直接影响钩子硬度、对白占比、人物口风的正式约束，压缩态也必须保留。
51. 详纲压缩如果把 `30` 集及以内的逐集覆盖压成大段范围占位，会直接削弱剧本阶段对逐集节奏和均衡性的感知；`30` 集及以内应优先保逐集可见，再谈二次细节压缩。
52. 长季 timeout 修复不能靠 `npm run build` 或源码搜索自证，必须补一轮对应季长的真实 fresh evidence，并且直接看 `runtime-diagnostics.log` 里的 `act_start timeoutMs` 和后半段 `responseChars`；只有日志确认吃到新 timeout、后半段不再超时，才算修复成立。
53. 只要剧本内容已经生成出来，质量信号就应该先驱动"下一道返修 Agent"，不能再被系统粗暴升级成"整轮失败不可见"。
54. `scriptFailureResolution` 只应该用于真失败：runtime/parse/persist/API 这类执行失败；内容写得不好不是失败，而是返修工序的输入。
55. 剧本页前台提示不能只做"成功/失败"二分；至少要能表达第三种状态：**内容已生成，系统正在继续调用 Agent 把它改好**。
56. 字数超限惩罚必须和字数不足惩罚对齐；如果一边重一边轻，模型就会按轻的那边跑。
57. prompt 里的字数约束不能用"尽量"；必须用"必须""硬红线""超了就不合格"这种硬词，否则模型会无视。
58. VO 漏出不能只靠 prompt 规则；如果 prompt 规则已经被多次证明无效，就必须在 repair 指令里加明确的修复动作，形成"检测→修复→验证"闭环。
59. 字数稳定不能只盯整集总字数，必须正式进入"场级预算控制"；不然 2-3 场集照样会靠单场塞爆写成 1700-1900 字。
60. 胖集主因优先按"单场塞太多回合"处理，不要误判成"场数太多"；瘦集主因优先按"sceneByScene 骨架偏薄"处理，不要只怪模型偷懒。
61. repair 不能只当通用修错器；进入长季稳定阶段后，它必须明确承担"削胖补瘦"的职责，而且优先依据当前集 `sceneByScene` 骨架补戏，不准凭空注水。
62. 场级预算必须写成硬数字，而不是"尽量控制"；只要还是"尽量"，模型就会继续把 2-3 场集写成大胖场。
63. 真实字数稳定修复必须先把极端样本压平：先消灭 `1800-2000+` 的极胖集和 `500-600` 的极瘦集，再谈整体美化。
64. 死代码清理不能只靠静态 grep 的"0 引用"拍板；**必须再跑一次 `typecheck/build` 看真实 import 链**。这轮 `resolve-lane-strategy.ts` 和 `select-script-generation-batch.ts` 就是 grep 误判、编译纠偏的反例。
65. 真死代码优先清"分析残留 / 格式辅助 / 诊断导出"这类**无正式消费方且不参与主链合同**的东西；正式生产文件除非编译和运行链都证实无人依赖，否则不要轻删整文件。
66. 代码质量整治要先做**依赖身份分层**，不能只看 `src/` 里的 import；根目录调试脚本、`tools/e2e`、历史设计文档也会制造"好像还在用 / 好像已经接生产"的假象。
67. 物理退场必须放在"正式链 / 原型链边界收清、依赖显式化、死导出清理"之后；不能跳过前两步直接删整模块。
68. 正式链和原型链长期并存时，**代码文件自己就要把边界写清楚**；否则下一轮很容易把 prototype helper 误写成 production 事实。
69. 能静态 import 的正式依赖就不要为了省事留动态 import；动态 import 会放大"搜索看不出来、编译才知道"的误判风险。
70. 停用模块如果只剩 `Temporary stop` 空壳、又没有任何消费方，就不要继续挂在现役目录里装可用，直接物理退场。
71. **内容质量问题不再由 hard guard 掌权**；judge / guard / 质量检测只负责报问题、分问题、把原稿交给下一道 Agent，不负责直接打死整条链。
72. **模型能力上限不是代码能修的**；钩子弱、对白退化、人物扁平这类创作问题，不能靠加 guard 或堆 prompt 条款来解决。
73. **换模型不等于换质量**；同一个 prompt 合同在不同模型上表现不同，但质量根因往往在 prompt 本身和上游详纲，不在模型选择。
74. **自定义 OpenAI 兼容端点必须先验 JSON 完整性**；如果稳定返回截断 JSON，就不适合当前结构化全链，不能靠重试硬扛。
75. **Qwen free 类慢模型不适合串行全链**；前置阶段（粗纲+人物）就超时的模型，不能拿来跑 10 集以上真实全链。
76. **继续堆 prompt 规则有明确收益递减点**；当一轮 prompt 改动后通过率反而下降，就必须换打法，不能原地继续加条款。
77. **字数合同不能为了"先跑完"而放宽**；正式口径一旦定为 `800-1800`，就应该靠字数 Agent 在原稿上继续改到位，不靠把合同改成 `600-2200` 迁就现状。
78. **长季剧本生成的质量策略应当是 Agent 串行返修式**；当前集写得不好，就让当前集继续走返修 Agent，不要把"当前集不过关"直接做成门禁式停机。
79. **承接链不能靠测试里的假字段自证**；如果正式产物只有 `sceneNo`，承接链就必须按 `sceneNo` 找上一集，不能测试里偷偷塞 `episodeNo` 再宣布连续性已接通。
80. **承接摘要必须优先基于 `screenplay` / `screenplayScenes`**；不能把拆分后的 `action/dialogue/emotion` 再喂回模型，否则后半程会被二次拆分、二次脏化的文本持续污染。
81. **发车前检查要先区分"代码真坏了"还是"测试还在盯旧文案"**；如果正式 prompt、定向单测和 build 都对齐当前口径，就不要为了迎合旧断言把已废弃的 prompt 文案加回去。
82. **长季当前集失败后的第 2/3 次机会，必须是"基于上一版成稿改稿"**；不能继续把失败重试当成从零重写。重试 prompt 必须带上上一版原文、当前字数、目标区间、压缩/扩写比例和最胖/最瘦场信息。
83. **字数合同必须在 threshold、正式 prompt、测试断言三处共用同一口径**；`guard=800-1800`、`prompt=700-1200` 这种"系统自己打自己"的状态，不是提示词细节问题，而是正式链缺陷，长测前必须先修。
84. **10 集真实 API 全过后，优先把它当"链路已收住"的 fresh evidence**；不要再拿更早那轮 8/10 或旧口径失败样本压过当前事实。
85. **10 集全过不等于 20 集中段也稳**；一旦 20 集在第 11 集以后出现 `4000+` 这种极端胖稿，要先当"中段上下文放大问题"处理，不能误写成"整条链又污染了"。
86. **胖稿改稿不能把压缩比例钳死在一个看起来安全、实际仍会超线的下限**；像 `4374 -> 65% = 2843` 这种目标，本质上是在提示词里要求模型继续失败。极端胖稿必须直接给绝对目标窗口和更低保留比例。
87. **正式合同可以是 `800-1800`，但提示词执行目标不必把上沿直接亮给模型**；如果真实 evidence 已说明模型看到 `1800` 会继续上飘，提示词就该收成"约 `1000` 字、尽量 `900-1200`"，把上沿留给系统守门，不留给模型拿来冲。
88. **字数返修必须是独立工序，不准继续和首稿写作混成同一任务口**；首稿负责放飞写，`char_count` 返修负责基于原稿删改或补厚。任务名、prompt 角色、调用入口都要显式区分，不然模型会重新把"返修"听成"再抽一版新稿"。
89. **质量判断以后优先做"工序分发"，不做"判官定生死"**；发现胖瘦、污染、格式、对白、钩子、节奏问题时，先问"该派哪个单职责 Agent 来改"，而不是先问"这一集该不该被卡死"。
90. **正式 batch 只负责产出"当前最佳稿"，不再承担内容判死权**；batch 内多次机会可以继续比稿，但最终只要拿到了当前最佳稿，就进入下一道 Agent 返修，不再因为内容问题直接整轮失败。
91. **postflight 的质量报告是观察口，不是拦截口**；`quality.pass / weakEpisodes` 负责告诉系统和用户"还有哪些集要继续修"，不再往 `issues` 里塞 `screenplay_quality_failed` 当阻断码。
92. **持久化真相必须优先保住"已有稿可见"**；只要磁盘里已经有剧本稿，就不能再被 runtime failure 或质量信号盖成纯失败态。真失败可以阻止"正式继续推进"，但不能把现成稿子抹成不可见。
93. **内容调优默认优先改 Agent，不优先改 gate / 门禁 / 硬编码阈值**；只要问题属于"写得不好，写多了，写少了，写偏了"，第一反应必须是"加或改一个单职责 Agent 去修"，不是先去别处补更多硬限制。
94. **单职责 Agent 必须吃原稿，不准零基础重写冒充修复**；Agent 输入必须至少包含：当前坏稿原文、明确目标区间或目标效果、当前具体错误点。没有原稿上下文的"再生成一次"，不算返修。
95. **同一类内容问题不能在 batch 内和 batch 后被双重串行返修**；如果当前集在 batch 里已经做过字数返修，就不能再让同一类字数问题在 batch 后整批重跑一遍。否则系统会进入"明明已经写出来了，但总调用时长被返修链拖爆"的假卡死。
96. **观察口和返修口必须对齐**；像 `hook_weak` 这类用户已经明确放掉、且只该观察的问题，不能继续偷偷接成自动重试或自动返修触发器。
97. **Agent-first 主链的首要验收不是"判得更细"，而是"整轮能返回、能落盘、能继续"**；如果返修链设计得太重，导致剧本其实已经生成出来了，但整轮迟迟不返回保存阶段，那也是正式链缺陷。
98. **质量信号必须分两层：actionable 和 observe-only**；screenplay-quality.ts 的 inspectScreenplayQualityEpisode 必须返回 actionableProblems（触发自动返修）和 observeOnlyProblems（只做观察）。pass 只看前者；后者永远不触发自动返修和门禁。
99. **统一 Agent 输入协议是所有单职责 Agent 的硬合同**；每个 Agent 的 prompt 必须包含：原稿全文 + 当前具体问题 + 明确目标 + 禁止事项。不满足这四条的 prompt 一律视为不合规，不能进生产路径。
100. **repair_batch 必须是可选的，不能卡死返回路径**；startScriptGeneration 在 batch 完成后必须能立即返回 postflight 和保存链；repair_batch 可以在后续异步步骤中继续，但不准把同步等待它当成正式返回条件。
101. **Agent prompt 里的合同必须写成"必须"，不能写成"约"或"尽量"**；模型对"约保留原稿 X%" 这类比例词的理解和系统不一样，会把它当成软建议；胖/瘦 prompt 必须直接给绝对字数区间（"必须压到 900-1200 字"），同时加"不准新增场景/人物/情节点"禁令，防止模型把改稿听成继续写新内容。
102. **剧本质量 Agent 必须修"戏剧工艺问题"，不能退回去修"小说措辞问题"**；连续性 Agent 优先修"有没有新推进、有没有结果落地、是不是原地打转"，人物弧线 Agent 优先修"主角选择/反派施压/配角杠杆"是否成立，主题 Agent 优先修"选择-代价-后果"是否把主题落到戏里，而不是靠关键词、口号或内心独白硬判。
103. **内容质量检测默认要 scene-aware、drama-aware**；只看"决定/终于/依然/不争"这类词会把剧本误判成小说。后续质量测量优先看：场次引擎、局面变化、角色功能、结果落地。
104. **主题 Agent 不准写死项目专属主题词**；即使当前项目长期是"谦卦"，正式工程入口也必须从 outline/theme 动态取主题，再把它翻译成角色选择和后果，不能把固定世界观硬编码进通用修复链。
105. **判断质量 Agent 有没有价值，先看它修的是不是当前主痛点，不先看有没有触发日志**；主链稳定后，Agent 的意义在于把"能看但不够稳"的成稿继续往上推，如果它修的不是当前最痛的那类内容问题，触发再多也只是空转。
106. **主题 Agent 不能因为"主题分低"就每集都扫**；它应该低频，只在关键集、批次末集、收口集这类真正需要把主题钉进动作和后果的地方出手。
107. **内容质量 Agent 的角色名推断不能只依赖上游角色卡**；如果 `generationInput.characters` 没把配角/对手传全，正式链要回退吃当前集的人物表和场次 roster，避免把"角色没传到"误判成"角色没戏"。
108. **剧本质量继续往上提时，先收"改局强度"，不要先加更多主题口号**；如果一集已经有推进，但局面变化还不够痛、不够狠，第一优先是把结果写得更硬、更落地，不是再补一句主题态度。
109. **主角选择是否成立，不能只看他有没有动作，要看他是不是为这个动作付了代价**；没有代价的"选择"很容易只是流程动作，不是真正的戏剧选择。
110. **配角有价值，不等于配角出场了，而是配角有没有改筹码**；只陪场、只安慰、只重复信息的配角，不算人物功能成立。
111. **真实验证里如果 `projects.json` 和 `script.json` 的字数统计口径不同，必须明确写清口径差异**；运行侧统计和落盘正文统计不能混着引用，不然下一轮又会把同一轮 evidence 读成两个结论。
112. **前半段集位如果已经明确主痛点，就只准打那一个痛点，不要再让多 Agent 级联稀释目标**；像第 `1` 集的结尾翻面不够狠、第 `4` 集的场次引擎偏弱，都应该聚焦 `continuity-agent`，而不是顺手把人物解释和主题补口一起带进去。
113. **第 1 集开门红最值钱的是"受压后当场反咬出结果"**；如果第 `1` 集还停在"被压住 + 准备追查"，那下一轮 repair 应优先逼出已经发生的翻面结果，不优先补主题态度或人物解释。
114. **第 4 集这类中前段承接集，最怕场次只有气压没有结果**；一旦 sceneEngine 弱，优先补"阻碍 -> 应对 -> 结果"的完整场次引擎，不要误修成对白更满、解释更多。
115. **七问是篇章级工具，agent-first，不是普通 prompt**；详细定义见 `AGENTS.md` 七问框架章节。核心判断：
   - 七问在**粗纲层**确定，不在详纲层——篇章边界在粗纲阶段就定下来了，到详纲调不过来
   - 七问只挂**篇章层**，不挂集层——同一篇章内所有集共享同一套七问答案
   - 七问是**用户参与决策**，不是系统自动生成—— `seven_questions_agent` 起草，用户确认或修改
   - 七问是**专用 agent**，走 Agent 协议：输入 -> 起草 -> 用户确认 -> 落盘，不是普通 prompt 调用
   - 不能逐集套七问——每集只是篇章里的一步执行动作，不是每集都要问一遍目标-阻碍-代价
116. **七问真相源唯一权威是 `outlineDraft.outlineBlocks[].sevenQuestions`**；顶层 `project.sevenQuestions` 已移除，不允许再作为独立字段回流
117. **七问工作流三接口**：生成初稿 → 用户确认 → 写入 `outlineBlocks` → 再生成粗纲；不能跳过确认直接写粗纲，也不能偷偷生成一版覆盖已确认七问
118. **旧入口如果还保留，只能退化成阻断壳，不能继续冒充正式主路**；像 `workspace:generate-outline-and-characters` 这种历史入口，如果产品已经改成“先确认七问”，那旧入口最多只负责明确报错，不准再偷偷生成
119. **前端 API 和 UI 不准用 `unknown` 把主线糊过去**；一旦主线已经确定，`storyIntent / outlineDraft / characterDrafts` 这类关键返回值必须有明确类型，不准长期靠 `as` 和 `unknown` 盲接
120. **无人消费的旧 hook / 旧方法不算“以后也许会用”，按噪音处理**；如果全仓没有正式消费方，就不要继续挂着旧粗纲入口、旧生成方法、旧桥接壳装现役
121. **临时排查脚本和运行残留目录不能长期混在主仓根目录里**；`check-* / debug-* / inspect-* / minimal-repro.* / .tmp-seeded-*` 这类东西只要不再服务当前主线，就应删除、归档或移到专门目录，不能继续污染验收判断
122. **历史阻断壳在代码里也要叫“历史阻断壳”**；如果某个旧入口只剩“接住旧调用并明确报错”的职责，那 DTO、preload 包装、上下文标签都要一起改口，不能继续沿用“正式生成入口”命名
123. **正式链 / 原型链 / 历史壳的边界要写进文件头，不靠人脑记忆维护**；像 `prototype-only`、`CURRENT PRODUCTION ENTRY`、`LEGACY / NON-PRODUCTION` 这种身份标识，应该直接写在最容易被误读的文件顶部
124. **归档区不要继续保留隐藏工作目录名冒充正常归档结构**；像 `.sisyphus` 这类隐藏工作目录，一旦确认只剩历史证据价值，就改成可见归档名，避免下一轮继续把它误判成现役工作面
125. **parser 剥掉 placeholder stub，不等于污染已经不存在**；只要原文里出现过 `人物：人物`、`人物：场景`、`△# 第X集` 这类 placeholder stub，质量检测和 batch guard 仍要留下 `template_pollution` 信号，不能因为正式场被 parser 保住就静默放行
126. **污染类问题进入 Agent-first 主链后，顺序必须是“先清污染，再修内容”**；`template_pollution` / `voice_over` 这类问题如果不先拿掉，continuity / character / theme Agent 吃到的就是脏稿，会把返修目标稀释掉
127. **正式主链默认先返回，再决定要不要等 `repair_batch`**；`repair_batch` 不能再作为同步返回前置条件，否则“明明已经生成出来却迟迟不返回/不落盘”的老问题会复发
128. **已有可见剧本时，后续批次失败也要产出 `ledger/postflight`**；不能因为后半程失败，就把前面已生成内容的账本和 postflight 一起清成 `null`
129. **把 `repair_batch` 接回正式入口只能证明“返修链没断”，不等于内容质量已经恢复**；真实 10 集复跑如果仍然大量胖稿，就要回到字数 Agent 和真实返修效果本身，不要再把主问题误写成入口返回路径
130. **`bestAttempt` 选稿不能只看失败码种类，还要看离合同还有多远**；如果两版都只剩 `char_count`，系统必须优先保留“差值更小”的那版，而不是因为失败码相同就把第一次更胖的旧稿一直留到最后
131. **单测证明 tie-break 修好了，不等于真实胖稿问题已经解决**；如果真实 10 集复跑仍然大面积偏胖，就说明主战场已经从“选错稿”继续收缩到“字数 Agent 本身有没有真压下去、以及压下去后有没有被后续工序重新写胖”
132. **内容 Agent 的收稿逻辑不能用“正文变了”冒充“结果更好了”**；continuity / character / theme / format 这些 Agent 产出的候选稿，只有在 guard 维度上更接近正式合同，才允许覆盖原稿。只要候选稿更胖、更差、或只是“改了但没变好”，就必须退回原稿
133. **真实样本如果显示“最坏胖稿和平均字数下降了，但过线率还没起来”，就说明这刀只修到了中间层**；不能把这种结果写成“主问题已解”，而要继续往下收缩到真正剩下的单点根因
134. **字数 Agent 不能只会“压下来”，还必须保住最低结构真相**；如果真实样本出现 `72` 字极瘦稿、重复场号、正文被削到只剩壳子，就说明压缩 prompt 缺少最低保真约束。下一步要补的是“反过压保护”，不是再继续单向加压
135. **如果给 rewrite prompt 补了最低保真约束，真实样本仍然反复出现极瘦稿、重复场号和 placeholder 壳，就不要再把“继续改 prompt”当主方案**；最后必须回到程序侧，对 rewrite 结果做结构保真验收和拒收
136. **如果程序侧已经补了 rewrite 拒收，真实 10 集仍大幅波动，就要承认 rewrite 已经不是最后主战场**；这时不能继续围着单集修稿链打转，而要回到首稿生成稳定性、本批上下文、上游脚手架这些更前一层的根因
137. **Agent-first 不等于“首稿也要一次写满全部工艺”**；首稿的职责应该收缩到“先写出稳定可修稿”：场次稳、承接稳、冲突稳、结果稳。人物弧光、主题锚定、连续性精修这些高级工艺，应优先交给后续单职责 Agent，而不是继续把 prompt 堆成大而全总导演
138. **production 剧本返修正式分两段**：`Phase A` 逐集即时修结构真相和胖瘦问题（`format_pollution -> scene_structure -> char_count`）；`Phase B` 批后再做轻量内容工艺修整（`continuity / character / theme`）。同一类问题不准跨两段重复掌权
139. **首稿 prompt 的价值是降低波动，不是提前替所有 Agent 干活**；如果 prompt 已经长到同时指挥结构、人物、主题、制度场、尾段哲学和大量世界观细则，就说明又把 Agent-first 做回 prompt-first 了
140. **检测层最终要收成“问题分发单”**；问题码、Agent 映射、observe-only 和 actionable 必须清楚分开。检测负责告诉系统“该派谁修”，不负责自己变成第二裁判口
141. **当前主线必须有单独主线文档**；任务清单、进度、完成定义统一写进 `docs/主线/YYYY-MM-DD-主线.md`。主线完成后归档到 `docs/归档/主线/`，下一轮再新建，不准长期靠口头滚动推进
142. **主线是否继续，不靠感觉，要靠最新真实 evidence 决定**；如果当前主线的关键验证项已经用真实样本跑通，就先考虑收口归档和切新主线，不要因为“还能再补一点”就无限续命
143. **前段样本过线不等于长季已稳**；像这次 `20` 集前 `15` 集全过、但 `16-20` 集批次仍因 `duplicate_scene_detected:scene_16_to_17` 失败，就必须把新失败类型单独立成新主线，不能继续挂在旧主线尾巴上
144. **Agent 改名不等于 Agent 真接线**；只要 runtime 派单里还残着旧字段名、旧变量名或旧断言，整条内容返修链就可能静默短路。换口径时必须同时检查：prompt 签名、派单条件、输入字段、测试断言、typecheck。
145. **`generationStatus` 只认 main 掌权**；只要 main 已经通过 `project-generation-status-hub` 广播任务状态，renderer 就不能再本地先 `setGenerationStatus(...)` 造一份“正在生成”。否则正式真相已经落盘完成，前台仍可能卡成 `4410/160 秒 · 99%` 这种假进度。
146. **新的结构化 Agent 一旦进生产链，就必须同步具备“诊断 + 有限次纠偏重试 + 明确错误尾注”**；不能前一层 `faction-matrix-agent` 已经有 parse issue 诊断，后一层 `character-profile-v2-agent` 还停在裸 `JSON.parse` + 裸错误码。
147. **判断“是不是旧报错残留”不能靠感觉，必须先看最新主进程日志和最新构建 chunk 名**；像这次 `character_profile_v2_parse_failed`，只有对上 `manual-app-launch-restart-20260413-2.log` 和新 `out/main` chunk，才算坐实是新报错。
148. **七问这类正式前置确认步骤，只要后端已经有独立保存接口，前台就必须同步做成独立 stage 或至少独立操作闭环**；不能继续把“保存”和“继续跑重任务”绑在一个按钮里。
149. **重度复合任务必须区分“单次 AI 调用超时”和“整条 workspace 任务预算”**；单个 agent 的 `90s/120s` 不能直接拿来当 `outline_and_characters` 这种复合管线的总等待边界。
148. **人物/阵营这类可天然切片的结构化生成，优先按上游业务分块，不要继续要求模型一次吐完整张大表**；一次大 JSON 越长，越容易在严格解析链里触发截断、缺字段和重试拖时长。
149. **并发池的目标是“限流保稳”，不是“越高越快”**；像人物 V2 这种按 faction 切片后的调用，先把最大并发锁在 `3` 这类温和值，再根据真实 evidence 调，不准重新放回无上限 `Promise.all`。
150. **倒计时结束后白板本质上是失败不可见，不算成功等待态**；只要某个分块已经明确 parse/runtime 失败，就必须带具体对象名直接抛给前台，不允许“看起来还在生成，结果一个字都没有”。
151. **天然重任务必须有自己的 task 身份和 timeout 预算**；像 `faction_matrix` 这种世界观大表，不能继续借 `rough_outline` 的壳跑，否则日志、预算和前台报错都会错位。
152. **Stage 3 上游一旦超时，必须按对象名直接上浮**；`faction_matrix_timeout:*` 这类错误不准再被外层包成空对象、白板或笼统超时，必须沿 IPC 链直接给前台。
153. **正式发车前要留下握手证据，但握手日志是证据口不是 gate**；像确认版七问这类前置真相，入口应记录 `goal/obstacle/...` 是否有值，证明“米已下锅”，但不能反过来靠日志决定要不要继续生成。
154. **结构化大表生成一旦连续遭遇 provider 断流，正式链必须先退避、再按天然业务边界继续切小**；像人物 V2 这种势力表，不准继续整势力硬扛到超时，应先做有限次指数退避，必要时再降级到单人物请求。
155. **E2E 证据目录必须默认自清，不准把 `tools/e2e/out` 当永久仓库**；`userdata-* / evidence-* / e2e-tmp-* / Cache_Data` 这类运行残留必须由工具链自动修剪，人工清场只能当补刀。

## 本轮新增原则（2026-04-17 HTTP API 前后端分离）

156. **HTTP API 路由路径必须显式唯一**；不能靠注册顺序碰运气，同层级两个路由不能抢同一个路径。遇到 404 先查路由冲突，再查拼写。
157. **防御性访问用 `?.` + `|| []` 模式**；业务允许字段为空时，`sevenQuestions?.sections || []` 是标准写法，不要用 `if (x && x.y)` 冗长判断，也不要裸访问。
158. **认证、计费、限流等横切关注点放在路由中间件层**；业务服务不关心用户身份和积分，测试业务服务时不用 mock 认证和计费逻辑。
159. **前端 HTTP API 调用统一收敛到 `api-client.ts`**；不准在组件里直接 `fetch('/api/...')`，Token 管理、错误处理、重试逻辑都在 api-client 统一处理。
160. **HTTP API 服务端和 Electron IPC 主进程可以共存**；共享业务逻辑放 `server/src/application/`，环境特有逻辑各自独立，不准把 Electron 依赖塞进 HTTP API 服务层。
161. **积分扣费必须在业务执行前检查、业务成功后执行**；余额不足返回 402，业务失败不扣费。扣费时机和金额写在路由层，不写在服务层。
162. **HTTP API 错误响应统一格式**：`{ error: string, message: string }`，错误码用 snake_case，message 用中文用户可读文案。不准返回裸 Error.stack 或英文技术错误。

## 本轮新增原则（2026-04-09）

154. **人物页分轨先做“派生 + 升级”，不急着改持久化合同**；`light character cards` 和 `faction roster` 当前只允许从 `entityStore` 派生出来看和升，不要为了人物扩容马上再发明一套新的保存结构。
171. **只要某一阶段已经有过正式结果，这一页就必须保留显式“重新生成”入口**；不能逼用户删项目、重开项目或去别的页绕一圈，才有机会重跑当前阶段。
172. **阶段重跑必须复用当前正式生成链，不准为了“加个重生按钮”再发明旁路入口**；粗纲/人物就继续走“确认版七问 -> 粗纲和人物”，详细大纲就继续走正式详纲入口。
173. **上游阶段一旦重跑成功，下游旧结果必须同步让位**；像粗纲/人物重生后，旧的详细大纲和剧本必须一起清空，不能继续冒充当前版本。
174. **势力位自动展开当前只允许停留在 renderer 派生观察层**；`placeholderSeats` 这类“宫主位 / 执事位 / 情报位”只能帮助用户看清势力缺口，暂时不写回 `entityStore`，也不直接喂给详细大纲和剧本，避免占位符重新长成第二套人物真相。
175. **一旦势力位已经作为 `slot` 轻量人物卡正式落进 `entityStore`，renderer 就不能再把它当纯 placeholder 看待**；人物页、详纲和剧本都应该吃这批已落底账的轻量卡，不再另复制一套前台席位真相。
176. **前台 `placeholderSeats` 只能补“还没落底账”的观察缺口，不能和正式 `slot` 轻量人物卡并列掌权**；只要底账里已经有 slot 卡，renderer 就优先展示底账席位，不再继续靠本地蓝图脑补第二份名单。

## 本轮新增原则（2026-04-07）

145. **三层批次正式口径统一是 `5/5/5`**；粗纲、详纲、剧本都按 `5` 集一批，不准一层 `5` 集、一层 `10` 集再靠后面兜。
146. **自动返修只改一次，而且必须是同一个专家按问题单改原稿**；检查器只负责报 `P0/P1` 硬毛病和具体原因，不负责让另一批专家接力瞎重写。
147. **内容问题默认不停机**；自动改完一轮后就继续往后写，用户如果觉得某一集还不行，再由前台“重写这一集”手动触发同一专家继续改。
148. **前台切页和项目主进度不是一回事**；`project.stage` 继续代表正式进度真相，但用户点回 `chat / outline / character` 时，前台 `currentStage` 必须跟着切过去，不能拿主进度真相把页面浏览能力卡死。
149. **七问确认后的粗纲/人物生成，只要内容已经产出来，就不能再被 `character_bundle_incomplete` 这种内容不完整信号整轮打死**；这类信号只记 diagnostic，前台先显示已有结果。
150. **确认信息落盘前就要把角色名清洗干净**；`storyIntent` 一旦写脏，后面七问、粗纲、人物、详纲都会一起吃脏数据，所以 `protagonist / antagonist / officialKeyCharacters / lockedCharacterNames` 必须在持久化前先归一。
151. **确认信息里的集数口径要按聊天里最后一个“正式声明”取值**；旧结构化摘要可以覆盖更早的闲聊，但不能压过用户后面明确说的“改成 30 集 / 现在就按 30 集做”。
152. **只要生成底稿里已经有 `【角色卡】`，正式人物名单就只认角色卡**；`officialKeyCharacters / lockedCharacterNames` 和七问后人物小传都要优先对齐角色卡名单，角色卡外的人不准继续混进正式人物链。
153. **共享“归一化”函数不能偷带默认集数去改写现有 outline 真相**；像 `ensureOutlineEpisodeShape()` 这种既被显示层调用、又被编辑层调用的函数，未显式传入 `count` 时必须先信任 outline 自身真实集数；否则 `30` 集会在 renderer 里被静默截成 `10` 集。

## 已坐实的典型错误

### 1. 把阶段完成当成问题解决

过去很多计划会写"本质问题已解决""主线已完成"，但后面新事实又证明旧旁路还在。

以后看到这些信号，要先怀疑完成定义是不是太窄：

- 只证明这一层能跑一次。
- 只证明一个入口能跑。
- 只证明一个样本通过。
- 只证明代码静态搜索暂时清零。

### 2. 把局部真测当整链成立

过去常犯的错：

- 首场好，就当整剧好。
- 剧本一层暂时顺了，就当上游已经稳了。
- 详纲能出，就当详纲定义对了。

以后验收必须先问：

- 上游输入是不是唯一真相。
- 下游是不是只吃这一套真相。
- 旧旁路有没有真的退场。

### 19. 把"搜不到引用"直接当死代码结论，也是假完成

这次死代码清理又坐实了一件事：

- 静态 grep 看起来 `0` 引用，不等于真实编译链里真的没人依赖。
- `resolve-lane-strategy.ts` 和 `select-script-generation-batch.ts` 就被 grep 漏掉了正式 import。
- 真正把误判揪出来的，不是继续搜，而是 `npm run typecheck:node` 和 `npm run build`。

以后处理方式固定为：

- 先用 grep / LSP 找候选。
- 再小步改动。
- 最后必须用真实编译结果确认哪些才是真死代码。
- 只要编译链打脸，就立刻回修，不把"之前搜过了"硬写成事实。

## 本轮新增原则（2026-04-06 深夜）

144. 长季批内复写如果 evidence 证明“失败集的 `promptLength` 比上一集突然暴涨一大截”，先怀疑 prompt recap 在灌正文，不要先怀疑 `duplicate detector` 误判。
145. 承接块、ledger 块、momentum 块都只能喂摘要级信息，不能把上一集整段剧本文字重新塞回 prompt；那等于给模型发一份现成复写稿。
146. `当前集任务` 必须排在前情 recap 前面；agent-first 的意思是先让模型看见“这集要干什么”，再看“之前发生了什么”。
147. `evidence.promptLength` 默认只代表“这次尝试发出去的 prompt 长度”，不天然等于 base prompt 长度；只要 episode 进入二次/三次改稿，就会被 rewrite prompt 放大。
148. 只要 evidence 已经跑到 `ep20`，而项目落盘仍停在 `generation in progress / scriptDraft=[]`，就不要继续把主问题误判成“内容没生成出来”；那是保存链或返回链问题。

### 20. 把"有人手工用"误判成"正式生产在用"，也是错

根目录调试脚本、`tools/e2e` 验证脚本、历史设计文档，也会持续引用正式模块，原型模块或历史 helper。

如果不先把它们和"正式生产依赖"分开，就会出现两种误判：

1. prototype / 临时脚本因为"有人在用"被错当成正式生产链
2. 某个模块因为只被调试脚本或原型测试消费，又被错当成现役正式能力

所以以后先问的不是"有没有引用"，而是"**这个引用属于哪种身份**"。

### 3. 哪层炸了就只修哪层

这会把下游表面问题当根因，越修越碎。

正确做法：

- 先看第一次越界的是哪层。
- 先看上游是不是已经把不该交的债交下来了。
- 先看有没有第二口径在偷跑。

### 4. 把优化做回限制

表面上说"优化不限制"，实际动作却变成：

- 多加规则。
- 多加说明。
- 多加守卫。
- 多加禁止项。
- 多加补丁块。

只要系统越来越像审查器，而不是越来越像创作脚手架，就说明方向又歪了。

### 5. 让旧文档继续像现役入口

这会直接制造第二口径。

典型表现：

- 新三件套已经改口了，旧任务卡还在说另一套主线。
- 旧 worklog 还在根目录，像当前入口。
- README、ARCHITECTURE、旧计划总表还在对外发声。

### 6. 假完成最容易藏在"结构收口"里

过去有过这种情况：文档写"唯一真相源已收口"，但后面又发现：

- 生成入口还有二次 summarize。
- 系统还会补写正式 facts。
- `activeCharacterBlocks` 还是双来源。
- 自动推进还在后台继续开工。

所以以后只要旧旁路还活着，就不准把"唯一真相源已收口"写进现役文档。

### 7. 只改现役文档，不做物理退场，也是假收口

典型表现：

- 现役文档已经说只认 5 份，但根目录旧 README、旧 worklog、旧计划目录还在原位。
- `docs/archive`、`docs/plans`、`docs/当前工作区` 这种旧入口名字还挂在 `docs` 根目录。
- 全仓一搜，旧 Markdown 还散在 `.codex`、`.sisyphus`、`tools/e2e`、`测试` 里。

以后只要做文档收口，必须补这一步：

- 旧文档真的搬走。
- 旧入口名字真的退场。
- 全仓搜索确认 `docs/归档/` 外只剩 5 份现役文档。

### 8. 只改代码不补四件套，也是假收口

典型表现：

- 代码这轮已经收住了，但 `项目结构.md` 还是旧的。
- `项目原则.md` 没记这轮新踩坑和新禁区。
- `AI自我学习.md` 没记这次为什么成、为什么败。
- `聊天记录.md` 还是旧主线，下一终端一进来就接错方向。

以后每轮任务收口，默认补这 4 步：

- 更新 `聊天记录.md` 的当前主线和最新进展。
- 更新 `项目原则.md` 的新原则、新禁区、新误判。
- 更新 `AI自我学习.md` 的新理解、新踩坑、新提醒。
- 更新 `项目结构.md` 的新结构落点、新风险区、新归档变化。

### 9. 把运行时依赖直接绑进业务模块，会把工程验证卡死

这次详纲单测的真实问题不是"测试环境不好"，而是业务模块自己直接 import 了 Electron 诊断日志。

典型表现：

- `generate-detailed-outline.ts` 这种本来该能纯 Node 跑的模块，直接 import Electron 运行时文件。
- 结果单测一加载模块就炸，连业务逻辑都还没开始测。

以后处理方式固定为：

- 纯逻辑模块只依赖合同、纯函数、注入接口。
- Electron 日志、窗口、IPC、文件路径这类运行时东西，只能放在外层入口接进去。

### 10. 原生 Node 测试链上的 shared / renderer 运行时文件，也不能随手写

这次 `detailed-outline-generation-notice.test.ts` 和 `resolve-detailed-outline-persistence.test.ts` 又坐实了一件事：

- 不是只有业务模块会卡纯 Node 单测。
- 共享守卫、失败提示、运行时边界文件，只要会被测试直接 import，一样会在模块加载阶段先炸。

典型表现：

- 运行时 import 还在省略 `.ts` 后缀。
- `AuthorityFailureError` 这类共享错误类用了 Node strip-only 不支持的 parameter property。

以后处理方式固定为：

- 原生 `node --test` 要直接加载的运行时文件，导入后缀写全。
- 这类文件避免使用 Node strip-only 不支持的 TS 语法。
- 先保证测试链能加载，再谈业务判断有没有通过。

### 12. 链式操作里参数传错会让"保存成功"显示成"保存失败"

这次 `scriptquality-matrix.mjs` 的 `runScriptChain` 里发现了一种新型误判：

- `saveScriptDraft` 和 `saveScriptRuntimeState` 都正确执行了（数据已写入磁盘，脚本数=10 可以证明）。
- 但 `loadProjectFromStore(caseOutDir, projectId)` 始终返回 null，导致 `draft=null`，runner 报告 `save_chain=❌`。
- 根因：`caseOutDir`（= `tools/e2e/out/`）和 `userDataDir`（= `tools/e2e/out/userdata-sq-{caseId}-{ts}/`）只差一个 `userdata-*` 子目录，但正确路径是后者。

教训：链式操作（写→等→读）里，读的路径必须和写的地方完全对应。参数名相似（`caseOutDir` vs `userDataDir`）不等于值相等。

以后看到"操作成功但验证失败"，要优先查读写路径是否真的对上了。

### 11. "算了但没用" 是最隐蔽的假放行

这次 `buildScriptGenerationExecutionPlan()` 又暴露了一种新型假放行：

典型表现：

- 函数内部先调用了 `validateStageInputContract()` 得到了真实的 stageValidation。
- 但最后 return 时直接硬编码 `ready: true / blockedBy: []`，完全丢弃 stageValidation 的结果。
- 表面上有校验，但校验结果根本没有影响输出。

这次剧本入口的问题不是"没有校验"，而是三处同时存在：

1. `buildScriptGenerationExecutionPlan()` —— 算了但没用，硬编码返回。
2. `evaluateStageAccess('script')` —— 无条件 ready: true，不管 plan 内容。
3. `workflow:start-script-generation` —— 没有 guardian，直接进生成。

以后看到"先调用了验证但最后硬编码返回"这种模式，要直接当成假放行处理。

### 13. rewrite 模式的拼接顺序必须是 `[新内容, base]`，不能是 `[base, 新内容]`

这次 P1-2 真实模型验证发现 rewrite 模式拼接 bug：

- `runScriptChain` 里 rewrite 逻辑：`base = script.filter(episodeNo > limit)`，`finalScript = [...base, ...newScenes]`。
- 结果：新生成的 episodes 1-10 被接在了 overflow episodes 11-12 **后面**，`judgeScript` 读到 `scriptDraft[0]` = overflow（无 dialogue）→ quality=弱。
- 修复：`finalScript = [...newScenes, ...base]`。

教训：rewrite 模式下新内容替换 episodes 1-limit，overflow 追加在最后。拼接顺序必须保证新内容在前。改 rewrite 逻辑时要同时验证 `scriptDraft[0]` 确实是新内容的第一条。

### 15. 真实 E2E seed 本身也必须满足官方质量裁判口径

这次 RS-A 真实模型验证发现：

- RS-A 的 plan/gen/save/runtime 全部通过，但 postflight 报 quality=弱。
- 根因不是系统 prompt，不是 batch，不是 save_chain，而是 seed 的 ep1 内容太薄（107 字符，无压力词，无钩子词）。
- `judgeScript` 正确地捕获了这个问题，但误判为"系统 postflight 质量问题"。
- 修复：补强 RS-A seed ep1（扩写至 571 字符，含"代价""下一招"等压力词和钩子词），RS-A real 立即 PASS。

教训：真实 E2E 测试失败时，要先确认 seed 内容是否满足 judgeScript 的质量门槛，再判断是系统/prompt 问题还是 seed 质量问题。测试样本缺陷不能拿来当系统缺陷的证据。

### 16. 把旧绿灯当成今天还能背书，也是第二层假完成

这次 2026-03-28 凌晨复核又坐实了一件事：

- 2026-03-27 文档里写过 `npm run build` / `npm run typecheck` 通过、3 个 real case PASS。
- 但当前代码和当前口径一复跑，`build/typecheck` 已被新 WIP 测试文件打红，`fs-b` / `rs-a` / `rw-b` 在 official quality 下也全部 FAIL。

以后处理方式固定为：

- 历史 PASS 只能当线索，不能当现役结论。
- 要指挥下一轮任务，必须先用当前代码、当前口径、当前命令复跑。
- 只要复跑结果变了，就直接改现役文档，不保留"昨天还是绿的"这种安慰性说法。

### 17. 同一集多版本并存时，默认取第一条，会把旧稿重新扶正

这次 rewrite 收口又坐实了一件事：

- 第一批重写后，`existingScript` 很可能变成 `[旧1,旧2,旧3,新1,新2,新3]`。
- 如果"查上一集"还是按 `find()` / 第一条命中，就会在第二批续写时重新吃回旧版。
- 表面像"模型又不稳了"，本质上是上下文装配把旧稿扶正了。

以后处理方式固定为：

- 同一 `episodeNo` 多版本并存时，必须显式 `latest wins`。
- batch 内新生成的内容优先级高于旧稿。
- 先排上下文装配，再下结论说"模型不稳定"。

### 18. 让生成合同、解析入口和质量合同互相打架，最后一定会把多场剧本压回单场

这次剧本 sceneCount 问题又坐实了一件事：

- prompt 明面上要求 2-4 场和 scene heading。
- 主生成入口却长期先走 A/D/E。
- repair 入口也只改 A/D/E，不刷新 `screenplay / screenplayScenes`。

结果就是：

- 模型即使写出了多个 heading，也可能在 `extractSceneSections()` 和单场重建里先被压扁。
- quality 最后报"场次数不在 2-4 场"，但真正的第一次越界发生在入口合同。

以后处理方式固定为：

- 先收一套正式交付格式。
- 先看入口路由是不是和质量合同一致。
- repair 只要改了正文，就同步刷新 `screenplay / screenplayScenes`，不准留陈旧结构继续掌权。

## 以后看到这些信号，立刻停下来

1. 同一件事在不同文档里有不同答案。
2. 同一字段在磁盘、主进程、renderer、兼容层各有一份。
3. 刚说已完成，下一轮又要用新计划重修同一根问题。
4. UI 展示字段反过来控制下游逻辑。
5. 兼容层、repair、fallback 开始偷偷改正式内容。
6. 自动推进开始在用户没确认时继续往下跑。
7. 测试脚本、导出脚本、临时工具开始绕开主链写正式结果。

## 有效做法顺序

以后遇到主链问题，默认先按这个顺序做：

1. 先确认当前现役文档是不是只有 5 份在掌权。
2. 先确认当前主线是不是只有一条。
3. 先确认用户输入是不是还在当唯一真相。
4. 先查旧兼容、旧 fallback、自动推进、旧任务卡有没有回流。
5. 再查具体技术点、提示词、超时、格式问题。
6. 只要命令会拉起 `vite-node`，就默认交给独立执行终端，不在当前指挥终端直接跑。

## 最后一句

这个项目最危险的不是偶尔失败，而是旧判断、旧入口、旧旁路还在继续掌权。

只要还有第二口径，就不算真的收口。
118. 正式任务名一旦收口，错误码、日志前缀、UI 解释文案也必须同步收口；不能运行时已经叫 `rough_outline`，对外还继续报 `outline_bundle_*`。
119. 唯一真相源一旦确认，项目主快照合同里就不要继续保留同名顶层字段；`deprecated` 不是收口，移除才是收口。
120. `tools/e2e` 一旦确认官方口、观察口、分析口分层，纯分析脚本就要物理归到 `analysis/`；否则同目录混放会继续制造“它也像官方入口”的错觉。
121. `tools/e2e` 里的共用基础件不能被误判成分析脚本；像 `tools/e2e/utils/evidence-routing.mjs` 这种一旦被官方口直接依赖，就统一留在 `tools/e2e/utils/`，不跟 runner 一起混搬。
122. 根目录临时脚本只要确认没有正式消费方，就不能继续挂在仓库根目录；要么直接删，要么统一沉到 `scripts/archive/` 之类的专门归档区。
123. `tools/e2e` 里的原型场级脚本也不能继续和现役 episode 级 runner 混放；明确是 prototype 的，就统一收进 `prototype/`。
124. 剧本主链里的 repair 如果要算“已完成”，就必须把修后稿随同主返回链一起落盘；只发一个“后台 repair 完成”的进度文案，不算收口。
125. 七问后如果详纲入口仍要求 formal facts 已确认，就不能把“粗纲和人物已生成”当成下游已放行；facts 还是 `draft` 时必须显式确权或补自动确权，不准靠用户下一轮撞错误码才发现。
126. 如果“用户已确认七问”本身已经是正式确认动作，那么同一条 outline 工序里新生成的主线 facts 就不能继续默认落成 `draft`；不准让上游确认语义和下游 formal fact 合同互相打架。
127. 剧本 prompt 里只摆“故事合同/关系施压/硬事实”资料块，不算真正承接；只要 audit 还在报 `relationship_shift_missing` 或 `hard_anchor_pending`，就必须把它们收成“本集必须发生的动作、改位、代价和结果”。
128. 真实全链验证脚本一旦继续掌权，就必须跟现役工序同步升级；正式主链改成 `七问草稿 -> 七问确认 -> 基于确认版七问生成粗纲人物` 后，旧脚本再直接调历史 `generateOutlineAndCharacters`，就不再算有效验收。
129. 如果 `10` 集 fresh 已绿、`30` 集只在末集留下 `f6_playability_postflight_weak` / `f6_continuation_postflight_weak` 这类低级红灯，就不要再回头怀疑前面的关系推进、硬锚点和 facts 工序；主线应直接转到末集 F6 收口，而不是把已坐实的合同层重新打散。
130. 如果真实 evidence 里已经看到同场号占位头（如 `人物：人物`、`△第30集`）混在正式场前面，就不要把末集 F6 红灯只归咎于“承接词根太弱”；先收解析真相，把 placeholder stub 从正式 sceneCode 上剥掉，再判断剩下的是 prompt 还是内容问题。
130. 某轮 fresh 30 集如果已经 `scriptPass=true`，也不能反过来假装 placeholder 污染已经全季清零；只要最新 `script.json` 里仍能搜到 `人物：人物`、`△# 第28集` 这类残留，就要明确记成“末集已绿，但中后段同类污染仍在”，不能拿一次总通过把局部残留抹掉。
131. placeholder stub 不只会长成纯文本版，也会长成 markdown 版；像 `人物：**人物**`、`△# 第28集`、`## 28-1 ...` 这种壳，一旦混进正式 screenplay，就要在 `normalizeText / fake roster / placeholder action` 三层一起收，不能只靠一个正则碰运气。

## 本轮新增原则（2026-04-06 晚）

132. 用户亲口给出的短剧写作方法论，是剧本层唯一创作口径；必须进正式合同，不能只写在 prompt 注释里。
133. 项目级创作控制必须有唯一真相源；当前主线目标收在 `storyIntent.shortDramaConstitution`，不准让 prompt、ledger、agent 各自再解释一套。
134. 集级创作控制必须有唯一真相源；当前主线目标收在 `episodeBeats[].episodeControlCard`，不准再靠 prompt 临场拼装“这集要干什么”。
135. `快节奏、强冲突、稳情绪` 属于创作宪法，不属于门禁；系统只能用它派单给 Agent，不准把它实现成 hard gate。
136. 首稿 prompt 只负责执行当前集控制包，不准继续兼任 showrunner、弧光裁判、主题裁判和情绪裁判。
137. 创作返修 Agent 必须按“推进 / 弧光 / 情绪 lane”三类单职责分工；继续保留 `continuity / character_arc / theme_anchor` 和新口径并行，等于双轨。
138. 如果用户要求 `Agent first`，就不要再用“先卡死、再失败、再人工决定”冒充 agent-first；内容差就继续派 Agent 修。
139. 长季 `duplicate / persist` 这类技术问题只能作为新架构验收项，不能继续压过当前创作主线。
140. 项目级创作真相源必须在 `confirm_story_intent` 时落盘；不能等到粗纲、详纲或剧本 prompt 再临时拼一套“创作宪法”。
141. 集级创作真相源必须在 `detailed_outline` 时落到 `episodeBeats[]`；不能等到剧本 prompt 再临场猜“这一集该怎么写”。
149. 剧本首稿 prompt 必须先看项目级创作宪法和当前集控制卡，再看 recap；不能让 `前情提要` 抢到控制包前面，否则 prompt 会重新退回 recap-first。
150. 只要 `短剧创作宪法 / 当前集控制卡 / sceneByScene` 已经同场出现，旧散规则就只能降级为辅助说明；发生冲突时，必须明确以这三层为准，不能继续双轨并列掌权。
151. 真 Agent 接线一旦完成，验收优先补无 fixture 的定向接线测试；不要让 `tools/e2e/out/...` 这类本地 evidence 目录变成唯一证据来源，否则缺目录会把“fixture 不在”误判成“生产没接上”。
152. 剧本阶段一旦确认要吃“两级控制包”，执行计划和 runtime input 就必须显式带这份合同；只让 prompt 去上游字段现查，仍然算隐式透传，不算正式收口。
153. runtime 控制包一旦存在，剧本 prompt 和返修 Agent 都必须优先读它；上游原字段最多保留为历史兼容信息，不能继续反向抢权。
154. 正式派单口径收口后，文件名、prompt builder 文件名和定向测试文件名也要物理跟着改；只改函数名不改文件名，仍然会让旧口径继续发声。
155. 只要当前新口径已经用真实 fresh run 跑出 `10/10` 全过，就先把它记成现役 fresh evidence；不要再让更早的旧失败样本继续压过当前事实。
156. 如果当前新口径下真实 `20` 集已经 `success=true` 且 `generatedScenes=20`，就不能再把主问题写成“内容问题会把整轮打成失败态”；这时应把判断收缩到具体剩余红点（例如单集胖稿）。
157. rewrite 选稿不能把“单个轻问题”机械判得比“严重离合同很远”更重；像第 `14` 集这种“首稿严重超胖，rewrite 已压回合同附近，只剩 `voice_over`”的场景，正式链必须保留更接近合同的 rewrite 稿，不能退回到胖旧稿。
158. 如果 `waitForProject / script_ready` 先超时，不要立刻把超时瞬间快照写成“保存链坏了”；必须先核对最终 `projects.json / script.json / visible.json`，确认 Electron 后台是否已继续跑完，再决定是不是产品问题。
159. 用户要求“只做指挥计划”时，主线文档就不能继续写成散点待办；必须改成全盘执行计划，明确工作面、边界、交付物、验收口径和停手条件，让执行人拿过去就能干。
160. 详细大纲写不了时，要直接把原因翻成前台动作；不能只让用户关掉失败提示后自己猜该回粗纲还是回人物。
161. 如果系统要求用户“回粗纲补关键设定”，那粗纲主页面就必须真的挂出可点击的关键设定确认入口；不能一边卡这条，一边把入口藏起来。
162. 关键设定确认可以作为粗纲层显式能力存在，但不能再拿它当详细大纲的硬门禁；粗纲正文已经成立时，详纲应允许先写。
163. 长剧人物系统默认按“势力先行”建模；先有势力、人物位和关系，再决定哪些人物升级成正式小传，不要让 prompt 临场现编整套人事结构。
164. `characterDrafts` 只代表完整人物小传，不代表全体人物名册；功能人物和阶段性人物默认允许只保留轻量人物卡。
165. 长剧人物规模只给区间建议，不设硬人数门槛；`20` 集约 `8-12` 个活跃人物、`40` 集约 `12-20` 个活跃人物这类口径只能指导设计，不能反过来做 gate。
166. 详纲和剧本默认只吃当前批次活跃人物包，不全量吃整季人物；否则人物一多就会同时把 prompt 撑爆、把门禁做回“全员先补完整小传”。
167. 确认信息一旦已经收成 `generationBriefText`，实体拆解就应在这一层正式落成 `entityStore`；不要把世界底账继续留给后续 prompt 临场猜。
168. `entityStore` 新增实体时，默认优先保留已有稳定 id 和 richer fields，只补新识别出的底账信息；不要每次确认信息都把旧实体整包重置。
169. 只要 renderer 还在直接调 `workflow:start-script-generation` / `workflow:rewrite-script-episode`，就必须把当前项目快照里的 `entityStore` 一起带进 request；否则 `activeCharacterPackage` 只是在共享层算出来了，正式产品入口并没有真的吃到。
170. 共享派生只是“有字段更准、没字段也能跑”的时候，不要顺手把这个字段抬成全链硬必填；像 `protagonist` 这种批次激活辅助信息，应当保持可选，避免为了一个派生提示把所有调用方一起改成强依赖。
171. 七问确认这类“上游已正式确认”的动作，不能只看前台返回成功；继续下游前必须能从正式项目快照回读到 `outlineDraft.outlineBlocks[].sevenQuestions`，回读不到就按保存失败处理，不能静默往下生成。
172. 普通 `saveOutlineDraft()` 如果 incoming outline 缺 `outlineBlocks`，默认要保留 existing 的篇章块和七问；空字段不能覆盖已确认的篇章级真相，否则就会把“已确认七问”擦成“未确认”死循环。
173. 旧项目一旦已经有确认版 `generationBriefText`，就不能因为 `entityStore` 是历史空值而继续把人物页显示成“轻量人物卡=0 / 势力=0”；读项目时必须先按已确认信息回填世界底账。
174. 势力拆解必须优先认真实势力名，不认糊成一整串的描述文本；像 `玄玉宫`、`天地会`、`青木堂` 这种要拆出来，`天地会青木堂`、`玄玉宫等七座道观` 这种脏串不能直接进正式底账。

175. `docs/项目结构.md` 必须是仓库总地图，不是最近几轮进展摘录；它至少要先写清正式主链、代码分层、正式线/观察线/原型线/历史壳、问题定位口和风险区，保证新终端看一遍就知道哪里该动、哪里别动、哪里先查。
176. 旧项目页面如果已经用手工联调坐实一条关键显示结论，下一步就要补稳定 `data-testid` 和 observation 回归脚本；不能长期只靠截图和人肉点击复核同一条 UI 事实。
177. `episode_control_agent` 不能只看宪法和 beat；只要控制卡要设计 `openingBomb / conflictUpgrade`，就必须显式对照当前人物小传做行为边界检查，防止控制卡把人物写崩。
178. 上游人物真相一旦变化，下游正在生成的剧本就必须立刻 stale warning 并停；不准继续沿用旧 `context.characters` 快照假装同一轮还能往下写。
179. `character fingerprint` 这种预警能力，只有真的接进运行时监听才算成立；“合同里能算出来”但启动链和 batch 链没监听，等于没有预警。
180. 人物行为边界必须进入正式创作宪法，而不只是一句 prompt 提醒；剧情冲突只能基于人物当下心理状态展开，不能靠降智、硬拧、突变去制造反转。
