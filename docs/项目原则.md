# 项目原则

## 核心原则

1. 用户输入是唯一真相，系统不替代创作裁判。
2. 发现双口径不打补丁，直接收一套。
3. 项目不要先补丁，要修源头问题。
4. 不允许第二写口、第二发布口、第二裁判口。
5. 系统只做引导、结构化输入、轻量校验和最小必要机制。

## 工程原则

1. server 是唯一正式业务入口。
2. PocketBase 是唯一正式项目真相存储。
3. renderer 正式业务只准走 HTTP `api-client`。
4. IPC 只保留桌面壳职责；HTTP 对齐后，对应 IPC 必须物理退场。
5. `src/shared/` 是前后端唯一共享合同层，严禁双轨副本。
6. 假迁移不算迁移——把 authority 从一个 main 文件挪到另一个 main 文件不是真迁移，必须后移到 server 或下沉到 src/shared/。
7. "server 有壳但 main 才是真逻辑"不算后移——server 端路由如果只是 TODO/占位，main 端旧实现就不是真正退场。

## 文档原则

1. 现役规则只认 `CLAUDE.md`；`AGENTS.md` 只做同步指针。
2. `docs/` 只保留 5 个活文档 + `当前主线.md`。
3. 其他文档全部归档到 `docs/archive/`，不再参与现行判断。
4. 每次代码改动后、每次出现新问题、每次出现新结论时，立即同步五件套。
5. 活文档尽量控制在 200 行以内，超过就压缩。

## 验证原则

1. 任何功能必须跑通再算完成。
2. 不许只看代码就说“完成”。
3. 默认至少看 `npm test`、`npm run build`、`npx tsc --noEmit --pretty`。
4. 提示词和文案质量测试，允许真实调用模型 API。
5. 第三方模型通道优先，第三方连不上才回退官方 DeepSeek。

## 架构原则

1. 回答架构问题前先读 `graphify-out/GRAPH_REPORT.md`。
2. 新项目默认前后端分离：前端一个目录、后端一个目录、生成逻辑和提示词放后端。
3. 多项目共享 PocketBase 时，必须按 `appId` 和 `<appId>_xxx` 业务表前缀做物理隔离。
4. `user_apps` 管准入，`user_wallets` 管积分，`<appId>_transactions` 管项目流水。
5. 项目积分绝不能写在全局 `users` 表里。

## 典型误判

1. 把某一层通过当整条链通过。
2. 把旧截图、旧日志、旧 PASS 当现役结论。
3. 发现双口径后继续保留兼容分支。
4. 回答架构问题时不看图谱，靠印象拍脑袋。
5. 文档口径已经换了，但旧文档没有物理归档。
