# AI 自我学习

这个文件只记录我从 AI 视角对项目的理解、踩坑、成败原因和后续提醒。

断线恢复时，先看 `AGENTS.md`，再看 `docs/聊天记录.md`，然后看这份文件。

---

## 我对这个项目的理解

这是一个 AI 剧本创作桌面软件，不是让 AI 直接一把写完全本，而是把创作拆成明确工序：

用户输入 → 创作信息总结/故事梗概 → 七问候选锁定 → storyIntent → 剧本骨架 → 人物小传 → 详细大纲 → 剧本

真正难的不是"怎么不报错"，而是：
- 用户输入必须始终是唯一真相
- 上游一旦确认，下面只能展开，不能偷偷改写
- 系统工程保护不能越界长成第二口径

---

## 为什么过去反复失败

1. **没有唯一掌权的记忆系统**：同时有太多文档在发声，旧任务卡、旧 worklog 没退场，新终端先看到过期但还像现役的东西。
2. **把"这轮范围内通过"当"本质问题已解决"**：旧旁路没死、旧兼容还在、新主线还没覆盖到下游。
3. **把阶段问题误判成单点 bug**：下游问题常是上游越界、双口径、补丁链残留的结果。
4. **把局部真测当整链成立**：首场顺不等于整季顺，单层顺不等于整链顺。
5. **计划本身也会制造第二口径**：分计划各讲各的，被事实推翻的还保留"已完成"口气。

## 已确认的有效做法

### 主权与记忆

- **先收主权，再修细节**：删旧入口、收死主写口、统一合同、暂停自动推进——这个顺序证明有效。
- **旧文档必须物理退场**：只靠嘴上说"过期了"没用，只要旧文档还像现役入口，新终端就会信它。
- **每轮收口必须补四件套**：聊天记录、项目原则、AI 自我学习、项目结构——缺一就会失忆。
- **文档收口带全仓自检**：只写对新文档不等于旧入口退场，必须搜全仓确认只剩 5 份现役。
- **假迁移不算迁移**：把 authority 从一个 main 文件挪到另一个 main 文件不是真迁移，必须后移到 server 或下沉到 src/shared/。validateStageInputContract 在 Wave 4 被用户当场纠正，改为共享模块后真正收口。

### 单一真相与双口径

- **唯一真相源确定后，顶层重复字段直接从主合同移除**：deprecated 注释不够，字段还在就会诱导第二真相源回流。
- **正式确认后回读验证**：确认动作走 HTTP 后，保存成功判断也必须走服务端回读，不能只靠本地乐观状态。
- **authority merge 防覆盖**：普通保存不能顺手把已确认的权威字段用空值覆盖掉。

### PocketBase 与存储

- **PocketBase JSON 字段直接传对象，不需要 stringifyJson()**：double-stringify 会导致 400 错误。
- **PocketBase JSON 字段 required=true 时空数组也算缺失**：用占位符 `[{ _dummy: true }]` 绕过。
- **项目云端落盘不能照搬本地整包思路**：主表放项目级真相，阶段数据独立分表，各自带 version。
- **前端不该知道数据库拆了几张表**：快照缝合必须留在服务端 mapper，不准推给前端拼。

### HTTP API 与前后端分离

- **认证/计费/限流放在路由中间件层**：业务服务不关心用户身份和积分。
- **前端统一封装 api-client**：所有 HTTP 调用收归一处，不准在组件里直接 fetch。
- **长时间后台任务必须 Start → 202 → 轮询**：不能同步挂起，否则连接超时。
- **迁一半会制造第二正式主链**：HTTP 链必须覆盖正式等价能力 + 接管正式真相存储 + 旧入口有退场计划。

### 验证纪律

- **内容质量必须跑真实 API**：mock、单测、结构校验只能验流程和合同，不替代内容质量验收。
- **历史 PASS 会过期**：只有当前代码 + 当前口径 + 当前命令三件套同时再过一次，才配写成现役可信结论。
- **fresh real run 是最终证据**：代码改动后不 build 不跑真实链，很容易把旧产物误当新事实。

## 最容易再犯的错

1. 看见超时/格式错/报错码，直接追技术点，不先查第二口径。
2. 看见某一层顺了，就下意识觉得主链快好了。
3. 看见旧计划写得很完整，就把它当成今天还有效。
4. 看见"唯一真相源已收口"这种旧结论，没先核对有没有新事实推翻。
5. 把历史参考文档当成现役入口。
6. 以为"现役文档写完了"就等于文档系统已经收口，没有再做物理迁移和全仓搜索。
7. 把历史 PASS 当现役，不复跑就信。
8. 把带 vite-node 的命令直接丢在当前指挥终端跑，把会话跑崩。
9. 精确进程管理是允许的，但杀进程前必须确认 PID、命令行和用途，不能误杀 AI 宿主、当前终端或项目外关键进程。
10. 跨层错误响应字段不定义正式类型，用类型断言临时处理（shouldLogin 问题）。

---

## 已验证的判断

1. 前后端分离是正确方向；HTTP API + PocketBase 让调试和部署都更简单。
2. 文档整理先于代码大改；先收口旧口径，再动手，少走很多弯路。
3. 单测/typecheck 只能验结构，不能验内容质量；质量验收必须真实调用大模型 API。
4. QA 必须跑真实 UI 流程并截图存证，不只依赖单测。
