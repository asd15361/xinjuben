# doc-sync（文档同步机制）

这个文件只回答一件事：
**我们这个仓库的通用文档，怎么和 `D:\vivecoding规则` 保持每日对齐。**

## 同步源
- 唯一外部同步源：`D:\vivecoding规则`
- 只有这个目录里的通用规则文档，才作为“母版规则仓库”。

## 同步原则
- 我们仓库里的通用文档，要向 `D:\vivecoding规则` 对齐。
- 不是只抄标题，而是要对比差异：
- 哪些文档新增了
- 哪些规则新增了
- 哪些规则改了表述或强度
- 哪些旧规则被删除或挪位了
- 对比完后，把应该同步的内容补进我们仓库。

## 每日开工前固定动作
1. 先看本文件里的“最近一次同步记录”日期。
2. 再看 `D:\vivecoding规则` 对应文档的 `LastWriteTime`。
3. 如果本文件记录不是今天，先做今天的同步，不准直接开工。
4. 如果今天已经同步过，而且源仓库今天也没有更新，就不重复同步。
5. 如果今天已经同步过，但源仓库今天后来又更新了，仍然要追加一次当天同步记录。

## 必同步文档范围
- `AGENTS.md`
- `2.rules.md`
- `3.agents.md`
- `4.worklog.md`
- `docs/operator-preferences（用户协作偏好）.md`
- `docs/prompt-writing（提示词写作方法）.md`
- `docs/system-authority（系统定义权与主权规则）.md`
- `docs/engineering-workflow（工程排障与根因修复）.md`
- `docs/intel-sop（榜单情报与样本学习）.md`
- `docs/当前工作区/active-task（当前任务卡）.md` 的通用机制部分

## 同步方法
- 先看源仓库今天改了哪些文件。
- 再逐个比：
- 我们缺不缺这个文档
- 我们有这个文档，但内容有没有落后
- 我们自己的项目特有内容，会不会和源规则打架
- 如果不打架：直接同步到最新。
- 如果会打架：保留项目特有内容，但要把源规则的本质要求补进来。
- 如果某条规则只适用于通用仓库、不适用于本项目：要写清为什么不全量照搬，不能悄悄跳过。

## 同步后的固定动作
- 更新本文件里的“最近一次同步记录”。
- 在 `4.worklog.md` 记一条“今天做过文档同步”。
- 如果这次同步改了首页导航、硬规则、主权规则或用户协作规则，要顺手检查当前任务卡是否需要一起更新。
