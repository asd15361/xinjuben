### 2026-03-29 下午：sceneCount 根因定位——bold-markdown heading bug

- 问题：真实模型输出 `**6-1 日**` / `**6-2 日**` / `**6-3 日**`（markdown bold 包裹 scene heading），导致 Route 1.5 regex `/(?:^|\n)\d+\-\d+\s+/` 无法匹配，`normalizedText` 为 `**` 包裹内容，全部被压成 1 场。

- 修复：
  1. `parseGeneratedScene` 添加 `normalizedText = text.replace(/\*\*([^\n]+)\*\*/g, '$1')` 预处理，剥除 ** 标记后再做 heading 检测。
  2. Route 2 的 `looksLikeScreenplayFormat` 检查也改用 `normalizedText`。
  3. `screenplay-quality.ts` 新增 `sceneCountFromScenes`，优先读 `screenplayScenes.length`（generation 原始），fallback 到 `parseScreenplayScenes(screenplay).length`。
  4. `executeScriptRepair` 的 screenplay/screenplayScenes 保护逻辑强化：只有 `looksLikeScreenplayFormat(rawText) === true` 才覆盖，A/D/E 格式 repair 输出保留 generation 结果。
  5. `RAW_TEXT_MAX` 2000 → 8000 并加 `truncated` 标记。
- 验收：`parseGeneratedScene.test.ts` 8/8 ✅，`npm run build` ✅，commit f4dda9a 已入库。

## 这轮新确认的重大发现

1. 剧本入口的假放行不是 renderer 一处的问题，而是 plan 构建 / evaluateStageAccess / start-script-generation 三处同时硬编码或跳过了 validateStageInputContract。
2. `buildScriptGenerationExecutionPlan()` 虽然先算了 stageValidation，但最后直接丢弃结果硬编码返回——这种"算了但没用"是隐蔽性最高的假放行。
3. `switchStageSession()` 用 renderer 传来的 `targetStage` 直接覆写 `project.stage`，是把 UI 导航意图当成权威 stage 来源，违反"main derives, renderer consumes"原则。
4. 历史验收绿灯如果不按当前代码、当前口径、当前命令再跑一遍，就不能继续当现役事实；3/27 的“3/3 real PASS”就是这次被 3/28 凌晨复跑推翻的。
5. 板块单测全绿不等于当前工程仍然是绿的；并行施工中的新测试文件一样能把 `typecheck/build` 当场打红。
6. 剧本碎片化不能一上来先怪模型不稳定；rewrite 前情取错、batch 上下文冻结、跨批次旧稿优先，这些结构 bug 会伪装成 AI 输出不稳。
7. `mock` 下 `postflight=FAIL` 如果只是 stub 太短，是质量门正常工作，不是流程失败；流程口和质量口必须分开验。
8. `episode_script` 的正式合同如果同时要求“2-4 场 + scene heading + A/D/E 三段输出”，最后一定会在入口路由处互相打架；这不是 prompt 小问题，是合同设计错位。
9. `parseScreenplayScenes()` 会不会认多场，不是当前第一关键；真正决定多场能不能保住的，是 `parse-generated-scene.ts` 和 `parse-ai-repaired-scene.ts` 到底先选 screenplay 还是先选 A/D/E。
10. repair 只改 `action / dialogue / emotion`，不刷新 `screenplay / screenplayScenes`，会把旧场次结构继续当现役真相喂给后面的 repair guard 和 quality 链。
