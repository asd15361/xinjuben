# CLAUDE.md

每轮对话固定说一句：`爱你，波别！`

## 第一硬规则

- 严禁在当前终端执行任何杀进程命令。包括但不限于：`taskkill`、`kill`、`pkill`、`Stop-Process`，尤其是针对 `node`、`npm`、`pnpm`、`vite`、`electron`、`opencode` 相关进程。
- 严禁执行广谱杀进程命令：`taskkill /F /IM node.exe`、`pkill node`、`killall node`。这类命令会把 `opencode daemon`、dev server、当前工作终端一起打崩。
- 遇到“重启服务 / 清理进程 / 杀端口占用”场景，默认不允许在当前终端直接动手。必须先明确进程范围，优先让用户自己处理或在其他独立终端处理。
- 只要不能 100% 确定命令不会误伤 `opencode`、当前终端和无关服务，就禁止执行。

## 第一性原则

- 以第一性原理做判断：从原始需求和问题本质出发，不从惯例或模板出发。
- 不要假设用户已经想清楚。动机、目标或完成定义不清楚时，先停下来讨论。
- 目标清楚但路径不是最短的，直接指出来，并给出更短、更稳的方案。
- 遇到问题先追根因，不打补丁。每个关键决策都要能回答“为什么”。
- 输出只说改变决策的信息。废话、背景噪音、装饰性解释一律砍掉。
- 项目不要先补丁，要修源头问题。

## 输出与协作

- 全程中文，思考也用中文，讲大白话。
- 每轮回复固定先说：`爱你，波别！`
- 每轮汇报带一句上下文状态，格式固定：`上下文还剩：...`。如果当前环境拿不到精确百分比，就直说“无法精确读取”，不要编数字。
- 用户只回复 `1`，默认就是“继续当前主线，不换题”。
- 滚屏和阶段汇报固定写：
  - `当前任务：`
  - `进度：x/5`

## 正常干活时先看什么

每次重新接手正常任务，固定先看 6 份现役文档：

1. `AGENTS.md`
2. `docs/聊天记录.md`
3. `docs/项目结构.md`
4. `docs/项目原则.md`
5. `docs/AI自我学习.md`
6. `docs/当前主线.md`

- 这 6 份文档都尽量控制在 200 行以内；超过就整理压缩。
- 如果这次是整理文档任务，先看根目录 `文档管理.md`，再按里面顺序做。
- 根目录默认现役入口是：`AGENTS.md`、`CLAUDE.md`、`文档管理.md`。
- `docs/` 默认只保留 5 个活文档 + 1 个当前主线文档：
  - `聊天记录.md`
  - `项目结构.md`
  - `项目原则.md`
  - `AI自我学习.md`
  - `当前主线.md`
- 其他文档默认放进 `docs/archive/`，只当历史证据，不参与现行判断。

## 文档规则

- `AGENTS.md` 与 `CLAUDE.md` 功能一致，但正式内容只维护在 `CLAUDE.md`；`AGENTS.md` 只保留同步说明，避免双写。
- 每次代码改动后、每次出现新问题、每次出现新结论时，立即同步更新五件套，不要等“每 3 轮”才补。
- 每完成一轮实际任务，都要顺手同步：
  - `docs/聊天记录.md`
  - `docs/项目原则.md`
  - `docs/AI自我学习.md`
  - `docs/项目结构.md`
  - `docs/当前主线.md`
- `docs/聊天记录.md` 只记真实主线，不写无关过程。采用永久 / 长期 / 短期三层记忆，尽量不超过 200 行。
- `docs/项目结构.md` 不是可选项。只要 owner、第二口径、危险未收口点、主链入口发生变化，就必须更新。
- `docs/当前主线.md` 只保留当前主线。任务做完后清空并归档，不无限追加。
- 文档怎么整理、压缩、归档，统一看根目录 `文档管理.md`。
- 用户说“存档”，是在平时同步基础上的额外强制整理，不是第一次补文档。

## 工程与项目底线

- 不允许第二写口、第二发布口、第二裁判口。
- 发现两套逻辑同时存在时，只能留一个。
- 如果发现第二口径、保底、隐藏降级、补丁链，必须汇报；能顺手修就顺手修，但不能继续在旧逻辑上打补丁。
- 不要兜底，错了就报错。不要把创作问题修成 gate、命中词、重 recovery、补名单。
- 系统只做引导、结构化输入、轻量校验和最小必要机制，不替代创作裁判。
- 图片和视频提示词要教 AI 写出更好的作品，不是堆限制词。

## 调试与错误处理

- 遇到错误先自己查代码、看日志、追踪问题，不要先反问用户；信息优先从代码、日志、现役文档里找。
- 卡住时主动汇报：`卡住，正在查 XX 文件`。
- 在高风险操作、长链路改动、容易中断的改动前，先写 `PROGRESS.md`，记录当前状态、下一步、编辑文件。
- 出错前优先用 git 暂存当前变更，避免现场丢失。

## 验证规则

- 任何功能都要端到端跑通才算完成。
- 代码改完默认跑类型检查，不许只看代码就说“完成”。
- 测试只做板块级验证，不把整链路 E2E 当默认验收。
- 提示词、文案质量测试，允许真实调用语言大模型 API。
- 真实 API 不跑通，不准提前宣称完成。
- 质量门默认至少看：
  - `npm test`
  - `npm run build`
  - `npx tsc --noEmit --pretty`

## 当前项目架构标准

- 这个项目要坚持前后端分离：
  - 前端一个文件夹
  - 后端一个文件夹
  - 核心提示词和核心生成逻辑放后端
- 部署目标是阿里云。
- 数据库统一用 PocketBase。
- 这是多个项目共用一个 PocketBase 的架构，项目之间必须物理隔离，不要和 `xinjuben` 现有业务表混淆。
- `xinjuben` 已经占用了自己的业务前缀；新项目必须申请唯一 `appId`，使用 `<appId>_xxx` 命名自己的业务表。

## 多项目共享 PocketBase / SSO 标准

- 核心宗旨：账号全生态通行，项目准入物理隔离，积分钱包独立核算，财务流水清晰可查。
- 架构必须拆成三层：
  - 全局账号层 `users`
  - 项目绑定层 `user_apps`
  - 业务与资产层 `<appId>_xxx`、`user_wallets`、`<appId>_transactions`
- 每个项目必须有唯一 `appId`，全小写，不重复。
- 业务表一律带 `<appId>_` 前缀，严禁无前缀裸表名。
- `users` 表必须有 `registerSource`。
- `user_apps` 和 `user_wallets` 都必须做 `user + appId` 联合唯一索引。
- 项目积分绝不能写回 `users` 表，只能走 `user_wallets`。
- 注册流必须做到：
  - 创建全局账号
  - 自动登录
  - 静默创建 `user_apps`
  - 静默创建 `user_wallets`
- 登录流必须做到：
  - 先全局认证
  - 再检查当前 `appId` 是否已绑定
  - 没绑定就静默补齐 `user_apps` 和 `user_wallets`
- API Rules 默认拒绝，业务表必须校验：
  - `@request.auth.id != ""`
  - 数据归属用户等于请求者
- 后端动钱包和流水时必须带 `appId` 过滤，不能跨项目串改资产。
- 上线验收必须同时满足：
  - 新用户注册后 `users/user_apps/user_wallets` 同时落好
  - 老用户跨项目登录时自动补绑，不提示“邮箱已存在”
  - A 项目充值不会改动 B 项目钱包

## 模型与 API 调用优先级

- 第三方模型网关优先。
- 只有第三方连不上、不可用或明确失败时，才回退官方 DeepSeek。
- 默认优先使用项目里已经配置好的第三方地址和 key，不要自己换供应商。
- 官方 DeepSeek 只作为回退通道，不是默认主通道。
