# 质量不过不整轮失败执行计划

## 目标

把当前“剧本质量不过 = 整轮 script generation 失败”的行为，改成：

1. **生成成功**时，仍然返回已生成内容、ledger、postflight
2. **内容质量不过**时，不把它伪装成流程成功，但也不再把整轮结果直接打死为失败
3. UI / 持久化 / 恢复链能区分：
   - 流程失败
   - 内容可交付但质量未过

## 已确认事实

1. `src/shared/domain/workflow/script-generation-orchestrator.ts`
   - 当前在 repair 后只要 `qualityReport.pass === false` 就直接 `return success: false`
2. `src/main/application/script-generation/start-script-generation.ts`
   - 已经会返回 `ledger + postflight`，但 orchestrator 侧没把这类结果当“可交付但有问题”对待
3. `finalize-script-postflight.ts`
   - 已经能产出完整的 `postflight.quality` 与 `screenplay_quality_failed` issue 列表
4. 当前用户方向已经明确：
   - 内容够用就继续走
   - 不要因为边角质量门一直把结果整轮打死

## 设计原则

1. **不做假成功**：质量不过仍然要明确暴露问题
2. **不丢结果**：已生成内容、ledger、postflight 必须保留
3. **不发明第二裁判口**：仍以 shared quality / postflight 为准
4. **最小改动**：先只改 script generation 结果语义与持久化/展示分流，不顺手大改产品流程

## 实施步骤

### Step 1：补清结果语义

改 contracts / result 使用口径，让 script generation 结果明确支持：

- `success=true` 但 `postflight.pass=false`
- 或引入等价的“completed_with_issues”语义字段

要求：

1. 不影响真正的 runtime failure / parse failure / guardian failure
2. 质量不过只落在 postflight/quality 层，不再冒充流程失败

### Step 2：改 orchestrator 主流程

文件重点：

- `src/shared/domain/workflow/script-generation-orchestrator.ts`

目标：

1. repair 后即使 `qualityReport.pass=false`，也不要直接 `return success: false`
2. 改为：
   - 保留 `generatedScenes`
   - 持久化 ledger / postflight
   - 返回可恢复、可查看的问题信息
3. 进度消息从“failed”改成更符合语义的完成态/带问题完成态

### Step 3：改 IPC / 持久化 / failureResolution 分流

重点检查：

- `src/main/ipc/workflow/script-generation-runtime-handlers.ts`
- `src/shared/domain/workflow/persisted-generation-truth.ts`
- 相关 `scriptFailureResolution` 使用点

目标：

1. 真失败才写 `scriptFailureResolution`
2. 质量不过但结果可交付时，不要把 project 置成“失败等待续跑”
3. 保证 renderer 能拿到 postflight 问题并展示

### Step 4：补测试

至少覆盖：

1. quality fail 时 orchestrator 不再整轮失败
2. 仍返回 `generatedScenes`
3. 仍保留 `ledger + postflight`
4. 真 runtime failure 语义不变
5. 进度板 / 持久化 / failureResolution 不误写成失败

### Step 5：验证

至少执行：

1. 相关 node tests
2. `npm run typecheck`
3. 如需，补一轮最小行为验证（quality fail but deliverable）

## 本轮不做

1. 不改内容质量规则本身
2. 不改弱钩子 detector 口径
3. 不把 scene-level prototype 接进正式生产
4. 不顺手重做整个 UI 交互

## 完成定义

满足以下条件才算完成：

1. quality fail 不再导致整轮 script generation 返回失败
2. 真失败（guardian/runtime/parse/persist）行为不变
3. renderer / project snapshot 不会把“有问题的可交付结果”误认成“失败中断”
4. 测试和 typecheck 通过
