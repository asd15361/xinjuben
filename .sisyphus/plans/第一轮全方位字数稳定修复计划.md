# 第一轮全方位字数稳定修复计划

## 目标

把当前剧本阶段“有的集 1900 多字、有的集 500 多字”的大幅波动压下来，收成更稳定的 2-4 场短剧体量。

## 现役样本与阈值基准

本轮直接用现有 50 集真实 evidence（`tools/e2e/out/userdata-xiuxian-full-real-50ep-mndh2yac/evidence/05-script.md`）做锚点。

### 胖集样本

- `18` 集：`1707` 字 / `3` 场
- `21` 集：`1503` 字 / `3` 场
- `40` 集：`1897` 字 / `2` 场

### 瘦集样本

- `32` 集：`638` 字 / `2` 场
- `33` 集：`665` 字 / `2` 场
- `34` 集：`597` 字 / `2` 场
- `41` 集：`631` 字 / `2` 场

### 本轮硬阈值（先按现有合同收，不另发明新合同）

- 整集上限：`1200` 字（现有合同）
- 整集下限：按 `sceneCount` 沿用 `EPISODE_CHAR_COUNT.min(sceneCount)`
- 2 场集目标：`850-1100` 字
- 3 场集目标：`900-1200` 字
- 4 场集目标：`1100-1200` 字

### repair 触发阈值（本轮新增执行阈值）

- **胖集**：`charCount > 1300`
- **严重胖集**：`charCount > 1600`
- **瘦集**：`charCount < 750`
- **严重瘦集**：`charCount < 650`

这些阈值只用于 repair 分流，不改正式 quality 合同。

## 当前已确认的根因

1. **胖集根因**：单场塞太多回合，不是场数多。
2. **瘦集根因**：上游 `sceneByScene` 骨架偏薄，下游只展开了一点。
3. **trim 太弱**：只修最后一场、最多压 30%，根本收不住 1700-1900 字大胖集。
4. **repair 不够像体量稳定器**：现在会修错误项，但不会系统性削胖补瘦。

## 本轮要做的 3 件事

### 1. 每场预算控制（主控制器）

在正式 prompt 里把每集预算拆成每场预算，不再只写“整集不超过 1200 字”。

落点文件：

- `src/main/application/script-generation/prompt/create-script-generation-prompt.ts`
- 如有必要，补 `buildEpisodeSceneDirectives` 相关测试

本轮直接写死的每场预算：

- 2 场集：每场目标 `420-560` 字；单场超过 `650` 字视为危险
- 3 场集：每场目标 `280-380` 字；单场超过 `450` 字视为危险
- 4 场集：每场目标 `220-300` 字；单场超过 `340` 字视为危险

每场回合数上限（写进 prompt）：

- 1 场最多 `1` 个核心推进回合
- 最后场允许 `1` 个结果落地 + `1` 个尾钩，但不准再接第三次翻盘
- 不准在同场里连写“追上 → 反打 → 再被堵 → 再交易”这种四连段

### 2. repair 真正削胖补瘦（主收口器）

落点文件：

- `src/main/application/script-generation/runtime/repair-generated-scenes.ts`
- `src/shared/domain/script/screenplay-minimal-trim.ts`
- `src/shared/domain/script/screenplay-repair-guard.ts`

#### 对胖集

- 触发条件：`charCount > 1300`
- 指令要明确要求：
  - 删重复动作
  - 删同义对白
  - 一场只保留 `1` 个核心推进回合
  - 去掉同场里的第二、第三次翻盘
  - 胖集优先把多余内容拆掉，不靠把关键结果砍没来控长

#### 对瘦集

- 触发条件：`charCount < 750`
- 指令要明确要求：
  - 补对手回应
  - 补动作结果
  - 补当场代价
  - 补至少一轮有效冲突交换
  - 不能靠加解释句、心理句、背景句来注水

### 2.1 repair 的最小识别依据

本轮不引入复杂语义识别器，先用当前可执行的工程口径：

- **重复动作**：同一场内连续出现同义动词/同一人连续追打、压、拽、扑、逼近等重复段落
- **同义对白**：同一场内连续 2 句以上语义接近、只是换说法的威胁/解释/质问
- **有效冲突交换**：至少包含“我方动作 → 对方回应 → 局面变化”三拍
- **当场代价**：受伤、失物、被堵、被拿捏、证据外流、时间被压缩至少一种

### 3. 瘦集骨架回查入口（主溯源器）

给瘦集保留一个回查口：

- 方便后面直接对照 `sceneByScene` 看是不是上游本来就给薄了
- 本轮先不重做详纲生成，但要把回查链路留清楚

落点文件：

- `src/main/application/script-generation/runtime/repair-generated-scenes.ts`
- 允许后续从 `detailedOutlineBlocks / segments` 回看当前 episode 的 `sceneByScene`

本轮要求：

- repair 提示里明确引用当前 episode 的 `sceneByScene`
- 让瘦集补足时优先补“sceneByScene 已经有但下游没展开出来”的东西
- 不是凭空加戏

## 本轮不做

1. 不继续死抠弱钩子 detector
2. 不重构 scene-level 正式主链
3. 不发明新 fallback / recovery 体系
4. 不把内容问题重新伪装成系统失败

## 预期完成标准

1. prompt 已明确每场预算与每场最多推进回合数
2. repair 已能区分“胖集削减”和“瘦集补足”
3. 测试与 typecheck 通过
4. 四件套同步本轮新口径
