# 代码质量整治收口计划（2026-03-31）

## 1. 目标

围绕以下 5 个方向，做一轮**不误伤正式主链**的代码质量整治：

1. 继续收口实验残留
2. 把正式链依赖关系再显式化
3. 减少“搜索看不出来、编译才知道”的隐性依赖
4. 把死导出、分析 helper、调试残留再做一轮系统清理
5. 把真正长期不用的辅助模块做物理退场

这轮不是大重构，不追求一次性“仓库焕新”。这轮只做：

- 先把正式链和实验链边界收清楚
- 先把误删风险降下来
- 先把真死残留继续清掉
- 让后续每一轮改动更容易判断“什么能删，什么不能删”

## 2. 当前已坐实的事实

### 2.1 正式链与实验链并存

- 正式剧本生成链仍是 episode-level：
  - `run-script-generation-batch.ts`
  - `create-script-generation-prompt.ts`
  - `parse-generated-scene.ts`
  - `finalize-script-postflight.ts`
- scene-level 仍是 prototype lane：
  - `create-scene-generation-prompt.ts`
  - `assemble-episode-scenes.ts`
  - `tools/e2e/test-scene-p*.mjs`

### 2.2 有一批“暂时停用/实验残留”仍在正式目录里

- `src/renderer/src/app/hooks/useProjectAutoGeneration.ts`
  - 当前实现只有注释：Temporary stop
- `src/main/application/workspace/decompose-chat-for-generation.ts`
  - 文件尾部有 re-export，后续要核对是否真有消费方

### 2.3 隐性依赖确实存在

- `project-files/read-index.ts`
  - 同时静态 import 和动态 import `write-shard.ts`
- `project-generation-status-hub.ts`
  - 通过 `await import('../../infrastructure/storage/project-store')` 读取正式存储入口
- 上一轮死代码清理已经坐实：
  - 光靠 grep 会误判 `resolve-lane-strategy.ts`
  - 光靠 grep 会误判 `select-script-generation-batch.ts`
  - 最终是 `typecheck/build` 揪回真实依赖

### 2.4 真死代码可以清，但必须先分层

已经验证过：

- `hard-hook.ts` 里有过真无人消费分析残留
- `screenplay-format.ts` 里有过真无人消费格式分析 helper

说明这条路线可行，但只能优先清“分析残留 / 辅助函数 / 非主链导出”，不能先上来删正式生产文件。

## 3. 这轮整治原则

1. **先显式化，再删除**
   - 先把“这是不是正式链”写清楚，再决定删不删
2. **先删分析残留，不先删正式文件**
3. **任何死代码判断必须过三关**
   - grep / LSP 候选
   - 代码上下文确认
   - `typecheck/build` 验证
4. **实验链不再和正式链并排发声**
5. **计划内每一步都必须可验证**

## 4. 分阶段执行

## Phase A：正式链 / 实验链地图收口

### 目标

把“现役正式链”和“实验/原型链”在代码层、文档层、测试层都明确分开。

### 先补一条依赖身份分层（A0）

这一步必须在真正删东西前先做，不然会继续把“有人手工用”误判成“正式生产在用”。

所有引用都先分成 5 类：

1. 正式生产依赖
2. prototype / 原型验证依赖
3. 测试依赖
4. 根目录临时调试 / 分析脚本依赖
5. 历史设计文档 / 历史证据引用

特别纳入盘点的，不只 `src/`：

- 根目录 `*.ts` / `*.js` 调试脚本
- `tools/e2e/*`
- 历史设计文档（例如逐场生成设计文档）

原因：这些东西虽然不属于正式生产链，但会持续制造“它好像已经接进生产”或“它其实还在用”的假象。

### 动作

1. 为以下文件补“正式 / 原型”边界说明（仅在必要处补最小注释，不大写文案）：
   - `create-scene-generation-prompt.ts`
   - `assemble-episode-scenes.ts`
   - `run-script-generation-batch.ts`
   - `create-script-generation-prompt.ts`
2. 盘点所有 `tools/e2e/test-scene-p*.mjs`
   - 明确归类为 prototype verification，不再让它们像正式验收入口
3. 盘点 `useProjectAutoGeneration.ts`
   - 如果确认继续停用且无人消费：物理退场
   - 如果是预留恢复点：改成明确的停用占位口径，不继续假装可用

### 成功标准

- 新人只看代码和现役文档，就能知道正式链是哪个、原型链是哪个
- 不再出现“把 prototype lane 误写成生产事实”
- 不再把根目录调试脚本、`tools/e2e` 验证脚本、历史设计文档的消费误判成正式生产依赖

### QA

- Grep 检查：`prototype|原型|正式链|production`
- 相关文件读审
- `typecheck` 不受影响

## Phase B：依赖关系显式化

### 目标

减少“搜索看不出来、编译才知道”的隐性依赖。

### 动作

1. 建一份正式链依赖清单（优先 script-generation / ai / storage）
   - 模块
   - 入口文件
   - 直接依赖
   - 是否存在动态 import
2. 针对以下高风险点做收口：
   - `project-files/read-index.ts` 对 `write-shard.ts` 的静态+动态双引用
   - `project-generation-status-hub.ts` 对 `project-store` 的动态 import
3. 能显式静态 import 的，优先改显式；必须动态 import 的，补清楚原因
4. 禁止“只是为了懒加载/图省事”保留非必要动态 import
5. 依赖表不只记 import 形态，还要记“依赖身份”：
   - 正式运行依赖
   - 测试依赖
   - 临时脚本依赖
   - 类型依赖

### 成功标准

- 至少 script-generation / ai / storage 三块的正式依赖关系能一眼看出
- 再做死代码判断时，不容易因为 import 形态而误判
- 再做死代码判断时，也不会把“临时脚本依赖 / 历史设计引用 / 测试依赖”混成“正式生产依赖”

### QA

- grep `await import\(|import\(` 复核
- `npm run typecheck:node`
- `npm run build`

## Phase C：死导出 / 分析 helper / 调试残留系统清理

### 目标

继续清理“文件还活着，但里面挂着不再服务正式链的东西”。

### 动作

1. 扫描这几类目标：
   - 只在定义处出现的导出
   - 仅测试消费、但不应作为公开 API 的导出
   - debug-only helper
   - analysis-only helper
2. 优先处理：
   - `main/application/*`
   - `shared/domain/script/*`
   - `shared/domain/workflow/*`
3. 处理顺序：
   - 先去 export
   - 再删实现
   - 最后删对应测试（如果测试只围着死代码转）

### 成功标准

- 公开导出面缩小
- 不再保留“只为分析或历史实验服务”的公开函数

### QA

- LSP diagnostics = 0
- 相关板块单测通过
- `typecheck/build` 通过

## Phase D：长期不用辅助模块物理退场

### 目标

把“已经长期停用、也不再属于正式链”的辅助模块真正移除，而不是继续挂在现役目录里制造噪音。

### 动作

1. 建一份候选清单：
   - 停用 hook
   - test-only orchestration helper
   - prototype-only runtime helper
   - 历史兼容桥接但已无消费方的文件
2. 每个候选都必须经过：
   - grep / LSP
   - 上下文读审
   - `typecheck/build`
3. 能删就物理删；不能删就明确标记用途和边界

### 成功标准

- 现役目录里的“空壳 / 停用 / 历史辅助模块”显著减少

### QA

- 删除后跑相关单测
- `npm run typecheck:node`
- `npm run build`

## 5. 推荐执行顺序

1. Phase A：先把正式/实验边界收清
2. Phase B：再收依赖显式化
3. Phase C：再做批量死导出清理
4. Phase D：最后做物理退场

原因：

- 如果边界没收清，后面会一直误删
- 如果依赖不显式，死代码判断会一直打脸
- 先删文件比先收边界更危险

## 6. 风险点

1. **grep 假阴性**
   - 看起来没引用，实际还在正式链中
2. **测试专用导出和正式公开导出混在一起**
3. **动态 import 掩盖真实依赖**
4. **prototype lane 虽未接生产，但仍被误当主线依据**

## 7. 本轮之后的验收口径

每做完一个 phase，必须同时满足：

1. 相关板块单测通过
2. `npm run typecheck:node` 通过
3. `npm run build` 通过
4. 四件套同步更新
