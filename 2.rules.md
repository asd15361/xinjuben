# 2. rules（基础硬规则）

本文件只回答一件事：
**不管现在做什么任务，哪些底线必须守住。**

如果你想看别的，请去这里：
- 长期协作方式、创作哲学、流程边界：`3.agents.md`
- 用户协作偏好、沟通方式、自动代劳边界：`docs/operator-preferences（用户协作偏好）.md`
- 调试、找根因、重构流程：`docs/engineering-workflow（工程排障与根因修复）.md`
- 榜单、情报中心、样本学习：`docs/intel-sop（榜单情报与样本学习）.md`
- 文档同步源、每日对齐流程、最近同步时间：`docs/doc-sync（文档同步机制）.md`
- 仓库目录怎么放、功能该落哪里：`docs/plans/仓库构架与功能落点蓝图（第一版）.md`
- 最近做了什么：`4.worklog.md`

## 一、总底线
- 我们做的是优秀软件，但交付必须稳定、清楚、可验证。
- 能自动完成的技术工作，默认由 AI 自己完成，不转交给用户。
- 大改动先出计划，小改动可以直改，但也要能快速回滚。
- 改完必须验证，不允许“我觉得差不多”。
- 安全、凭证、生产配置这类高风险内容，必须谨慎处理。
- 解开补丁，找到本质。
- 创作问题默认交给 AI 主体去写，系统只允许做引导、结构化输入、轻量校验和最小必要机制。
- 能用提示词、脚手架、工序边界解决的，绝不改成兜底、阻塞 gate、命中词校验、重型 recovery 或正文抽名补名单。
- 每个自然日开工前，先检查 `D:\vivecoding规则` 与 `docs/doc-sync（文档同步机制）.md`。
- 如果同步记录不是今天，必须先完成当天文档同步，再开始当天正式开发。
- 同步不是只看时间，还要对比差异：新增文档、新增规则、规则变更都要补进本仓库。
- 当天已经同步过，就不重复同步，但后续若源仓库当天再次更新，仍要补同步记录。

## 二、AI 默认自己做的事
- 跑命令、改配置、重启服务、跑测试、做界面验证、收集报错、完成可自动化的浏览器操作和部署检查。
- 能由 AI 自动完成的，不要让用户手动做。

## 三、什么情况必须先出计划
出现这些情况，就不能边改边试：
1. 跨模块重构
2. 跨阶段或跨流程重构
3. 一次改 3 个及以上文件
4. 会影响生成链路、缓存、验收、回滚路径
5. 明显已经不是小修，而是在动系统边界

正确顺序：
1. 讲清目标
2. 讲清改动范围
3. 讲清风险和回滚点
4. 讲清验证方式
5. 确认后再动手

下面这些情况可以直接改：
- 单文件
- 小范围
- 不改主流程
- 出问题能快速撤回

## 四、改完必须形成闭环
标准闭环：

```text
分析需求 -> 修改代码或文档 -> 跑验证 -> 看界面或结果 -> 发现问题再调整 -> 再验证
```

要求：
- 不做“改完就走”的半成品交付。
- 命令、构建、API、部署、测试失败时，先做根因分析。
- 最多允许 3 次有依据的重试。
- 3 次后还失败，就明确告诉用户卡点、原因和下一步最可行方案。
- 收尾必须包含“资源清理”：
- 用完工具后，关闭对应 MCP 会话。
- 不重复启动同类 MCP 实例；如果已有同类实例在跑，优先复用或先确认是否需要关闭旧实例。
- 关掉这轮临时拉起的代理、浏览器、MCP 工具、测试进程、临时 dev server。
- 不能把 `chrome-devtools-mcp`、`@upstash/context7-mcp`、`@playwright/mcp`、`mcp-server-git`、headless browser、watchdog 这类进程残留在后台不管。
- 清理时按进程树回收，不只杀单个 PID。
- 如果这轮用过代理或浏览器工具，结束前必须检查一次相关后台进程数量；发现明显孤儿进程，必须主动清理。
- 如果宿主进程仍在保活，不能假装“清理完成”。
- “功能做完了”不算真正结束；“功能做完 + 验证完 + 资源清干净”才算结束。

## 五、安全底线
- 真实账号密码不能提交到仓库。
- 凭证优先放本地 `.env` 或其他安全位置。
- 涉及账号、权限、支付、生产配置，先说清再动。
- 不私自修改 superuser 或其他高权限凭据。
- 不把敏感信息写进长期规则文档、worklog、memory 或计划文档。

## 六、什么叫可以交付
至少满足这些：
- build 能过
- 页面能操作
- 主流程能跑通
- console 没有阻断性报错
- API 主调用链正常
- 文档前后一致，不互相打架

做不到这些，就不要轻易说“已经好了”。

## 七、代码结构硬阈值
- 新文件硬上限：`220` 行。
- `180` 行以上进入预警区，新增内容前必须先判断要不要拆。
- 老文件超过 `260` 行视为“结构债务文件”，默认只允许修 bug 或拆分，不允许继续叠加新功能。
- 单函数硬上限：`40` 行；超过 `25` 行就优先考虑拆子函数。
- 嵌套层级必须小于等于 `3` 层。
- 分支数量必须小于等于 `3` 个。
- 根装配文件（如 `main/index.ts`、`renderer/src/App.tsx`、`preload/index.ts`）只允许做组合与注册，不允许承载业务总控。
- 一个文件只允许一个主职责；如果已经承担“计划 + 执行 + 审计”这类多职责，必须拆。
- 超过阈值就停下来找根因，不准继续往里塞。
- 默认只修原有功能，不主动借修 bug 之名新增新功能。
- 如果原有问题已经不能靠小修收住，必须先停下来，改做重构计划，再继续动代码。

## 八、仓库边界硬规则
- `renderer` 只允许放展示、交互、本地编辑态和 feature hook，不允许藏 AI 路由、ledger、audit、resume、failure 等主链规则。
- `preload/api/*.ts` 只允许做 IPC 桥接，不允许写业务编排。
- `main/index.ts` 只允许做启动和装配，不允许继续堆 handler 细节与 use case 逻辑。
- `shared/domain` 只能放纯规则，禁止依赖 React、Electron、IPC、fetch、文件系统。
- 进入“结构债务”状态的文件，只允许拆分和修复，不允许继续追加新功能。

## 九、正式事实权责硬规则
- 正式事实的定义权只属于创作者，不属于 AI、审计器、ledger 或系统推断器。
- 正式事实的正式声明动作只能发生在 `outline` 工序。
- 系统拥有正式事实校验权，但校验权不等于定义权。
- 系统拥有升格判定权，但升格确认权属于创作者。
- AI 只能输出“候选正式事实”或“建议正式事实”，不能直接写入 `confirmed formal facts`。
- ledger / audit / repair 只能围绕已确认的正式事实运行，不能偷偷发明新的正式事实。

## 十、API 与配置
- 所有 API 配置默认放 `.env`。
- API 不可用时，先查 `.env`、路由和可用性，再判断要不要改代码。
- 验证模型时，优先用低成本或不扣费方式。
- 当前可用模型、路由和异常现象，记进工作记录。

## 十一、数据与权限（如果这轮任务涉及）
- 每条业务数据都要能追到来源和责任边界。
- 默认不公开业务数据。
- 新增数据结构时，同时说明字段、索引、规则和验证步骤。
- 涉及共享实例或多应用复用时，先定义隔离方式。

## 十二、沟通要求
- 全程中文。
- 用大白话解释复杂问题。
- 每次任务结束尽量说清：做了什么、为什么这样做、这是快速处理还是根因优化、下一步建议先做什么。
- 如果这轮清理过后台进程，要明确告诉用户“清了哪些类型、为什么清”。

## 十三、每次结束前自查
- 有没有隐藏 bug？
- 有没有权限或凭证风险？
- 有没有漏配置或漏版本？
- 有没有 UI 崩坏或 console 报错？
- 有没有把该靠提示词解决的问题写成硬编码？
- 有没有把创作问题偷偷改成兜底、阻塞 gate、命中词校验、重型 recovery 或正文抽名补名单？
- 有没有越过流程边界乱改？
- 有没有把建议写成死限制？
- 有没有让系统替代创作裁判？
- 有没有把本该拆出去的逻辑继续塞进债务文件？
- 有没有让系统偷偷拥有了正式事实定义权？
- 有没有留下这轮拉起但已不再需要的代理、MCP、浏览器、watchdog、临时 node/python 进程？

## 十四、机器和文件系统边界
- 不要卸载用户电脑上的东西。
- 不要乱删除用户文件。
- 默认只动当前项目目录里的内容。
- 如果要碰项目外的文件，必须有明确理由并尽量克制。

## 十五、这类内容以后写哪里
- 长期协作哲学：`3.agents.md`
- 用户协作偏好：`docs/operator-preferences（用户协作偏好）.md`
- 工程调试和根因修复：`docs/engineering-workflow（工程排障与根因修复）.md`
- 榜单、样本、情报中心：`docs/intel-sop（榜单情报与样本学习）.md`
- 正式事实、权责、升格规则：`docs/plans/正式事实定义权补完计划（第一版）.md`
- 一次性任务细节：`4.worklog.md` 或 `docs/plans/`
- 失效规则：移入归档

## 十六、业务完成标准
- 用户真正要的结果已经出来。
- 主流程从真人视角走通。
- 没有明显逻辑跑偏。
- 不是表面修复冒充根因修复。

## 十七、收尾表达要求
- 收尾时先说明完成没有，再说明建议什么。
- 如果当前需求没有完成，不要让建议抢到前面。
- 建议要用大白话说明作用，不能只说技术动作。
- 如果建议属于主线任务之外的扩展项，要明确说这是额外项，不是当前需求本体。

## 十八、建议表达要求
- 用户不懂技术时，AI 该建议的时候必须建议。
- 建议不能只报技术动作，要说明作用和后果。
- 建议优先讲根因层后果，不讲空泛表面话。
- 每条建议尽量标清：`强建议` 或 `可选优化`。
